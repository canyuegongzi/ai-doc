# 限流与重试

> 状态：第一版草案

## 1. 文档定位

本文档用于定义模型接入层中的限流与重试设计，重点回答：

- 为什么模型调用必须做限流与重试
- 哪些错误可以重试，哪些不应重试
- 如何避免因错误重试引发成本放大和雪崩
- 限流与重试如何与路由、fallback、审计和治理协同

## 2. 为什么模型层必须重视限流与重试

模型调用与普通 HTTP 调用相比，存在几个更突出的特点：

- 成本直接与调用次数相关
- 延迟通常更高
- 上下游服务更复杂
- 供应商限流更常见
- 流式响应更容易出现中断
- 重试可能放大成本与副作用

因此，如果模型层没有限流与重试策略，系统很容易在高峰、抖动或供应商异常时出现：

- 服务雪崩
- 成本爆炸
- 重复生成
- 请求超时堆积
- 下游工具链路连锁失败

## 3. 当前行业最佳实战与共识

### 3.1 重试应是策略能力，而不是简单 while 循环

当前主流 API 文档和工程实践都不建议“无脑重试”。正确做法是：

- 识别错误类型
- 识别幂等性
- 识别是否已产生副作用
- 结合指数退避和最大重试次数

### 3.2 限流应按多维度做，而不是只有全局 QPS

企业系统中的限流维度至少可能包括：

- 模型级
- 租户级
- 场景级
- 用户级
- 工具级
- 预算级

### 3.3 fallback 和重试应配合使用

如果某家供应商持续故障，继续重试同一路径往往没有意义，应进入 fallback 或直接降级。

## 4. 限流设计

### 4.1 限流目标

限流的目的不只是保护供应商接口，更是保护本平台的：

- 成本预算
- 稳定性
- 调用队列
- 关键场景优先级

### 4.2 推荐的限流维度

建议至少支持：

- provider-level rate limit
- model-level rate limit
- tenant-level quota
- scene-level concurrency
- user-level request cap
- budget-based throttling

### 4.3 优先级策略

在资源紧张时，建议明确优先级，例如：

- 诊断和生产辅助任务优先于低价值试探性请求
- 在线请求优先于离线批处理
- 付费租户优先于普通租户（如果适用）

## 5. 重试设计

### 5.1 可重试错误

通常包括：

- 网络抖动
- 短暂超时
- 供应商 429 限流
- 临时 5xx 错误
- 可恢复的流式中断

### 5.2 不应重试的错误

通常包括：

- 鉴权失败
- 参数校验失败
- 明确不支持的能力调用
- 输出 schema 与输入完全不兼容的错误
- 已经造成外部副作用的不可幂等调用

### 5.3 推荐重试策略

建议采用：

- 指数退避
- 抖动
- 最大重试次数
- 超时上限
- retry budget

### 5.4 与幂等性的关系

生成类模型调用通常是“逻辑幂等但结果不稳定”的特殊情况。

需要明确：

- 重试是否允许不同结果
- 重试是否会造成重复计费
- 对带工具调用的响应是否还能安全重试

因此，模型调用重试最好结合 request id、trace id 和响应状态做判定。

## 6. 流式调用中的特殊问题

Streaming 调用会带来更复杂的中断和重试问题。

典型问题包括：

- 中途断流
- 已输出部分结果
- usage 尚未返回
- tool call 已部分产生

因此 streaming 重试需要特别谨慎：

- 纯文本场景可考虑从头重试
- 已触发 tool call 的场景应避免盲目重试
- 需要记录 partial output 和 terminal state

## 7. 重试、fallback、熔断的关系

三者应清晰区分：

### 7.1 重试

适用于短暂可恢复错误。

### 7.2 fallback

适用于当前模型或供应商不适合继续处理，需要切换备选路径。

### 7.3 熔断

适用于某路径在一段时间内持续异常，需要短暂停止流量进入。

如果这三者混在一起，系统会很难治理。

## 8. 成本控制视角下的重试设计

模型层重试必须纳入成本控制。

需要明确：

- 单请求最大重试次数
- 单请求最大 token retry budget
- 单租户失败风暴保护
- 超预算后的自动降级

否则在异常时段，重试很容易从稳定性机制变成成本放大器。

## 9. 面向低代码平台的典型策略

### 9.1 在线问答场景

- 低延迟优先
- 小次数快速重试
- 超时后可直接返回降级结果

### 9.2 Schema 生成场景

- 可允许更长超时
- 输出校验失败时可做有限重试
- 结构化输出失败时可换更稳模型

### 9.3 配置诊断场景

- reasoning 请求通常成本较高
- 不应无限重试
- 若 retrieval 或 tool 失败，应优先定位失败层而非简单重跑全链路

### 9.4 批量 embedding 场景

- 更适合队列化 + 分批重试
- 应显式控制 batch size 与 quota

## 10. 常见误区

### 10.1 429 就立即重试很多次

这通常只会让限流问题更严重。

### 10.2 不区分可重试和不可重试错误

这样会造成无意义调用和成本浪费。

### 10.3 只做重试，不做熔断和 fallback

一旦供应商持续异常，系统仍然会陷入抖动。

### 10.4 不记录重试信息

没有 retry count、latency、error type、cost delta，后续几乎无法优化策略。

## 11. 一句话总结

限流与重试的核心，不是“尽量多试几次”，而是通过多维限流、错误分类、指数退避、retry budget、fallback 与熔断协同，在稳定性和成本之间建立可控平衡，让模型层在异常条件下仍保持可运营性。
