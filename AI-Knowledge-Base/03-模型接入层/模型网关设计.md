# 模型网关设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义企业 AI 平台中的模型网关（Model Gateway）设计，回答：

- 为什么模型接入必须网关化
- 模型网关到底解决哪些问题
- 一个可持续演进的模型网关应具备哪些核心能力
- 模型网关在企业级 RAG、Agent、多模态和低代码场景中扮演什么角色

## 2. 为什么模型接入不能停留在“直接调 API”

在 AI 应用早期，很多团队都会从直接调用某一家模型 API 开始，这是合理的起点。但随着场景增多，直接调用模型接口会迅速暴露问题。

常见问题包括：

- 不同场景绑定不同模型供应商，后续难切换
- LLM、Embedding、Rerank、Vision 接口风格不一致
- 不同模型参数、限流、错误码和流式协议差异大
- 成本无法统一统计
- 故障时缺少 fallback 和降级能力
- 模型版本变化对业务造成连锁冲击
- 场景开发团队各自维护一套模型适配逻辑

因此，模型接入如果不网关化，后续很快就会从“开发方便”变成“治理混乱”。

## 3. 行业最佳实战中的共识

结合当前行业公开资料，模型网关已经越来越成为企业 AI 平台的默认组成部分。

### 3.1 模型接口正在统一，但供应商差异仍然显著

OpenAI 的 Responses API 已经把 text、image、structured outputs、built-in tools、function calling 等能力统一到单一接口；Anthropic、Google、Bedrock 等也在持续加强消息式接口和 tool use 支持。

但即使如此，供应商之间依然存在明显差异：

- 模型命名与能力边界不同
- 参数体系不同
- multimodal 输入格式不同
- tool use 协议不同
- 速率限制和成本模型不同

这意味着企业仍然需要一层统一抽象与路由控制。

### 3.2 推理模型、便宜模型、多模态模型并存已成常态

当前主流 AI 系统通常不会只使用一种模型，而是会组合：

- 通用对话模型
- 推理模型
- Embedding 模型
- Rerank 模型
- Vision / multimodal 模型
- 语音模型

因此，模型网关不只是接一个 LLM，而是接一组模型能力。

### 3.3 路由、降级、成本和观测已成为模型层必备能力

Bedrock 的 inference profiles 已经把跨 Region 路由、成本跟踪、吞吐治理前置为产品能力；OpenAI 和 Anthropic 也都在文档中持续强调 tool-capable models、reasoning models、不同能力模型的选择与切换。

这说明模型层已经从“调用通道”升级为“流量与能力治理层”。

## 4. 模型网关的定义

模型网关可以理解为：

位于业务应用与底层模型供应商之间的一层统一能力入口，负责抽象模型能力、路由请求、治理调用、追踪成本，并屏蔽不同供应商和不同模型之间的接口差异。

用一句话概括：

模型网关的本质，是 AI 时代的模型接入控制平面。

## 5. 模型网关解决什么问题

### 5.1 统一接入问题

为 LLM、Embedding、Rerank、Vision、语音等模型提供统一入口，减少业务方直接依赖底层 SDK。

### 5.2 多供应商切换问题

让系统不必在应用层硬编码绑定单一供应商模型。

### 5.3 模型路由问题

根据场景、成本、延迟、上下文长度、能力要求选择最合适的模型。

### 5.4 成本治理问题

统一统计：

- token 消耗
- 调用次数
- 场景成本
- 租户成本
- 模型级成本

### 5.5 稳定性问题

通过：

- 限流
- 重试
- 熔断
- fallback
- 降级

提升整体系统可用性。

### 5.6 观测与审计问题

让每一次模型调用都可追踪、可回放、可分析。

## 6. 模型网关不解决什么问题

### 6.1 不负责业务知识理解

模型网关负责模型调用，不负责企业知识工程和检索逻辑。

### 6.2 不负责场景流程控制

模型网关不应承担 Workflow、Agent 状态推进或工具编排职责。

### 6.3 不替代 Prompt、RAG 和 Tool 设计

它是底层基础设施，不是场景逻辑层。

## 7. 模型网关的核心能力

### 7.1 统一能力目录

至少应统一管理：

- chat / response generation
- embeddings
- rerank
- vision / multimodal understanding
- speech / audio（若有）

### 7.2 统一模型元数据

每个模型至少应有：

- modelId
- provider
- capabilityType
- contextWindow
- input/output modality
- pricing metadata
- latency profile
- availability status
- routing tags

### 7.3 统一调用接口

例如统一为：

- `generate()`
- `embed()`
- `rerank()`
- `analyzeVision()`

无论底层是 OpenAI、Anthropic、Gemini、Bedrock 还是自部署模型，业务方都通过统一接口调用。

### 7.4 参数归一化

不同供应商对如下参数命名和语义可能不同：

- temperature
- max_tokens
- top_p
- reasoning effort
- tool choice
- structured outputs
- image input format

模型网关需要定义自己的统一参数层，并在适配层中做映射。

### 7.5 路由策略

至少应支持：

- 按场景路由
- 按模型能力路由
- 按成本路由
- 按租户路由
- 按可用性路由
- 按延迟路由

### 7.6 fallback 与降级

当主模型不可用、过载或成本过高时，系统应能：

- 切换同能力备选模型
- 降级为更便宜模型
- 降级为更短上下文模式
- 对高成本场景做限流或拒绝

### 7.7 限流与配额

需要支持：

- 模型级限流
- 租户级限流
- 场景级限流
- 预算控制

### 7.8 观测与成本统计

每次调用至少应记录：

- traceId
- provider
- modelId
- request type
- latency
- input / output tokens
- estimated cost
- error code
- retry count

## 8. 模型网关的推荐架构

可以将模型网关进一步拆成几个子模块。

### 8.1 Capability Registry

记录系统支持的模型能力和模型目录。

### 8.2 Provider Adapter

屏蔽不同供应商 SDK / API 差异。

### 8.3 Routing Engine

根据策略决定用哪个模型。

### 8.4 Policy Layer

处理：

- 配额
- 限流
- fallback
- 风险限制
- 场景白名单

### 8.5 Telemetry Layer

统一记录 metrics、logs、trace、cost。

### 8.6 Cache / Replay Layer（可选）

对于部分场景，可支持：

- 响应缓存
- embedding 缓存
- 调用回放

## 9. 模型路由应如何设计

模型路由是模型网关最关键的能力之一。

推荐至少从以下维度做判断：

### 9.1 按能力路由

例如：

- reasoning-heavy task -> 推理模型
- low-cost FAQ -> 轻量模型
- multimodal understanding -> vision model
- retrieval pipeline -> embedding + rerank model

### 9.2 按场景路由

例如：

- 组件问答 -> 通用 chat model + retrieval
- schema 生成 -> 更强结构化输出模型
- 运维诊断 -> reasoning model

### 9.3 按 SLA 路由

例如：

- 高吞吐低延迟场景 -> 中小模型
- 复杂高价值场景 -> 更强模型

### 9.4 按成本预算路由

不同租户、业务线或时段可以采用不同成本策略。

## 10. 模型网关在当前行业中的前沿变化

### 10.1 从“LLM proxy”演进为“AI control plane”

早期模型网关更像反向代理；现在更像控制平面，要同时管理：

- 多模型目录
- 多能力编排
- route / fallback / quotas
- cost / trace / policy

### 10.2 原生支持 tool-capable models 与 responses-style interface

随着 OpenAI Responses API、Anthropic tools、Gemini function calling 等能力趋于成熟，模型网关不再只包一层 text completion，而是要兼容：

- structured outputs
- multimodal input
- built-in tools
- external tools
- stateful interactions

### 10.3 与 Bedrock / multi-cloud / self-hosted 并行共存

企业越来越常见的形态是：

- 一部分走云 API
- 一部分走 Bedrock / Vertex / Azure OpenAI
- 一部分走自部署或私有推理服务

模型网关因此需要天然支持多环境、多云、多部署形态。

## 11. 面向低代码平台如何理解模型网关

在低代码平台场景中，模型网关应至少支撑以下模型能力组合：

- 通用问答模型
- 推理模型
- Embedding 模型
- Rerank 模型
- 多模态模型

并支持对不同场景做路由，例如：

- 组件知识问答 -> chat + retrieval
- Schema 生成 -> 结构化强模型
- 配置诊断 -> reasoning model
- 截图 / 设计稿理解 -> vision model

这样低代码平台就不需要每个场景自己直接接模型，而是统一从网关申请能力。

## 12. 常见误区

### 12.1 只做一个简单代理层

如果模型网关只是转发 HTTP 请求，它很快就会失去价值。

### 12.2 只统一 LLM，不统一 embedding / rerank / vision

真实 AI 系统通常依赖多类模型能力，统一抽象不应只停留在对话模型。

### 12.3 没有成本与 trace 就先大规模推广

这样很容易出现模型成本不可见、调用失败难定位、租户无法对账等问题。

### 12.4 路由规则完全手写硬编码

更合理的做法是把路由规则配置化，并与场景、能力和成本策略绑定。

## 13. 一句话总结

模型网关的本质，是为企业 AI 系统提供统一模型能力入口、路由控制、成本治理和可观测性控制的模型控制平面，让模型接入从“供应商 API 调用”升级为“平台能力管理”。
