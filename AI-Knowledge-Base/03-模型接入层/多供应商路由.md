# 多供应商路由

> 状态：第一版草案

## 1. 文档定位

本文档用于定义企业 AI 平台中的多供应商路由设计，重点回答：

- 为什么模型层要做多供应商路由
- 路由策略应按什么维度设计
- 如何在能力、成本、稳定性与合规之间做平衡
- 多供应商路由如何服务 RAG、Agent 和低代码场景

## 2. 为什么必须考虑多供应商路由

早期很多系统默认绑定单一供应商模型，这在验证阶段是合理的。但一旦进入平台化阶段，单供应商绑定会迅速带来问题：

- 不同模型在推理、结构化输出、多模态、成本上差异明显
- 单一供应商故障会直接影响全系统
- 不同场景需要不同性能 / 成本平衡点
- 合规、地域和私有化要求会导致部署形态多样化

因此，多供应商路由不是锦上添花，而是成熟模型层的重要组成部分。

## 3. 当前行业最佳实战与趋势

### 3.1 模型能力已经高度分化

当前主流模型在以下方面差异非常大：

- reasoning 能力
- tool use 能力
- 多模态能力
- 结构化输出稳定性
- 上下文长度
- 价格
- 延迟

这意味着“一个模型走天下”的策略在平台阶段越来越不经济。

### 3.2 云 API、多云平台和自部署并存已经是常态

企业系统越来越常见以下混合形态：

- OpenAI / Anthropic / Google API
- Bedrock / Vertex / Azure OpenAI
- 开源模型私有部署
- 专用 embedding / rerank 服务

因此，多供应商路由应被设计为常态能力，而不是例外处理。

### 3.3 路由已经从手工选模型变成策略系统

Bedrock inference profiles、OpenAI 不同模型家族、Anthropic 与 Gemini 的差异化路线都说明：

路由不再只是“开发者选一个模型”，而越来越像“平台根据策略做能力匹配”。

## 4. 多供应商路由解决什么问题

### 4.1 能力匹配问题

不同场景适合不同模型：

- FAQ 问答 -> 轻量低成本模型
- 复杂诊断 -> reasoning model
- 截图理解 -> multimodal model
- 检索 -> embedding / rerank model

### 4.2 成本优化问题

通过路由，可以把高成本模型用在高价值任务上，把低成本模型用于大流量轻任务。

### 4.3 可用性问题

当某家供应商故障、限流或性能退化时，系统可以切换到备选模型。

### 4.4 合规与部署问题

某些数据或租户可能必须走：

- 私有化模型
- 指定地区
- 指定云平台

多供应商路由是实现这些策略的基础。

## 5. 路由设计的核心维度

### 5.1 按能力路由

基于模型能力标签做选择，例如：

- `supportsReasoning`
- `supportsStructuredOutput`
- `supportsToolUse`
- `supportsVision`
- `supportsLongContext`

### 5.2 按场景路由

基于业务场景做选择，例如：

- 组件问答
- Schema 生成
- 配置诊断
- 运维分析
- 截图理解

### 5.3 按成本路由

基于预算和单位成本做选择，例如：

- 默认模型
- 高级模型
- 超预算降级模型

### 5.4 按延迟路由

实时高并发场景通常要求更快的模型，复杂离线分析场景可以接受更高延迟。

### 5.5 按可用性路由

当主模型出现：

- 超时
- 配额不足
- 区域故障
- 服务波动

时，自动切换到备用模型。

### 5.6 按合规与租户路由

例如：

- 某些租户必须走私有化模型
- 某些数据不得出境
- 某些场景必须走指定供应商

## 6. 推荐的路由策略结构

建议把路由拆成三层。

### 6.1 静态能力目录

描述每个模型：

- provider
- capability
- supported features
- cost profile
- latency profile
- deployment scope

### 6.2 策略匹配层

根据请求上下文和规则，筛选候选模型。

### 6.3 动态决策层

结合实时状态做最终选择，例如：

- 当前可用性
- 当前负载
- 当前预算使用情况
- 当前限流状态

## 7. fallback 与主路由的关系

主路由解决“优先选谁”，fallback 解决“选中的不可用怎么办”。

两者应明确分开设计。

推荐至少支持：

- 同能力同等级 fallback
- 同能力低成本 fallback
- 同能力低性能 fallback
- 跨供应商 fallback

## 8. 低代码平台中的典型路由策略

### 8.1 组件问答

优先：轻量 chat model + retrieval

### 8.2 Schema 生成

优先：结构化输出稳定的模型

### 8.3 配置诊断

优先：reasoning model

### 8.4 页面截图分析

优先：multimodal / vision model

### 8.5 大规模索引构建

优先：高性价比 embedding 模型

这说明模型路由应直接绑定场景能力，而不是只在底层技术层面做切换。

## 9. 常见误区

### 9.1 多供应商路由等于随机切换供应商

真正的路由应由策略驱动，而不是简单 round-robin。

### 9.2 只按价格做路由

如果忽略能力匹配，成本可能看似下降，但总体效果会急剧恶化。

### 9.3 没有统一 capability metadata 就做路由

这样很快会变成一堆经验型硬编码。

### 9.4 把 fallback 当成完整路由体系

fallback 很重要，但只是路由体系的一部分。

## 10. 一句话总结

多供应商路由的核心，不是让平台同时接很多模型，而是通过能力、场景、成本、延迟、可用性和合规等多维策略，为每一次请求选择最合适的模型路径，从而把模型层真正做成可运营、可治理的能力调度系统。
