# 模型统一抽象

> 状态：第一版草案

## 1. 文档定位

本文档用于定义企业 AI 平台中“模型统一抽象”的设计思路，重点回答：

- 为什么需要在模型网关之上做统一抽象
- 统一抽象应该抽象到什么粒度
- 如何避免把所有模型能力强行抽成最小公约数，导致能力损失
- LLM、Embedding、Rerank、Vision 等不同能力应该如何抽象

## 2. 为什么必须做统一抽象

当系统只接一个模型时，直接用供应商 SDK 看起来很高效。但一旦系统开始接入：

- 多家供应商
- 多种模型能力
- 多种部署形态
- 多个业务场景

如果没有统一抽象，业务层会迅速被底层模型细节污染。

典型问题包括：

- 同一类能力在不同供应商之间参数完全不同
- 上下文格式和工具协议不一致
- 业务层需要知道每家供应商的错误码和流式协议
- 切换模型时需要修改大量场景代码
- 同一个场景难以做动态路由

统一抽象的目标，就是让业务系统面向“能力”编程，而不是面向“供应商 API”编程。

## 3. 统一抽象的设计目标

一个好的模型统一抽象，至少应满足以下目标：

### 3.1 能屏蔽供应商差异

业务方应主要关心“我要什么能力”，而不是“这家 API 细节是什么”。

### 3.2 能表达能力差异

统一抽象不能把所有模型都压扁成最小公约数，否则会丢失：

- reasoning effort
- tool use
- structured outputs
- multimodal inputs
- long context
- rerank scoring

### 3.3 能支持路由与降级

统一抽象应便于模型网关做替换、fallback 和策略路由。

### 3.4 能保持扩展性

未来模型能力会继续变化，因此抽象不应过于僵硬。

## 4. 统一抽象的基本原则

### 4.1 先按能力类型抽象，再在能力类型下做细分

比起做一个万能 `Model` 接口，更合理的方式是按能力拆分：

- ChatModel / ResponseModel
- EmbeddingModel
- RerankModel
- VisionModel
- SpeechModel（如有）

### 4.2 抽象公共部分，但保留能力扩展位

统一抽象应包括：

- 通用输入输出结构
- 通用 metadata
- 通用错误模型
- 通用 tracing fields

同时允许保留 provider-specific 或 capability-specific 扩展字段。

### 4.3 不要为了统一而牺牲核心能力

例如不能因为有些模型不支持 structured outputs，就把所有模型接口都退化成纯文本输出。

### 4.4 统一抽象服务于业务稳定，而不是追求接口完美

抽象的目标是降低耦合和提高治理能力，而不是做一个理论上最漂亮但实际难落地的接口体系。

## 5. 推荐的抽象层次

建议将统一抽象拆成三层。

### 5.1 Capability Layer

定义能力类型。

例如：

- `chat`
- `embedding`
- `rerank`
- `vision`
- `speech`

这一层回答的是：系统想调用哪一类模型能力。

### 5.2 Contract Layer

定义每类能力的输入输出契约。

例如：

- 请求参数
- 响应结构
- 流式输出协议
- 错误模型
- trace metadata

这一层回答的是：这一类能力应该以什么统一协议被调用。

### 5.3 Provider Adapter Layer

负责把统一契约映射到底层供应商实现。

这一层回答的是：OpenAI、Anthropic、Gemini、Bedrock、自部署模型各自怎么适配。

## 6. Chat / Response Model 抽象

这是最核心也最容易做坏的一类抽象。

### 6.1 建议的职责

统一表达：

- 文本 / 多模态输入
- 文本或结构化输出
- system / user / assistant / tool messages
- tool definitions
- response format / schema
- streaming
- usage / token metadata

### 6.2 不建议过度收敛的点

以下能力不应被过度抹平：

- reasoning-specific settings
- tool choice policy
- response format support
- built-in tools / remote MCP
- multimodal input support

更合理的方式是：

- 提供统一主接口
- 为高级能力提供 optional capability flags 或 extension fields

### 6.3 推荐抽象思路

与其抽成“永远只支持文本 prompt + 文本 response”，更建议抽成“统一响应生成接口”，允许：

- text input
- image input
- stateful conversation
- structured output
- tool use

这样更接近当前主流 Responses-style interface 的方向。

## 7. Embedding Model 抽象

### 7.1 核心职责

统一表达：

- 输入内容列表
- query/document/task type
- 输出向量
- 向量维度
- usage metadata

### 7.2 关键差异点

不同 embedding 模型可能存在：

- query/document 模式区分
- dimensions 可裁剪
- multilingual 支持差异
- multimodal embedding 支持差异
- single-vector / multi-vector 差异

因此统一抽象时，要允许：

- task type
- modality type
- dimensions
- embedding mode

这类扩展位存在。

## 8. Rerank Model 抽象

### 8.1 核心职责

统一表达：

- query
- documents / candidates
- topK
- 排序分数
- 排序后的候选

### 8.2 为什么应单独抽象

Rerank 与 LLM 和 Embedding 的计算形态不同，如果被混进 chat model 抽象中，后续会很难治理与替换。

### 8.3 关键抽象点

- score semantics
- result ordering
- metadata passthrough
- batch behavior

## 9. Vision Model 抽象

### 9.1 核心职责

统一表达：

- 图像 / PDF / 截图等视觉输入
- 与文本的联合输入
- 视觉理解输出
- 可选结构化输出

### 9.2 当前为什么越来越重要

随着 GPT-4o、Gemini、Qwen-VL、Claude vision、Jina multimodal embedding 等路线成熟，视觉理解已经不再是附加能力，而是越来越多企业场景的正式输入源。

在低代码平台里，vision 抽象未来尤其可能用于：

- 截图理解
- 设计稿理解
- 页面异常截图诊断
- 图表 / 报表理解

## 10. 统一错误模型与元数据抽象

除了能力接口，统一抽象还应覆盖：

### 10.1 错误模型

建议统一至少区分：

- auth error
- quota / rate limit
- timeout
- provider unavailable
- validation error
- unsupported capability
- internal adapter error

### 10.2 usage metadata

建议统一记录：

- input tokens
- output tokens
- latency
- cost estimate
- model id
- provider
- trace id

这样后续 tracing、cost analysis、fallback decision 才有统一基础。

## 11. 能力声明机制

统一抽象除了接口，还应有 capability declaration，用来描述某模型支持什么能力。

例如：

- supportsStreaming
- supportsStructuredOutput
- supportsToolUse
- supportsVisionInput
- supportsReasoningEffort
- maxContextWindow
- supportsDimensionsOverride

这样路由系统和业务系统都能基于能力进行选择，而不是依赖人工记忆。

## 12. 当前行业中的抽象趋势

### 12.1 从 Chat Completion 抽象转向 Response Abstraction

OpenAI 当前明显在推动从 Chat Completions 迁移到 Responses API，这意味着统一抽象也应从“聊天接口”升级到“统一响应生成接口”。

### 12.2 从单模态抽象转向多模态统一接口

现代模型越来越倾向 text + image + tool use + structured output 的统一接口，因此抽象层不能只为纯文本而设计。

### 12.3 从单模型抽象转向能力目录抽象

真正的平台设计重点不再是“封装一个模型 SDK”，而是建立一个 capability catalog + contract system。

## 13. 面向低代码平台如何设计统一抽象

在低代码平台里，统一抽象应特别支持以下几类能力：

- `chat`：问答、总结、解释、建议
- `structured generation`：Schema、DSL、配置生成
- `embedding`：组件、模板、页面、案例检索
- `rerank`：高质量候选排序
- `vision`：截图、设计稿、视觉文档理解

并确保场景开发方不需要关心：

- 底层是哪家模型
- 输入格式如何转
- token 如何记
- fallback 怎么切

## 14. 常见误区

### 14.1 把所有模型硬抽成一个万能接口

结果往往是接口看起来统一了，但高级能力全部丢失。

### 14.2 完全不做抽象，直接在业务层写供应商 SDK

短期快，长期一定失控。

### 14.3 只抽象输入输出，不抽象 metadata 和 errors

这样后续 tracing、cost governance 和 routing 很难统一。

### 14.4 抽象层不包含 capability 声明

这会让路由系统只能靠经验硬编码判断。

## 15. 一句话总结

模型统一抽象的核心，不是把所有模型压扁成一个最小公约数接口，而是建立“按能力类型分层、按契约统一、保留高级特性扩展位”的抽象体系，让业务面向能力开发，平台面向供应商适配和治理演进。
