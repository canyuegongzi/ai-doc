# 成本统计与配额治理

> 状态：第一版草案

## 1. 文档定位

本文档用于定义模型接入层中的成本统计与配额治理设计，重点回答：

- 为什么 AI 系统必须在模型层做成本可见化
- 成本应按哪些维度统计
- 配额治理应如何与路由、限流、降级和租户体系结合
- 如何把成本治理从“月底对账”提升为“运行时控制能力”

## 2. 为什么 AI 平台必须先做成本治理

AI 系统与传统业务系统相比，有一个非常突出的特点：模型调用本身直接消耗费用，而且费用通常与：

- token
- 图像输入
- 音频输入
- 模型档位
- 推理时间
- rerank / embedding 调用量

直接相关。

如果没有成本统计与配额治理，常见后果包括：

- 无法解释钱花在哪
- 高成本场景持续扩张
- 某些租户或场景不受控地耗尽预算
- 架构优化缺少依据
- 高级模型被滥用

## 3. 当前行业最佳实战与趋势

### 3.1 成本治理已成为平台级能力

当前主流模型厂商都在持续强化 usage visibility、model pricing transparency 和 quota controls，这说明行业已经形成共识：

AI 调用成本不能只由业务团队自己感知，而应成为平台统一可观测能力。

### 3.2 成本问题不只是财务问题，而是架构问题

一个场景成本过高，往往反映的是：

- 模型路由不合理
- Prompt 太长
- 检索上下文过度膨胀
- 重试策略失控
- 没有缓存
- 用了不必要的高阶模型

因此成本统计本质上是优化系统设计的输入。

### 3.3 配额治理正在从静态阈值走向动态策略

当前更成熟的做法不是“每月给一个总额度就结束”，而是：

- 按租户 / 用户 / 场景分层治理
- 支持实时预算消耗感知
- 支持超预算降级与限流
- 支持按优先级保留关键能力

## 4. 成本统计解决什么问题

### 4.1 让成本可见

至少要知道：

- 哪个模型最花钱
- 哪个场景最花钱
- 哪个租户最花钱
- 哪类请求平均成本最高

### 4.2 让成本可归因

不仅知道“总共花了多少钱”，还要知道：

- 花在 LLM 还是 embedding / rerank / vision
- 花在 retrieval 还是生成
- 花在正常请求还是重试失败请求

### 4.3 让成本可控制

系统应能基于成本信息实时采取：

- 降级
- 限流
- 路由切换
- 配额告警
- 批处理节流

## 5. 推荐的成本统计维度

建议至少支持以下维度。

### 5.1 按 provider 统计

例如：

- OpenAI
- Anthropic
- Google
- Bedrock
- 自部署模型服务

### 5.2 按 model 统计

例如：

- reasoning model
- standard chat model
- embedding model
- rerank model
- vision model

### 5.3 按 capability 统计

例如：

- chat
- embedding
- rerank
- vision
- tool-augmented generation

### 5.4 按 scene 统计

例如：

- FAQ 问答
- 组件推荐
- Schema 生成
- 配置诊断
- 运维分析

### 5.5 按 tenant / user / project 统计

用于配额管理和内部结算。

### 5.6 按 trace / request 统计

用于单请求成本分析和异常排查。

## 6. 推荐采集的成本数据

每次模型调用建议至少记录：

- provider
- modelId
- capabilityType
- requestId / traceId
- tenantId / userId / sceneType
- input tokens
- output tokens
- image units / audio units（如有）
- retries
- latency
- estimated cost
- timestamp

## 7. 配额治理设计

### 7.1 配额类型

建议区分：

- 硬配额
- 软配额
- 瞬时速率配额
- 周期预算配额

### 7.2 推荐配额维度

至少支持：

- tenant quota
- user quota
- scene quota
- capability quota
- model-tier quota

### 7.3 超配额后的策略

可选策略包括：

- 拒绝请求
- 切换到低成本模型
- 延迟到异步队列
- 仅保留关键场景通行
- 通知管理员或租户负责人

## 8. 成本治理与路由的关系

成本治理不应是单独账本，而应参与路由决策。

例如：

- FAQ 默认走便宜模型
- 高价值场景才允许走 reasoning model
- 某租户预算接近上限时自动切低成本模型
- 长文本场景优先先压缩再调用高价模型

## 9. 成本治理与降级的关系

降级是成本治理的重要执行手段之一。

典型组合包括：

- 超预算 -> 降级模型档位
- 高峰期 -> 限制高成本场景
- 成本异常上升 -> 禁用高成本 optional feature

## 10. 成本治理与评测的关系

成本统计只有和质量评测结合才有意义。

因为平台真正关心的是：

- 单位成本带来的效果提升是多少
- 更贵模型是否真的值得
- rerank / long context / reasoning 带来的增益是否匹配支出

因此，成本治理应与 evals 联合分析，而不是单纯压低成本。

## 11. 面向低代码平台的典型治理策略

### 11.1 组件问答

- 默认使用低成本模型
- retrieval 上下文做严格裁剪
- 高频缓存优先

### 11.2 Schema 生成

- 对复杂高价值任务允许使用更强模型
- 但必须结合结构化校验，避免重复无效生成造成浪费

### 11.3 配置诊断

- reasoning model 只在需要时使用
- 尽量先靠规则、检索和轻量分析缩小问题空间

### 11.4 批量 embedding / 索引构建

- 单独预算池
- 分批处理
- 速率上限
- 支持暂停与恢复

## 12. 常见误区

### 12.1 只看模型单价，不看全链路成本

很多成本浪费来自：

- 长上下文膨胀
- 检索噪声
- 重试风暴
- 结构化失败反复重跑

### 12.2 只按总量做预算，不按场景拆分

这样很难定位真正的高成本问题点。

### 12.3 没有实时策略，只做月度统计

这样统计结果虽然有，但无法在运行时控制问题。

### 12.4 为了省钱一刀切降级所有场景

成本优化必须和业务价值、风险等级、质量要求联动。

## 13. 一句话总结

成本统计与配额治理的核心，是让 AI 平台能够按 provider、model、scene、tenant、request 等多维度看清成本、归因成本并实时控制成本，使模型层从“黑盒花钱”变成“可观测、可预算、可动态调度”的运营对象。
