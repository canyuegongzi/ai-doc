# 索引更新机制

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的索引更新机制，重点回答：

- 知识库如何从“静态构建”升级为“持续更新系统”
- 更新、删除、重建、版本切换应如何设计
- 近两年更成熟的企业知识系统在增量更新方面有哪些实践变化

## 2. 为什么索引更新机制是知识工程生产化关键

很多 demo 级系统在初次建库时效果不错，但一进入真实使用就很快失效。根因通常不是 retrieval 算法本身，而是索引更新机制不成熟。

典型问题包括：

- 文档更新了，索引没更新
- 文档删除了，旧 chunk 还在召回
- 元数据变了，但 retrieval 仍按旧标签工作
- embedding 模型升级了，但旧向量和新向量混杂
- 不同场景对“最新”定义不同，系统无法表达

因此，索引更新机制不是运维小事，而是知识系统真实性与可信度的基础。

## 3. 近两年的关键变化

### 3.1 从“离线批处理重建”走向“增量 upsert + 文档管理”

LlamaIndex 的 document management pipeline 明确引入了 doc hash、upsert、delete、sync 这些能力，说明行业实践已经不再满足于一次性建库。

### 3.2 更新机制开始和文档管理、权限同步、版本治理绑定

更成熟的知识系统不会只更新向量，还会同步更新：

- metadata
- 权限信息
- source reference
- citation 信息
- 文档版本状态

### 3.3 索引更新越来越像数据工程而不是脚本任务

随着数据源增多、更新频率增大、RAG 场景变复杂，索引更新越来越需要：

- 事件驱动
- 失败重试
- 状态管理
- 版本切换
- 可观测性

这使它更接近实时数据工程系统。

## 4. 索引更新解决什么问题

### 4.1 保证知识新鲜度

让检索结果尽量反映当前最新的有效知识。

### 4.2 保证知识一致性

避免源数据和索引状态长期偏离。

### 4.3 保证治理正确性

当权限、版本、生命周期变化时，索引也应同步变化。

### 4.4 保证平台可运营

让知识系统可以长期运行，而不是每次靠人工全量重建。

## 5. 推荐的更新模式

### 5.1 全量重建

适用于：

- 初次建库
- 小规模数据源
- embedding 大版本变更
- 索引结构大调整

优点：

- 实现简单
- 一致性直观

缺点：

- 成本高
- 耗时长
- 容易影响在线系统

### 5.2 增量 upsert

适用于：

- 文档增量变化
- 高频更新数据源
- 正常生产运行

优点：

- 成本低
- 更新快
- 更适合生产系统

### 5.3 删除同步

当源对象被删除、下线或失效时，索引必须同步删除或标记不可用。

### 5.4 版本化更新

适用于：

- embedding 模型升级
- chunk 策略变更
- 索引结构升级

通常需要支持旧版本和新版本并行一段时间。

## 6. 推荐的更新流水线

建议将索引更新拆成以下阶段：

1. 变更检测
2. 变更分类
3. 影响范围计算
4. 对象重解析 / 重切分
5. 重新向量化
6. 索引 upsert / delete
7. 引用映射更新
8. 状态记录与指标更新

## 7. 变更检测机制

常见方式包括：

### 7.1 时间戳检测

适用于来源系统有可靠 `updatedAt` 字段。

### 7.2 内容 hash 检测

适用于需要精确判断内容是否真的变化。

### 7.3 事件驱动检测

适用于来源系统可发 webhook 或事件。

### 7.4 版本号检测

适用于文档、配置、schema 有明确版本字段。

实际系统中，通常应组合使用而不是单一依赖。

## 8. 更新粒度设计

### 8.1 文档级更新

适合结构简单、文档整体变化频繁不大的场景。

### 8.2 chunk 级更新

适合希望减少无效重建成本的场景。

### 8.3 对象级更新

适合结构化对象场景，例如：

- 单组件定义变更
- 单页面模板变更
- 单逻辑节点定义变更

近两年更成熟的知识系统越来越倾向于细粒度更新，而不是每次全对象重建。

## 9. 更新机制中的一致性问题

更新系统必须考虑以下一致性风险：

- 新 metadata 已更新，但旧向量还没删
- 新 chunk 已入库，但旧 chunk 还在召回
- citation 指向旧对象
- 权限范围已变，但 filter 还没同步

因此，建议至少保证：

- 写入顺序可控
- 中间状态可追踪
- 失败时可恢复
- 更新结果可验证

## 10. 版本切换与重建策略

当发生以下情况时，通常不宜只做普通增量更新：

- embedding 模型大版本变更
- chunk 策略大幅变化
- metadata 结构重构
- 索引类型从 dense-only 切到 hybrid

更合理的做法通常是：

- 构建新版本索引
- 做灰度流量切换
- 验证后切主
- 回收旧索引

这和数据库 schema / search index migration 的思路更接近。

## 11. 更新失败处理与可观测性

索引更新是长期运行任务，必须具备：

- 更新状态表
- 失败重试
- 死信队列
- 同步延迟监控
- 更新成功率监控
- stale data 检测
- 版本切换日志

没有这些能力，系统会逐渐积累隐性脏索引。

## 12. 与权限和治理的关系

索引更新不仅更新内容，还应同步：

- permissionScope
- tenantScope
- sensitivityLevel
- deletedFlag
- status

否则最容易出现的问题是：

内容更新正确了，但治理边界仍然是旧的。

## 13. 面向低代码平台的更新重点

在低代码平台场景中，建议重点关注以下更新类型：

### 13.1 组件协议更新

影响：

- 组件问答
- 组件推荐
- Schema 生成约束

### 13.2 页面模板更新

影响：

- 页面生成
- 模板推荐
- 最佳实践检索

### 13.3 错误案例更新

影响：

- 配置纠错
- 运维诊断

### 13.4 发布规范与运维规则更新

影响：

- Agent 执行前规范查询
- 发布辅助与诊断链路

这些知识通常都具有“更新后应尽快生效”的需求。

## 14. 常见误区

### 14.1 用全量重建当长期默认策略

在规模和更新频率上来后，这通常不可持续。

### 14.2 只更新新增，不处理删除和失效

这会导致知识库越来越脏。

### 14.3 embedding 升级时混用旧新索引

如果不做版本治理，检索表现会很不稳定。

### 14.4 更新系统没有状态追踪

出了问题后几乎无法排查是采集错、解析错还是索引错。

## 15. 一句话总结

索引更新机制的核心，是把知识库从一次性构建结果变成可持续同步、可增量 upsert、可删除、可版本切换、可观测的长期运行系统；近两年的最佳实践也越来越清晰，成熟知识系统必须具备文档管理、细粒度更新和索引版本治理能力。
