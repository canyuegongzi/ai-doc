# 清洗与标准化

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的清洗与标准化（cleaning & normalization）设计，重点回答：

- 为什么文档在解析后还必须经过清洗与标准化
- 哪些噪声最容易损害后续检索与生成质量
- 企业知识系统应如何把原始内容转化为稳定、可索引、可引用、可治理的知识对象

## 2. 为什么清洗与标准化是知识工程核心环节

很多团队会把知识工程理解为：

采集 -> 解析 -> 切分 -> 向量化

但在真实系统里，如果跳过清洗与标准化，后续往往会出现：

- chunk 质量很差
- metadata 语义混乱
- 同一对象有多种命名方式
- 文本里充满页码、页眉、目录噪声
- 表格与代码块被污染
- 相同内容重复入库

因此，清洗与标准化的作用是把“可读内容”变成“可被系统稳定消费的知识单元”。

## 3. 近两年知识工程中的明显变化

### 3.1 清洗不再只是去脏字符，而是知识对象整形

过去很多清洗逻辑停留在：

- 去空格
- 去换行
- 去 HTML 标签

但近两年的企业 RAG 实践越来越清楚：真正重要的是把内容整理成统一知识对象形态。

例如：

- 统一标题层级
- 修正段落边界
- 去除模板噪声
- 提取字段定义
- 统一术语别名
- 保留结构单元

### 3.2 标准化开始服务下游多个系统，而不只服务检索

现在的知识对象往往不仅服务 RAG，还服务：

- 结构化生成
- Agent 规范查询
- 评测集构造
- 案例沉淀
- 规则校验

这意味着标准化后的对象要更稳定、可追踪、可复用。

### 3.3 多模态与半结构化文档要求更精细的清洗策略

对于：

- 扫描 PDF
- 图文混排文档
- 表格和图表
- 代码与配置片段

清洗已经不只是“删字符”，而是“保结构、降噪声、保语义边界”。

## 4. 清洗与标准化解决什么问题

### 4.1 噪声问题

去除对检索和生成没有帮助甚至有害的内容。

### 4.2 一致性问题

统一术语、命名、结构表达和对象格式。

### 4.3 冗余问题

消除重复内容、重复块、模板尾部、目录重复等。

### 4.4 可治理问题

确保对象能被可靠追踪、更新、引用和过滤。

## 5. 清洗的对象层次

建议按四个层次处理。

### 5.1 字符层

处理：

- 编码问题
- 异常字符
- 多余空白
- 错乱换行
- 控制字符

### 5.2 文本块层

处理：

- 重复段落
- 页眉页脚
- 目录噪声
- 页码残留
- 无意义模板文案

### 5.3 结构层

处理：

- 标题层级恢复
- 段落边界修正
- 列表项归并
- 表格行列结构保留
- 代码块边界修复

### 5.4 语义层

处理：

- 同义术语统一
- 字段别名映射
- 对象名规范化
- 版本信息抽取
- 规则项拆分

## 6. 推荐的清洗步骤

### 6.1 基础文本清洗

包括：

- 去无效空白
- 去乱码
- 统一编码
- 统一标点风格（按需）

### 6.2 噪声块清洗

包括：

- 去页眉页脚
- 去目录残片
- 去页面导航元素
- 去重复模板提示

### 6.3 结构修复

包括：

- 合并被错误拆开的段落
- 修复标题与正文关系
- 修复代码块断裂
- 恢复表格结构或表格摘要

### 6.4 语义标准化

包括：

- 统一组件命名
- 统一字段命名
- 统一业务术语
- 统一时间和版本格式

### 6.5 冗余与重复处理

包括：

- 块级重复检测
- 文档级重复检测
- 模板重复片段去重

## 7. 标准化的核心目标

标准化不是让所有内容看起来一样，而是让知识对象以稳定、可预测的格式进入后续系统。

建议至少在以下方面标准化。

### 7.1 对象结构标准化

例如统一为：

- title
- content blocks
- block types
- metadata
- source references

### 7.2 术语标准化

例如统一：

- 组件名称
- 模块名称
- 页面类型名称
- 字段名称
- 配置项名称

### 7.3 时间与版本标准化

例如统一：

- updatedAt
- version
- publishVersion
- schemaVersion

### 7.4 引用定位标准化

例如统一：

- documentId
- sectionId
- blockId
- pageNumber

## 8. 面向不同数据源的清洗策略差异

### 8.1 文档型数据源

重点：

- 去噪声
- 保标题层级
- 保章节引用能力

### 8.2 结构化配置型数据源

重点：

- 去无意义字段
- 保留关键字段与关系
- 生成面向检索的摘要

### 8.3 代码型数据源

重点：

- 保留语法块
- 保留注释
- 不要把格式清洗得失去语义边界

### 8.4 日志与案例型数据源

重点：

- 提取关键事件
- 去掉无关上下文噪声
- 保留时间、模块、错误码等关键字段

### 8.5 视觉文档型数据源

重点：

- 清理 OCR 噪声
- 保留版面结构线索
- 保留图表与表格语义

## 9. 清洗如何服务后续 chunking

优质 chunking 依赖优质清洗。清洗阶段应尽量为切分提供：

- 稳定段落边界
- 明确标题层级
- 独立代码块
- 独立表格块
- 去重后的纯净知识块

如果清洗做不好，后续 chunking 很容易把：

- 标题和正文切开
- 表格拆坏
- 重复内容切成多个块
- 代码注释与代码脱离

## 10. 清洗如何服务 metadata

标准化阶段应尽量同步抽出可用 metadata，例如：

- 组件名
- 页面类型
- API 路径
- 模块名
- 错误码
- 数据源名
- 标签

这样 metadata 就不是完全后置猜测，而是从清洗阶段就开始建立。

## 11. 面向低代码平台的清洗重点

在低代码平台场景中，建议重点做以下标准化：

- 统一组件名与别名
- 统一 Schema 字段命名
- 统一 setter / slot 术语
- 统一页面模板类别
- 抽取错误案例的错误码、报错描述、修复动作
- 对配置样本生成简洁摘要，方便后续检索

这类标准化对推荐、生成和诊断都会直接增益。

## 12. 常见误区

### 12.1 清洗只做字符处理

这远远不够，真正价值在结构和语义层。

### 12.2 为了干净把结构信息也删掉

这会让后续 chunking 和 retrieval 丢失关键语义边界。

### 12.3 不做术语统一

在企业知识中，同一个对象多种叫法是检索质量下降的重要原因之一。

### 12.4 标准化过度

如果把不同对象过度压平，反而会损失领域语义和结构特色。

## 13. 一句话总结

清洗与标准化的核心，不是把文本变干净，而是把异构原始内容整形成稳定、可追踪、可切分、可检索、可治理的知识对象；近两年的知识工程实践也越来越清楚，真正有价值的不是“清理字符”，而是“构建高质量知识单元”。
