# 索引版本管理

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的索引版本管理（index versioning）设计，重点回答：

- 为什么索引必须有明确版本概念
- 在 embedding 升级、chunk 策略变更、metadata 重构和索引结构升级时，如何避免直接破坏在线效果
- 索引版本应如何与更新机制、检索链路、评测和回滚能力协同

## 2. 为什么索引必须做版本管理

在 demo 阶段，很多知识库只有一个“当前索引”。

但在真实系统中，以下变化都可能导致索引逻辑发生根本变化：

- embedding 模型升级
- chunk 策略调整
- metadata 字段变更
- dense-only 切到 hybrid
- 权限模型重构
- 解析策略升级

如果没有版本管理，常见后果是：

- 旧向量和新向量混杂
- 检索结果不可比较
- 回滚困难
- 评测无法做新旧对比
- 故障时无法快速恢复旧版本

因此，索引版本管理是知识系统可演进性的基础设施。

## 3. 近两年的关键变化

### 3.1 embedding 和 retrieval 迭代速度越来越快

随着 embedding 模型更新频繁、hybrid / rerank / long-context retrieval 不断演化，索引不再是“建好就不动”的静态资产，而是持续演进对象。

### 3.2 企业知识系统越来越需要灰度和双轨运行

更成熟的实践通常不会直接替换旧索引，而是：

- 构建新版本索引
- 做离线评测
- 做灰度流量验证
- 再切主
- 必要时回滚

这和搜索系统、特征系统、数据库 schema migration 的演进思路越来越相似。

### 3.3 版本管理正在从“命名习惯”升级为正式控制面能力

不仅要给索引起版本号，更要管理：

- version metadata
- active / shadow state
- migration history
- rollback strategy
- lineage

## 4. 什么变化必须触发新索引版本

建议以下情况至少考虑新版本索引。

### 4.1 embedding 模型变更

例如：

- 模型家族变化
- 向量维度变化
- query/document 策略变化

### 4.2 chunk 策略变化

例如：

- 固定窗口切分改为结构感知切分
- 段落级改为对象级切分
- overlap 策略调整

### 4.3 metadata 结构变化

例如：

- 新增关键过滤字段
- 权限字段模型变化
- sourceType 重新分类

### 4.4 索引形态变化

例如：

- dense-only -> hybrid
- 单 collection -> 多 namespace
- 索引引擎切换

### 4.5 解析与标准化逻辑变化

例如：

- 解析器升级
- 表格 / 代码块处理逻辑变化
- 文档摘要策略变化

## 5. 推荐的版本标识设计

建议每个索引版本至少记录：

- indexVersion
- embeddingModelVersion
- chunkStrategyVersion
- parsingVersion
- metadataSchemaVersion
- indexType
- buildTime
- buildScope
- status
- lineage

这样后续才有可能定位：

一次效果变化究竟来自哪一层知识工程变更。

## 6. 推荐的版本状态模型

建议至少区分以下状态：

- `building`
- `shadow`
- `active`
- `deprecated`
- `rollback-ready`
- `archived`

其中：

- `shadow` 表示已构建完成但尚未正式接流量
- `active` 表示线上主版本
- `rollback-ready` 表示可以快速恢复使用

## 7. 推荐的版本切换策略

### 7.1 双轨并行构建

旧版本继续服务，新版本后台构建。

### 7.2 离线评测

对新版本做：

- recall 对比
- rerank 后效果对比
- 引用质量对比
- metadata filter 正确性验证

### 7.3 灰度验证

将部分流量切到新版本，对比：

- 命中率
- 用户反馈
- 成本变化
- 延迟变化

### 7.4 主版本切换

验证通过后切主。

### 7.5 回滚预案

若新版本异常，应可迅速回滚到旧版本。

## 8. 版本管理与更新机制的关系

索引更新机制负责“把变化同步进去”，索引版本管理负责“当变化足够大时，如何安全演进系统”。

可以理解为：

- 增量更新解决日常同步
- 版本管理解决结构演进

两者不能互相替代。

## 9. 版本管理与评测的关系

索引版本管理必须和 evals 绑定，否则版本号只是标签，没有实际决策意义。

建议至少对每个新版本保留：

- 版本评测报告
- 主要差异点
- 主要收益
- 已知退化点
- 是否通过上线门槛

## 10. 版本管理与回溯能力

每个检索结果最好都能追溯到它来自哪个索引版本。

这对以下场景很重要：

- 线上问题排查
- 结果解释
- A/B 对比
- 审计
- 故障回滚

## 11. 面向低代码平台的索引版本管理重点

在低代码平台场景中，以下变化尤其值得触发新版本：

### 11.1 组件协议体系升级

会直接影响组件检索、配置生成和诊断。

### 11.2 页面 Schema 表达方式变化

会影响模板检索和结构化生成约束。

### 11.3 错误案例标准化规则调整

会影响诊断场景效果。

### 11.4 权限字段与租户隔离模型变更

会影响所有 retrieval filter 正确性。

## 12. 常见误区

### 12.1 embedding 模型升级后直接覆盖旧索引

这样一旦线上退化，很难快速回滚。

### 12.2 不记录索引构建 lineage

后续很难判断当前索引具体由哪些策略组成。

### 12.3 只有版本号，没有灰度与评测

这只是命名，不是真正的版本管理。

### 12.4 旧索引立即删除

一旦新版本异常，会失去最重要的恢复手段。

## 13. 一句话总结

索引版本管理的核心，是把知识系统的结构演进纳入正式控制面：任何会显著影响检索语义、过滤逻辑、chunk 结构或引用能力的变化，都应通过“构建新版本 -> 评测 -> 灰度 -> 切主 -> 可回滚”的方式推进，而不是直接覆盖线上索引。
