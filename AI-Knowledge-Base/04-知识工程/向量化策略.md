# 向量化策略

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的向量化策略（embedding strategy），重点回答：

- 什么内容应该被向量化，什么内容不应该简单直接向量化
- 向量化对象、模型、维度、模式和任务类型应如何选择
- 近两年 embedding 能力快速演进后，企业知识工程的向量化策略应如何升级

## 2. 为什么向量化不是“调用 embedding 接口”这么简单

很多团队在做知识库时，会把向量化理解成：

- 取文本
- 调 embedding 模型
- 存入向量库
- 完成

但在真实系统中，向量化策略直接决定：

- 检索效果
- 索引成本
- 更新成本
- 多语言表现
- 结构化对象的可检索性
- 混合检索能力上限

因此，向量化不是技术细节，而是知识工程中的一级设计问题。

## 3. 近两年的关键变化

### 3.1 向量化对象已经从“纯文本段落”扩展到“多类型知识对象”

当前更成熟的系统不再只向量化长文本段落，而会向量化：

- FAQ 问答对
- 组件说明
- 结构化对象摘要
- 代码块
- 表格单元
- 错误案例
- 视觉文档内容

这意味着向量化策略必须和 objectType 强绑定，而不是默认所有输入都是 paragraph。

### 3.2 dense-only 不再是默认最优

近两年的企业检索实践已经越来越清楚：

- dense 检索擅长语义召回
- sparse / lexical 检索擅长关键词、术语、编号、精确匹配
- hybrid 往往更稳

因此，向量化策略通常需要与稀疏表示策略联合设计，而不是只看 dense embedding。

### 3.3 向量化与 dimensions / multi-vector / multimodal 越来越相关

随着 OpenAI dimensions override、BGE-M3、Jina v4、Cohere Embed 4、Qwen3 Embedding 等路线成熟，向量化策略已经不只是“模型选哪个”，还涉及：

- 维度裁剪
- query/document 模式
- multi-vector
- multimodal embedding

## 4. 向量化解决什么问题

### 4.1 让语义检索成为可能

将知识对象编码为语义向量，使系统可以按含义而不是只按关键词匹配。

### 4.2 让异构表达之间可以对齐

不同说法、不同语言、不同风格描述可以映射到相近空间。

### 4.3 让复杂知识对象进入统一检索框架

例如：

- 文本段落
- 配置摘要
- 组件能力描述
- 错误案例描述

都可以通过向量化进入相似检索链路。

## 5. 什么内容应该向量化

适合向量化的对象通常具备以下特征：

- 存在语义表达
- 用户可能用自然语言去查它
- 需要和其他对象做相似匹配
- 适合被 embedding 模型压缩成语义表示

典型包括：

- 文档段落
- FAQ 问答对
- 组件描述
- 页面模板说明
- API 说明摘要
- 错误案例摘要
- 代码说明片段

## 6. 什么内容不应直接原样向量化

以下内容如果直接原样向量化，常常效果不佳：

- 超长原始 JSON
- 噪声很大的日志全文
- 纯表格原始矩阵
- 高度重复模板内容
- 没有语义摘要的结构化配置对象

这类对象更适合：

- 先摘要
- 先结构化抽取
- 先规则化标准化
- 再决定是否向量化

## 7. 推荐的向量化对象层次

建议至少区分以下几类向量化对象。

### 7.1 文本块向量化

适用于：

- 章节段落
- 说明文档
- FAQ 内容

### 7.2 结构化对象摘要向量化

适用于：

- 页面 Schema 摘要
- 组件能力摘要
- 数据源定义摘要
- 逻辑节点摘要

### 7.3 案例单元向量化

适用于：

- 错误案例
- 复盘条目
- 工单摘要

### 7.4 多模态对象向量化

适用于：

- 图文混排文档
- 截图说明
- 视觉文档

## 8. 向量化策略的关键设计维度

### 8.1 模型选择

需要考虑：

- 语言覆盖
- 长输入支持
- 维度
- 成本
- 私有化部署能力
- query/document distinction
- multimodal support

### 8.2 维度选择

维度影响：

- 检索效果
- 存储成本
- ANN 查询性能
- 网络传输成本

高维不一定永远更好。平台应根据场景和预算做平衡。

### 8.3 输入模式选择

例如：

- query mode
- document mode
- code mode
- multimodal mode

### 8.4 single-vector 还是 multi-vector

对于一些复杂场景，multi-vector 可以保留更多细粒度表示，但成本和索引复杂度也更高。

### 8.5 dense 还是 hybrid

很多企业场景更适合：

- dense + sparse
- dense + metadata filter
- dense + rerank
- dense + sparse + rerank

## 9. 向量化前的预处理建议

向量化前应尽量确保对象已经经过：

- 清洗与标准化
- 结构边界修复
- 术语统一
- metadata 补齐

对于结构化对象，建议额外考虑：

- 生成摘要文本
- 保留关键字段词汇
- 对字段和值做语义压缩

## 10. 面向不同对象的向量化策略建议

### 10.1 说明文档类

可直接按 chunk 进行向量化。

### 10.2 结构化对象类

建议先生成“语义摘要 + 关键字段摘要”再向量化。

### 10.3 表格类

建议按行组、字段定义或表格摘要向量化，而不是整张表直接向量化。

### 10.4 日志类

建议先抽取错误事件、错误码、模块和摘要，再向量化。

### 10.5 代码类

建议保留代码块边界，并可加入注释摘要或函数说明。

## 11. 向量化与 retrieval 的关系

向量化策略必须和 retrieval 一起设计。

如果后续 retrieval 采用：

- metadata pre-filter
- hybrid search
- rerank
- sourceType routing

那么向量化时就应提前考虑：

- 哪些对象进入 dense index
- 哪些对象进入 sparse index
- 哪些对象需要同时保留 lexical signal
- 哪些对象只适合作为 metadata filter 对象

## 12. 向量化与成本的关系

向量化本身也是持续成本项，尤其在：

- 大规模重建
- 高频增量更新
- 多版本索引并存
- 高维向量存储

场景下更明显。

因此，向量化策略应同时考虑：

- 哪些对象值得长期向量化
- 哪些对象应按需构建
- 哪些对象适合摘要后再向量化

## 13. 面向低代码平台的向量化建议

在低代码平台中，建议重点向量化：

- 组件功能说明
- 组件 props / setter / slot 摘要
- 页面模板说明
- Schema 样本摘要
- 逻辑节点定义摘要
- 错误案例摘要
- 数据源说明

不建议直接原样向量化：

- 超长原始 schema
- 杂乱日志全文
- 无清洗的配置快照

## 14. 常见误区

### 14.1 所有内容都直接原样向量化

这通常会造成高噪声检索。

### 14.2 只关注 embedding 模型，不关注对象表示方式

很多效果问题并不在模型，而在向量化对象设计本身。

### 14.3 dense-only 思维

现代企业检索很多时候更适合 hybrid 或 multi-stage retrieval。

### 14.4 维度和成本完全不纳入设计

这会导致索引规模快速失控。

## 15. 一句话总结

向量化策略的核心，不是把所有文本都送进 embedding 模型，而是根据对象类型、任务目标、检索模式和成本边界，设计“什么内容被如何表示”的整体方案；近两年的实践也越来越说明，真正有效的向量化，一定是 object-aware、task-aware、hybrid-aware 的。
