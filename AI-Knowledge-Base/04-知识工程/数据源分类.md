# 数据源分类

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 知识工程体系中的数据源分类方法，回答：

- 企业 AI 系统中的知识到底来自哪些数据源
- 为什么要先做数据源分类，而不是直接谈向量化和检索
- 不同数据源在采集、解析、更新、权限和索引方式上有什么本质差异

## 2. 为什么知识工程要先做数据源分类

知识工程最常见的误区之一，是一开始就把所有内容都当成“文档”。

但在真实企业系统中，知识来源往往高度异构，包括：

- 长文档
- 页面配置
- 组件协议
- 表格
- 代码仓库
- 工单记录
- 日志
- API 定义
- 数据库元信息
- 图片、PDF、截图

如果不做数据源分类，后续会出现几个严重问题：

- 不同类型内容被用相同 chunk 策略粗暴处理
- 元数据设计混乱
- 权限过滤无法准确落地
- 更新策略一刀切
- 检索质量不稳定

因此，数据源分类不是整理目录，而是知识工程的第一步建模工作。

## 3. 当前行业最佳实战与近两年变化

### 3.1 知识源正在从“文本文件集合”演进为“企业信息资产集合”

过去很多 RAG 系统只处理：

- PDF
- Word
- Markdown
- FAQ 文本

但近两年企业知识系统的主流趋势已经明显变化。主流实践越来越强调：

- 结构化数据与非结构化数据并存
- 视觉文档与文本内容并存
- 代码、配置、日志、表格都应进入知识体系
- 任务执行系统需要消费更结构化、更高质量的知识对象

这意味着“文档库”已经不足以概括知识工程的输入范围。

### 3.2 agentic RAG 反向推动数据源建模更细

近两年随着 agentic RAG、tool-augmented RAG 和复杂检索链路发展，知识工程不再只服务问答，还服务于：

- 任务规划
- 工具调用前的规范查询
- 结构化生成
- 诊断与修复

这使得数据源分类必须更细，因为不同任务需要不同类型的知识对象。

### 3.3 文档理解正在从纯文本解析走向多模态理解

OpenAI File Search、Cohere 企业文档解析实践、Jina 与多模态 embedding 路线都说明：

- PDF 不再只是“提文本”
- 表格、图表、布局、图片、截图也属于知识输入
- 文档结构本身就是知识的一部分

因此，数据源分类必须显式容纳视觉文档与混合文档。

## 4. 数据源分类的总体原则

### 4.1 先按知识对象类型分类，再按存储位置分类

不要只按“来自哪套系统”分类，更要按“它本质是什么对象”分类。

### 4.2 先按用途分类，再按技术格式分类

例如页面 Schema 虽然最终可能存成 JSON，但在知识工程里它不是普通 JSON 文件，而是平台领域对象。

### 4.3 先按更新方式和权限边界分类

这些属性会直接影响：

- ingestion 频率
- 索引方式
- metadata 设计
- 检索过滤

## 5. 推荐的数据源一级分类

建议将企业 AI 知识源至少分为八大类。

### 5.1 文档型数据源

典型包括：

- 产品文档
- 使用手册
- 设计规范
- FAQ
- 培训材料
- 会议纪要
- 制度说明

特点：

- 以长文本为主
- 有章节、段落和标题结构
- 常用于 FAQ 问答和知识解释

### 5.2 结构化业务对象型数据源

典型包括：

- 页面 Schema
- 组件定义
- Setter 协议
- 数据源配置
- 逻辑编排节点
- API 定义
- 数据模型定义

特点：

- 结构化强
- 字段语义明确
- 不适合被简单当成长文处理
- 常用于生成增强、推荐和诊断

### 5.3 半结构化文档型数据源

典型包括：

- HTML 页面
- Confluence / 语雀页面
- 混排 PDF
- 带表格说明文档
- 邮件线程

特点：

- 结构不完全规整
- 常包含标题、表格、列表、链接、图片
- 解析质量直接影响后续检索质量

### 5.4 代码与技术资产型数据源

典型包括：

- 代码仓库
- 配置文件
- SQL 脚本
- CI/CD 配置
- SDK 示例
- 接口实现样例

特点：

- 语法结构强
- 更适合代码块级或文件级切分
- 常用于代码检索、配置诊断和生成辅助

### 5.5 运行与运维型数据源

典型包括：

- 构建日志
- 发布记录
- 错误日志
- 监控事件
- 运行状态快照
- 告警记录

特点：

- 时效性强
- 生命周期短
- 常用于诊断、排障和运维辅助
- 更接近“运行数据知识源”

### 5.6 交互与经验型数据源

典型包括：

- 工单记录
- 问答记录
- 历史处理案例
- 故障复盘
- 人工修复经验

特点：

- 语义价值高
- 非结构化程度较高
- 非常适合做案例检索和经验增强

### 5.7 多模态与视觉文档型数据源

典型包括：

- 扫描版 PDF
- 图片化文档
- 截图
- 设计稿
- 图表
- 报表截图

特点：

- 文本并不总是直接可用
- 布局和视觉结构本身很重要
- 越来越适合进入现代知识工程链路

### 5.8 外部系统与实时接口型数据源

典型包括：

- 实时数据库查询
- 搜索系统
- 第三方 API
- 业务服务实时数据

特点：

- 不一定预先入库
- 更接近在线知识查询或工具调用
- 与传统离线索引式知识源不同

## 6. 推荐的数据源维度标签

除了一级分类，还建议为每个数据源对象补充一组标准标签。

### 6.1 来源标签

例如：

- sourceSystem
- sourceType
- ownerTeam
- syncMode

### 6.2 业务标签

例如：

- bizDomain
- moduleType
- appCode
- tenantCode
- sceneTags

### 6.3 内容标签

例如：

- contentFormat
- structureLevel
- modality
- language
- docType

### 6.4 治理标签

例如：

- permissionScope
- sensitivityLevel
- lifecycleType
- updateStrategy

这些标签后续会直接决定 Metadata 设计与检索过滤能力。

## 7. 面向知识工程的分类维度建议

每个数据源至少应从以下六个维度描述：

### 7.1 内容结构

它是：

- 纯文本
- 半结构化文档
- JSON / YAML / schema
- 代码
- 表格
- 图像 / PDF /视觉文档

### 7.2 更新频率

它是：

- 高稳定
- 周期更新
- 高频更新
- 实时变化

### 7.3 权限边界

它是：

- 公开知识
- 租户隔离知识
- 角色受限知识
- 高敏感知识

### 7.4 典型用途

它更常用于：

- FAQ
- 搜索推荐
- 生成辅助
- 执行前校验
- 诊断
- 运维分析

### 7.5 解析策略

它更适合：

- 段落切分
- 标题层级切分
- 表格解析
- 代码块切分
- 结构化字段抽取
- 视觉布局解析

### 7.6 索引策略

它更适合：

- 向量索引
- 稀疏索引
- 结构化索引
- 图索引
- 混合索引

## 8. 当前企业知识工程中最容易被忽视的数据源

近两年在企业实践中，最容易被低估但价值很高的数据源包括：

### 8.1 历史案例与工单

它们通常最贴近真实问题和真实处理路径。

### 8.2 配置对象与 Schema 样本

在生成型与诊断型场景中，这类知识往往比长文档更有价值。

### 8.3 运行日志与状态记录

在排障和运维 AI 场景中，这些是最关键的数据源之一。

### 8.4 视觉文档

很多企业知识并不在可直接复制文本的文档中，而在截图、表格和扫描件里。

## 9. 面向低代码平台的数据源映射

在低代码平台场景中，建议重点关注以下数据源分类。

### 9.1 领域定义型

- 组件协议
- Setter 定义
- 插槽规则
- 数据源定义
- 逻辑节点定义

### 9.2 配置样本型

- 页面 Schema 样本
- 表单模板
- 查询模板
- 布局模板

### 9.3 运维诊断型

- 构建日志
- 发布记录
- 错误案例
- 历史修复路径

### 9.4 知识说明型

- 平台使用文档
- FAQ
- 组件说明
- 内部规范

这说明低代码平台的知识工程绝不能只从“产品文档”出发，而必须从“平台对象 + 配置对象 + 运行对象 + 文档对象”四类资产共同建模。

## 10. 常见误区

### 10.1 把所有数据源都当成文档

这会直接损害知识工程质量。

### 10.2 只做静态文档，不接运行数据和案例数据

这样系统只会“讲道理”，不会“解决真实问题”。

### 10.3 不做权限与生命周期分类

后续检索、治理和索引更新会非常被动。

### 10.4 只按技术格式分类，不按业务对象分类

这会让数据源与场景脱节。

## 11. 一句话总结

数据源分类的核心，不是给资料分文件夹，而是为知识工程建立“按对象、结构、更新、权限、用途和索引方式”组织知识资产的基础模型；近两年的趋势也已经非常明确，企业知识源正在从纯文本文件扩展到结构化对象、运行数据、案例资产和多模态文档的统一集合。
