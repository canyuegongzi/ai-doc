# 索引构建

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的索引构建设计，重点回答：

- 知识对象在完成清洗、切分和向量化后，应该如何组织为可检索索引
- dense、sparse、hybrid、metadata filter 等能力应如何组合
- 近两年向量索引与混合检索系统的工程实践有哪些重要变化

## 2. 为什么索引构建不是“存进向量库”

很多系统把索引构建简单理解为：

- chunk -> embedding -> upsert vector db

但在真实知识系统中，索引构建至少要回答：

- 存哪些向量
- 存哪些稀疏表示
- metadata 怎么组织
- 多租户怎么隔离
- 哪些对象用 dense，哪些对象用 sparse 或 hybrid
- 是否需要对象级索引与 chunk 级索引并存
- 如何支持引用、删除、更新和版本切换

因此，索引构建本质上是“检索结构设计”，而不是单纯存储步骤。

## 3. 近两年的关键变化

### 3.1 hybrid index 正在成为主流工程选择

Pinecone、Weaviate、Qdrant 等近年的文档和产品路线都在强化 hybrid search 能力，这说明行业已经形成一个明确判断：

仅靠 dense 向量检索，很难稳定覆盖企业场景中的术语、编号、字段名、错误码、组件名等精确匹配需求。

### 3.2 索引类型开始更强调动态和多形态共存

Weaviate 的 vector index 文档已经明确支持 HNSW、Flat、Dynamic index 等不同策略，这反映出索引设计不再是“一次选型终身不变”，而是开始强调：

- 数据规模变化
- 多租户隔离场景
- 吞吐与成本平衡
- 索引结构动态切换

### 3.3 向量索引已不再是唯一检索结构

更成熟的企业知识系统通常会组合：

- dense index
- sparse index / lexical index
- metadata filter index
- 外部结构化索引
- 对象映射索引

这意味着“索引构建”应按检索策略整体设计，而不是只谈 vector DB。

## 4. 索引构建的核心目标

### 4.1 提供高质量召回基础

支持：

- 语义检索
- 关键词检索
- 混合检索
- metadata filter

### 4.2 提供可治理的知识结构

支持：

- 权限过滤
- 引用定位
- 删除同步
- 版本管理
- 审计定位

### 4.3 提供可扩展的存储结构

支持：

- 新数据源接入
- 增量更新
- 多索引共存
- 多租户隔离

## 5. 推荐的索引组成

建议把索引至少设计为以下几个逻辑部分。

### 5.1 Dense Index

用于语义检索。

### 5.2 Sparse / Lexical Index

用于关键词、编号、字段名、错误码等精确匹配。

### 5.3 Metadata Filter Layer

用于：

- tenant filter
- sourceType filter
- bizDomain filter
- updatedAt filter
- permission filter

### 5.4 Object Mapping Layer

用于建立：

- documentId -> chunkIds
- objectId -> chunkIds
- chunkId -> source reference
- chunkId -> version

的映射关系。

### 5.5 Citation / Source Reference Layer

用于支持引用回溯和来源展示。

## 6. Dense / Sparse / Hybrid 的选择建议

### 6.1 Dense

适合：

- 自然语言语义检索
- 模糊表达理解
- 同义表达匹配

### 6.2 Sparse

适合：

- 精确术语
- 编号
- 错误码
- API path
- 字段名
- 组件名

### 6.3 Hybrid

适合：

- 大多数企业检索场景
- 既有语义表达又有精确术语的场景
- 低代码平台这类结构化 + 文档化混合场景

从近两年的工程趋势看，hybrid 越来越像默认推荐，而不是高级玩法。

## 7. 索引粒度设计

索引粒度需要至少区分以下几种。

### 7.1 文档级索引

适合粗粒度导航和文档级聚合。

### 7.2 chunk 级索引

适合大多数 RAG 检索场景。

### 7.3 对象级索引

适合：

- 组件对象
- Schema 对象
- API 对象
- 逻辑节点对象

### 7.4 字段级索引

适合高度结构化、字段语义非常关键的场景。

成熟知识系统通常不是单一粒度，而是多粒度并存。

## 8. 多租户和权限型索引设计

企业场景中，索引构建必须从一开始考虑权限与租户问题。

常见策略包括：

- 共享索引 + metadata filter
- 分租户 namespace
- 分租户 collection
- 高敏感数据单独索引

选择时要综合考虑：

- 数据规模
- 更新频率
- 租户数量
- 检索延迟
- 权限复杂度

## 9. 近两年的工程实践建议

### 9.1 小规模多租户可考虑更轻量索引策略

Weaviate 动态 index / flat index 等路线提示了一个重要思路：

在数据量小但租户多的场景里，并不一定总要为所有场景都用重型 HNSW。

### 9.2 hybrid 不一定非要两个系统拼接

Pinecone 的单 hybrid index 路线说明，dense + sparse 可以在一个索引体系里更低运维成本地结合。

### 9.3 metadata filter 要作为索引设计的一部分，而不是后补功能

如果 metadata filter 后补，后续性能和治理能力都会受限。

## 10. 引用与回溯能力设计

索引构建不应只关注检索命中，还应支持：

- 引用原始来源
- 定位原始块
- 回溯原始文档和页面
- 恢复 chunk 所属标题路径和页码

这意味着索引中应显式保存：

- source reference
- title path
- page / section info
- source uri

## 11. 面向低代码平台的索引建议

在低代码平台场景中，建议至少考虑以下索引形态：

### 11.1 组件知识索引

同时支持：

- 组件名称精确匹配
- 功能描述语义匹配
- props / setter / slots 检索

### 11.2 Schema 模板索引

支持：

- 页面类型匹配
- 模板语义检索
- 区块级或组件级 schema 检索

### 11.3 错误案例索引

支持：

- 错误码精确匹配
- 报错语义匹配
- 修复案例检索

### 11.4 运维记录索引

支持：

- 时间过滤
- 租户过滤
- 版本过滤
- 关键字 + 语义混合检索

## 12. 常见误区

### 12.1 只有 dense index，没有其他结构

这样在真实企业检索中很容易漏掉关键精确匹配结果。

### 12.2 索引只考虑召回，不考虑删除与版本

长期运行后会出现大量脏数据和引用错误。

### 12.3 不保留 source mapping

这会导致 citation、审计和回放能力很弱。

### 12.4 多租户场景只靠应用层过滤

这样容易产生权限风险，也不利于性能优化。

## 13. 一句话总结

索引构建的核心，不是把向量存进去，而是按 dense、sparse、metadata 和对象映射等多种结构把知识组织成可召回、可过滤、可更新、可引用、可治理的检索基础设施；近两年的实践也越来越明确，企业知识系统的索引设计，正在从“向量库存储”演进为“多结构混合检索架构”。
