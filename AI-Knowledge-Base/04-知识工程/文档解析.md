# 文档解析

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的文档解析（document parsing）设计，重点回答：

- 为什么文档解析是知识工程里最容易被低估、但最影响最终效果的环节
- 文档解析应该解析什么，不应该只停留在“抽文本”
- 近两年文档解析领域出现了哪些重要变化

## 2. 为什么文档解析是知识工程成败分水岭

在很多 RAG 系统中，团队会把重点放在：

- 选 embedding 模型
- 选向量库
- 选 rerank 模型
- 设计 Prompt

但实际效果不佳时，根因往往出在更前面的文档解析阶段。

原因是：

如果解析阶段没有把文档真实结构、语义边界和元信息保留下来，后续再强的检索和生成能力也只能在错误输入上工作。

可以简单理解为：

文档解析决定知识对象长什么样，而不是只是把文件“读出来”。

## 3. 当前行业最佳实战与近两年变化

### 3.1 文档解析已经从 OCR / 文本提取进化为 Document Intelligence

近两年一个很明显的变化是，企业文档解析不再只关注“把字符识别出来”，而越来越强调：

- 标题层级
- 段落结构
- 表格结构
- 图表与视觉区域
- 页眉页脚噪声
- 多栏布局
- 文档逻辑单元

Cohere 的企业 PDF 解析实践、Unstructured 的 partition 路线、视觉文档模型和多模态解析能力都在反映这一点。

### 3.2 视觉文档解析越来越重要

现在很多企业知识并不在干净的 markdown 文本中，而在：

- 扫描 PDF
- 带图表的报告
- 含表格的规范文档
- 图文混排说明
- 截图化流程文档

这意味着解析系统必须能处理视觉结构，而不能只做纯文本抽取。

### 3.3 解析开始为后续 chunking、metadata、retrieval 提供结构信号

更成熟的知识工程实践，已经不再把 parsing、chunking、metadata 当成彼此独立的后置步骤，而是将 parsing 视为后续所有知识建模的结构基础。

## 4. 文档解析到底要解析什么

一个真正可用的解析系统，至少需要提取以下信息层次。

### 4.1 原始文本层

包括：

- 正文文本
- 标题文本
- 列表文本
- 注释文本
- 表格文本

### 4.2 结构层

包括：

- 标题层级
- 段落边界
- 页面边界
- 表格边界
- 代码块边界
- 引用块边界
- 图表区域

### 4.3 版面层

包括：

- 多栏布局
- 页眉页脚
- 页码
- 左右区域
- 视觉邻近关系

### 4.4 语义单元层

包括：

- 一条规则
- 一个字段定义
- 一段 API 说明
- 一个组件说明
- 一个错误案例
- 一个表格记录

### 4.5 元信息层

包括：

- 文档标题
- 作者
- 更新时间
- 来源系统
- 权限标签
- 所属模块
- 文档类型

## 5. 解析对象的主要类型

不同内容类型，需要不同解析策略。

### 5.1 原生文本类

例如：

- Markdown
- TXT
- HTML
- Wiki 页面

解析重点：

- 结构保留
- 噪声去除
- 标题层级

### 5.2 半结构化文档类

例如：

- PDF
- Word
- PPT 导出文档
- 富文本页面

解析重点：

- 版面识别
- 表格解析
- 段落与标题恢复
- 图片 / 图表占位保留

### 5.3 结构化配置类

例如：

- JSON Schema
- YAML
- API spec
- 组件协议

解析重点：

- 字段语义
- 层级关系
- 可复用结构单元抽取

### 5.4 代码类

例如：

- TS / JS / Java / Python
- SQL
- 配置脚本

解析重点：

- 文件结构
- 函数 / 类 / 代码块边界
- 注释与代码关联

### 5.5 视觉文档类

例如：

- 扫描 PDF
- 图片化文档
- 报表截图
- 页面截图

解析重点：

- OCR
- 布局恢复
- 图像区域理解
- 视觉结构抽取

## 6. 推荐的解析输出结构

解析结果不应只是一个大字符串。

建议至少输出：

- `documentId`
- `sourceMetadata`
- `blocks`
- `blockType`
- `blockText`
- `hierarchy`
- `pageInfo`
- `layoutInfo`
- `structuredFields`（如有）
- `parsingWarnings`

其中 `blocks` 可以是后续 chunking 的重要输入。

## 7. 为什么解析结果必须保留块级结构

块级结构通常比整篇文本更有价值，因为它能帮助后续：

- 按标题切分
- 按段落切分
- 按代码块切分
- 按表格切分
- 按字段定义切分
- 做 metadata 对齐

如果解析阶段直接把所有内容拍扁成大段文本，后续 chunking 会变得非常粗糙。

## 8. 表格、代码和 PDF 是三类高风险解析对象

### 8.1 表格

表格不仅包含文本，还包含列、行和语义对齐关系。

如果解析丢了表结构，后续检索和回答很可能失真。

### 8.2 代码

代码的语义边界并不等于自然语言段落，必须保留函数、类、注释、配置块等结构。

### 8.3 PDF

PDF 最大的问题通常不在文本抽取本身，而在：

- 版面恢复
- 多栏顺序
- 表格和图片处理
- 页眉页脚噪声
- 文档逻辑结构恢复

## 9. 当前近两年的重点变化

### 9.1 基于规则的 parser 正在和多模态模型结合

传统 parser 更擅长速度和稳定性，多模态模型更擅长复杂布局理解。更成熟的体系通常会把两者结合，而不是只押宝其中一个。

### 9.2 解析开始服务 agentic retrieval

随着 agentic RAG 出现，解析结果不再只是“用于回答”，还可能用于：

- 任务规划前查规范
- 多阶段检索
- 工具选择前查协议
- 结构化生成约束

这要求解析结果更细粒度、更结构化。

### 9.3 “语义单元解析”比“纯文本抽取”更重要

当前更先进的知识工程，重点开始从“拿到文本”转向“拿到可被模型和系统消费的知识单元”。

## 10. 解析质量如何影响后续环节

### 10.1 对 chunking 的影响

解析差 -> chunking 边界差 -> 检索上下文差

### 10.2 对 metadata 的影响

如果结构单元不清楚，很难正确挂载：

- 标题
- source section
- table name
- field name
- component name

### 10.3 对 retrieval 的影响

检索结果是否命中，很多时候首先取决于解析是否保住了有价值的语义单元。

### 10.4 对 citation 的影响

如果解析没有保留页码、章节、来源块信息，后续引用能力会很弱。

## 11. 面向低代码平台的解析重点

在低代码平台场景中，文档解析不能只看产品文档，还要重点处理：

- 组件协议文档
- Schema 样本
- setter 说明
- slot 定义
- 错误案例文档
- 发布与运维说明
- 代码与配置片段

这些对象都要求解析结果尽量保留：

- 字段级边界
- 组件级边界
- 结构单元
- 示例代码或 JSON 片段

## 12. 常见误区

### 12.1 把解析理解成“提纯文本”

这会直接损失结构与语义边界。

### 12.2 对所有文档使用同一种 parser

不同对象差异很大，解析策略也应不同。

### 12.3 解析结果不保留块级结构和页级信息

后续 chunking、citation、问题定位都会受损。

### 12.4 不处理版面噪声

页眉、页脚、页码、目录噪声很容易污染知识库。

## 13. 一句话总结

文档解析的核心，不是把文件内容读出来，而是把原始文档转化为保留结构、语义单元、版面线索和元信息的知识对象；近两年的发展也已经很明确，企业知识工程正在从“文本提取”升级为“多模态文档理解与知识单元构建”。
