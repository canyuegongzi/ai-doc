# 权限继承与过滤

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的权限继承与过滤设计，重点回答：

- 企业知识对象在进入索引前后，如何继承原系统权限
- 为什么权限问题不能留到应用层最后再补
- 在 RAG、检索、Agent 和低代码平台场景下，权限过滤应如何落地

## 2. 为什么权限继承是企业知识工程的生死线

在个人 demo 或公开知识场景中，很多系统会默认所有内容都可被检索和生成使用。

但在企业环境中，这种默认前提通常是错误的。企业知识天然存在边界，例如：

- 不同租户之间隔离
- 不同角色之间隔离
- 开发、测试、生产环境之间隔离
- 平台管理员与普通用户可见范围不同
- 文档可见，但部分字段或附件不可见

如果知识工程阶段没有把权限继承和过滤设计进去，系统可能出现：

- RAG 检索泄露权限外内容
- Agent 在错误知识基础上做决策
- 不同租户内容互相污染
- 审计时无法解释某条内容为何被返回

可以直接说：

企业知识工程如果没有权限继承与过滤，就不能算可上线系统。

## 3. 近两年的关键变化

### 3.1 企业 RAG 已经从“检索对不对”转向“检索对不对且有没有越权”

随着 RAG 真正进入企业场景，业界越来越清楚：

高质量检索不仅要看命中率，还要看权限正确性。

也就是说：

召回到对的内容但用户无权查看，同样是错误结果。

### 3.2 权限开始被视为 retrieval pipeline 的正式输入

近两年的更成熟实践会把：

- user context
- tenant scope
- role scope
- document scope
- field scope

纳入 retrieval filter，而不是后置做简单裁剪。

### 3.3 Agent 场景使权限问题进一步升级

当知识不只用于回答，而是用于：

- 工具选择
- 动作建议
- 执行判断

权限问题的风险会更高，因为错误知识可能进一步放大为错误行动。

## 4. 权限继承解决什么问题

### 4.1 保持源系统权限语义

知识对象被采集、切分、向量化后，仍应尽量保留原系统里的权限含义，而不是变成“脱离源系统的自由文本”。

### 4.2 让检索和生成建立在正确可见范围内

只有在权限边界内召回的内容，才能进入上下文。

### 4.3 让审计和治理有据可查

系统要能解释：

- 这条结果为什么被用户看到
- 它来自哪个源对象
- 命中了什么权限范围

## 5. 权限继承的基本原则

### 5.1 权限应尽量从源系统继承，而不是知识系统重新发明一套

知识系统应尽量复用或映射原有：

- 组织结构
- 角色体系
- 租户边界
- 文档访问范围
- 资源级 ACL

### 5.2 权限信息应尽早进入知识对象元数据

权限不应只在在线请求时临时推导。更稳的做法是：

- 在采集阶段保留权限信息
- 在 chunk / object 元数据中携带权限字段
- 在索引阶段确保可过滤

### 5.3 检索过滤应前置，而不是只在最终结果层后置裁剪

如果先召回再删掉，很可能：

- 浪费召回预算
- 影响排序质量
- 污染上下文构造
- 给后续 Agent 带来错误中间状态

因此，权限过滤应尽量在 retrieval pipeline 的前置或中间阶段生效。

## 6. 推荐的权限层次

建议至少区分以下几层。

### 6.1 租户级权限

用于保证多租户内容天然隔离。

### 6.2 应用 / 工作空间级权限

用于保证不同业务空间之间隔离。

### 6.3 资源级权限

例如：

- 某篇文档
- 某个页面
- 某个组件定义
- 某份发布记录

### 6.4 块级 / 字段级权限

在一些高敏感场景下，整篇对象可见，但其中部分字段或片段不可见。

### 6.5 行为级权限

某些知识可查看，但不可用于某些执行型场景决策。

这类约束在 Agent 场景中尤其重要。

## 7. 推荐的权限 metadata 字段

建议知识对象至少保留：

- tenantCode
- appCode / workspaceId
- permissionScope
- roleScope
- sensitivityLevel
- ownerType / ownerId
- visibilityType
- sourceAclVersion

如果存在更细粒度控制，可进一步补充：

- fieldScope
- sectionScope
- environmentScope

## 8. 权限过滤在 retrieval 中应如何落地

推荐至少有三道过滤。

### 8.1 pre-retrieval filter

在召回前基于 metadata 做过滤，例如：

- tenantCode = 当前租户
- appCode = 当前工作空间
- visibilityType in allowed set

### 8.2 candidate filtering

在召回候选后，进一步检查更细粒度权限。

### 8.3 context-level enforcement

在 context builder 阶段再次确保：

- 不同来源内容没有越权混入
- 不可见字段已移除或脱敏
- citation 不指向不可见对象

## 9. 过滤策略设计建议

### 9.1 allowlist 优先于 denylist

在高风险环境中，更推荐基于明确可见范围构造 allowlist，而不是先放开再排除。

### 9.2 metadata filter 优先于 prompt 提示

不要指望模型根据 Prompt 自觉不引用越权内容。权限应由系统过滤，而不是靠提示词自律。

### 9.3 检索和生成都要考虑权限

不仅召回要过滤，最终输出也要防止：

- 转述越权信息
- 引用不可见来源
- 聚合得出不可见结论

## 10. 字段级与块级权限的特殊问题

很多企业内容存在这种情况：

- 文档本身可见
- 部分字段不可见
- 某段附录不可见
- 某环境配置仅特定角色可见

此时通常不能只靠文档级权限。

更稳的做法是：

- 在解析或清洗阶段拆出独立块
- 块级携带权限字段
- 检索时按块过滤

## 11. 与索引设计的关系

权限继承会直接影响索引设计：

- 是否按租户分 namespace
- 是否按敏感级别分 collection
- metadata filter 是否足够高效
- 是否需要 field / section 级对象拆分

这说明权限不是检索层后置问题，而是索引结构前置问题。

## 12. 面向低代码平台的权限继承重点

在低代码平台场景中，建议重点继承以下边界：

### 12.1 页面与应用边界

- 当前用户是否可见该页面配置
- 当前用户是否可见该应用下的数据源定义

### 12.2 租户边界

- 不同租户页面、组件配置、日志和案例必须严格隔离

### 12.3 环境边界

- 开发 / 测试 / 生产环境信息应区分

### 12.4 运维与配置边界

- 运维日志、构建信息、发布详情通常不应对普通用户开放

## 13. Agent 场景下的权限特殊性

当知识被 Agent 使用时，权限边界不应只定义“是否能看到”，还要定义“是否能据此行动”。

例如：

- 某日志信息可查看，但不能据此自动触发生产操作
- 某配置定义可检索，但不能直接作为写入依据

因此，Agent 场景中的权限继承最好与风险等级和 Tool 权限模型联动。

## 14. 常见误区

### 14.1 先把所有内容都索引进去，再在回答时过滤

这对企业系统风险太高，也会污染检索与排序质量。

### 14.2 只做文档级权限，不做 chunk / object 级权限

在复杂场景下很快不够用。

### 14.3 权限信息不随更新同步

源系统 ACL 变化后，如果索引侧不更新，会形成隐性越权风险。

### 14.4 把权限问题交给 Prompt

Prompt 不是权限系统。

## 15. 一句话总结

权限继承与过滤的核心，是让知识对象在进入采集、索引、检索、上下文构建和生成全过程中都维持与源系统一致的可见边界；近两年的企业实践也越来越明确，权限应是 retrieval pipeline 的正式输入，而不是后置补丁。
