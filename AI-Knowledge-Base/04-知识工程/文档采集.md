# 文档采集

> 状态：第一版草案

## 1. 文档定位

本文档用于定义知识工程中的文档采集（ingestion）设计，重点回答：

- 企业知识源如何被持续、稳定、可治理地接入知识系统
- 文档采集与文档解析、索引构建之间如何分工
- 近两年知识采集管线有哪些新的工程趋势

## 2. 为什么文档采集不是“把文件传上来”

很多早期 RAG 系统把 ingestion 理解为：

- 上传文件
- 切块
- 向量化
- 完成

但在企业环境中，真正的文档采集远不止于此。它通常需要解决：

- 数据源发现
- 增量同步
- 去重与版本识别
- 权限继承
- 内容变更追踪
- 失败重试
- 生命周期管理
- 索引更新触发

因此，文档采集本质上是一条持续运行的数据接入管线，而不是单次上传动作。

## 3. 当前行业最佳实战与近两年变化

### 3.1 ingestion 已从离线导入演进为持续同步管线

LlamaIndex 的 ingestion + docstore 管理实践已经明确把文档哈希、去重、upsert、变更检测纳入正式流程；OpenAI 的 retrieval/file search 路径则把“文件接入 -> 自动切分 -> 向量化 -> 可检索”进一步产品化。

这说明行业共识正在形成：

知识采集不应是一次性初始化，而应是持续同步系统。

### 3.2 文档采集越来越重视文档管理而不是仅仅导入

近两年更成熟的知识系统通常会显式管理：

- 文档 ID
- 来源系统
- 版本号
- 内容 hash
- 同步时间
- 删除状态
- 权限映射

这使得知识库更像“受管文档资产”，而不是“临时向量集合”。

### 3.3 采集对象正从文件扩展到内容流

现在的企业知识源不只来自文件，还来自：

- Wiki 页面
- 代码仓库
- 数据库记录
- 工单系统
- 日志系统
- API 响应
- 对象存储

因此，文档采集实际上已经演进为“知识对象采集”。

## 4. 文档采集的职责边界

### 4.1 应负责什么

- 发现数据源
- 拉取内容
- 标准化基础元信息
- 识别增量变化
- 推送到解析与索引流程
- 记录采集状态
- 处理失败重试
- 同步删除与失效信息

### 4.2 不应负责什么

- 不直接负责复杂文档理解
- 不直接负责最终检索排序
- 不直接负责最终回答生成

这些分别属于解析、检索和生成阶段。

## 5. 采集方式分类

### 5.1 全量采集

适用于：

- 初次建库
- 小规模数据源
- 低频更新的数据

优点：

- 简单直接

缺点：

- 成本高
- 易重复处理
- 不适合大规模高频更新场景

### 5.2 增量采集

适用于：

- 文档有更新时间
- 内容变化有事件来源
- 数据量大且持续更新

优点：

- 成本更低
- 更适合生产环境

### 5.3 事件驱动采集

适用于：

- 文档系统支持 webhook
- 业务系统能发出变更事件

优点：

- 时效性强
- 更接近实时知识同步

### 5.4 周期轮询采集

适用于：

- 外部系统没有事件能力
- 需要兼容遗留系统

优点：

- 通用性强

缺点：

- 延迟较高
- 可能带来重复扫描成本

## 6. 推荐的采集管线阶段

建议把采集流程拆成以下阶段。

### 6.1 Source Discovery

识别有哪些数据源、有哪些对象需要接入。

### 6.2 Fetch

从数据源拉取原始内容。

### 6.3 Normalize Metadata

统一基础元信息，例如：

- sourceId
- sourceType
- resourceId
- updatedAt
- owner
- tenantScope
- permissionScope

### 6.4 Change Detection

通过时间戳、版本号、content hash 或事件判断是否真的发生变化。

### 6.5 Queue / Dispatch

将待处理对象投递给解析、切分和索引后续流程。

### 6.6 Status Tracking

记录每个对象的采集状态、失败原因和最后成功时间。

## 7. 采集系统中必须保留的元数据

建议每个被采集对象至少保留：

- sourceSystem
- sourceType
- resourceId
- version
- contentHash
- fetchedAt
- updatedAt
- tenantCode
- permissionScope
- deletedFlag
- parsingStatus
- indexingStatus

这些字段后续对增量更新、权限继承和问题排查都非常关键。

## 8. 去重、版本与 upsert 设计

近两年更成熟的知识系统，通常都会显式支持：

### 8.1 文档唯一标识

确保同一资源不会因为重复采集产生多份逻辑副本。

### 8.2 内容哈希

用于判断内容是否真正变化。

### 8.3 upsert 语义

支持内容更新时局部重建，而不是每次全量清空重建。

### 8.4 失效与删除同步

如果源文档被删除或下线，知识系统中也应同步失效或删除。

## 9. 权限继承与租户同步

文档采集阶段必须尽早纳入权限信息，而不能等到检索阶段才补。

至少应考虑：

- 哪个租户可见
- 哪个角色可见
- 哪个业务空间可见
- 哪类字段需要脱敏

如果采集阶段完全不保留这些信息，后续检索过滤很难可靠实现。

## 10. 失败处理与可观测性

采集系统是长期运行系统，因此必须具备：

- 失败重试
- 死信队列或错误池
- 状态面板
- 采集延迟监控
- 同步成功率监控
- 数据漂移和缺失监控

这部分常被忽略，但实际上决定知识库是否可运营。

## 11. 当前近两年的重点变化

### 11.1 视觉文档采集越来越重要

知识采集对象不再只是文本文件，越来越多企业开始把：

- 扫描 PDF
- 报表截图
- UI 截图
- 图表图像

纳入知识处理链路。

### 11.2 文档管理和向量化逐步解耦

近一阶段更成熟的实践，会把：

- 采集与文档管理
- 解析与 chunking
- embedding 与索引

拆成可独立演进的阶段，而不是一个黑盒脚本串起来。

### 11.3 采集系统正变成知识工程的“事件总线入口”

后续不仅索引更新要依赖采集事件，甚至：

- eval 样本更新
- 缓存失效
- 场景知识热更新
- 审计记录

都可能依赖 ingestion 事件流。

## 12. 面向低代码平台的采集对象建议

在低代码平台中，文档采集不应只针对传统文档，建议优先采集：

- 组件文档
- 组件协议与 schema 定义
- 页面 Schema 样本
- Setter 定义
- 插槽规则
- 发布规范
- 构建与错误案例
- 平台使用 FAQ

并为不同对象设计不同的采集与更新机制。

## 13. 常见误区

### 13.1 把 ingestion 写成一次性脚本

这会导致后续无法增量更新、无法追踪状态、无法运营。

### 13.2 不记录 content hash 和 resourceId

会直接影响去重、版本和 upsert 能力。

### 13.3 采集阶段不保留权限信息

后续检索与审计成本会显著增加。

### 13.4 全量重建当成默认策略

对大规模企业知识源来说，这通常成本过高且不够稳。

## 14. 一句话总结

文档采集的核心，不是把文件导入向量库，而是构建一条能够持续发现、同步、去重、版本化、继承权限并驱动后续解析与索引更新的知识接入管线；近两年的趋势也表明，ingestion 正在从“离线导入步骤”演进为“知识工程的持续运行系统”。
