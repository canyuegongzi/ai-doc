# 古诗词RAG-MVP复盘

## 1. 背景
古诗词 RAG MVP 是一个典型的早期检索增强生成实验。它的目标并不是构建完整的知识平台，而是验证一个问题：当知识规模有限、语料相对标准化时，RAG 能否帮助模型更准确地识别和回答古诗词相关问题。

这个案例虽然简单，但非常适合作为知识库中的基础复盘样板。原因在于它能清晰暴露出 RAG 的几个核心问题：
- 召回是否准确
- 纯向量检索在短文本场景下是否可靠
- Prompt 是否真的能约束模型只基于候选回答
- 结构化输出和兜底策略是否必要

## 2. 实验目标
这次 MVP 的目标主要包括：
- 根据用户输入的诗句片段召回最可能的诗词条目
- 在候选范围内让模型判断最匹配结果
- 尽量输出稳定、可消费的结构化结果
- 验证纯向量检索在短句检索任务中的效果边界

它更偏“检索匹配型 RAG”，而不是长篇知识问答型 RAG。

## 3. 输入与数据范围
实验数据主要是古诗词条目，通常包括：
- 标题
- 作者
- 朝代
- 正文内容
- 可能还有标签、主题、注释等附加信息

用户输入通常是：
- 一句完整诗句
- 一段诗句片段
- 拼写不完整的模糊片段

这类输入有一个特点：
- 非常短
- 高歧义
- 对精确匹配和关键词顺序较敏感

## 4. 初始方案
早期 MVP 往往采用比较直接的链路：
- 对诗词文本做 embedding
- 把用户问题也转成 embedding
- 从向量库里取 topK
- 把 topK 候选交给大模型判断
- 输出最可能的结果

一个典型初始方案可能是：
- 向量库：本地轻量向量存储或 Chroma
- Embedding：通用文本 embedding 模型
- 生成模型：通用 chat / completion 模型
- topK：5 或 10
- Prompt：要求模型只能在候选中判断

## 5. 初始结果
这个方案在一些长句、主题明确的输入上通常可以工作，但在短句和高相似诗句场景中很快暴露问题：
- 召回结果主题相关但不是原句
- 短句相似度过高，容易把同主题内容召回来
- 模型在候选中做“合理猜测”，产生看似自然但不准确的判断
- 输出结构不稳定，不利于程序消费

这说明纯向量检索对短文本匹配任务并不稳。

## 6. 核心问题分析
这次实验里最重要的几个问题包括：
- 短文本语义信息不足，embedding 容易受到主题词影响
- 诗句类任务对字词顺序和局部重合度非常敏感
- 如果 topK 候选不够准，后续生成模型再聪明也很难挽回
- Prompt 约束虽有帮助，但不能替代检索质量
- 如果没有结构化输出，结果不利于自动评估和后续处理

## 7. 优化措施
后续优化通常会包括以下方向：
- 增加 exact match 或 fuzzy match 兜底
- 把关键词/BM25 检索与向量检索混合
- 对候选集合做 rerank
- Prompt 中明确限制“只能在候选中判断，不可补全外部知识”
- 输出 JSON 结构，包括匹配条目、置信说明和候选来源

如果数据规模不大，这类场景往往更适合“规则 + 关键词 + 向量”的混合方案，而不是把全部希望压在 embedding 上。

## 8. 关键收获
这个实验最重要的结论不是“RAG 有效”，而是：
- RAG 不是向量检索的同义词
- 召回质量决定整体效果上限
- 短文本匹配类任务对 lexical signal 更敏感
- Prompt 约束和结构化输出是必要但不充分条件
- 小型 MVP 也应该尽早建立可评估的输出格式

## 9. 对后续知识库建设的启发
这个案例对后续企业知识库建设有几个直接启发：
- 不能默认所有场景都适合纯 embedding 检索
- 知识对象的特性决定检索策略
- query type 不同，召回策略也应不同
- 结构化输出和评测应该尽早纳入 MVP
- 规则系统并不是“过时方案”，很多时候是必要补充

## 10. 适合沉淀的经验模板
从这个案例里可以沉淀出一组通用经验模板：
- 短文本检索优先考虑 hybrid recall
- 对高歧义输入增加 exact / fuzzy match
- 在生成前先约束候选集合
- 输出尽量结构化，便于评测和复盘
- 对召回失败和生成失败分别分析，不要混在一起

## 11. 后续可继续验证的问题
- 不同 embedding 模型在短文本古诗词场景的差异
- BM25 与向量检索的最佳融合方式
- rerank 模型是否能显著提高短句识别准确率
- topK 设置如何影响最终结果稳定性
- 是否需要专门的诗句对象切分方式

## 12. 小结
古诗词 RAG MVP 是一个很小但很典型的案例。它清楚地说明：RAG 的难点并不在“把数据向量化”，而在“选择合适的召回策略、约束生成边界并建立可复盘的输出结构”。对后续企业场景而言，这个案例的价值在于帮助我们尽早摆脱“向量化即完成 RAG”的误区。
