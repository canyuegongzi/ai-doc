# Schema生成实验

## 1. 背景
Schema 生成实验的目标，是验证低代码平台是否能够基于自然语言需求、字段结构、模板知识和组件协议，生成一份可渲染、可编辑、可继续迭代的页面 Schema 草稿。

这是平台 AI 中最容易被关注、也最容易被高估的场景之一。因为“生成一个页面”看起来直观，但真正难的不是让模型输出 JSON，而是让输出结果能够符合平台协议、匹配业务语义，并在设计器中继续可维护。

## 2. 实验目标
本次实验通常主要验证：
- 自然语言需求能否转成页面草稿
- 检索增强是否能提高组件选择和结构稳定性
- 生成结果是否可渲染、可编辑、可继续修改
- 生成后是否需要大量人工修正
- 结构化输出和校验机制是否足以兜住常见错误

## 3. 典型输入
实验输入往往包括：
- 页面需求描述
- 表单或数据表字段定义
- 页面类型说明
- 可能的参考模板
- 风格或端类型约束

这些输入本质上同时包含：
- 语义需求
- 领域对象信息
- 结构化约束

## 4. 初始方案
最早的方案通常是：
- 写一个长 Prompt
- 把页面 Schema 协议和组件列表作为上下文输入
- 直接要求模型输出完整 JSON

这个方法在 Demo 阶段可能能出结果，但很快会暴露问题：
- 结构不稳定
- 组件名称或字段路径写错
- 局部节点不合法
- setter / slot / props 配置不完整
- 数据绑定关系缺失
- 页面能看但不可继续编辑

## 5. 发现的问题
实验中常见问题包括：
- 模型能生成看起来像 Schema 的 JSON，但不符合正式协议
- 组件选择偏“语言上合理”，不一定符合真实能力边界
- 字段类型和组件映射经常出现细节错误
- 一次性生成整页时，错误会在多个层次累积
- 没有后置校验时，很难判断哪些地方真的错了

## 6. 优化思路
比较有效的优化路径通常包括：
- 引入页面模板和历史 schema 检索
- 检索组件物料定义、setter 和 slot 协议
- 分阶段生成，而不是一步到位生成全页
- 使用结构化输出约束
- 在生成后执行校验和自动修复
- 明确标记缺失信息，让用户补充，而不是强行猜测

## 7. 更合理的链路
一个更稳定的实验链路通常是：
1. 识别页面类型与目标对象
2. 检索相似页面模板与组件组合
3. 生成页面骨架
4. 根据字段结构补全组件和绑定
5. 补充数据源和交互骨架
6. 运行校验器
7. 自动修复局部问题或提示人工补充

这样做的核心是把“生成”变成“生成 + 检索 + 校验 + 修复”的组合问题。

## 8. 实验结论
这类实验通常会得出几个很稳定的结论：
- 纯 Prompt 直接生成完整 Schema 不稳定
- 模板和案例召回显著提升结果质量
- 结构化输出可以降低格式错误，但不能替代协议校验
- 分层生成比整页一步生成更可靠
- “可编辑率”和“可渲染率”比“看起来像不像 JSON”更重要

## 9. 对平台能力建设的启发
这个实验说明：
- 页面 Schema 模型必须正式建模
- 组件物料、setter、slot、数据源知识必须结构化
- 需要专门的 schema validator 和 repair 机制
- 生成场景必须与推荐、问答、配置诊断打通
- 后续评测应围绕渲染成功率、编辑成功率和人工修改量建立

## 10. 建议沉淀的复盘点
建议在知识库中继续沉淀：
- 哪类页面最适合先做自动生成
- 哪些组件最容易生成错误
- 哪些字段映射规则最容易失败
- 模板召回对生成质量的影响
- 生成后校验的常见错误分布

## 11. 后续可验证的问题
- 分阶段生成的最佳拆分粒度是什么
- 是否需要针对不同页面类型设计独立 Prompt 和模板
- 历史页面样例对生成效果提升有多大
- 结构化输出与 validator 的最佳协作方式是什么
- 是否可以逐步加入逻辑 DAG 草稿生成

## 12. 小结
Schema 生成实验说明，低代码平台中的“页面生成”本质上是一个高度约束的结构化工程问题，而不是开放式文本生成问题。只有把检索、模板、对象知识、结构化输出、校验和修复组成闭环，这个场景才会真正可用。否则它很容易停留在演示层，而难以进入生产使用。
