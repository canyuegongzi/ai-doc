# 配置纠错Agent实验

## 1. 背景
配置纠错 Agent 实验的目标，是验证低代码平台是否能够结合页面 Schema、组件物料协议、数据绑定关系、逻辑配置和运行时报错信息，对常见配置问题进行定位、解释和修复建议生成。

这个实验比纯问答更接近真实生产价值，因为很多平台问题并不是“文档不会看”，而是“页面就是不工作了”。只要 AI 能帮助快速定位错误、解释原因并给出可信修复建议，就能显著降低对平台专家的依赖。

## 2. 实验目标
实验主要关注以下问题：
- 是否能从报错信息和配置对象中定位问题
- 是否能识别常见的配置错误类型
- 是否能给出可采纳的修复建议
- 是否能在边界内生成自动修复草稿
- 是否能把诊断结果结构化输出，便于审计和复盘

## 3. 实验输入
典型输入包括：
- 页面 schema 或局部配置片段
- 组件物料协议
- setter / slot / props 定义
- 数据源模型与绑定关系
- 逻辑 DAG 配置
- 错误日志、trace、异常堆栈或用户描述现象

这说明该实验不是单一文本问答，而是“结构化配置 + 错误证据 + 领域知识”的联合诊断问题。

## 4. 初始方案
初始方案往往比较简单：
- 用户输入报错信息
- RAG 检索相关组件文档和 FAQ
- 模型给出可能原因与建议

这种方案能覆盖部分显性问题，但很快会遇到上限：
- 只看文档，不看真实 schema，定位往往不准
- 日志语义与页面配置没有绑定，导致建议偏泛化
- 多种错误可能共享相似症状，模型容易猜测
- 没有校验和补丁生成能力，难以形成闭环

## 5. 发现的问题
实验中常见问题包括：
- 没有读取真实配置对象时，建议容易停留在“可能是这个原因”
- 单靠错误日志很难区分 props 类型问题、slot 结构问题还是数据绑定问题
- 相同报错背后可能有完全不同根因
- 如果没有规则校验器，模型提出的修复建议也可能不合法
- 如果没有修复边界，自动修改配置风险较高

## 6. 优化方向
更成熟的实验方案通常会加入：
- 先做问题分类，再决定检索和诊断链路
- 读取真实 schema 节点和相关物料协议
- 结合规则校验器定位明显错误
- 召回历史相似错误案例
- 生成结构化诊断结果和修复补丁草稿
- 对高风险修复只输出建议，不自动执行

## 7. 典型诊断链路
一个更合理的实验链路通常是：
1. 输入报错与上下文
2. 识别问题类别
3. 读取相关 schema、物料、数据源、逻辑节点
4. 检索规则与历史案例
5. 运行协议/结构校验
6. 生成原因排序和修复建议
7. 如果可验证，生成局部 patch
8. 输出结果并等待人工确认或继续执行

## 8. 实验中常见错误类型
在低代码平台中，纠错实验里最容易出现的几类问题包括：
- 字段绑定路径不存在
- props 类型不匹配
- setter 输出结构不合法
- slot 节点嵌套错误
- 数据源字段引用不存在
- 逻辑节点引用了失效对象
- 页面 schema 节点层级错误
- 组件版本升级后的兼容性问题

这些问题很适合作为第一批诊断标签和评测样本。

## 9. 实验结论
这类实验通常会得出几个关键结论：
- 配置纠错不能只靠 RAG，必须读取真实配置对象
- 规则校验器和领域知识对诊断成功率影响很大
- 问题分类能显著提升后续链路稳定性
- 自动修复应从低风险、可验证的问题开始
- 结构化诊断输出非常重要，便于追踪采纳率和误诊率

## 10. 对平台能力建设的启发
这个实验说明平台需要补的不是单一 Prompt，而是一整条诊断链：
- 配置对象读取能力
- 问题分类能力
- 规则校验能力
- 错误案例知识库
- patch 生成与验证能力
- 审计与 HITL 能力

这也是为什么配置纠错更适合 Agent 架构，而不是停留在问答层。

## 11. 建议沉淀的复盘点
建议在知识库中进一步沉淀：
- 常见错误分类体系
- 错误与根因的映射关系
- 哪类错误适合自动修复
- 哪类错误需要人工补充上下文
- 诊断结果的结构化输出模板
- patch 生成与验证的安全边界

## 12. 后续可验证的问题
- 历史错误案例对诊断准确率提升有多大
- schema validator 与模型推理如何协作最有效
- 是否能建立统一的错误 taxonomy
- 自动 patch 的最小安全范围是什么
- 如何把用户采纳反馈回流成诊断优化样本

## 13. 小结
配置纠错 Agent 实验说明，低代码平台中的错误诊断不是“把报错扔给模型解释”这么简单，而是一条需要对象读取、规则校验、案例检索和受控修复协作的工程链路。这个实验的价值在于帮助平台明确：真正可落地的 AI 诊断能力，必须扎根于领域对象和运行时证据，而不能只靠语言模型猜测。
