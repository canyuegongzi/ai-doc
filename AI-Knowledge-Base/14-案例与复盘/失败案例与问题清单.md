# 失败案例与问题清单

## 1. 目的
失败案例与问题清单的作用，不是记录“做得不够好”，而是把已经暴露的问题系统化，避免团队在后续建设 AI 能力时重复踩坑。对知识库而言，这类文档很重要，因为很多真正有价值的经验并不来自成功案例，而是来自失败之后对边界和原因的澄清。

## 2. 为什么要单独沉淀失败案例
AI 项目里最常见的问题不是完全不可用，而是：
- 在 Demo 阶段看起来有效
- 一进入真实业务就开始不稳定
- 难以解释为什么失败
- 后续优化全靠体感，而没有问题清单驱动

因此，失败案例要单独沉淀，至少有三个价值：
- 帮助识别系统能力边界
- 帮助形成评测样本和验收门槛
- 帮助决定下一阶段优先补哪块基础设施

## 3. 适合记录的失败类型
建议把失败问题分成几类，而不是全部混在一起：
- 知识问题：知识缺失、知识过期、知识切分错误、metadata 缺失
- 检索问题：召回不到、召回偏题、排序不准、权限过滤失败
- 生成问题：结构错误、格式不稳定、幻觉、遗漏关键字段
- Tool / Agent 问题：选错工具、参数错误、重复执行、状态丢失
- 治理问题：越权、脱敏失败、审计缺失、审批边界不清
- 工程问题：超时、重试风暴、缓存污染、版本切换异常

## 4. 典型失败案例模板
每个失败案例建议至少记录：
- 背景：什么场景、什么输入、什么目标
- 现象：具体出现了什么错误或偏差
- 影响：对用户或系统产生了什么影响
- 根因：最终确认的原因是什么
- 临时处理：当时如何止损
- 长期修复：后续需要补哪些能力
- 可复用结论：以后遇到类似问题应如何判断

## 5. 当前阶段可重点收集的问题
结合已有场景，当前阶段建议优先收集这些问题：
- RAG 只用向量检索导致短文本召回不准
- 组件推荐只看语义描述，忽略字段结构和协议约束
- Schema 生成结果格式正确但协议不合法
- 配置纠错只看日志，不看真实 schema 对象
- Agent 没有明确工具边界，出现误调用或重复调用
- 检索结果未做权限过滤，存在越权风险
- 发布或运维场景回答没有时间线和引用依据

## 6. 为什么问题清单要可追踪
问题清单不是一次性文档，而应可持续迭代。建议每个问题至少带：
- 问题编号
- 所属场景
- 严重程度
- 当前状态
- 是否已有 workaround
- 是否已转化为评测样本
- 是否已进入规则或产品修复计划

例如可以记录为：

```ts
interface AiIssueItem {
  issueId: string;
  title: string;
  scenario: string;
  severity: 'low' | 'medium' | 'high';
  status: 'open' | 'mitigated' | 'resolved';
  rootCause?: string;
  workaround?: string;
  linkedEvalCase?: string[];
}
```

## 7. 对后续建设的价值
失败案例和问题清单最大的价值，在于它能直接驱动后续建设顺序。例如：
- 如果很多问题来自召回偏差，就优先补 retrieval 和 eval
- 如果很多问题来自结构化输出不稳，就优先补 schema 校验和 repair
- 如果很多问题来自 Agent 越权，就优先补权限、审批和审计
- 如果很多问题来自运行链路不可追踪，就优先补 trace 和 replay

也就是说，问题清单本身就是路线图的一部分。

## 8. 建议的维护方式
建议后续把失败案例维护成两层：
- 案例层：详细复盘具体一次失败
- 问题清单层：把共性问题抽象为可追踪条目

这样既能保留上下文，也能形成长期治理视角。

## 9. 常见误区
- 只记录“效果不好”，不记录具体现象和根因
- 把所有问题混在一起，不分检索、生成、Agent、治理、工程
- 问题出现后没有转化为评测样本和规则优化
- 只在事故复盘时记录，平时不维护
- 问题单没有状态管理，后续没人跟进

## 10. 小结
失败案例与问题清单是 AI 知识库中非常重要的一层。它们帮助平台把“踩过的坑”转化成可复用经验、评测样本和治理优先级。对企业 AI 系统来说，真正的成熟不是没有失败，而是每次失败都能变成下一次更稳的基础。
