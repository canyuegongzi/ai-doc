# 组件推荐实验

## 1. 背景
组件推荐实验的目标，是验证低代码平台能否根据用户的业务描述、字段特征和页面场景，推荐合适的组件候选，而不是要求用户先自己熟悉全部物料目录。

相比通用问答，这个实验更接近平台 AI 的业务价值核心，因为它直接影响：
- 组件资产复用率
- 用户搭建效率
- 新手上手成本
- 后续 Schema 生成的稳定性

## 2. 实验目标
本次实验通常希望验证以下问题：
- 根据场景描述能否找到合适的组件候选
- 根据字段结构能否做更准确的组件匹配
- 推荐结果是否能得到用户采纳
- 仅靠向量检索是否足够，还是需要标签和规则参与

## 3. 实验输入
实验输入一般有两类：
- 自然语言场景描述
  - 例如“适合审批表单中的多选组织架构组件”
- 结构化字段或页面上下文
  - 例如字段类型、是否多选、是否树形、是否远程搜索、是否移动端

这意味着组件推荐并不是单纯文本相似检索，而是“语义 + 结构”的联合匹配问题。

## 4. 知识与候选集准备
实验需要准备的知识通常包括：
- 组件物料定义
- 分类与标签
- props / slots / setter 信息
- 适用场景说明
- 示例 schema
- 常见替代组件
- 历史使用和采纳记录

如果组件知识仅停留在文档文本层，推荐结果往往会偏表面化。

## 5. 初始方案
常见的初始方案是：
- 把组件描述文档做向量化
- 用户问题做 embedding
- 用向量检索召回 topK 组件
- 由模型生成推荐理由

这个方案的优点是快，但很容易暴露问题：
- 语义相近但能力边界不同的组件混在一起
- 字段级约束无法仅靠描述文本体现
- 标签、端类型、交互方式等条件没有被有效利用
- 推荐解释看起来合理，但实际不适合当前字段或场景

## 6. 发现的问题
实验中通常会暴露以下问题：
- 只用 embedding 容易推荐“看起来相关”的组件，而不是“真正适配”的组件
- 不同组件的边界差异往往体现在 props / setter / slot 协议中，而不是简介文案里
- 字段类型、数据量、层级结构、端类型等条件如果不参与排序，会明显影响准确率
- 没有替代组件和推荐理由时，用户难以建立信任

## 7. 优化方向
更成熟的推荐方案往往会采用：
- 场景语义召回 + 规则过滤
- 组件标签、字段类型、端类型、能力标签参与 metadata filter
- 对召回组件做 rerank
- 输出主推荐和替代推荐
- 关联示例 schema 和典型使用场景

在一些平台中，还可以补充：
- 历史采纳率
- 组件质量评分
- 组件版本适配度
- 租户可见性与权限范围

## 8. 实验结论
这类实验的核心结论通常是：
- 组件推荐不是单纯向量检索问题
- 结构化物料知识的价值比描述文档更高
- 推荐系统需要“语义召回 + 规则过滤 + rerank + explanation”组合
- 解释能力会显著影响用户采纳率
- 页面推荐和组件推荐最好打通，而不是割裂建设

## 9. 对后续方案的启发
这个实验对平台建设的启发包括：
- 组件知识对象必须结构化沉淀
- 组件标签体系和能力 taxonomy 很重要
- 推荐链路应与问答、生成、诊断共享底层知识对象
- 用户采纳反馈要进入后续 rerank 优化
- 推荐不应停留在“给一个名字”，而应给“候选 + 理由 + 示例”

## 10. 建议沉淀的复盘点
建议在知识库中继续沉淀：
- 哪些场景最适合组件推荐
- 哪些字段条件最影响推荐准确率
- 哪些组件容易被混淆
- 推荐理由模板怎么设计更容易被采纳
- 推荐结果如何衔接到 Schema 生成

## 11. 后续可验证的问题
- 历史使用数据是否足以改善 rerank
- 字段级特征与文本语义的最优融合方式
- 是否需要为组件推荐设计专门 embedding 视图
- 替代组件展示形式如何影响用户选择
- 推荐与页面模板推荐的协同方式如何设计

## 12. 小结
组件推荐实验说明，低代码平台的推荐问题本质上是“对象匹配问题”，而不是简单语义搜索问题。真正有效的方案需要同时理解场景语义、结构化字段、组件能力边界和实际采纳反馈。这也是为什么推荐能力应作为平台 AI 的正式能力沉淀，而不是临时功能。
