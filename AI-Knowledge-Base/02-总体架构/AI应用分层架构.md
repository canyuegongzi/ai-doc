# AI应用分层架构

> 状态：第一版草案

## 1. 文档定位

本文档用于从分层视角描述 AI 应用系统的整体组织方式，回答“企业级 AI 应用应该拆成哪些层、每层负责什么、层与层之间如何协同”这几个问题。

如果说 [AI应用总体架构.md](D:/low-code/ai-doc/AI-Knowledge-Base/00-总览与路线/AI应用总体架构.md) 提供的是总纲视角，那么本文档提供的是更清晰的结构化分层视图，便于后续展开到模型接入、知识工程、RAG、Agent、Tool、治理与工程实现。

## 2. 为什么 AI 应用需要分层架构

AI 应用天然容易被误解成“一个模型接口 + 一段 Prompt + 一些文档”。

但真实企业系统通常同时涉及：

- 用户入口
- 业务编排
- 检索链路
- 工具调用
- 状态管理
- 模型推理
- 权限控制
- 评测观测
- 成本治理

如果不做分层，后续会出现：

- 模块职责混乱
- 场景能力难复用
- 调试与治理困难
- 需求变化时耦合严重

分层架构的目的，不是为了画漂亮架构图，而是为了让 AI 能力真正具备可复用、可替换、可治理、可演进的结构基础。

## 3. 分层设计原则

设计 AI 应用分层架构时，建议遵循以下原则：

### 3.1 按职责分层，不按技术名词分层

例如“RAG 层”“Agent 层”只是描述能力形态，更底层仍然要拆到：

- 输入入口
- 编排
- 检索增强
- Tool / Action
- 模型接入
- 治理

### 3.2 把稳定层和变化层分开

- 稳定层：模型网关、知识工程、Tool 协议、治理体系
- 变化层：具体场景 Prompt、业务流程、应用入口、模板与案例

### 3.3 把认知增强和执行增强分开

- 认知增强主线：知识处理 -> 检索 -> 上下文构建 -> 生成
- 执行增强主线：任务理解 -> 工具调用 -> 状态推进 -> 完成交付

### 3.4 把能力层和治理层分开

能力负责“做成事”，治理负责“做得稳、做得可控”。

## 4. 推荐的七层架构

结合当前主流 AI 工程实践与企业平台落地需求，建议将 AI 应用划分为七层主结构。

### L1 交互与应用入口层

负责承接用户、系统和外部接口的请求。

典型入口包括：

- 智能问答入口
- 搜索入口
- 页面生成入口
- 设计器内智能侧边栏
- 运维控制台智能入口
- API / SDK / Webhook

核心职责：

- 接收请求
- 采集上下文
- 识别来源系统
- 透传身份和租户信息
- 记录场景类型

这一层面向用户和业务系统，是 AI 能力进入业务链路的边界层。

### L2 场景编排层

负责根据业务场景决定请求应该进入哪种处理模式。

常见模式包括：

- 直接问答
- 检索增强问答
- 搜索推荐
- 结构化生成
- Tool 调用流程
- Workflow Agent

核心职责：

- 识别任务类型
- 选择处理链路
- 拼装场景 Prompt
- 决定是否进入 RAG
- 决定是否进入 Agent / Workflow

这一层是业务场景和底层能力之间的桥梁层。

### L3 Agent / Workflow Runtime 层

负责承载多步骤任务执行与状态推进。

核心职责：

- 任务理解
- 局部规划
- 状态管理
- 工具调度
- 中间结果记录
- 人工确认
- 错误恢复
- 终止控制

这一层特别适合承接：

- 配置诊断
- 生成后校验
- 多源查询
- 受控执行流程

### L4 检索与认知增强层

负责在模型生成前补足知识上下文。

核心职责：

- Query Rewrite
- Query Expansion
- 意图识别
- 实体抽取
- 多路召回
- Hybrid Search
- Rerank
- Context Builder
- Citation 组织

这一层是 RAG 的核心能力层，重点是认知增强，而不是简单向量检索。

### L5 Tool / Action 能力层

负责把外部系统能力标准化暴露给模型、Workflow 和 Agent 使用。

典型能力包括：

- 查询页面
- 查询组件
- 检查 Schema
- 获取数据源信息
- 创建草稿
- 查询构建日志
- 发送通知

核心职责：

- Tool 协议定义
- 参数校验
- 执行调度
- 超时与重试
- 风险标记
- 幂等与审计

### L6 知识与数据处理层

负责把企业原始数据转化为 AI 可消费知识。

核心职责：

- 数据源接入
- 文档解析
- 清洗标准化
- Chunk 切分
- Metadata 抽取
- 向量化
- 索引构建
- 增量更新
- 权限同步

这一层是 RAG 和知识增强系统的基础设施层。

### L7 模型接入与推理层

负责统一接入各类模型能力。

包括：

- LLM
- Embedding 模型
- Rerank 模型
- Vision 模型
- 多模态模型
- 语音模型

核心职责：

- 模型抽象
- 多供应商接入
- 路由策略
- 成本统计
- 限流与重试
- 熔断与降级
- 模型 fallback

## 5. 横向治理底座

除了七层主链路，还需要一组横向能力贯穿全局。

这些能力不属于单一层，但作用于所有层。

### 5.1 权限与安全治理

包括：

- 身份透传
- 租户隔离
- 数据权限过滤
- 敏感信息脱敏
- 高风险动作审批

### 5.2 Prompt 与配置资产管理

包括：

- Prompt 模板管理
- 变量注入
- 版本管理
- A/B 测试
- 场景配置管理

### 5.3 Evaluation 与 Observability

包括：

- 离线评测
- 在线监控
- Trace
- 质量分析
- 工具成功率
- 幻觉分析

### 5.4 成本与配额治理

包括：

- token 统计
- 模型成本控制
- 场景限流
- 配额治理

### 5.5 审计与回放

包括：

- 请求链路留痕
- 工具调用记录
- 关键决策记录
- 高风险动作追踪

## 6. 两条核心能力链

从架构视角，可以把 AI 应用理解为两条核心能力链共同协作。

### 6.1 认知增强链

数据源 -> 知识处理 -> 检索增强 -> 上下文构建 -> 模型生成

这条链解决的是“模型知道什么”的问题。

### 6.2 执行增强链

任务理解 -> Tool / Action -> 状态推进 -> Workflow / Agent -> 结果交付

这条链解决的是“系统能做什么”的问题。

这两条链并不是分离的，而是在真实业务场景中不断交叉。

## 7. 一次请求是如何流经各层的

以“帮我诊断这个页面为什么渲染失败”为例，一次请求通常会经历：

1. 入口层接收请求，识别用户身份、页面上下文和租户信息。
2. 场景编排层识别为“诊断类任务”。
3. Runtime 层决定先检索相关组件协议和历史案例。
4. 检索增强层完成 Query Rewrite、召回、重排和上下文构建。
5. Runtime 层再调用校验工具、日志查询工具。
6. Tool 层执行查询并返回结果。
7. LLM 基于上下文和工具结果生成诊断结论与修复建议。
8. Guardrails 和输出校验判断结果是否合规。
9. 结果返回入口，同时写入 Trace、审计和反馈闭环。

这个过程说明，AI 应用不是单次模型调用，而是一个跨层协同系统。

## 8. 为什么不能把 RAG 或 Agent 单独视为系统全貌

### 8.1 只做 RAG 不够

如果只有 RAG，系统通常能回答，但未必能完成任务、调用工具、推进流程或执行校验。

### 8.2 只做 Agent 也不够

如果只有 Agent 而没有知识工程、Tool 协议、权限治理和评测体系，系统会变得高风险且不稳定。

### 8.3 真正有效的是分层协同

RAG 是认知增强模块，Agent 是执行增强模块，Tool 是动作接口，Workflow 是执行框架，治理体系是控制底座。

## 9. 面向企业场景的分层意义

这套分层架构对于企业应用的意义在于：

- 能让多个业务场景共享底层能力
- 能让模型替换、工具扩展、知识更新独立演进
- 能把治理能力前置，而不是上线后补救
- 能让 AI 能力从 demo 走向平台化

## 10. 面向低代码平台如何映射这套分层

在低代码平台场景中，可以这样映射：

- L1：设计器入口、问答入口、运维入口
- L2：页面生成场景、组件问答场景、配置诊断场景
- L3：Workflow Agent、生成后校验流程、诊断流程
- L4：组件协议检索、Schema 模板检索、案例检索
- L5：页面查询工具、Schema 校验工具、日志查询工具、草稿创建工具
- L6：页面 Schema 样本、组件物料文档、Setter 定义、错误案例知识库
- L7：LLM、Embedding、Rerank、多模态模型

这样可以确保低代码 AI 不只是一个聊天窗口，而是平台能力的系统化扩展。

## 11. 常见误区

### 11.1 直接按产品功能堆能力，不做分层

这样后续很难沉淀平台复用能力。

### 11.2 把模型接入层当成全部 AI 架构

模型只是底座之一，远远不等于完整 AI 应用系统。

### 11.3 把治理能力放到最后补

在企业场景中，这会导致系统越扩展越难收束。

### 11.4 让每个场景各自实现一套 RAG 和 Tool 逻辑

这会快速形成重复造轮子和维护失控。

## 12. 一句话总结

AI 应用分层架构的核心，是把用户入口、场景编排、Agent / Workflow、检索增强、Tool / Action、知识处理和模型接入组织成可协同、可治理、可复用的系统结构，从而支撑企业级 AI 能力长期演进。
