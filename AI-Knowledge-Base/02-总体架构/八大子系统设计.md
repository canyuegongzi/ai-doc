# 八大子系统设计

> 状态：第一版草案

## 1. 文档定位

本文档用于回答一个更平台化的问题：如果要把 AI 能力沉淀为企业可复用的共用基础设施，最值得抽象成哪些核心子系统。

与分层架构不同，分层关注“系统如何按层组织”，而子系统关注“哪些能力应该独立沉淀为平台模块”。

## 2. 为什么需要子系统视角

企业 AI 建设如果只停留在场景层，容易出现：

- 每个场景重复接模型
- 每个场景各写一套 Prompt
- 每个场景各做一套文档处理
- 每个场景各做一套工具协议
- 评测和观测完全分散

从行业最佳实战看，真正能规模化复用的团队，往往都会逐步形成一组稳定的共用 AI 子系统。

## 3. 行业最佳实战中的共识

结合 OpenAI、Anthropic、LangGraph 以及主流企业 AI 平台实践，可以提炼出几个共识：

### 3.1 模型能力必须网关化

因为不同模型在：

- 成本
- 推理能力
- 多模态能力
- 工具使用能力
- 上下文长度

上差异很大，企业需要统一抽象和路由，而不是业务场景直接硬编码绑定单一模型。

### 3.2 Prompt 正在从代码片段变成资产对象

随着 Prompt versioning、Prompt objects、eval-driven prompt iteration 的成熟，Prompt 已经越来越适合独立沉淀为可管理资产。

### 3.3 知识处理与检索必须解耦

知识加工管线和在线检索引擎虽然相关，但职责不同：

- 前者负责把原始数据变成知识资产
- 后者负责把用户问题变成高质量上下文

### 3.4 Tools 与 Runtime 是 agentic system 的两个核心基础设施

Anthropic 和 OpenAI 的最新资料都在反复强调：

- tool quality 决定 agent quality
- runtime / orchestration 决定 agent stability

### 3.5 Evals 与 Observability 已经成为平台级能力

不仅是做一个 dashboard，而是要把 tracing、质量评测、成本分析和回放能力沉淀成共用服务。

## 4. 推荐的八大子系统

### 4.1 Model Gateway

这是 AI 平台的模型接入总入口。

核心职责：

- 接入 LLM、Embedding、Rerank、Vision、多模态模型
- 提供统一调用接口
- 模型路由
- fallback 与降级
- 限流与配额
- 成本统计
- 供应商切换

为什么必须单独成系统：

因为模型更新速度快、供应商众多、场景需求差异大，如果没有网关层，后续切换和治理成本会急剧上升。

### 4.2 Prompt Center

这是 Prompt 资产管理中心。

核心职责：

- Prompt 模板管理
- 变量注入
- 版本控制
- 场景模板复用
- Prompt A/B 测试
- 与 evals 结果关联

为什么越来越重要：

当前行业已经越来越清楚，Prompt 不是临时字符串，而是长期可演进资产。

### 4.3 Knowledge Pipeline

这是知识加工子系统。

核心职责：

- 数据源接入
- 文档解析
- 清洗标准化
- Chunk 策略
- Metadata 构建
- 向量化
- 索引更新
- 权限同步
- 结构化知识抽取

为什么必须独立：

因为知识工程质量决定 RAG 上限，而这部分工作明显不同于在线问答或在线检索。

### 4.4 Retrieval Engine

这是在线认知增强引擎。

核心职责：

- Query Rewrite
- Query Expansion
- Recall
- Hybrid Search
- Rerank
- Context Builder
- Citation 组织
- retrieval 级别缓存

为什么必须独立：

因为“把知识加工好”和“把查询理解好”是两种完全不同的能力，不应强耦合在同一个模块里。

### 4.5 Tool Registry / Action Center

这是所有外部动作能力的统一注册与调度中心。

核心职责：

- Tool 定义与注册
- 参数 Schema
- 工具文档
- 风险分级
- 权限边界
- 执行策略
- 幂等与重试策略
- 工具目录管理

为什么必须独立：

因为 tools 不应散落在各业务服务里直接拼接给模型，而应作为标准化接口层沉淀下来。

### 4.6 Agent Runtime / Workflow Engine

这是复杂任务执行的运行时子系统。

核心职责：

- Workflow 编排
- Agent 状态机
- 持久化执行
- 人工确认
- Memory 管理
- 失败恢复
- 长任务管理
- 终止控制

为什么必须独立：

因为复杂任务系统的稳定性，已经不再只是模型能力问题，而是运行时基础设施问题。

### 4.7 Evaluation & Observability Center

这是 AI 质量与运行控制中心。

核心职责：

- 离线评测
- 在线质量指标
- Trace
- 工具成功率分析
- 幻觉分析
- 成本分析
- 行为回放
- Prompt / model / tool 效果对比

为什么必须独立：

当前主流最佳实战都在强调：没有 evals 和 tracing，就很难把 AI 系统真正推入生产环境。

### 4.8 AI Application Layer

这是最终对业务提供价值的应用层。

典型应用包括：

- AI 问答助手
- AI 页面生成助手
- AI 组件推荐助手
- AI 配置纠错助手
- AI 逻辑编排助手
- AI 运维助手

为什么仍要单独列出来：

因为平台能力的最终目标不是搭平台本身，而是稳定支持更多高价值应用落地。

## 5. 八大子系统之间的关系

可以把这八个子系统理解为一条闭环：

1. Model Gateway 提供基础模型能力。
2. Prompt Center 管理场景化指令资产。
3. Knowledge Pipeline 生产知识资产。
4. Retrieval Engine 在请求时组织认知上下文。
5. Tool Registry 提供动作能力。
6. Agent Runtime 组织复杂任务执行。
7. Evaluation & Observability Center 负责质量与运行闭环。
8. AI Application Layer 面向业务场景交付价值。

## 6. 为什么这八个子系统是当前更合理的拆法

### 6.1 它兼顾了“认知、执行、治理”三大主线

- 认知主线：Knowledge Pipeline + Retrieval Engine
- 执行主线：Tool Registry + Agent Runtime
- 治理主线：Model Gateway + Prompt Center + Eval/Obs Center

### 6.2 它适合渐进建设

并不是必须一次性建设完八个系统。

很多团队可以按顺序演进：

- 先 Model Gateway + Knowledge Pipeline + Retrieval Engine
- 再 Prompt Center + Tool Registry
- 再 Agent Runtime
- 再 Evaluation & Observability Center
- 最后大规模扩展 AI Application Layer

### 6.3 它适合平台化复用

这八个模块既能支撑单一场景，也能随着场景增多沉淀为通用平台能力。

## 7. 当前行业前沿与这八大子系统的映射

### 7.1 OpenAI 路线映射

近一阶段 OpenAI 明显在强化：

- Responses API 作为 agentic primitive
- tool use
- MCP 接入
- reasoning + tools
- prompt / schema 生成

这对应：

- Model Gateway
- Tool Registry
- Prompt Center
- Agent Runtime

### 7.2 Anthropic 路线映射

Anthropic 的公开资料反复强调：

- augmented LLM
- simple composable workflows
- tools quality
- eval-driven iteration
- harness for long-running agents

这对应：

- Retrieval Engine
- Tool Registry
- Agent Runtime
- Evaluation & Observability Center

### 7.3 LangGraph 路线映射

LangGraph 重点强调：

- durable execution
- human-in-the-loop
- memory
- tracing
- production-ready stateful runtime

这对应：

- Agent Runtime / Workflow Engine
- Evaluation & Observability Center

### 7.4 企业平台路线映射

当前很多企业内部 AI 平台的实际演进路径，也都在逐步形成：

- 模型网关
- Prompt 资产库
- 知识加工平台
- 在线检索服务
- Tool 注册中心
- Agent / workflow 编排引擎
- 评测观测中心
- 多场景应用层

## 8. 子系统建设优先级建议

### 第一阶段

优先建设：

- Model Gateway
- Knowledge Pipeline
- Retrieval Engine

因为这是最基本的认知底座。

### 第二阶段

继续建设：

- Prompt Center
- Tool Registry

因为开始需要把场景模板和动作接口标准化。

### 第三阶段

继续建设：

- Agent Runtime / Workflow Engine

因为只有当前两阶段足够稳定时，复杂任务执行才值得引入。

### 第四阶段

完善建设：

- Evaluation & Observability Center

虽然这部分应尽早考虑，但一般在前几层开始形成真实链路后才会加速系统化建设。

### 第五阶段

规模化扩展：

- AI Application Layer

把平台能力复用到更多高价值场景中。

## 9. 面向低代码平台如何映射八大子系统

### Model Gateway

统一接入问答、生成、推理、Embedding、Rerank、多模态模型。

### Prompt Center

沉淀：

- 页面生成 Prompt
- 组件推荐 Prompt
- 配置诊断 Prompt
- 运维分析 Prompt

### Knowledge Pipeline

沉淀：

- 页面 Schema 样本
- 组件协议
- Setter / slot 规则
- 发布规范
- 错误案例

### Retrieval Engine

支持：

- 组件问答
- Schema 样本召回
- 页面模板推荐
- 诊断前案例检索

### Tool Registry

统一管理：

- `getPageInfo`
- `searchComponent`
- `validateSchema`
- `getBuildLog`
- `createPageDraft`

### Agent Runtime

支撑：

- 页面生成工作流
- 配置纠错工作流
- 运维辅助排障流

### Evaluation & Observability Center

跟踪：

- 生成质量
- 召回质量
- 工具成功率
- 平台场景成本
- 审计与回放

### AI Application Layer

最终形成：

- 平台知识问答助手
- 页面生成助手
- 组件推荐助手
- 配置纠错助手
- 运维助手

## 10. 常见误区

### 10.1 直接把所有能力做进一个“大 AI 服务”

这样短期看似快，长期一定难以扩展。

### 10.2 没有 Prompt Center

Prompt 会很快散落在业务代码中，无法做版本管理和效果追踪。

### 10.3 没有 Tool Registry

工具会逐渐变成一堆不可治理的隐式接口。

### 10.4 把 evals 和 tracing 当成上线后再说

现在行业最佳实战已经很明确，这类能力必须尽早纳入平台设计。

## 11. 一句话总结

八大子系统设计的核心，是把 AI 应用中最值得复用的能力沉淀为模型网关、Prompt 中心、知识加工、检索引擎、工具中心、运行时、评测观测中心和业务应用层八类平台模块，从而支撑企业 AI 能力长期平台化演进。
