# 数据流与调用链路

> 状态：第一版草案

## 1. 文档定位

本文档用于描述 AI 应用在运行时的数据流转与调用链路，重点回答：

- 一次请求从入口进入后会经过哪些环节
- 不同类型请求的链路有何差异
- 模型、检索、工具、运行时和治理能力在链路中如何协同
- 为什么数据流设计会直接影响系统稳定性、成本、可观测性和治理能力

与分层架构文档相比，本文档不再强调“系统分成哪几层”，而是强调“这些层在真实请求中如何被串起来”。

## 2. 为什么 AI 系统必须重视调用链路设计

传统业务系统的链路通常比较清晰，例如：

用户请求 -> 服务处理 -> 数据库查询 -> 返回结果

而 AI 系统往往会变成：

用户请求 -> 场景识别 -> Prompt 组装 -> 检索 -> 重排 -> 上下文构建 -> 模型推理 -> 工具调用 -> 再推理 -> 结构化校验 -> 审计与追踪 -> 返回结果

这意味着：

- AI 系统链路更长
- 中间状态更多
- 调用分支更多
- 隐性成本更多
- 失败点更多

如果链路设计不清晰，后续很容易出现：

- 无法定位问题出在哪一层
- 无法判断成本耗在什么环节
- 无法控制高风险动作
- 无法复盘一次请求发生了什么

## 3. 企业 AI 调用链路的基本共识

结合近一阶段行业最佳实战，可以看到几个共同方向。

### 3.1 一次请求通常不是一次模型调用

OpenAI、Anthropic、LangGraph 的公开实践都已经明确说明，生产级 AI 系统通常由多步处理链路构成，而不是单个 prompt-response 回合。

### 3.2 检索、工具、状态和治理必须进入主链路

它们不是附加组件，而是运行时链路中的正式节点。

### 3.3 Trace 必须贴着调用链路设计

如果调用链路没有显式节点，后续就很难做 tracing、replay、审计和评测定位。

### 3.4 不同任务类型应使用不同链路模板

例如：

- FAQ 型场景
- 生成型场景
- 诊断型场景
- Workflow Agent 场景

它们的链路模式不应完全一样。

## 4. 一次 AI 请求的通用调用链路

从总体上看，一次典型企业 AI 请求通常会经过以下阶段：

1. 请求进入
2. 上下文装配
3. 场景识别
4. 路由与链路选择
5. 检索增强或工具准备
6. 模型推理
7. 工具调用与状态推进
8. 输出校验与治理处理
9. 返回结果与记录反馈

下面分别展开。

## 5. 阶段一：请求进入

这一阶段发生在应用入口层。

典型输入包括：

- 用户自然语言请求
- 页面上下文
- 当前应用或租户信息
- 会话 ID
- 调用来源
- 当前任务上下文

这一阶段必须同步采集的元信息包括：

- userId
- tenantId
- appId / workspaceId
- requestId / traceId
- sceneType
- sourceChannel
- timestamp

这些信息后续会贯穿整个调用链路。

## 6. 阶段二：上下文装配

在进入实际 AI 链路前，系统通常需要把当前请求相关上下文补齐。

常见内容包括：

- 会话历史
- 当前任务状态
- 用户偏好
- 页面或资源上下文
- 入口层已知参数

这一阶段的关键不是“把所有内容都拼进去”，而是按场景筛选最必要上下文。

如果上下文装配失控，会直接导致：

- token 成本过高
- 上下文污染
- 模型混淆
- 权限边界不清

## 7. 阶段三：场景识别

系统需要识别这次请求属于哪种任务模式。

常见模式包括：

- 直接问答
- FAQ / RAG
- 搜索推荐
- 结构化生成
- 配置诊断
- Workflow Agent
- Tool-based Agent

场景识别后的结果，会决定后续链路走向。

从最佳实战角度，这一阶段应尽量显式，而不要完全交给模型隐式决定。

## 8. 阶段四：路由与链路选择

不同任务类型，对应不同链路模板。

例如：

### 8.1 直接问答链路

适用于：

- 简单解释
- 文本改写
- 摘要
- 分类

链路特点：

- 无检索
- 无工具
- 无复杂状态
- 更强调 Prompt 和结构化输出

### 8.2 RAG 问答链路

适用于：

- 知识问答
- 产品说明
- 组件文档问答
- FAQ

链路特点：

- 先检索，再生成
- 需要 query rewrite、recall、rerank、context builder
- 需要 citation 和回答依据

### 8.3 生成增强链路

适用于：

- 页面 Schema 生成
- 配置生成
- 模板生成

链路特点：

- 先召回模板、协议和样例
- 再生成结构化结果
- 再做校验与修复

### 8.4 诊断链路

适用于：

- 配置错误分析
- 日志定位
- 渲染问题诊断

链路特点：

- 先检索规则和案例
- 再调用查询或校验工具
- 最后汇总问题与建议

### 8.5 Workflow Agent 链路

适用于：

- 多步骤任务
- 受控执行流程
- 需人工确认的任务

链路特点：

- 包含状态机
- 可能多次检索、多次模型调用、多次工具调用
- 需要中间状态持久化

## 9. 阶段五：检索增强与知识注入

当场景需要外部知识时，系统通常进入检索增强子链路。

典型步骤包括：

1. Query Rewrite
2. Query Expansion
3. Recall
4. Hybrid Search
5. Rerank
6. Context Builder
7. Citation 装配

这一阶段的输出不是最终答案，而是高质量上下文。

最佳实战中，一个重要原则是：

不要把检索结果直接原样拼进 Prompt，而要做显式上下文治理。

## 10. 阶段六：模型推理

模型推理阶段通常发生在以下几种时机：

- 直接生成答案
- 基于检索上下文生成回答
- 判断是否需要工具调用
- 基于工具结果做下一步决策
- 生成结构化输出

当前行业实践已经越来越强调：

- 不同步骤可使用不同模型
- 推理模型、便宜模型、Embedding 模型、rerank 模型应分工使用
- 应尽量根据任务复杂度做 model routing

因此，模型推理本身也经常是一个动态路由过程，而不是固定绑定单一模型。

## 11. 阶段七：工具调用与状态推进

当任务需要访问外部系统或执行动作时，链路会进入 Tool / Action 子链路。

典型步骤包括：

1. 选择工具
2. 生成参数
3. 参数校验
4. 权限校验
5. 执行工具
6. 返回结果摘要
7. 更新任务状态
8. 决定是否继续下一步

这一阶段的关键是：

- 所有动作必须显式留痕
- 高风险动作必须可拦截
- 失败必须可恢复
- 调用结果必须可以回灌到后续推理

## 12. 阶段八：输出校验与治理处理

无论是问答结果、生成结果还是执行结果，都不应直接裸返回。

通常还需要经过：

- 格式校验
- 结构化校验
- 业务规则校验
- 权限检查
- 敏感信息过滤
- 风险动作确认
- fallback 或修复流程

这一步的本质是把“模型输出”变成“业务可交付结果”。

## 13. 阶段九：返回结果与反馈闭环

最终返回给用户的结果，通常至少包含：

- 主结果
- 引用或依据
- 风险提示或不确定性提示
- 可能的下一步动作

与此同时，系统内部应同步记录：

- trace
- 日志
- token 成本
- 工具调用记录
- 结果质量标签
- 用户反馈

这部分数据会回流到：

- Evaluation
- Prompt 优化
- Retrieval 优化
- Tool 优化
- 场景迭代

## 14. 四类典型链路模板

### 14.1 FAQ / 知识问答链路

入口 -> 场景识别 -> Query Rewrite -> Recall -> Rerank -> Context Builder -> LLM 回答 -> Citation -> 返回

### 14.2 生成增强链路

入口 -> 场景识别 -> 检索模板与规范 -> LLM 生成结构化草稿 -> 校验工具 -> 修复或回退 -> 返回

### 14.3 诊断分析链路

入口 -> 场景识别 -> 检索规则与案例 -> 日志/校验工具 -> LLM 汇总诊断 -> 返回建议

### 14.4 Workflow Agent 链路

入口 -> 场景识别 -> 进入 Runtime -> 检索 / 模型 / 工具多轮协作 -> 人工确认（可选） -> 完成结果 -> 审计与回放

## 15. 链路设计中的关键最佳实践

### 15.1 让链路节点显式化

不要把所有逻辑都塞在一段 Prompt 或一段代码里。节点越显式，后续越容易：

- trace
- 评测
- 调试
- 回放
- 审计

### 15.2 让不同任务走不同链路模板

不是所有请求都应该进入统一超长 Agent 链路。

### 15.3 让数据和状态可回流

每个关键节点都应能记录：

- 输入
- 输出
- 耗时
- 成本
- 错误
- 决策原因

### 15.4 让治理能力嵌入链路，而不是链路外补丁

权限、审批、结构化校验、风险分级应成为正式节点，而不是最后加一层 if 判断。

## 16. 面向低代码平台的调用链路映射

在低代码平台场景中，可以把典型调用链路具体映射为：

### 16.1 页面生成

需求输入 -> 检索页面模板 / 组件协议 / 数据源定义 -> 生成 Schema 草稿 -> 调用校验工具 -> 修复 -> 返回

### 16.2 配置诊断

报错输入 -> 检索组件规范 / 错误案例 -> 查询页面配置 / 日志 -> 诊断分析 -> 输出建议

### 16.3 组件推荐

场景描述 -> 检索组件库 / 历史案例 -> 排序 -> 生成推荐说明 -> 返回

### 16.4 运维辅助

问题输入 -> 检索规范 -> 调用日志 / 发布 / 构建查询工具 -> 汇总分析 -> 必要时进入人工确认或后续流程

## 17. 常见误区

### 17.1 只设计模型调用，不设计完整调用链路

这会让系统一旦复杂起来就失去可控性。

### 17.2 链路全部动态化

如果什么都交给模型临时决定，系统会很难调试和治理。企业系统应优先有显式链路模板。

### 17.3 不记录中间节点数据

这样后续几乎无法做 tracing、evals 和 replay。

### 17.4 检索、工具、治理各自独立，不在主链路里协同

这会导致系统看起来模块很多，但整体不可运营。

## 18. 一句话总结

AI 应用的数据流与调用链路，本质上是把入口、上下文、检索、模型、工具、状态和治理节点串成一条可观测、可回放、可治理的执行路径；链路设计得越清晰，系统越容易稳定落地和持续优化。
