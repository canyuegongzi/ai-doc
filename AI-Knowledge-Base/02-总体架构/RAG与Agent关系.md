# RAG与Agent关系

> 状态：第一版草案

## 1. 文档定位

本文档用于回答一个在 AI 架构设计中最容易被混淆的问题：RAG 和 Agent 到底是什么关系，它们分别解决什么问题，为什么不应被理解为两个彼此孤立的系统。

在很多讨论中，RAG 和 Agent 往往被并列成两个热门方向，但如果不把两者关系讲清楚，后续架构设计很容易出现：

- 同一问题重复建设两套系统
- 把 RAG 做成“简单问答模块”
- 把 Agent 做成“无知识基础的万能执行体”
- 不知道在具体场景中应优先建设哪一类能力

## 2. 最核心的结论

如果只保留一句话，可以这样理解：

RAG 主要解决“知道什么”的问题，Agent 主要解决“接下来怎么做”的问题。

进一步说：

- RAG 是认知增强模块。
- Agent 是执行增强模块。
- Workflow 是连接认知与执行的稳定框架。

## 3. RAG 的职责边界

RAG 的核心任务是：

- 获取外部知识
- 理解用户查询
- 从知识源召回相关信息
- 构建高质量上下文
- 基于上下文生成回答、建议或生成约束

因此，RAG 更偏向：

- 知识服务
- 事实补充
- 上下文增强
- 生成前认知准备

RAG 的重点是：

- 知识质量
- 召回质量
- 重排质量
- 上下文质量

## 4. Agent 的职责边界

Agent 的核心任务是：

- 理解任务目标
- 决定执行路径
- 调用工具
- 观察结果
- 持续推进状态
- 完成任务闭环

因此，Agent 更偏向：

- 执行协作
- 多步骤任务处理
- 工具调度
- 状态推进
- 人机协同

Agent 的重点是：

- 任务边界
- Tool 协议
- 状态管理
- 风险控制
- 错误恢复

## 5. 为什么 RAG 和 Agent 经常被一起提及

因为很多真实任务同时需要：

- 先知道正确知识
- 再执行正确动作

例如：

- 先查组件协议，再生成页面配置
- 先查 API 文档，再调用系统接口
- 先查错误案例，再做诊断和修复建议
- 先查平台规则，再执行低风险操作

在这些任务里，RAG 和 Agent 不是替代关系，而是顺序协作关系。

## 6. 两者的典型协作方式

### 6.1 RAG 作为 Agent 的前置认知模块

这是最常见的形态。

流程通常是：

1. Agent 理解当前任务。
2. 判断需要补充外部知识。
3. 调用检索能力或 RAG 子链路。
4. 获取相关文档、规则、案例或协议。
5. 再基于知识做后续决策与执行。

这种情况下，RAG 相当于 Agent 的“外部记忆扩展”或“认知前置模块”。

### 6.2 RAG 作为 Agent 的执行过程支撑

在复杂任务中，Agent 不一定只检索一次，而可能在多个步骤中反复调用 RAG。

例如：

- 第一步检索页面模板
- 第二步检索组件协议
- 第三步检索历史案例
- 第四步检索发布规范

这种情况下，RAG 不只是前置模块，而是 Agent 运行时的持续知识支撑能力。

### 6.3 Agent 作为 RAG 的动作扩展层

RAG 本身通常停留在“回答”和“生成建议”阶段。

当用户希望系统继续执行时，例如：

- 创建页面草稿
- 触发配置检查
- 查询运行状态
- 汇总多源结果

就需要 Agent 进一步把检索结果转化为动作。

## 7. 什么时候以 RAG 为主，什么时候以 Agent 为主

### 7.1 以 RAG 为主的场景

适用于：

- 文档问答
- FAQ
- 组件知识检索
- 搜索推荐
- 生成前知识补充

这类场景的核心问题是“找知识”和“基于知识回答”。

### 7.2 以 Agent 为主的场景

适用于：

- 多步骤诊断
- 工具协作查询
- 草稿生成与校验闭环
- 受控执行流程
- 运维辅助流程

这类场景的核心问题是“推进任务完成”。

### 7.3 混合场景

企业里大部分高价值场景其实是混合形态。

例如：

- 页面生成助手：需要检索模板和协议，也需要多步骤生成与校验
- 配置纠错助手：需要查规范和案例，也需要调用校验工具和形成修复路径
- 运维助手：需要查日志与知识，也需要多工具协同分析

## 8. 为什么不应把 RAG 和 Agent 做成两套互不关联的系统

### 8.1 会导致知识能力和执行能力割裂

如果 RAG 系统只服务问答，而 Agent 系统没有复用同一套检索和知识底座，就会出现：

- 知识重复建设
- 数据源不一致
- 权限和元数据重复处理
- 场景效果不一致

### 8.2 会导致架构能力重复

例如：

- 两边都实现一套文档解析
- 两边都实现一套 metadata 处理
- 两边都实现一套 query 理解逻辑

### 8.3 会导致治理和评测割裂

如果 RAG 和 Agent 各自演化，后续评测、审计、成本和权限治理会非常分散。

## 9. 更合理的架构关系

更合理的做法是：

- 建立统一知识工程底座
- 建立统一检索引擎
- 建立统一 Tool / Action 中心
- 建立统一 Workflow / Agent Runtime
- 由场景编排层决定当前请求主要走 RAG 模式、Agent 模式还是混合模式

也就是说：

RAG 和 Agent 应该共享基础设施，只是在不同场景中扮演不同主角色。

## 10. Workflow 在两者之间的桥梁作用

Workflow 是 RAG 与 Agent 能稳定协同的关键中间层。

因为它可以：

- 定义什么时候先检索
- 定义什么时候调用工具
- 定义什么时候做校验
- 定义什么时候进入人工确认
- 定义什么时候终止任务

在企业场景中，很多真正可落地的系统，并不是“完全自由 Agent + 一个检索插件”，而是“Workflow 框架下的 RAG + Tool + Agent 协同”。

## 11. 面向低代码平台如何理解两者关系

在低代码平台场景中，两者的关系可以非常具体地表达为：

### 11.1 页面生成助手

- RAG 提供模板、组件协议、Schema 样本
- Agent 负责根据需求组织生成步骤、调用校验工具并产出草稿

### 11.2 配置纠错助手

- RAG 提供错误案例、组件规范、字段规则
- Agent 负责串联日志查询、Schema 校验、问题分析和修复建议

### 11.3 平台运维助手

- RAG 提供历史知识、规范和说明
- Agent 负责多工具查询、汇总和受控执行

这说明在低代码平台中，RAG 更像“平台知识引擎”，Agent 更像“平台任务协作器”。

## 12. 常见误区

### 12.1 认为 RAG 和 Agent 是两条互斥路线

实际上，大多数高价值场景都需要两者配合。

### 12.2 认为上了 Agent 就不需要 RAG

没有外部知识增强的 Agent，往往在企业场景中效果非常不稳定。

### 12.3 认为 RAG 已经足够，不需要执行系统

如果目标是让 AI 参与任务闭环，只做 RAG 通常不够。

### 12.4 认为 Agent 本身就会“记得一切”

Agent 依然需要：

- RAG
- Memory
- Tool
- Workflow
- Guardrails

共同支撑。

## 13. 一句话总结

RAG 和 Agent 不是两个彼此竞争的方向，而是企业级 AI 架构中的两条核心能力链：RAG 负责知识增强，Agent 负责任务执行，真正可落地的系统通常通过 Workflow 将两者组织为统一协同体系。
