# 六层架构设计

> 状态：第一版草案

## 1. 文档定位

本文档用于给出一版更适合企业 AI 平台落地的“六层架构设计”，重点不是罗列所有模块，而是提供一套更适合工程实施、团队协作和平台演进的能力拆分方式。

与 [AI应用分层架构.md](D:/low-code/ai-doc/AI-Knowledge-Base/02-总体架构/AI应用分层架构.md) 相比，本文档更强调“能力层次”和“建设优先级”；与 [八大子系统设计.md](D:/low-code/ai-doc/AI-Knowledge-Base/02-总体架构/八大子系统设计.md) 相比，本文档更强调“系统应该按哪几层来组织”。

## 2. 为什么还需要一版六层架构

七层架构有助于完整描述系统边界，但在实际推进中，很多团队更需要一版更聚焦、可实施、可映射到研发组织和平台能力建设的结构。

六层架构的价值在于：

- 更适合做建设优先级划分
- 更适合对应平台能力中心
- 更适合指导服务拆分
- 更适合和业务场景做映射

## 3. 行业最佳实战中的共识

结合近一阶段主流工程实践，可以看到几个较稳定的共识。

### 3.1 不要直接从“Agent 框架”开始

Anthropic 在 2024 年底的《Building Effective AI Agents》中明确强调，最成功的系统通常不是最复杂的框架，而是“简单、可组合”的模式，并建议优先从简单方案开始，只有在确实需要时再增加 agentic complexity。

### 3.2 Agent 的基础是增强型 LLM，而不是神秘黑盒

Anthropic 在同一篇文章里把“augmented LLM”定义为 agentic system 的基础积木，强调 retrieval、tools、memory 是 LLM 的增强能力，而不是外加魔法层。

这意味着架构设计应先把模型、检索、工具、记忆和治理这些基本能力层打稳，再谈更高阶 Agent。

### 3.3 长时任务需要状态化运行时

LangGraph 官方文档把 durable execution、human-in-the-loop、comprehensive memory、debugging 作为长运行 Agent 的核心基础设施。这说明当前行业主流已经把“状态化运行时”视为复杂 AI 系统的必要组成。

### 3.4 Tools、guardrails、evals 已经成为一等公民

OpenAI 在 2025 年的 agent guide 和 Responses API 更新中，明确把 tools、instructions、guardrails 作为 agent design foundation；Anthropic 2025 年关于 tool 设计与 agent evals 的文章，也都说明工具质量和评测闭环已经从“附加项”变成“生产落地核心项”。

### 3.5 企业系统优先是 workflow + agent 的混合形态

Anthropic、OpenAI、LangGraph 的公开资料有一个共同点：都在强调 workflow、tool use、human oversight、durable execution，而不是鼓励一开始就做完全自由自治系统。

因此，更适合企业落地的不是“纯 Agent 架构”，而是“可控流程 + 局部 agentic decision”的混合架构。

## 4. 推荐的六层架构

基于以上共识，建议把企业 AI 应用能力抽象为以下六层。

### L1 体验与场景层

这是最贴近业务和用户的一层。

典型对象包括：

- 智能问答助手
- 页面生成助手
- 组件推荐助手
- 配置诊断助手
- 运维分析助手
- API / SDK 能力入口

核心职责：

- 提供面向具体场景的交互入口
- 承接用户任务和业务动作
- 组织场景 Prompt、场景模板和 UI 反馈方式
- 沉淀场景级指标和反馈

这一层本质上是“应用产品层”。

### L2 编排与运行时层

这是 AI 应用的流程控制核心层。

核心能力包括：

- 路由与任务识别
- Workflow 编排
- Agent Runtime
- 状态机
- 持久化执行
- Human-in-the-loop
- 失败恢复
- 长流程任务管理

这一层负责回答：

- 当前任务应该走哪条链路
- 哪一步需要检索
- 哪一步需要工具
- 哪一步需要人工确认
- 什么时候结束或转人工

从行业实践看，这一层正是当前 agentic system 的关键基础设施层。

### L3 认知增强层

这是面向知识增强的能力层。

核心能力包括：

- Query Rewrite
- Query Expansion
- Recall
- Hybrid Search
- Rerank
- Context Builder
- Citation
- Memory retrieval

这一层的目标是把“用户问题”转化为“模型可利用的外部认知上下文”。

在很多实际系统中，这一层往往既服务问答，也服务 Agent。

### L4 动作与系统接口层

这是 AI 与外部系统交互的动作层。

核心能力包括：

- Tool 定义
- Tool 注册
- Tool 参数 Schema
- 权限校验
- 风险分级
- Tool 调用编排
- 幂等控制
- 外部 API / DB / service adapter

这一层的关键不在于“能不能调 API”，而在于能否把动作抽象成标准协议，并纳入治理。

### L5 知识与数据底座层

这是支撑认知增强和部分记忆能力的知识底座。

核心能力包括：

- 数据源接入
- 文档解析
- 清洗标准化
- Chunk 设计
- Metadata 设计
- 向量化
- 索引构建
- 索引更新
- 权限同步
- 结构化知识抽取

从最佳实战看，这一层经常被低估，但它实际决定了 RAG 和领域智能的上限。

### L6 模型与治理基础层

这是系统最底部但又横向影响全局的一层。

核心能力包括：

- LLM / Embedding / Rerank / Vision 模型接入
- 模型路由
- 限流、熔断、降级
- 成本与配额
- Guardrails
- Evaluation
- Observability
- 审计与安全治理

虽然很多内容属于横向治理，但在工程实施中，通常需要与模型接入和运行基础设施一起建设，因此适合作为最底座的一层统一考虑。

## 5. 六层之间的核心关系

### 5.1 上层定义业务目标

L1 决定用户真正看到的能力形态，L2 决定任务如何被组织成流程或运行时状态。

### 5.2 中层完成认知与动作协同

L3 负责“知道什么”，L4 负责“能做什么”，两者共同支撑 Runtime 层完成任务。

### 5.3 下层提供稳定底座

L5 提供知识资产和数据基础，L6 提供模型能力与治理控制。

可以用一句话概括：

上层是场景，中层是能力协同，下层是基础设施与治理。

## 6. 为什么这版六层更适合企业落地

### 6.1 更适合建设优先级排序

企业通常可以按如下顺序推进：

1. 先建 L6 模型与治理基础
2. 再建 L5 知识与数据底座
3. 再建 L3 认知增强层
4. 再建 L4 动作接口层
5. 再建 L2 编排与运行时层
6. 最后大规模扩展 L1 场景层

### 6.2 更适合组织协作

这一拆法通常也更容易映射到团队职责：

- 平台团队：L5-L6
- AI 应用中台：L2-L4
- 业务团队：L1

### 6.3 更适合从简单系统演进到 agentic system

如果一开始只有：

- L1 场景层
- L3 检索增强
- L6 模型基础

就可以先做 FAQ / RAG 系统。

随着 L4 和 L2 的逐步成熟，就可以演进到 Workflow Agent。

## 7. 各层典型技术与模式映射

### L1 体验与场景层

典型模式：

- AI copilot
- 智能侧边栏
- 智能搜索
- 表单式交互 + AI 补全

### L2 编排与运行时层

典型模式：

- Router
- Sequential workflow
- Parallel workflow
- Evaluator-optimizer
- Planner-executor
- Stateful graph runtime

Anthropic 的 workflow 分类和 LangGraph 的 durable execution / HITL 能力，都是这一层的重要参考。

### L3 认知增强层

典型模式：

- Hybrid RAG
- Retrieval pipeline
- Long-context + retrieval hybrid
- Retrieval with rerank
- Citation-aware answering

### L4 动作与系统接口层

典型模式：

- Function calling
- Tool registry
- MCP integration
- Action gateway
- Risk-tiered tool access

### L5 知识与数据底座层

典型模式：

- Knowledge pipeline
- ETL for unstructured content
- Metadata-aware indexing
- Vector + sparse + graph coexistence

### L6 模型与治理基础层

典型模式：

- Model gateway
- Multi-model routing
- Guardrails layer
- Evaluation & tracing
- Safety and audit layer

## 8. 面向低代码平台如何映射六层架构

在低代码平台里，这六层可以更具体地表达为：

### L1

- 页面生成入口
- 组件问答入口
- 配置纠错入口
- 运维分析入口

### L2

- 页面生成 Workflow
- 配置诊断 Runtime
- 运维辅助执行流

### L3

- 组件协议检索
- 页面模板检索
- Schema 样本检索
- 案例检索

### L4

- 查询页面工具
- 查询组件工具
- 校验 Schema 工具
- 创建草稿工具
- 查询构建日志工具

### L5

- 页面 Schema 库
- 组件物料知识库
- Setter / slot 规则库
- 错误案例库
- 数据源定义库

### L6

- 多模型网关
- Embedding / Rerank 接入
- Trace / eval / guardrails
- 租户与权限治理

## 9. 常见错误架构形态

### 9.1 只有场景层，没有底座层

结果是每个场景都单独接模型、单独做检索、单独写 Prompt，后续无法平台化。

### 9.2 只有知识层，没有运行时层

结果是系统只能做回答，无法做复杂任务闭环。

### 9.3 只有 Agent 层，没有动作接口和治理层

结果是系统表面很聪明，但不可控、不可调试、不可审计。

### 9.4 把治理做成附属项

当前行业最佳实战已经足够明确：evals、guardrails、tool quality、tracing 都必须前置考虑。

## 10. 一句话总结

六层架构的核心思路，是把 AI 应用拆成“场景、运行时、认知增强、动作接口、知识底座、模型与治理基础”六个能力层，用较少但稳定的分层方式支撑从 RAG 到 Workflow Agent 再到平台化 AI 系统的渐进演进。
