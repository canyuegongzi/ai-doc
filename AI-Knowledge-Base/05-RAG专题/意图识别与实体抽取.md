# 意图识别与实体抽取

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 RAG 在线链路中的意图识别与实体抽取设计，重点回答：

- 为什么现代 RAG 不应只把 query 当成一串文本，而要识别其任务意图和关键实体
- 意图识别、实体抽取与 rewrite / expansion / recall 的关系是什么
- 在最新的 query understanding 和 agentic retrieval 架构中，这一层为什么越来越重要

## 2. 为什么 Query Understanding 必须显式包含意图与实体

企业用户的自然语言问题往往不是纯文本问题，而是隐含了：

- 查询目标
- 场景类型
- 关键业务对象
- 过滤条件
- 时间范围
- 风险边界

例如：

“帮我看看这个审批表单为什么这个字段没出来”

这句话里至少隐含了：

- 意图：诊断
- 对象：审批表单
- 实体：字段
- 问题类型：渲染 / 绑定异常

如果系统只把这当成一句普通 query 去检索，很可能无法充分利用这些信息。

## 3. 近两年的关键变化

### 3.1 Query understanding 已从 query rewrite 扩展为结构化语义解析

Azure AI Search 2026 年的 RAG 与 agentic retrieval 文档都明确强调，现代 RAG 面对的是 complex, conversational, vague questions，信息检索系统必须理解意图，而不仅是匹配词。

这意味着 query understanding 层越来越像“轻量语义解析层”。

### 3.2 agentic retrieval 对实体和意图的结构化表示需求更高

当系统需要：

- 生成多个 focused subqueries
- 在多 source 中选择检索路径
- 按 objectType 或 sourceType 做路由

就必须更显式地知道：

- 当前是在查文档、找案例、查对象定义，还是做诊断
- query 中有哪些实体
- 实体之间是什么关系

### 3.3 企业场景使实体抽取比通用问答更关键

在企业与低代码平台场景中，真正决定 retrieval 质量的，往往是是否识别出：

- 组件名
- 页面名
- 字段名
- 错误码
- API path
- schema node
- tenant / app / environment

这些都属于实体级信号，而不是一般自然语言语义。

## 4. 意图识别解决什么问题

### 4.1 决定检索链路类型

系统需要知道当前问题是：

- FAQ 问答
- 搜索推荐
- 生成增强
- 配置诊断
- 运维分析
- Agent 执行前规范查询

这会直接影响：

- recall 策略
- rerank 目标
- context builder 组织方式
- answer generation 模式

### 4.2 决定后续 query planning 方向

例如：

- 解释型问题偏单查询
- 诊断型问题偏案例 + 规则召回
- 复合问题更适合多查询拆分

## 5. 实体抽取解决什么问题

### 5.1 明确 query 的关键对象

例如识别：

- 组件名
- 页面名
- 错误码
- 字段名
- API 名
- 模板名

### 5.2 支持 metadata filter

有了实体后，可以更精准地做：

- sourceType filter
- objectType filter
- appCode / tenantCode filter
- module routing

### 5.3 支持 query expansion 和 source-aware recall

实体往往决定：

- 哪些同义词需要扩展
- 哪类索引更值得优先搜索
- 哪些案例更相关

## 6. 常见意图类型

建议至少区分以下几类意图。

### 6.1 解释型

回答“是什么、为什么、怎么用”。

### 6.2 搜索导航型

回答“找什么、推荐什么、去哪里看”。

### 6.3 生成增强型

回答“生成前需要参考什么模板、规则、样例”。

### 6.4 诊断型

回答“为什么出错、哪里有问题、怎么修”。

### 6.5 执行增强型

回答“执行某个动作前需要查什么规范、案例、约束”。

## 7. 常见实体类型

在企业和低代码平台中，建议至少识别以下实体类别：

- componentName
- pageName / pageType
- schemaNode
- fieldName
- setterType
- slotName
- apiName / apiPath
- errorCode
- tenantCode
- appCode
- environment
- version

## 8. 推荐的结构化输出

建议 query understanding 阶段至少输出：

- intentType
- entities
- filters
- confidence
- rewrittenQuery
- optionalExpandedQueries

例如：

```json
{
  "intentType": "diagnosis",
  "entities": {
    "componentName": "Table",
    "fieldName": "status",
    "errorCode": null
  },
  "filters": {
    "sourceType": ["componentDoc", "errorCase"],
    "appCode": "purchase-flow"
  }
}
```

这种结构会显著提升后续检索链路可控性。

## 9. 实现路径建议

### 9.1 规则驱动识别

适用于：

- 错误码
- 组件名词典
- API path
- tenant / app / environment

优点：

- 稳定
- 高精度

### 9.2 模型驱动识别

适用于：

- 模糊意图
- 非标准表达
- 多轮上下文补全

### 9.3 混合式识别

企业场景通常推荐：

- 规则先抓强结构实体
- 模型再补意图和模糊实体

## 10. 与 rewrite / expansion / recall 的关系

### 10.1 意图识别影响 rewrite 方式

不同意图需要不同 rewrite 策略。

### 10.2 实体抽取影响 expansion 方式

实体决定：

- 是否要保留特定术语
- 是否要扩展别名和近义词

### 10.3 意图与实体共同影响 recall 路由

它们决定：

- 查哪类 source
- 查哪类 object
- 是否走 hybrid / multi-query / case recall

## 11. 与 agentic retrieval 的关系

在 agentic retrieval 场景中，意图与实体识别的重要性更高，因为系统可能需要：

- 先决定要不要拆子问题
- 再决定每个子问题查哪类 source
- 再决定哪些 query 需要 lexical 强匹配，哪些需要 semantic 强匹配

如果没有结构化意图和实体信息，检索规划会明显更弱。

## 12. 面向低代码平台的典型场景

### 12.1 组件问答

意图：解释 / 搜索导航

实体：组件名、props 名、setter 名、slot 名

### 12.2 页面生成需求

意图：生成增强

实体：页面类型、业务模块、表单类型、审批模式、数据源对象

### 12.3 配置诊断

意图：诊断

实体：组件名、字段名、错误码、schema 节点、页面名

### 12.4 运维分析

意图：诊断 / 执行增强

实体：发布版本、构建任务、环境、租户、页面 ID

## 13. 常见误区

### 13.1 把所有 query 都当成普通问答处理

### 13.2 不识别实体，直接做 dense retrieval

### 13.3 意图识别只用于 UI，不进入 retrieval pipeline

### 13.4 实体识别结果不结构化，后续无法复用

## 14. 一句话总结

意图识别与实体抽取的核心，是把用户自然语言问题从“文本串”升级为“带任务意图、关键对象和过滤条件的结构化查询意图”；近两年的最新 RAG 架构也越来越明确，query understanding 必须具备这层语义解析能力，才能真正支撑复杂企业检索和 agentic retrieval。
