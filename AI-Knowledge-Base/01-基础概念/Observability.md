# Observability

> 状态：第一版草案

## 1. 定义

Observability 可以理解为：系统在运行过程中，通过日志、指标、链路追踪、事件记录和状态数据，让开发者和运维者能够看见 AI 应用实际发生了什么、为什么发生以及问题可能出在哪里的能力。

用一句话概括：

Observability 的本质，是让 AI 系统不再是黑盒，而是一个可被观察、可被诊断、可被解释的运行系统。

## 2. 它解决什么问题

### 2.1 解决运行过程不可见的问题

AI 系统通常涉及：

- 模型调用
- 检索链路
- Tool 调用
- Agent 状态推进
- 输出校验
- 成本消耗

如果这些过程不可见，系统一旦表现异常，很难知道问题出在哪里。

### 2.2 解决故障定位困难的问题

常见问题包括：

- 为什么回答突然变差
- 为什么某类请求成本异常升高
- 为什么 Tool 调用失败
- 为什么 Agent 死循环或步数过多
- 为什么某次结果没有命中正确知识

### 2.3 解决线上运行无法持续优化的问题

没有运行可观测性，就很难积累真实世界的数据来支持后续优化。

## 3. 它不解决什么问题

### 3.1 不等于效果评估

Observability 主要关注系统运行过程和运行状态，而不直接回答质量是否达标。

质量判断属于 Evaluation。

### 3.2 不等于单纯打日志

普通日志只是可观测性的一个部分。更完整的 Observability 还包括：

- 指标
- 链路追踪
- 事件流
- 状态快照
- 成本统计

### 3.3 不天然带来治理

可观测性让问题被看到，但如何限制风险、修复错误、审批高风险动作，还需要治理体系和运行控制策略。

## 4. AI 系统中需要观察什么

### 4.1 请求入口信息

例如：

- 用户身份
- 租户信息
- 场景类型
- 请求来源
- 当前上下文

### 4.2 检索链路信息

例如：

- Query Rewrite 结果
- 召回候选内容
- Rerank 结果
- 最终上下文内容

### 4.3 模型调用信息

例如：

- 使用了哪个模型
- 输入输出 token
- 延迟
- 重试情况
- 失败原因

### 4.4 Tool 调用信息

例如：

- 调用了哪个工具
- 输入参数是什么
- 是否成功
- 返回结果摘要
- 调用耗时

### 4.5 Agent / Workflow 状态信息

例如：

- 当前状态
- 已执行步骤
- 节点跳转路径
- 失败节点
- 人工确认节点

### 4.6 输出与治理信息

例如：

- 输出是否合法
- 是否命中 Guardrails
- 是否触发风险审批
- 最终返回内容摘要

### 4.7 成本与资源信息

例如：

- 单请求成本
- 模型成本分布
- 不同场景资源消耗
- 高成本请求聚类

## 5. Observability 的核心组成

### 5.1 Logs 日志

记录离散事件和执行细节。

### 5.2 Metrics 指标

记录可聚合的数值，例如：

- 请求量
- 平均延迟
- 失败率
- token 消耗
- 工具成功率

### 5.3 Traces 链路追踪

记录一次请求从入口到输出的完整链路。

对 AI 系统尤其重要，因为它能把模型调用、检索步骤、工具调用和状态转换串起来。

### 5.4 Events 事件

记录关键业务节点，例如：

- 命中审批
- 触发人工确认
- 进入失败恢复
- 命中高风险动作限制

## 6. 为什么 AI 系统比传统系统更需要 Observability

因为 AI 系统天然更复杂、更不确定。

### 6.1 链路更长

一次请求可能包含检索、模型、工具、校验、状态切换等多个步骤。

### 6.2 不确定性更高

同一类输入在不同模型、不同上下文下可能产生不同结果。

### 6.3 问题根因更隐蔽

回答错误可能来自知识缺失、召回问题、上下文污染、模型理解偏差、工具失败、规则误判等多种因素。

### 6.4 成本更敏感

AI 系统通常直接消耗模型成本，因此可观测性不仅是稳定性问题，也是资源治理问题。

## 7. Observability 与 Trace 的关系

Trace 是 Observability 的核心组成之一，但不是全部。

可以理解为：

- Trace 负责回答“一次请求具体经历了什么”
- Observability 负责回答“整个系统运行得怎么样、哪里出了问题、趋势如何变化”

## 8. 常见误区

### 8.1 只记录最终输出，不记录中间链路

这会让问题诊断能力非常弱。

### 8.2 只记录技术日志，不记录业务语义

AI 系统的可观测性必须同时保留技术视角和业务视角，例如场景类型、任务意图、风险等级。

### 8.3 没有成本可见性

AI 系统如果看不见 token 和模型成本，后续扩展很容易失控。

## 9. 与其他概念的关系

### 9.1 与 Evaluation 的关系

Observability 负责看运行过程，Evaluation 负责看质量效果。

### 9.2 与 Guardrails 的关系

Guardrails 负责控制边界，Observability 负责记录边界命中情况和执行过程。

### 9.3 与审计的关系

Observability 提供技术运行视角，审计更强调责任归属、动作证据和合规要求。两者相关但不等同。

## 10. 面向低代码平台如何理解 Observability

在低代码平台场景中，重点要观察：

- 页面生成流程走到了哪一步
- 哪些组件或模板被召回
- 哪个校验节点失败
- 哪个工具导致配置修复建议中断
- 哪类页面或场景成本异常高
- 发布辅助流程中是否触发审批和风险限制

这类信息决定后续能否真正把平台智能能力做成可运营系统。

## 11. 一句话总结

Observability 的本质，是通过日志、指标、追踪和事件，让 AI 系统的运行链路、异常原因、资源消耗和状态变化变得可见，从而支持诊断、治理和持续优化。
