# Memory

> 状态：第一版草案

## 1. 定义

在 AI 应用中，Memory 通常指系统为模型或 Agent 持续保留、组织和使用历史信息的机制。

这些信息可能来自：

- 当前会话上下文
- 历史交互记录
- 任务执行过程
- 用户偏好
- 外部知识库
- 长期业务状态

用一句话概括：

Memory 的本质，是让 AI 不只依赖当前这一轮输入，而是能够在合适边界内利用过去的信息持续完成任务。

## 2. 它解决什么问题

### 2.1 解决上下文断裂问题

如果系统每一轮都只看当前输入，很多任务会因为缺失历史上下文而无法顺利推进。

例如：

- 多轮对话中用户省略前文信息
- 任务执行中需要引用前一步结果
- 用户在会话中逐步 уточ化需求

### 2.2 解决任务连续性问题

复杂任务往往跨越多个步骤，执行过程中需要保存中间状态。

例如：

- 已检索到哪些结果
- 已调用过哪些工具
- 当前执行到了哪一步
- 是否已经发生过失败与重试

### 2.3 解决个性化与历史经验复用问题

某些场景下，系统需要保留用户偏好、默认选项、历史常用路径等信息，以提升体验和效率。

### 2.4 解决长期知识利用问题

有些信息不适合只放在当前 Prompt，而需要以更长期、可检索的形式沉淀下来。

## 3. 它不解决什么问题

### 3.1 不等于知识库

Memory 和知识库有关，但不是同一个概念。

知识库通常强调稳定的外部知识资产；Memory 更强调与用户、任务、执行过程相关的动态历史信息。

### 3.2 不等于上下文窗口本身

把更多历史消息直接塞进上下文，不等于真正设计了 Memory。

真正的 Memory 需要考虑：

- 记什么
- 保留多久
- 如何压缩
- 如何检索
- 如何清理

### 3.3 不天然提升正确性

记住更多信息并不一定让系统更准。如果 Memory 没有治理好，反而可能带来：

- 噪声积累
- 上下文污染
- 过期信息干扰
- 错误偏好延续

## 4. Memory 的常见类型

### 4.1 会话记忆

保存当前对话中的上下文信息。

适用于：

- 多轮问答
- 逐步 уточ化需求
- 当前交互会话中的连续理解

### 4.2 任务记忆

保存某个任务执行过程中的状态和中间结果。

适用于：

- Workflow
- Agent 执行闭环
- 多步骤工具调用

### 4.3 用户记忆

保存与特定用户相关的偏好、历史行为或常见习惯。

适用于：

- 个性化推荐
- 常用配置偏好
- 默认交互习惯

### 4.4 长期记忆

保存跨会话、跨任务仍然有价值的信息。

适用于：

- 长期项目上下文
- 历史决策记录
- 重要经验沉淀

### 4.5 外部知识型记忆

通过知识库、检索系统、案例库等方式保存更长期和可检索的信息。

这一类记忆与 RAG 高度相关。

## 5. Memory 的核心组成

### 5.1 写入机制

系统需要明确什么时候把什么信息写入 Memory。

### 5.2 存储机制

不同类型的 Memory 可以采用不同存储方式，例如：

- 会话缓存
- 结构化数据库
- 文档存储
- 向量索引
- 事件日志

### 5.3 检索机制

系统需要决定在什么时机、以什么条件读取历史信息。

### 5.4 压缩与摘要机制

随着历史增长，Memory 需要压缩、摘要和筛选，否则上下文会无限膨胀。

### 5.5 生命周期管理

系统需要管理：

- 保留多久
- 何时失效
- 何时归档
- 何时删除

## 6. Memory 在 Agent 中的作用

对 Agent 来说，Memory 通常不是可有可无，而是执行闭环的重要组成部分。

典型作用包括：

- 保存任务目标
- 保存步骤执行状态
- 保存工具调用结果
- 保存失败和重试信息
- 保存人工确认结果
- 保存最终输出与审计线索

没有合适的 Memory，Agent 很难支撑稳定的多步骤任务。

## 7. Memory 与 RAG 的关系

Memory 和 RAG 经常会被混在一起，但两者关注点不同。

### 7.1 RAG 更强调外部知识增强

RAG 关注的是如何从知识源中检索与当前问题相关的信息。

### 7.2 Memory 更强调历史状态延续

Memory 关注的是系统如何记住与当前用户、当前任务、过去步骤相关的信息。

### 7.3 两者可以协同

在复杂系统中，Memory 可以与 RAG 一起工作：

- 先从 Memory 中恢复当前任务状态
- 再用 RAG 查外部知识
- 再继续决策与执行

## 8. 常见误区

### 8.1 认为 Memory 越多越好

过量保留历史会显著增加噪声和上下文污染风险。

### 8.2 认为聊天历史就是完整 Memory

对多步骤任务而言，仅保留聊天记录远远不够，还需要结构化任务状态和执行记录。

### 8.3 忽视过期与冲突信息

Memory 中的信息如果不治理，很容易出现历史信息过期、偏好冲突、错误状态被继续沿用的问题。

## 9. 面向低代码平台如何理解 Memory

在低代码平台场景中，Memory 可以重点作用于：

- 用户当前页面设计目标
- 页面生成过程中的中间结构
- 组件选择历史与偏好
- 配置诊断过程中的步骤记录
- Agent 执行过程中的工具调用轨迹

这类信息往往既不适合只靠一次 Prompt 保存，也不适合全部混入长期知识库，因此需要专门的 Memory 设计。

## 10. 一句话总结

Memory 的本质，是为 AI 系统提供对历史上下文、任务状态和用户信息的持续记忆能力，让系统能够在多轮交互和多步骤执行中保持连续性，而不是每次都从零开始。
