# Guardrails

> 状态：第一版草案

## 1. 定义

Guardrails 可以理解为一组用于约束 AI 系统输入、输出、行为和执行边界的控制机制。

这些机制的目标，是让 AI 系统在真实业务环境中更加：

- 安全
- 可控
- 稳定
- 可审计
- 可接入业务系统

用一句话概括：

Guardrails 的本质，是为 AI 系统建立“护栏”，避免其在开放生成和动态执行中越界、失控或输出不可用结果。

## 2. 它解决什么问题

### 2.1 解决输出不受控问题

模型天然倾向于生成开放文本，但很多业务场景需要结构稳定、内容受限的输出。

例如：

- 必须返回合法 JSON
- 必须符合 Schema 约束
- 必须引用来源
- 不能输出敏感信息

### 2.2 解决行为越界问题

在 Tool Calling 和 Agent 场景中，系统需要控制：

- 哪些工具可以调用
- 哪些动作必须审批
- 哪些输入应被拒绝
- 哪些场景应中止执行

### 2.3 解决风险与合规问题

企业环境中必须考虑：

- 敏感信息泄露
- 越权访问
- 高风险操作误执行
- 不可追责的自动决策

### 2.4 解决工程接入问题

如果没有 Guardrails，很多 AI 输出无法直接接入业务系统，因为它们可能：

- 格式不合法
- 结构不稳定
- 缺失关键字段
- 不满足业务规则

## 3. Guardrails 的主要作用层面

### 3.1 输入护栏

约束用户输入、系统输入和上下文内容。

例如：

- 敏感输入检测
- 非法请求拦截
- Prompt 注入识别
- 超范围请求识别

### 3.2 输出护栏

约束模型最终输出。

例如：

- JSON Schema 校验
- 敏感内容检测
- 引用要求
- 结果格式修复

### 3.3 行为护栏

约束模型或 Agent 的行动边界。

例如：

- 工具白名单
- 风险动作审批
- 最大执行步数限制
- 人工确认节点

### 3.4 数据护栏

约束系统可访问和可返回的数据范围。

例如：

- 权限过滤
- 租户隔离
- 字段脱敏
- 数据域限制

## 4. Guardrails 的常见实现方式

### 4.1 结构化输出约束

通过 JSON Schema、函数调用协议、DSL 约束模型输出结构。

### 4.2 规则校验

在模型输出后，通过规则或校验器检查结果是否合法。

### 4.3 风险分级与审批

对于高风险行为，不直接执行，而是进入人工确认或审批流。

### 4.4 运行时拦截

在 Agent 执行过程中对：

- 步数
- 工具类型
- 参数范围
- 调用结果

进行动态检查与拦截。

### 4.5 安全过滤

对输入、上下文、输出和日志中的敏感信息进行过滤或脱敏。

## 5. Guardrails 与 Prompt 的关系

Prompt 可以起到一定约束作用，但 Prompt 不是完整 Guardrails。

原因在于：

- Prompt 只是软约束
- 模型不一定始终严格遵守
- Prompt 无法替代权限系统、规则引擎和审批机制

因此，真正可靠的 Guardrails 应尽量采用“显式机制 + 协议约束 + 系统检查”，而不是只依赖提示词。

## 6. Guardrails 与安全治理的关系

Guardrails 是安全治理的一部分，但不等于全部安全治理。

更完整的治理体系还包括：

- 身份透传
- 租户隔离
- 权限模型
- 审计日志
- 风险动作审批
- 成本与配额治理

Guardrails 更偏向“运行时边界控制”。

## 7. Guardrails 在 Agent 中的重要性

Agent 越强，Guardrails 越重要。

因为 Agent 具有：

- 多步骤执行能力
- 工具调用能力
- 条件分支能力
- 可能带来副作用的行为能力

如果没有 Guardrails，Agent 可能出现：

- 越权调用工具
- 重复执行
- 误触发高风险动作
- 在错误前提下持续推进任务

## 8. 常见误区

### 8.1 认为 Guardrails 只是内容审查

内容过滤只是 Guardrails 的一部分。更重要的是结构约束、行为控制、权限边界和风险治理。

### 8.2 认为 Prompt 已经足够当护栏

Prompt 可以辅助引导，但不应承担关键控制职责。

### 8.3 认为 Guardrails 只在上线前需要考虑

Guardrails 应在系统设计早期就纳入，而不是上线前补一层过滤。

## 9. 面向低代码平台如何理解 Guardrails

在低代码平台中，Guardrails 需要重点覆盖：

- Schema 输出结构校验
- 组件协议约束
- 插槽与 Setter 规则约束
- 高风险配置变更控制
- 发布前检查
- 多租户与权限边界保护

这意味着低代码场景中的 Guardrails，不只是“防止乱说”，更重要的是“防止乱配、乱改、乱执行”。

## 10. 一句话总结

Guardrails 的本质，是通过结构约束、规则校验、行为限制和安全控制，为 AI 系统建立稳定、可控、可审计的运行边界，让其能够安全进入真实业务环境。
