# Tool-Calling

> 状态：第一版草案

## 1. 定义

Tool Calling 可以理解为：模型在生成过程中，不直接凭空完成所有任务，而是按照预定义协议调用外部工具，以获取信息、执行动作或完成校验，然后再基于工具结果继续生成或决策。

用一句话概括：

Tool Calling 的本质，是让模型具备“调用外部能力”的接口，而不是把所有问题都交给模型内部完成。

## 2. 它解决什么问题

### 2.1 解决模型无法直接访问外部系统的问题

模型本身不能直接访问数据库、调用 API、读取日志、查询页面配置、校验 Schema，也不能直接执行发布、通知等系统动作。

Tool Calling 通过标准化接口，把外部系统能力暴露给模型使用。

### 2.2 解决知识与行动之间的断层问题

很多任务不是知道答案就结束，而是需要基于知识继续行动。

例如：

- 查某页面的发布记录
- 校验一段 Schema 的结构
- 查询某组件协议定义
- 生成页面草稿后执行规则检查

这些都需要工具参与。

### 2.3 解决结构化交互问题

相比自由文本交互，Tool Calling 通常要求：

- 明确的工具名称
- 明确的输入参数
- 明确的输出结构
- 明确的调用约束

这使得模型与系统之间的协作更稳定、更可控。

## 3. 它不解决什么问题

### 3.1 不天然解决任务规划问题

Tool Calling 只提供“如何调用工具”的能力，不负责决定整体任务应该怎么拆、先后顺序是什么、失败后怎么恢复。

这些问题属于 Workflow 或 Agent Runtime。

### 3.2 不天然解决权限与风险问题

模型能够调用工具，不代表它有权限调用所有工具，也不代表工具调用天然安全。

工具调用仍然需要：

- 身份透传
- 权限校验
- 风险分级
- 审计记录
- 高风险审批

### 3.3 不天然解决知识理解问题

Tool Calling 解决的是访问外部能力，不是认知增强。查询前仍然可能需要 RAG 提供知识补充。

## 4. Tool Calling 的核心组成

### 4.1 工具定义

每个工具都应具备明确协议，至少包括：

- 名称
- 描述
- 输入参数 Schema
- 输出结构
- 调用限制
- 风险等级

### 4.2 工具选择

模型或流程系统需要决定当前任务应该调用哪个工具。

### 4.3 参数生成

模型需要将自然语言意图转成结构化参数。

### 4.4 调用执行

系统负责真正执行工具，并将结果返回给模型或流程引擎。

### 4.5 结果观察

工具结果通常不是最终答案，而是下一步决策的输入。

## 5. Tool 与函数调用的关系

在很多模型平台中，Tool Calling 常以 function calling、structured tool use、actions 等形式出现。

可以把它统一理解为：

模型输出的不是最终自然语言，而是一个受限的“动作意图”，由系统按协议执行。

## 6. Tool Calling 的常见类型

### 6.1 查询型工具

例如：

- 查询页面信息
- 查询组件定义
- 查询发布状态
- 查询日志片段

### 6.2 检索型工具

例如：

- 搜索文档
- 搜索组件
- 搜索 Schema 模板

### 6.3 校验型工具

例如：

- Schema 校验
- 配置规则检查
- 参数合法性校验

### 6.4 生成型工具

例如：

- 生成页面草稿
- 生成配置片段
- 生成逻辑节点草稿

### 6.5 执行型工具

例如：

- 创建任务
- 提交草稿
- 触发低风险操作
- 发送通知

执行型工具风险最高，必须重点治理。

## 7. Tool Calling 的工程价值

### 7.1 让 AI 系统具备真实操作能力

没有 Tool Calling，很多 AI 系统只能停留在建议层。

### 7.2 让自然语言请求与系统接口建立桥梁

Tool Calling 把自然语言意图映射为系统可执行的结构化动作。

### 7.3 让模型输出更容易被约束和审计

相比自由文本，工具调用的参数、结果和风险边界更容易被记录和治理。

## 8. 常见误区

### 8.1 把 Tool 当成普通函数暴露给模型就结束了

真正可用的 Tool 系统还需要权限、风险、审计、超时、重试、幂等和错误恢复设计。

### 8.2 认为 Tool Calling 就等于 Agent

Tool Calling 只是执行能力的接口形式，Agent 还需要任务理解、状态管理、规划和终止机制。

### 8.3 认为工具越多越好

过多无边界的工具会显著增加模型决策难度、执行风险和维护成本。

## 9. 与其他概念的关系

### 9.1 与 Agent 的关系

Tool 是 Agent 的动作接口，Agent 通过 Tool 实现外部交互。

### 9.2 与 Workflow 的关系

Workflow 决定什么阶段调用什么 Tool，Tool 执行具体动作。

### 9.3 与 RAG 的关系

RAG 负责补知识，Tool Calling 负责做动作。两者经常在一个任务中配合使用。

## 10. 面向低代码平台如何理解 Tool Calling

在低代码平台中，Tool Calling 应重点围绕平台对象设计，例如：

- `searchComponent`
- `searchSchemaTemplate`
- `getPageInfo`
- `validateSchema`
- `getDataSourceDefinition`
- `createPageDraft`
- `checkPublishRisk`

此时 Tool Calling 的关键不只是“能调用”，而是：

- 工具协议是否稳定
- 输入输出是否结构化
- 风险边界是否明确
- 是否能被 Workflow 和 Agent 复用

## 11. 一句话总结

Tool Calling 的本质，是通过标准化协议把外部系统能力暴露给模型和 Agent 使用，让 AI 从“只能生成文本”扩展为“可以借助系统能力完成任务”。
