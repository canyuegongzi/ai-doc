# Evaluation

> 状态：第一版草案

## 1. 定义

Evaluation 可以理解为：通过一套明确的方法、样本、指标和流程，系统性评估 AI 应用在某类任务上的效果、稳定性和改进空间。

它关注的问题不是“感觉怎么样”，而是“在什么样的输入下，系统表现如何、问题出在哪里、优化后是否真的变好了”。

用一句话概括：

Evaluation 的本质，是把 AI 效果从主观体感变成可分析、可比较、可持续优化的对象。

## 2. 它解决什么问题

### 2.1 解决效果评估模糊的问题

如果没有评测体系，团队对 AI 效果的判断往往依赖：

- 个别 Demo
- 主观印象
- 少量案例
- 零散用户反馈

这会导致优化方向非常不稳定。

### 2.2 解决“问题到底出在哪一层”的判断问题

AI 系统通常包含多个环节，例如：

- Query 理解
- 召回
- 重排
- 上下文构建
- 模型生成
- 工具调用
- 结果校验

如果没有评测，很难知道问题到底出在检索、生成、工具还是规则。

### 2.3 解决优化是否有效的问题

当你更换模型、修改 Prompt、调整 Chunk 策略、增加 Metadata、引入 Rerank 或重写工具协议时，需要有方法验证优化是否真的带来提升。

## 3. 它不解决什么问题

### 3.1 不等于线上监控

Evaluation 主要关注“质量评估和效果判断”，而不是持续运行监控。

线上运行状态、调用链路、错误和成本变化属于 Observability 范畴。

### 3.2 不等于人工体验反馈本身

人工反馈是重要输入，但不能替代系统化评测。

### 3.3 不天然给出解决方案

评测告诉你“哪里有问题、变化是否有效”，但如何优化仍然需要结合架构、工程和场景分析来判断。

## 4. Evaluation 的核心对象

AI 系统中的可评测对象通常包括：

- 问答准确性
- 检索命中情况
- 排序效果
- 生成质量
- 结构化输出合法性
- Tool 调用成功率
- Agent 任务完成率
- 幻觉情况
- 人工满意度

## 5. Evaluation 的常见类型

### 5.1 离线评测

使用预先准备好的样本集进行批量评估。

特点：

- 可重复
- 可对比
- 适合回归验证
- 适合方案选型

### 5.2 在线评测

在真实用户请求和线上数据中观察效果。

特点：

- 更接近真实使用情况
- 可发现离线样本没有覆盖的问题
- 通常与反馈机制结合

### 5.3 人工评测

由人工对输出质量、可用性、可信度等维度进行判断。

适用于：

- 难以完全自动化打分的场景
- 复杂生成任务
- 主观体验较强的任务

### 5.4 自动评测

通过规则、指标、比对算法或模型辅助判断结果质量。

适用于：

- 结构化输出校验
- 检索命中情况判断
- 答案与标准集对比
- 格式与规则一致性检查

## 6. RAG 场景下如何做 Evaluation

RAG 不应只看最终回答是否“像对的”，还应拆层评估。

### 6.1 Query 理解评测

评估改写后的查询是否更接近真实意图。

### 6.2 Recall 评测

评估相关知识是否被召回。

### 6.3 Rerank 评测

评估相关片段是否排在更靠前的位置。

### 6.4 Context Builder 评测

评估传入模型的上下文是否相关、是否冗余、是否缺少关键片段。

### 6.5 Answer 评测

评估最终回答是否正确、完整、可引用、无幻觉。

这说明：

RAG 的评测必须分层，不应只看最后一句回答。

## 7. Agent 场景下如何做 Evaluation

Agent 的评测往往比普通问答更复杂，因为它涉及执行链路。

可以关注以下维度：

- 任务完成率
- 工具选择正确率
- 参数填写正确率
- 执行路径合理性
- 步数与成本
- 失败恢复效果
- 高风险行为控制情况

在 Agent 场景中，任务是否完成往往比输出措辞是否优美更重要。

## 8. Evaluation 的核心组成

### 8.1 评测目标

先明确评测要回答什么问题，例如：

- 当前 RAG 是否有效
- 新 Prompt 是否优于旧 Prompt
- 新模型是否值得替换
- Tool 调用是否足够稳定

### 8.2 评测样本

样本应覆盖典型场景、边界情况、常见错误和高价值任务。

### 8.3 指标体系

不同任务需要不同指标，例如：

- Recall 命中率
- TopK 命中率
- 结构化输出合法率
- 幻觉率
- 任务成功率
- 用户接受率

### 8.4 评测流程

需要明确：

- 谁执行评测
- 何时评测
- 如何记录结果
- 如何做版本对比

### 8.5 反馈闭环

评测结果最终应进入优化闭环，而不是只生成一份报告。

## 9. 常见误区

### 9.1 只做主观体验，不做系统评测

这会导致优化方向持续漂移。

### 9.2 只看最终答案，不看中间链路

对 RAG 和 Agent 来说，这会掩盖问题根因。

### 9.3 只有离线评测，没有线上反馈

离线样本重要，但无法覆盖真实世界全部复杂性。

### 9.4 指标很多，但没有决策意义

指标必须能支持具体判断，否则只是噪声。

## 10. 与其他概念的关系

### 10.1 与 Observability 的关系

Evaluation 关注“效果好不好”，Observability 关注“运行时发生了什么”。两者不同，但必须协同。

### 10.2 与 Guardrails 的关系

Guardrails 是运行边界控制，Evaluation 负责判断当前能力是否达到预期质量。

### 10.3 与案例复盘的关系

案例复盘是重要评测输入，评测结果也应反过来丰富案例库。

## 11. 面向低代码平台如何理解 Evaluation

在低代码平台场景中，Evaluation 应重点关注：

- 组件推荐是否合理
- Schema 生成是否结构合法
- 配置诊断是否命中真实问题
- 生成建议是否可落地
- 工具调用是否稳定
- 页面与组件知识召回是否准确

这些评测维度通常不能只看自然语言质量，而应更关注结构、规则和任务成功情况。

## 12. 一句话总结

Evaluation 的本质，是通过样本、指标和流程，把 AI 系统的效果从“凭感觉判断”转化为“可衡量、可比较、可持续优化”的工程对象。
