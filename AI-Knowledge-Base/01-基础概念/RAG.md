# RAG

> 状态：第一版草案

## 1. 定义

RAG，Retrieval-Augmented Generation，通常译为“检索增强生成”。

它的基本思想是：在模型生成答案之前，先从外部知识源中检索与当前问题相关的内容，再将这些内容作为上下文输入模型，从而让模型基于外部知识进行回答、总结、生成或决策。

用一句话概括：

RAG 的核心不是“让模型自己回忆”，而是“让模型在生成前先查资料”。

## 2. 它解决什么问题

RAG 主要解决以下几类问题。

### 2.1 模型知识过期问题

通用大模型的训练数据有时间边界，因此对最新知识、最新规则、最新文档并不可靠。

RAG 通过接入外部知识源，让模型可以基于更新后的数据回答问题。

### 2.2 私有知识缺失问题

企业内部文档、系统配置、产品规范、组件协议、历史案例等私有知识通常不在模型预训练范围内。

RAG 可以把这些知识转化为可检索上下文，补足模型的业务知识盲区。

### 2.3 领域语义不足问题

模型懂通用语言，但未必理解具体领域对象。

例如在低代码平台场景里，模型默认并不知道：

- 页面 Schema 的结构
- 组件物料的配置协议
- 插槽和 Setter 的规则
- 数据源与逻辑编排的关系

RAG 可以帮助模型在回答和生成前获得领域上下文。

### 2.4 事实依据不足问题

纯模型生成很容易出现“说得像对，但没有依据”的问题。

RAG 通过引入来源明确的上下文，可以提升回答的可追溯性，并支持引用出处。

## 3. 它不解决什么问题

RAG 很重要，但不是万能方案。它天然不解决以下问题。

### 3.1 不天然解决任务执行问题

RAG 的主要职责是增强认知，不是直接执行任务。

如果要查系统状态、调用 API、创建页面草稿、发送通知、执行发布，就需要 Tool、Workflow 和 Agent 能力。

### 3.2 不天然解决权限治理问题

即使有 RAG，如果没有额外的权限过滤和身份透传机制，模型仍然可能拿到不该拿到的内容。

因此，权限控制必须由知识处理和检索系统显式承担，不能指望模型自觉处理。

### 3.3 不天然解决输出稳定性问题

RAG 只能改善知识依据，不保证输出一定结构化、稳定、可接入系统。

结构化输出、JSON Schema、结果校验等问题需要单独设计。

### 3.4 不天然解决复杂多步骤推理和执行闭环

即使模型看到了检索内容，也不代表它一定能完成复杂任务拆解、工具调用、状态管理和错误恢复。

这些问题属于 Agent / Workflow 范畴。

## 4. RAG 的核心组成

一个完整的 RAG 系统通常至少由以下几部分组成。

### 4.1 Query 理解

用户原始问题往往不是最适合检索的查询形式，因此需要先做查询理解和改写。

常见能力包括：

- Query Rewrite
- Query Expansion
- 意图识别
- 实体抽取
- 多轮上下文补全

### 4.2 Recall 召回

从知识源中找出候选内容。

召回方式通常包括：

- 向量召回
- 关键词召回
- BM25
- Hybrid Search
- Metadata Filter
- 图谱或关系召回
- 代码检索

### 4.3 Rerank 重排

召回结果通常会包含噪声，因此需要进一步排序。

常见方式包括：

- Cross-encoder rerank
- 基于元数据加权
- 基于时间新鲜度排序
- 基于业务优先级排序

### 4.4 Context Builder 上下文构建

不是把召回结果原样塞给模型，而是要做上下文治理。

常见工作包括：

- 去重
- 去噪
- 合并同主题片段
- 截断无关内容
- 保留来源信息
- 控制 token 长度

### 4.5 Answer Generation 答案生成

在构造好的上下文基础上，由模型生成回答、摘要、推荐结果或结构化输出。

### 4.6 Citation 与依据回传

在企业场景中，很多时候不仅要给答案，还要说明“依据来自哪里”。

因此，一个完整的 RAG 系统通常需要支持来源保留与引用展示。

## 5. RAG 的完整链路

一个工程化的 RAG 链路通常是：

文档采集 -> 文档解析 -> 清洗标准化 -> Chunk 切分 -> Metadata 抽取 -> 向量化与索引 -> Query 理解 -> 多路召回 -> 重排 -> 上下文组装 -> 模型生成 -> 引用回传 -> 评测与反馈

这说明 RAG 绝不只是“把文档丢进向量库”这么简单。

## 6. RAG 的前置基础

RAG 要发挥价值，前提不是先选某个向量库，而是先把以下基础打好。

### 6.1 知识工程基础

如果文档没有被正确解析、切分、标注和更新，再强的检索模型也很难救回来。

### 6.2 元数据基础

很多业务场景下，Metadata 的重要性不低于正文内容。

例如：

- 文档类型
- 业务线
- 租户
- 应用编号
- 组件编号
- 更新时间
- 作者
- 权限范围

这些信息直接决定检索范围、排序逻辑和上下文筛选。

### 6.3 权限基础

企业 RAG 必须考虑谁能看到什么文档、什么片段、什么租户下的数据。

### 6.4 评测基础

如果没有评测集和反馈闭环，就很难知道检索是否真的有效。

## 7. RAG 的常见类型

RAG 不只有一种形态。按照场景不同，可以拆成几类。

### 7.1 FAQ 型 RAG

适用于：

- 产品问答
- 组件文档问答
- 平台帮助问答
- 制度与规范问答

特点：

- 问答目标明确
- 文档相对稳定
- 强调准确和可引用

### 7.2 搜索推荐型 RAG

适用于：

- 组件推荐
- 页面模板推荐
- API 推荐
- 文档导航

特点：

- 不一定直接生成长文本
- 更强调召回质量与排序质量
- 接近搜索引擎增强形态

### 7.3 生成辅助型 RAG

适用于：

- Schema 生成
- 表单生成
- 配置生成
- 文档草稿生成

特点：

- 检索结果不是最终答案，而是生成约束
- 更依赖结构化上下文和领域样本

### 7.4 执行增强型 RAG

适用于：

- Agent 调用工具前先查资料
- 生成配置前先查组件协议
- 执行操作前先查规范和历史案例

特点：

- RAG 是执行前的认知增强模块
- 与 Tool 和 Agent 强耦合

## 8. RAG 在企业中的真正价值

RAG 在企业中的价值，不只是“让模型答得更对”，而是帮助建立一套可追溯、可更新、可落地的知识服务能力。

可以概括为四点。

### 8.1 把静态资料转化为动态认知能力

让文档、规范、配置、案例不只是存着，而是能被实时理解和调用。

### 8.2 把私有知识转化为 AI 能力底座

让企业自己的知识资产真正参与 AI 生成和决策过程。

### 8.3 把领域知识转化为业务语义增强

让 AI 不只是懂自然语言，还能懂平台对象和业务上下文。

### 8.4 把知识服务从“搜索”升级为“检索 + 生成 + 引用”

相比传统搜索，RAG 可以更直接地组织答案和推荐结果，但前提是知识链路足够扎实。

## 9. RAG 的常见误区

### 9.1 把 RAG 等同于向量检索

向量检索只是召回手段之一，远远不等于完整 RAG。

### 9.2 只关注 embedding，不关注知识质量

如果文档脏、Chunk 切错、Metadata 缺失，embedding 本身无法解决问题。

### 9.3 只做召回，不做重排和上下文治理

直接把 TopK 全塞进 Prompt，通常会带来噪声和上下文污染。

### 9.4 没有权限过滤就直接接入企业知识

这在企业内部是高风险设计。

### 9.5 没有评测就凭感觉优化

这样很难知道问题出在召回、重排、上下文构建还是模型生成。

### 9.6 误以为 RAG 可以替代 Tool 和 Agent

RAG 解决的是“知道”，不是“执行”。

## 10. RAG 与其他概念的关系

### 10.1 与 Embedding 的关系

Embedding 是实现语义检索的重要基础之一，但 RAG 不等于 Embedding。

### 10.2 与向量库的关系

向量库是 RAG 的存储与检索组件之一，但不是 RAG 全部。

### 10.3 与搜索的关系

RAG 可以理解为“搜索能力 + 上下文构建 + 生成能力”的组合形态。

### 10.4 与 Prompt 的关系

Prompt 决定模型如何使用检索结果，但 Prompt 本身不是 RAG 的全部。

### 10.5 与 Agent 的关系

RAG 为 Agent 提供认知增强，Agent 则在需要时把 RAG 结果转化为行动依据。

### 10.6 与 Workflow 的关系

在很多企业场景中，RAG 往往不是单独存在，而是嵌入到某个更大的 Workflow 里。

## 11. 面向低代码平台时如何理解 RAG

在低代码平台场景中，RAG 不应只被理解为“文档问答系统”，而应理解为平台知识增强引擎。

它可以服务于：

- 组件知识问答
- 页面模板推荐
- Schema 生成增强
- 配置纠错诊断
- Agent 执行前的规范查询

此时，RAG 的知识源不只是说明文档，还包括：

- 页面 Schema 样本
- 组件协议
- Setter 定义
- 插槽规则
- 数据源定义
- 逻辑编排节点说明
- 平台错误案例

这意味着低代码 RAG 的重点，往往不是长文档问答，而是围绕结构化对象进行检索与生成增强。

## 12. 一句话总结

RAG 的本质，是通过外部知识检索和上下文构造，让模型在生成前先获得可信的业务知识，从而提升回答、推荐、生成和决策的准确性；它解决的是“模型知道什么”的问题，而不是“模型能执行什么”的问题。
