# Workflow

> 状态：第一版草案

## 1. 定义

Workflow 可以理解为：将一个任务拆分为若干个有明确顺序、输入输出和控制边界的步骤，并按照预定义或半预定义方式执行的流程结构。

在 AI 应用中，Workflow 的作用是把模型、检索、工具、校验、人工确认等能力组织成一个稳定的执行路径。

用一句话概括：

Workflow 的本质，是把不确定的 AI 能力放进一个相对确定的流程框架里。

## 2. 它解决什么问题

### 2.1 解决复杂任务难以一次完成的问题

很多任务不是一次模型输出就能完成，而需要多个步骤串联。

例如：

- 先识别任务类型
- 再检索相关知识
- 再生成结果草稿
- 再调用校验工具
- 必要时进入人工确认
- 最终输出结果

Workflow 负责把这些步骤串起来。

### 2.2 解决 AI 执行不稳定的问题

纯模型决策路径可能较随机，难以复现。Workflow 通过明确节点和边界，让系统行为更稳定。

### 2.3 解决治理和审计问题

有了 Workflow，系统更容易回答：

- 当前执行到哪一步
- 哪一步失败了
- 哪一步调用了什么工具
- 哪一步需要人工确认
- 整个流程如何回放

## 3. 它不解决什么问题

### 3.1 不天然解决知识缺失问题

Workflow 再清晰，如果没有知识工程和 RAG 支撑，模型在某些节点仍然可能缺少必要上下文。

### 3.2 不天然解决工具设计问题

Workflow 能编排工具，但工具本身的协议、权限、风险、幂等仍需要单独设计。

### 3.3 不意味着完全没有智能决策

Workflow 并不等于纯规则流程。很多 Workflow 中仍然会让模型在节点内做局部判断和生成。

## 4. Workflow 的核心组成

### 4.1 节点

每个节点代表一个明确步骤，例如：

- 任务理解
- 检索知识
- 调用工具
- 生成结果
- 结果校验
- 人工确认

### 4.2 边

边表示节点之间的流转关系，决定下一步走向。

### 4.3 输入输出

每个节点应有明确输入和输出，避免流程中间态失控。

### 4.4 控制条件

例如：

- 成功则进入下一步
- 失败则重试
- 高风险则转人工
- 无结果则换检索路径

### 4.5 状态记录

流程执行中需要记录节点状态、中间结果、错误和重试信息。

## 5. Workflow 在 AI 应用中的典型形态

### 5.1 固定流程 Workflow

所有步骤都预定义，模型只在局部节点参与。

适合：

- 结构清晰
- 风险较高
- 需要严格可控

### 5.2 条件分支 Workflow

流程主体固定，但在局部节点根据条件分支。

适合：

- 有明显路径选择
- 需要一定灵活性
- 仍然要求较高可控性

### 5.3 Agent-in-Workflow

由 Workflow 提供主框架，在局部复杂节点中引入 Agent 做决策。

适合：

- 既要控制整体边界
- 又要保留局部智能弹性

## 6. 为什么 Workflow 是企业早期更推荐的形态

在企业场景中，相比开放式 Agent，Workflow 具有明显优势：

- 更稳
- 更可观测
- 更容易调试
- 更方便审计
- 更容易插入审批和人工确认
- 更适合权限和风险边界控制

因此，企业早期通常应优先建设 Workflow 能力，再逐步引入更复杂的 Agent 化机制。

## 7. Workflow 与 Agent 的关系

Workflow 和 Agent 可以看作“框架”和“局部智能”的关系。

- Workflow 负责定义执行边界和主路径
- Agent 负责在部分节点中进行动态决策

在很多企业系统中，最实用的方案不是二选一，而是“Workflow 承载主干，Agent 提供局部弹性”。

## 8. Workflow 与 Tool Calling 的关系

Workflow 决定何时调用 Tool，Tool 执行具体动作。

Tool Calling 是动作层协议，Workflow 是流程层组织方式。

## 9. Workflow 与状态机的关系

Workflow 往往需要借助状态机思想来管理执行过程，例如：

- INIT
- UNDERSTAND
- RETRIEVE
- ACT
- VALIDATE
- CONFIRM
- COMPLETE
- FAILED

显式状态机能够显著提升流程可观测性和失败恢复能力。

## 10. 常见误区

### 10.1 认为 Workflow 就等于传统 BPM

AI Workflow 与传统流程类似，但会在节点内引入模型生成、检索增强和工具协作，因此更加动态。

### 10.2 认为有 Workflow 就不需要 Agent

Workflow 适合承载主流程，但复杂任务中仍然可能需要 Agent 做局部决策。

### 10.3 认为 Workflow 一定不灵活

合理设计的 Workflow 可以在保持边界清晰的同时，提供足够的分支和弹性。

## 11. 面向低代码平台如何理解 Workflow

在低代码平台中，Workflow 很适合承接以下任务：

- 需求描述 -> 检索组件与模板 -> 生成 Schema 草稿 -> 校验 -> 输出建议
- 报错信息 -> 检索协议与案例 -> 调用校验工具 -> 生成诊断结果
- 平台问题查询 -> 多源查询 -> 汇总结果 -> 输出建议或转人工

这类场景都具有明显的步骤特征，因此非常适合优先采用 Workflow 方式落地。

## 12. 一句话总结

Workflow 的本质，是为 AI 应用提供一个稳定、可控、可观测的任务执行框架，把模型、检索、工具、校验和人工协同组织成可落地的业务流程。
