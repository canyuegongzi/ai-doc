# Agent

> 状态：第一版草案

## 1. 定义

Agent 可以理解为一种“以目标为中心、能够在上下文中持续决策并调用外部能力完成任务”的 AI 应用形态。

与一次性生成答案不同，Agent 的核心不只是输出文本，而是围绕目标进行一系列动作，例如：

- 理解任务
- 制定计划
- 调用工具
- 观察结果
- 调整下一步动作
- 最终完成交付

用一句话概括：

Agent 的本质，是让模型从“回答者”扩展为“任务执行中的决策者”。

## 2. 它解决什么问题

Agent 主要解决以下几类问题。

### 2.1 多步骤任务问题

很多企业场景不是一次回答就能结束，而是需要连续多个步骤才能完成。

例如：

- 先理解用户需求，再查询知识，再生成页面草稿
- 先分析错误信息，再调用校验工具，再给出修复建议
- 先查询状态，再汇总结果，再触发后续流程

这类问题需要系统具备连续决策与步骤衔接能力。

### 2.2 工具协作问题

模型本身不能直接访问数据库、调用 API、读日志、校验 Schema、创建任务，因此需要借助工具完成真实动作。

Agent 的价值之一，就是在任务上下文中决定什么时候用什么工具，以及如何使用工具结果继续推进任务。

### 2.3 动态路径选择问题

一些任务的执行路径无法完全事先写死。

例如：

- 某个查询结果为空时是否换一种查法
- 某个校验失败后是否重试或换工具
- 某些情况下是否需要先查知识再执行
- 高风险操作是否需要人工确认

Agent 通过动态决策处理这些分支。

### 2.4 任务闭环问题

企业场景中，用户需要的通常不是“解释”，而是“结果”。

Agent 的目标，是将问答、检索、工具、状态管理和执行结果组织成一个闭环，而不是只输出一个可能有用的建议。

## 3. 它不解决什么问题

Agent 很重要，但不能把所有问题都交给 Agent。

### 3.1 不天然解决知识缺失问题

如果没有知识工程和 RAG 支撑，Agent 仍然可能在错误知识基础上做决策。

因此，Agent 不替代 RAG，反而经常依赖 RAG 作为认知增强模块。

### 3.2 不天然解决权限和风险问题

Agent 可以调用工具，但不意味着它天然知道什么工具能调、什么动作不能做。

权限控制、风险分级、审批流、审计机制必须由系统显式设计，而不能交给 Agent 自发完成。

### 3.3 不天然解决输出稳定性问题

即使 Agent 能完成任务，输出结果仍然可能不稳定、不结构化、不可直接接入系统。

因此，Agent 仍然需要依赖结构化输出约束、参数 Schema、结果校验和错误处理机制。

### 3.4 不意味着越“聪明”越好

企业场景下，Agent 的核心不是追求无限自治，而是追求可控、可解释、可恢复、可审计的任务执行能力。

## 4. Agent 的核心组成

一个工程化 Agent 通常由以下几个部分构成。

### 4.1 目标理解

把用户自然语言请求转换成明确任务意图。

常见类型包括：

- 问答型
- 检索型
- 生成型
- 查询型
- 操作型
- 流程型

### 4.2 Planner / 决策机制

用于决定当前任务应该如何推进。

常见形式包括：

- 显式计划
- 局部决策
- 分步推理
- Router + Specialist
- Workflow 节点选择

### 4.3 Tool / Action 系统

Agent 必须通过 Tool 与外部世界交互。

典型工具包括：

- 检索工具
- 查询工具
- 校验工具
- 生成工具
- 通知工具
- 运维工具

### 4.4 Observation / 结果观察

每次工具调用或步骤执行后，Agent 都需要读取结果并决定下一步。

### 4.5 状态管理

复杂 Agent 需要记录：

- 当前任务状态
- 已执行步骤
- 中间结果
- 错误原因
- 重试次数
- 是否需要人工确认

### 4.6 Memory

Agent 在执行过程中，通常需要维护不同层级的记忆，例如：

- 会话记忆
- 任务记忆
- 长期知识记忆

### 4.7 终止条件

一个健壮的 Agent 必须知道什么时候结束，而不是无限循环。

终止条件通常包括：

- 目标已完成
- 达到最大步数
- 出现不可恢复错误
- 命中风险边界
- 转人工处理

## 5. Agent 的基本工作方式

一个典型的 Agent 工作循环可以概括为：

理解目标 -> 制定下一步 -> 调用工具或检索 -> 观察结果 -> 更新状态 -> 决定是否继续 -> 输出结果或转人工

这说明 Agent 本质上是一种“任务驱动的循环式系统”，而不是一次性文本生成。

## 6. Agent 的常见类型

根据控制方式和复杂度不同，Agent 可以分成几类。

### 6.1 Workflow Agent

最推荐作为企业早期落地形态。

特点：

- 流程边界清晰
- 步骤可观测
- 适合插入人工确认
- 更稳、更可控

适用于：

- 配置校验
- 问题诊断
- 受控草稿生成
- 多步骤查询流程

### 6.2 Tool-based Agent

允许模型根据任务选择工具，但仍在受限边界内执行。

特点：

- 灵活性更高
- 对 Tool 协议和状态管理要求更高
- 更依赖治理能力

适用于：

- 多源查询
- 诊断分析
- 执行前决策辅助

### 6.3 Multi-Agent

由多个 Agent 承担不同角色并协作完成复杂任务。

特点：

- 适合复杂任务拆分
- 架构复杂度显著提高
- 沟通成本、调试成本、治理成本更高

适用于：

- 足够复杂且确实存在明确角色分工的任务

企业中不建议过早进入这一阶段。

### 6.4 Human-in-the-loop Agent

在关键节点引入人工确认、人工审批或人工接管。

特点：

- 更适合高风险场景
- 强调人机协同而不是完全自动化

## 7. Agent 与 Workflow 的关系

Workflow 和 Agent 不是对立关系。

更准确地说：

- Workflow 负责定义边界、节点和执行框架
- Agent 负责在框架内做局部判断和任务推进

在企业场景中，最实用的方式通常不是“完全自由 Agent”，而是“Workflow 承载流程，Agent 承担局部决策”。

## 8. Agent 与 Tool 的关系

没有 Tool，Agent 很难真正做事。

Tool 是 Agent 的“手脚”，负责访问外部系统和执行具体动作。

Agent 与 Tool 的关系可以理解为：

- Agent 决定何时调用什么工具
- Tool 负责执行动作并返回结果
- 系统负责对 Tool 调用做校验、权限控制、审计和风险管理

## 9. Agent 与 RAG 的关系

Agent 和 RAG 经常被一起提及，但它们解决的问题不同。

### 9.1 RAG 解决“知道什么”

RAG 负责把外部知识引入模型上下文，提升认知与事实依据。

### 9.2 Agent 解决“接下来怎么做”

Agent 负责围绕目标组织步骤、调用工具、推进任务闭环。

### 9.3 两者通常协同工作

在实际系统中，常见形态是：

- Agent 先用 RAG 查规范或知识
- 再依据检索结果调用工具
- 然后继续决策下一步

因此更合理的理解是：

RAG 是 Agent 的知识增强模块，Agent 是 RAG 的执行扩展模块。

## 10. Agent 的常见误区

### 10.1 把 Agent 理解为“会自己思考的模型”

工程上更准确的理解应是：Agent 是一个围绕任务进行状态推进、工具调用和结果观察的系统，而不是单纯的模型人格化表达。

### 10.2 把 Agent 等同于循环调用模型

如果没有显式状态、工具协议、错误恢复和终止条件，简单循环并不等于可用 Agent。

### 10.3 过早追求完全自治

在企业场景中，过早追求开放式自治通常会带来：

- 风险不可控
- 行为不可预测
- 调试困难
- 成本失控
- 权限问题突出

### 10.4 认为 Agent 可以替代治理

Agent 越强，越需要治理，而不是越不需要治理。

### 10.5 认为 Agent 一定比 Workflow 高级

企业场景中，Workflow 往往比开放式 Agent 更稳、更实用。不是越自由就越先进，而是越适合业务边界越好。

## 11. 企业里真正需要的 Agent 能力

企业内部最需要的 Agent，一般不是“无所不能”的 Agent，而是具备以下特点的 Agent：

- 任务边界清晰
- 工具边界清晰
- 权限边界清晰
- 执行轨迹可回放
- 高风险动作可审批
- 错误可恢复
- 成本可追踪

换句话说，企业真正需要的是“可控 Agent”，而不是“最自由 Agent”。

## 12. 面向低代码平台如何理解 Agent

在低代码平台场景中，Agent 可以重点服务于以下任务：

- 根据需求生成页面草稿
- 根据错误日志和协议诊断配置问题
- 根据目标生成逻辑编排草稿
- 根据上下文推荐组件和页面模板
- 在受控条件下执行查询、校验和低风险操作

此时，Agent 的关键不是聊天能力，而是：

- 是否理解低代码领域对象
- 是否能调用平台内工具
- 是否能保持结构化结果
- 是否能被规则和权限约束

因此，低代码 Agent 更接近“平台智能助手”或“平台任务协作器”，而不是泛化聊天机器人。

## 13. 一句话总结

Agent 的本质，是围绕目标持续决策、调用工具、观察结果并推进任务闭环的执行型 AI 系统；它解决的是“模型如何在边界内做事”的问题，而不是单纯“模型如何回答”的问题。
