# LLM

> 状态：第一版草案

## 1. 定义

LLM，Large Language Model，通常译为“大语言模型”或“大模型”。

从工程视角看，LLM 是一类基于大规模参数、大规模数据和大规模计算训练出来的通用序列建模系统，核心任务通常是根据已有上下文预测下一个 token，并在此基础上表现出理解、生成、归纳、改写、推理、规划、代码生成、工具使用等能力。

如果只保留一句最核心的话：

LLM 的本质，是一个在海量数据上训练出来的通用语言与符号处理引擎，而不是单一任务模型。

## 2. 为什么 LLM 是 AI 应用的核心

在当前 AI 应用体系中，LLM 之所以成为核心，不是因为它天然知道一切，而是因为它第一次把很多原本彼此分离的能力统一到了同一种接口之下：自然语言输入，自然语言或结构化输出。

它带来的关键变化包括：

- 把问答、摘要、分类、抽取、翻译、改写等能力统一到同一种模型范式中。
- 把代码、表格、文档、图片、音频等多种信息逐步纳入统一处理框架。
- 把原先需要大量任务专用模型的能力，转化为“预训练 + 提示 + 工具 + 后训练”的通用系统能力。
- 把软件系统的人机交互入口，从菜单、表单、按钮，扩展为自然语言和多模态交互。

因此，在企业架构里，LLM 不是“一个功能点”，而是 AI 应用系统的认知内核。

## 3. LLM 解决什么问题

### 3.1 通用语言理解与生成问题

LLM 擅长处理自然语言相关任务，例如：

- 问答
- 总结
- 改写
- 分类
- 信息抽取
- 文本生成
- 翻译

### 3.2 知识压缩与迁移问题

LLM 将大量世界知识、语言模式、代码模式和通用推理模式压缩进统一参数空间中，使模型能够在不同任务间迁移能力。

### 3.3 通用接口问题

LLM 最大的工程意义之一，是把“复杂能力”统一封装成一个几乎通用的调用接口。

对应用系统来说，这种统一接口比单点能力更重要，因为它让 AI 更容易嵌入原有系统。

### 3.4 弱结构任务处理问题

很多业务问题很难提前写死规则，例如：

- 用户意图识别
- 文档理解
- 模糊需求转结构化结果
- 多源信息总结
- 复杂文本与代码解释

LLM 对这类弱结构任务具有明显优势。

## 4. LLM 不解决什么问题

### 4.1 不天然具备最新知识

LLM 的知识受训练数据截止时间限制，因此不能天然掌握最新业务信息、最新企业知识和实时状态。

这也是为什么企业系统需要 RAG、检索、工具调用和外部数据接入。

### 4.2 不天然具备真实执行能力

LLM 可以生成操作建议，但它本身不能直接读取企业数据库、执行 API、发布页面或修改系统配置。

真正的执行需要 Tool、Workflow 和 Agent Runtime。

### 4.3 不天然可靠

LLM 能力强，但并不意味着它天然稳定、无幻觉、无越权、可直接进入生产环境。

企业使用时必须补上：

- 结构化输出
- Guardrails
- 权限控制
- 审计
- 评测与观测

### 4.4 不等于“智能系统的全部”

LLM 是认知内核，但一个真实 AI 应用还需要：

- 知识工程
- 检索增强
- Tool 系统
- 工作流编排
- 状态管理
- 治理与安全

## 5. LLM 的基本工作原理

### 5.1 核心训练目标

大多数现代 LLM 的基础目标仍然是自回归预测，即根据上下文预测下一个 token。

虽然目标形式看起来简单，但在足够大的模型规模、数据规模和训练计算下，这种目标会涌现出远超简单补全的能力。

### 5.2 Transformer 是主流基础架构

现代 LLM 的主流基础架构仍然是 Transformer。2017 年《Attention Is All You Need》提出 Transformer，奠定了后续大模型发展的主干路线。

这一点意味着：

- 注意力机制仍是主流核心
- 长上下文、并行训练、多模态扩展都建立在这一主干上
- 后续很多创新，本质是对 Transformer 的扩展、稀疏化、稳定化或工程化

### 5.3 预训练 + 后训练是主流范式

当前行业主流大模型通常包括两大阶段：

1. 预训练
基于超大规模文本、代码和多模态数据学习通用表示与生成能力。

2. 后训练
通过指令微调、偏好学习、强化学习、工具使用训练、安全训练等方式，把通用模型变成更可用的助手模型、推理模型或代理模型。

## 6. LLM 的核心能力结构

从能力结构上，可以把 LLM 理解成六类能力的组合。

### 6.1 语言能力

包括：

- 理解语义
- 生成文本
- 改写与压缩
- 多轮对话
- 多语言表达

### 6.2 知识能力

包括：

- 通用世界知识
- 代码知识
- 领域模式知识
- 文档风格与结构知识

### 6.3 推理能力

包括：

- 数学推理
- 代码推理
- 多步分析
- 条件判断
- 假设比较

但需要注意，推理能力并不是“纯粹逻辑引擎”，而是强依赖训练、后训练和推理时计算资源的综合结果。

### 6.4 规划能力

包括：

- 任务拆解
- 路径选择
- 多步骤生成
- 工具使用决策

### 6.5 工具使用能力

随着 Tool Calling、函数调用和代理训练的发展，LLM 越来越多地表现出：

- 识别何时需要调用工具
- 生成结构化参数
- 基于工具结果继续推理

### 6.6 多模态能力

最新一代模型越来越强调：

- 文本
- 图像
- 音频
- 视频
- 代码仓库
- 文件

在同一模型或同一系统中的统一处理能力。

## 7. 当前行业前沿趋势

结合 2024 到 2025 年的主流公开研究和官方技术报告，LLM 已经从“更大的文本模型”明显演进到“推理 + 多模态 + 长上下文 + 工具使用 + Agent 适配”的新阶段。

### 7.1 趋势一：推理型模型成为主线

近一阶段最显著的变化之一，是“reasoning model”从边缘能力转为主线能力。

OpenAI 在 2024 年 9 月发布的 o1 研究中明确提出，模型性能会随着训练时强化学习计算和测试时思考时间增加而持续提升；到 2025 年 4 月，OpenAI 在 o3 / o4-mini 中进一步强调了“训练模型不仅会使用工具，而且会推理何时使用工具”。

这意味着前沿 LLM 已不再只追求“直接给答案”，而是越来越强调：

- 更长的推理链
- 更高的 test-time compute
- 更强的复杂问题求解能力
- 更强的工具协同决策能力

从架构认知上，这代表“预训练规模”之外，推理时计算和强化学习后训练已经成为新的核心变量。

### 7.2 趋势二：长上下文从加分项变成基础能力

长上下文已经从 2023 年的差异化卖点，逐步变成 2024-2025 年前沿模型的重要基础能力。

Gemini 1.5 在 2024 年 3 月技术报告中强调，模型在长上下文检索任务中在至少 1000 万 tokens 范围内仍可保持极高检索能力；Gemini 2.5 在 2025 年报告中继续把 1M+ tokens 长上下文与工具使用、推理能力并列为下一代 agentic system 的核心基础。

这带来的直接变化是：

- 更多任务可以直接把大规模材料放入上下文
- 代码仓库级理解成为可能
- 视频、音频、长文档的统一处理能力增强
- 部分复杂 agent scaffold 可以被长上下文简化

但这并不意味着 RAG 会消失。更准确地说，长上下文提高了系统上限，但企业场景依然需要 RAG 处理权限、更新、成本和知识组织问题。

### 7.3 趋势三：多模态从“附加能力”走向“原生能力” 

GPT-4o 在 2024 年系统卡中把文本、图像、音频、视频输入统一到同一个模型框架中；Gemini 2.5 则进一步把“原生多模态 + 长上下文 + tool use + thinking”放在同一代模型定义里。

这意味着 LLM 的边界正在变化：

- LLM 不再只是文本模型
- 模型开始处理屏幕、图像、图表、音频、视频等复杂输入
- 多模态理解正在直接影响 Agent、computer use、代码辅助和企业办公场景

对企业应用而言，这种变化意味着未来很多“文档理解”问题会进一步扩展成“文档 + 图片 + 表格 + 录音 + 截图”的统一理解问题。

### 7.4 趋势四：工具使用从接口能力走向原生能力

过去 Tool Calling 更多被视作 API 接口层能力；但近一阶段，前沿模型越来越把 tool use 视作模型能力的一部分。

OpenAI 在 2025 年 4 月强调 o3 / o4-mini 能 agentically 使用 ChatGPT 内的工具；Gemini 2.5 技术报告也把 native tool use 作为模型家族的关键特征之一。

这代表行业关注点正在从：

- 模型能不能调用工具

转向：

- 模型能不能判断何时调用工具
- 模型能不能跨工具协调
- 模型能不能在长任务中基于反馈调整工具使用路径

这直接推动了 Agent 系统从“框架拼接”走向“模型能力 + 系统编排”协同。

### 7.5 趋势五：MoE 与效率工程成为主流路线

当前前沿模型越来越重视“能力/成本”而不只是参数总量。

Meta 的 Llama 3 技术报告仍以大规模 dense Transformer 为代表；而 Gemini 2.5 和 DeepSeek-V3 则公开强调 sparse MoE 路线。DeepSeek-V3 在 2024 年 12 月技术报告中提出 671B 总参数、每 token 激活 37B 参数，并结合 MLA、MoE 和训练效率设计来提升性价比。

这说明行业正在形成一个更明确的判断：

- 大模型竞争不再只是“谁参数更多”
- 更重要的是每单位成本下的能力
- 稀疏激活、KV cache 压缩、训练稳定性、推理效率、蒸馏和模型分层将越来越重要

### 7.6 趋势六：后训练的重要性持续上升

越来越多公开资料表明，前沿模型能力不再只是由预训练决定。

Gemini 2.5 报告明确强调了更强的 RL、可验证奖励、模型辅助奖励和更复杂环境中的多步 tool use 学习；OpenAI o1 / o3 系列则直接把“会思考更久”“会在工具间决策”建立在强化学习扩展上。

这意味着：

- 预训练决定基础上限
- 后训练决定助手能力、推理能力、工具能力和安全表现
- 企业实际使用效果越来越依赖 post-training 而不只是 base model

### 7.7 趋势七：Agent 友好性正在成为模型设计目标

Anthropic 在 2024 年 12 月《Building Effective AI Agents》中明确区分了 workflow 和 agent，并提出生产落地中最成功的系统通常是简单、可组合的模式，而不是过度复杂的框架。

同时，Gemini 2.5、OpenAI o3/o4-mini、Claude 3.5 系列都在不同程度上强调：

- 推理能力
- tool use
- 长上下文
- coding
- computer use / agentic task

这些信号说明，大模型行业正在逐步把“Agent 适配能力”视为模型竞争力的一部分。

## 8. 当前 LLM 的主流技术范式

结合当前行业研究，可以把现代 LLM 的技术范式概括为以下组合：

- Transformer 主干
- 大规模预训练
- 指令微调
- 偏好对齐与强化学习
- 长上下文扩展
- 多模态训练
- Tool use / function calling 适配
- 稀疏化与效率优化
- 安全训练与系统级防护

因此，今天讨论 LLM，已经不能只停留在“参数量 + 上下文长度”的老式视角，而应把它视为一个由模型结构、训练配方、后训练策略、推理策略和系统接口共同决定的综合系统。

## 9. LLM 的核心边界

尽管前沿能力快速提升，LLM 在企业应用中仍然存在明确边界。

### 9.1 知识边界

模型无法天然掌握企业实时知识，必须依赖 RAG、数据库、搜索、工具和权限化知识底座。

### 9.2 事实边界

模型可能生成看似合理但不可靠的内容，因此需要引用、校验和评测。

### 9.3 执行边界

模型不应直接被等同为执行系统；执行必须通过 Tool、Workflow、审批和审计进行约束。

### 9.4 成本边界

更强模型往往意味着更高延迟和更高成本，因此企业中通常需要模型分层与路由策略。

### 9.5 安全边界

长上下文、多模态和工具调用同时提升能力，也同时放大越权、注入、误执行和信息泄露风险。

## 10. 企业架构中应如何理解 LLM

在企业应用体系里，LLM 更适合作为一个“认知中枢”，而不是孤立功能。

建议从以下方式理解：

- LLM 是模型接入层的核心对象之一
- LLM 与 Embedding、Rerank、Vision 等能力共同构成模型底座
- LLM 需要通过 RAG 获得外部知识
- LLM 需要通过 Tool 获得真实执行能力
- LLM 需要通过 Workflow / Agent Runtime 进入可控任务闭环
- LLM 需要通过 Guardrails、Evaluation、Observability 被治理

也就是说：

LLM 是核心，但绝不是全部。

## 11. 面向低代码平台如何理解 LLM

在低代码平台场景中，LLM 不应只被理解为一个“聊天接口”，而应被理解为平台智能化的通用认知引擎。

它可以服务于：

- 组件知识理解
- 页面 Schema 生成
- 配置诊断
- 逻辑编排建议
- 平台问答
- 运维辅助分析

但它要真正产生业务价值，必须和以下内容结合：

- 低代码领域知识
- 结构化输出协议
- 校验工具
- 检索增强
- 权限体系
- 发布与运维流程

因此，对低代码平台而言，LLM 的角色更接近“领域智能内核”，而不是独立产品形态。

## 12. 与其他概念的关系

### 12.1 与 Prompt 的关系

Prompt 是调动 LLM 能力的主要接口之一，但 Prompt 不等于 LLM 本身。

### 12.2 与 Embedding 的关系

Embedding 模型负责向量表示与语义检索，通常与 LLM 配合构成 RAG 体系。

### 12.3 与 RAG 的关系

RAG 用于补足 LLM 的知识边界，是企业应用中最常见的认知增强手段之一。

### 12.4 与 Agent 的关系

Agent 往往以 LLM 作为决策核心，但额外增加了工具、状态、规划和执行闭环。

### 12.5 与 Guardrails 的关系

Guardrails 负责约束 LLM 输入、输出和行为边界，使其可以进入真实业务环境。

## 13. 常见误区

### 13.1 把 LLM 理解成“更大的聊天机器人”

聊天只是表现形式之一。LLM 的真正价值在于它是一个通用认知接口。

### 13.2 把参数量当成唯一指标

当下行业已经清楚表明：后训练、推理时计算、工具使用能力、上下文能力和成本效率同样关键。

### 13.3 认为模型越强就越不需要系统设计

模型越强，往往越需要好的知识底座、工具协议、治理机制和应用编排。

### 13.4 认为长上下文会完全替代 RAG

长上下文会改变部分系统设计，但企业场景里的知识更新、权限过滤、成本控制仍决定 RAG 长期存在。

### 13.5 认为 Agent 能力只是框架问题

当前前沿模型已经显示，tool use、thinking、long context 和 multimodality 正在逐步成为模型原生能力的一部分。

## 14. 一句话总结

LLM 的本质，是基于大规模预训练和后训练构建出来的通用认知引擎；而 2024-2025 年行业前沿的核心方向，已经从“做更大的文本模型”转向“更强推理、更长上下文、更原生多模态、更稳定工具使用、更高性价比和更适合 agentic system 的模型体系”。
