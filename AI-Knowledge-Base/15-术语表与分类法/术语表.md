# 术语表

## 1. 目的
术语表用于统一整个 AI 知识库中的核心概念、缩写、对象名称和常用表达，避免同一个词在不同文档中含义漂移，也避免相近概念被混用。

对企业级 AI 知识库来说，术语统一非常重要。因为后续不只是人要阅读这些文档，RAG、Agent、检索、评测、规则引擎和 Prompt 模板也都会依赖统一术语。如果术语体系不稳定，后续知识检索、场景建模和工程实现都会变得混乱。

## 2. 使用原则
术语表建议遵循以下原则：
- 同一个概念只保留一个主名称
- 可以记录别名，但要明确推荐用词
- 定义应简洁，优先强调边界而不是百科式解释
- 术语之间尽量建立关系，避免孤立定义
- 与平台领域对象有关的术语要和通用 AI 术语分开管理

## 3. 核心术语

### LLM
大语言模型。指经过大规模预训练和后训练后，可用于理解、生成、推理和工具调用的通用模型能力主体。

### Embedding
向量表示。指把文本、对象或多模态内容转换成可用于相似度计算和检索的向量表示。

### Prompt
提示词或提示上下文。指用来控制模型行为、上下文、输出格式和任务边界的输入编排方式。

### RAG
检索增强生成。指先检索外部知识，再将结果作为上下文或约束输入模型完成回答、生成或执行增强的系统模式。

### Query Rewrite
查询改写。指对用户原始问题进行重述、补全或标准化，以提高后续召回质量。

### Query Expansion
查询扩展。指围绕原始查询补充同义词、相关表达或子查询，以扩大召回覆盖范围。

### Recall
召回。指从候选知识集合中取回可能相关内容的过程。

### Hybrid Search
混合检索。指将 dense retrieval、sparse retrieval、metadata filter 等多种检索方式组合使用。

### Rerank
重排。指对初始召回结果做二次排序，以提升最终相关性。

### Context Builder
上下文构造器。指负责对召回结果做去噪、压缩、聚合和结构化拼装的模块。

### Citation
引用。指回答或生成结果所基于的来源证据及其指向方式。

### Agent
智能体。指围绕目标持续决策、调用工具、观察结果并推进任务闭环的系统。

### Tool
工具。指被模型或 Agent 调用的标准化动作对象，通常具备名称、描述、参数 schema、权限与风险信息。

### Action
动作。指具有真实副作用的执行对象，如发布、发送、写入、删除、通知等。Action 往往是 Tool 的高风险子集。

### Workflow
工作流。指由显式步骤、节点、条件和控制关系组成的受控执行流程。

### Planner
规划器。指将目标拆解成步骤、子任务或执行路径的决策模块。

### Memory
记忆。指系统在多轮交互或长期任务中保留的状态、偏好、历史和可复用上下文。

### Guardrails
护栏或约束。指围绕输入、输出、工具调用、权限和行为边界建立的控制机制。

### Evaluation
评测。指用于衡量 AI 系统效果、稳定性和正确性的测试与分析体系。

### Observability
可观测性。指通过日志、指标、Trace 和事件追踪理解系统运行过程的能力。

### Trace
链路追踪。指对一次请求或任务在多个模块中的执行过程做结构化记录。

### Human-in-the-Loop
人在回路。指在任务执行过程中引入人工审批、确认、补充或接管的机制。

## 4. 平台领域术语

### 页面 Schema
低代码页面的结构化定义，描述页面组件树、配置、数据绑定、交互和相关元信息。

### 组件物料
平台可管理、可配置、可复用的组件能力对象，通常包含 props、slots、setter、事件和场景知识。

### Setter
设计态配置器，用于在编辑器中帮助用户配置组件属性、结构和行为。

### Slot
插槽。指组件中可承载内容或子结构的命名位置和约束协议。

### 数据源
页面或逻辑运行过程中访问外部或内部数据的标准化对象，通常包含请求、响应、绑定和运行策略。

### 逻辑编排 DAG
用于描述页面逻辑、事件响应、数据流和动作执行关系的有向无环图模型。

### 发布流程
指页面从草稿态进入构建、审批、部署、生效和回滚的受控链路。

### 多租户模型
平台在同一套基础设施上服务多个租户时，对对象隔离、共享和作用域边界的建模方式。

## 5. 推荐写法说明
知识库后续文档中建议优先使用以下写法：
- 使用 `RAG`，少用“向量检索系统”泛指整个检索增强体系
- 使用 `Tool` 表示标准化动作对象，使用 `Action` 表示高副作用执行对象
- 使用 `页面 Schema` 而不是模糊称呼“页面 JSON”
- 使用 `组件物料` 而不是笼统称为“组件配置”
- 使用 `逻辑编排 DAG` 而不是简单称呼“流程图”

## 6. 维护建议
术语表建议持续维护，尤其在以下情况下要更新：
- 新引入新的平台能力类型
- 某个词在多个文档中被混用
- 需要补充领域对象的正式命名
- 外部行业术语进入内部体系时需要做统一映射

## 7. 小结
术语表是整个知识库的语言控制面。它的价值不在于定义有多长，而在于能否让不同模块、不同文档和不同系统使用同一套语义边界。术语越统一，后续知识沉淀、检索增强和工程实现越稳定。
