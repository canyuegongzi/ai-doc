# Tool风险等级

## 1. 目的
Tool 风险等级用于统一定义工具和动作的风险分类口径，帮助平台在 Tool 暴露、Agent 执行、审批控制、审计记录和自动化边界上采用一致规则。

如果没有统一风险等级，平台很容易出现：
- 低风险工具被过度限制
- 高风险工具被误暴露给模型
- Agent 自动执行边界模糊
- 审批策略和审计策略混乱

## 2. 分级原则
Tool 风险等级建议至少综合以下维度判断：
- 是否有真实副作用
- 副作用是否可回滚
- 影响范围大小
- 是否涉及敏感数据
- 是否跨租户或跨环境
- 是否容易误用
- 是否会触发外部系统动作

## 3. Low：低风险工具
### 定义
主要用于读取、查询、分析或模拟，不直接改变系统状态。

### 典型示例
- 查询页面详情
- 查询组件定义
- 检索知识库
- 获取发布记录
- 读取日志摘要

### 控制要求
- 正常权限校验
- 常规审计
- 一般不需要审批

## 4. Medium：中风险工具
### 定义
会生成建议、草稿、补丁或可选变更，但通常不会立即对生产系统产生不可逆影响。

### 典型示例
- 生成 Schema 草稿
- 生成配置修复 patch
- 创建待审核发布任务
- 生成逻辑图草稿

### 控制要求
- 需要更明确的输入校验
- 通常建议进入人工确认或 review
- 应保留 diff、原因和引用依据

## 5. High：高风险工具
### 定义
会直接对系统、数据、用户或外部系统产生真实副作用，且误执行成本较高。

### 典型示例
- 发布页面
- 删除配置
- 修改生产数据
- 发送通知/邮件
- 触发外部系统写操作
- 回滚线上版本

### 控制要求
- 强权限校验
- 通常需要审批或二次确认
- 必须完整审计
- 最好支持 dry-run、receipt 和 rollback 信息

## 6. Critical：关键风险工具
### 定义
会对平台级稳定性、安全性或大范围生产环境造成重大影响的工具。

### 典型示例
- 跨租户批量修改
- 大范围删除或重置
- 修改全局权限策略
- 全平台配置中心写入
- 强制切换关键运行环境

### 控制要求
- 默认不向模型直接暴露
- 只能通过严格的人审或专用运维流程执行
- 需要最高等级审计和多重确认

## 7. 风险等级不是静态常量
同一个 Tool 的风险等级可能随着上下文变化而变化，例如：
- 在测试环境是 Medium，在生产环境是 High
- 对单页面操作是 Medium，对批量操作是 High
- 读取普通配置是 Low，读取敏感数据是 High

因此，风险等级最好是“基础等级 + 运行时提升”的组合。

## 8. 风险等级的使用位置
风险等级应贯穿以下环节：
- Tool Catalog 展示
- Tool Discovery 裁剪
- Planner 和 Runtime 决策
- Approval 流程
- Audit 和 Replay
- Evals 和误操作分析

## 9. 与 Agent 分级的关系
Tool 风险等级和 Agent 类型分级并不相同，但两者强相关：
- 低等级 Agent 更适合只接触 Low / Medium 工具
- 高自由度 Agent 更需要严格限制 High / Critical 工具
- 多 Agent 协同场景下，应更谨慎暴露高风险动作

## 10. 推荐维护方式
建议每个 Tool 至少具备：
- baseRiskLevel
- riskFactors
- requiredApproval
- allowedEnvironments
- idempotency
- rollbackHint

## 11. 小结
Tool 风险等级的价值，在于把“是否能让 AI 调这个工具”从经验判断变成可执行规则。只有风险口径统一，权限、审批、审计和自动化边界才会真正稳定。
