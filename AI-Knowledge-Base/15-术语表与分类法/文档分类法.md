# 文档分类法

## 1. 目的
文档分类法用于定义知识库中的文档应该如何分类、命名和归档，确保后续内容新增时不会逐步失控，也方便后续做检索、标签、结构化抽取和自动化加工。

对 AI 知识库来说，分类法不仅影响人类阅读，也影响：
- RAG 的索引构建
- Metadata 的标准化
- 知识对象抽取
- Prompt 中的知识装配
- 评测集和案例库的组织方式

## 2. 分类原则
知识库文档分类建议遵循以下原则：
- 优先按知识对象和作用分类，而不是按写作时间分类
- 同一文档尽量只属于一个主分类
- 同类文档结构尽量一致，便于后续模板化维护
- 分类既要服务当前阅读，也要服务未来机器消费

## 3. 一级分类说明
当前知识库建议使用以下一级分类：
- 总览与路线
- 基础概念
- 总体架构
- 模型接入层
- 知识工程
- RAG 专题
- RAG 场景化
- Agent 专题
- Tool 与 Action 体系
- 安全与治理
- 评测与观测
- 工程实现
- 领域知识
- 业务场景方案
- 案例与复盘
- 术语表与分类法
- 模板
- 待整理与输入区

这些分类实际上对应了从“认知”到“架构”到“工程”再到“场景与经验”的完整链路。

## 4. 文档类型分类
从知识对象角度，可以把文档进一步分成以下类型：
- Concept：概念类文档
- Architecture：架构类文档
- Engineering：工程设计类文档
- Domain：领域对象类文档
- Scenario：业务场景类文档
- Case：案例复盘类文档
- Taxonomy：分类法和术语类文档
- Template：写作模板类文档
- Backlog：待整理输入类文档

## 5. 各类文档的典型特征

### Concept
回答“是什么、解决什么、不解决什么、和谁相关”。
典型目录：`01-基础概念`

### Architecture
回答“整体怎么设计、模块如何协同、边界如何划分”。
典型目录：`00-总览与路线`、`02-总体架构`

### Engineering
回答“系统如何落地、接口怎么定义、状态怎么流转”。
典型目录：`03-模型接入层`、`11-工程实现`

### Domain
回答“在具体业务平台里，这个对象是什么样的”。
典型目录：`12-领域知识-低代码平台`

### Scenario
回答“一个具体业务场景如何应用这些能力”。
典型目录：`06-RAG场景化`、`13-业务场景方案`

### Case
回答“真实实验中做了什么、出了什么问题、得出了什么结论”。
典型目录：`14-案例与复盘`

### Taxonomy
回答“怎么统一术语、怎么统一分类口径”。
典型目录：`15-术语表与分类法`

## 6. 推荐 Metadata 字段
为了后续更方便做检索和结构化抽取，建议每篇文档至少具备以下 metadata：
- title
- category
- docType
- topic
- tags
- relatedDocs
- status
- updatedAt
- owner

例如：

```yaml
---
title: Schema生成增强
category: 业务场景方案
docType: Scenario
topic:
  - 低代码
  - Schema生成
  - RAG
tags:
  - 页面生成
  - 结构化输出
status: draft
---
```

## 7. 分类法与命名的关系
为了保持统一，命名建议遵循：
- 文件名直接使用主概念或主场景名称
- 避免使用过于口语化或模糊的标题
- 同一层级尽量保持粒度一致
- 专题类文档用“主题名”命名
- 场景类文档用“场景名”命名
- 案例类文档用“对象 + 实验/复盘”命名

## 8. 对后续 RAG 的价值
文档分类法对后续 RAG 有几个直接价值：
- 可以按 docType 过滤召回结果
- 可以在不同任务中优先使用不同类型文档
- 可以让 Agent 在规划时识别“概念文档”和“案例文档”的不同价值
- 可以把分类本身作为 metadata 参与 rerank 和 context builder

## 9. 常见误区
- 按目录存放了文档，但没有形成真正的文档类型体系
- 同一个文档同时承担概念、架构、案例和路线图多种角色
- 只做人读目录，不做 metadata 设计
- 命名粒度不一致，导致检索与维护困难

## 10. 小结
文档分类法是知识库从“目录树”走向“结构化知识系统”的关键一步。目录决定了文档放在哪里，分类法决定了文档是什么。只有两者都清晰，知识库后续才适合做长期维护、检索增强和自动化加工。
