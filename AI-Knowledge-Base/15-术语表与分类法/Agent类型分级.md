# Agent类型分级

## 1. 目的
Agent 类型分级用于统一知识库中对不同 Agent 形态的称呼、能力边界和治理要求，避免后续把所有带工具调用的系统都笼统称为 Agent。

企业里真正需要的不是“最自由的 Agent”，而是“在不同任务和风险下使用合适等级的 Agent”。因此，分级比空泛讨论是否使用 Agent 更有实际价值。

## 2. 分级原则
Agent 分级建议优先按以下维度判断：
- 自主决策程度
- 工具调用自由度
- 状态与记忆复杂度
- 副作用风险
- 治理与审计要求

## 3. Level 0：Prompt Workflow
### 定义
由固定流程驱动，模型只承担局部生成或分类任务，不做自主规划和工具选择。

### 特点
- 流程完全预定义
- 没有自主工具选择
- 风险最低
- 最容易上线

### 典型场景
- FAQ 问答
- 模板化生成
- 单步结构化输出

## 4. Level 1：Workflow Agent
### 定义
主流程预定义，模型可在少数节点内做局部决策或内容生成。

### 特点
- 骨架受控
- 节点内有智能化能力
- 可观测、可回放
- 非常适合企业第一阶段落地

### 典型场景
- Schema 生成增强
- 配置纠错流程
- 审批摘要与发布分析

## 5. Level 2：Tool-Based Agent
### 定义
模型可以在给定工具集合内自主选择工具、组织调用顺序并推进任务。

### 特点
- 决策自由度更高
- 对工具协议、权限、风险控制要求显著提高
- 需要状态机、审计和 HITL 支撑

### 典型场景
- 运维诊断
- 配置修复建议
- 数据查询与跨对象分析

## 6. Level 3：Planner-Centric Agent
### 定义
具备显式规划能力，能把复杂目标拆成多步子任务，并动态调整执行路径。

### 特点
- 依赖 planner 和 runtime
- 状态更复杂
- 更适合长任务与复杂问题
- 对评测和回放要求更高

### 典型场景
- 复杂逻辑编排辅助
- 多阶段诊断与修复
- 跨系统任务编排

## 7. Level 4：Multi-Agent System
### 定义
多个 Agent 分工协作，每个 Agent 负责不同角色或子问题，由 router、planner 或 coordinator 进行组织。

### 特点
- 架构复杂度最高
- 不一定比单 Agent 更好
- 仅在任务足够复杂、角色边界明确时才有价值

### 典型场景
- 复杂方案生成
- 超长链路分析
- 多角色审查与复核

## 8. 为什么要分级
Agent 类型分级的价值在于：
- 不同场景可以选择不同复杂度的方案
- 治理、审计、权限和审批要求可以按级别匹配
- 评测方法和上线门槛可以分层设计
- 可以避免一上来追求高自由度自治 Agent 的误区

## 9. 分级与治理的关系
随着等级升高，通常会发生这些变化：
- 自由度增加
- 副作用风险增加
- 状态复杂度增加
- 误调用和越权风险增加
- 对 runtime、trace、replay、approval 的依赖增加

因此，Agent 等级越高，治理成本越高。

## 10. 推荐使用建议
对大多数企业平台场景，推荐优先顺序是：
- 先做 Level 0 / Level 1
- 再按实际价值进入 Level 2
- 只有在确有必要时再考虑 Level 3 和 Level 4

## 11. 小结
Agent 类型分级的核心作用，不是给系统贴标签，而是帮助平台明确：当前任务需要多大程度的自治、需要多强的治理，以及是否真的值得引入更复杂的 Agent 形态。分级越清楚，演进路线越稳。
