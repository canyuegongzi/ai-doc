# RAG场景分类

## 1. 目的
RAG 场景分类用于统一知识库中关于 RAG 的业务分类方式，避免后续把所有检索增强场景混成一类，也帮助后续在架构、评测、知识建模和工程实现上做更清晰的区分。

在企业里，RAG 并不是单一产品形态。不同场景对：
- 查询理解
- 知识对象
- 召回方式
- 输出形式
- 治理要求

都有显著差异。因此，对 RAG 做场景分类是必要的。

## 2. 分类原则
RAG 场景分类建议优先从“检索结果被如何使用”来划分，而不是只按数据来源或界面形式划分。因为决定架构差异的，往往不是数据本身，而是检索结果在业务流程中的角色。

## 3. 一级分类
当前知识库建议把 RAG 场景分成四类：
- FAQ 型 RAG
- 搜索推荐型 RAG
- 生成辅助型 RAG
- 执行增强型 RAG

这四类基本覆盖了企业和平台场景中的主流形态。

## 4. FAQ 型 RAG
### 定义
用于回答知识问答类问题，以准确、可引用的回答为主要目标。

### 典型场景
- 制度问答
- 平台使用帮助
- 组件文档问答
- API 文档问答

### 核心特点
- 文档较稳定
- 输出强调正确性和 citation
- 风险相对较低
- 通常是最容易优先落地的一类

## 5. 搜索推荐型 RAG
### 定义
用于帮助用户发现对象、筛选候选和给出轻量推荐，不一定输出长篇答案。

### 典型场景
- 组件推荐
- 页面模板推荐
- API 推荐
- 案例导航

### 核心特点
- 更像“搜索 + 排序 + explanation”
- 排序和 metadata 过滤非常关键
- 输出常常是候选对象列表，而不是完整自然语言答案

## 6. 生成辅助型 RAG
### 定义
用于在生成类任务中提供约束、模板和参考上下文，检索结果不是最终答案，而是生成过程的输入。

### 典型场景
- 页面 Schema 生成
- 表单配置生成
- DSL 生成
- 查询条件或配置片段生成

### 核心特点
- 检索结果更像参考包和约束条件
- 强依赖结构化知识对象
- 生成后往往还需要校验和修复

## 7. 执行增强型 RAG
### 定义
用于在 Agent 或 Workflow 执行前、中、后提供知识增强，帮助系统做更准确的决策和动作。

### 典型场景
- 调工具前查 API 协议
- 发布前查规则
- 配置诊断前查历史案例
- 运维分析前查告警规则或处理手册

### 核心特点
- 与 Tool / Agent 强耦合
- 检索结果直接影响执行路径
- 治理和审计要求更高

## 8. 为什么要区分这四类
区分这四类 RAG 的价值在于：
- 它们的知识对象不同
- query understanding 策略不同
- recall / rerank 策略不同
- 上下文构造方式不同
- 输出结构不同
- 成功指标也不同

如果不区分，后续很容易出现“一套 RAG 打天下”的误区。

## 9. 与评测的关系
不同类型 RAG 的评测重点也不同：
- FAQ 型更看回答正确率、citation 覆盖率
- 搜索推荐型更看候选命中率、点击率、采纳率
- 生成辅助型更看生成结果可用率、错误率、人工修改量
- 执行增强型更看任务成功率、误执行率、证据充分性

## 10. 与知识建模的关系
不同类型 RAG 对知识建模的要求不同：
- FAQ 型更依赖稳定文档和段落切分
- 搜索推荐型更依赖对象标签、分类和排序信号
- 生成辅助型更依赖模板、schema 片段、结构化协议
- 执行增强型更依赖规则、案例、动作边界和流程知识

## 11. 对平台建设的启发
平台在建设 RAG 能力时，不应从“一套通用检索服务”出发，而应考虑：
- 哪些场景属于哪一类
- 是否需要复用同一底层索引但分不同 pipeline
- 哪些场景应先做 FAQ，哪些应直接做生成增强
- 是否需要专门为 Agent 构建执行增强型检索链路

## 12. 小结
RAG 场景分类的核心价值，在于帮助平台从“泛检索能力”走向“按任务建检索系统”。只有把 FAQ、推荐、生成和执行四类场景分清楚，后续的知识工程、链路设计、治理和评测才会真正有针对性。
