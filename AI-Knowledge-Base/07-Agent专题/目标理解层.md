# 目标理解层

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 Agent 系统中的目标理解层（goal understanding layer），重点回答：

- Agent 如何从用户自然语言、系统事件或任务输入中理解“真正目标”
- 目标理解与 query understanding、任务模型、planner 之间是什么关系
- 为什么目标理解层在现代 agentic system 中是正式架构层，而不是 prompt 前处理细节

## 2. 为什么目标理解层重要

很多 agent 失败，并不是因为模型不会推理，而是因为系统从一开始就误解了目标。

例如：

- 用户只是想查规范，系统却进入执行链路
- 用户想要草稿，系统却给了最终答案
- 用户想诊断问题，系统却当成一般 FAQ 问答处理

因此，目标理解层的作用是：

在进入检索、规划和执行之前，先把“想做什么”搞清楚。

## 3. 近两年的关键变化

### 3.1 agent design foundations 正在前移到目标识别

OpenAI 的 agent guide 明确强调：选择合适 use case 与 agent 形态是第一步，而不是直接进入复杂 orchestration。这意味着“目标理解与任务路由”已经成为 agent 系统最早的控制点之一。

### 3.2 routing workflow 在企业 agent 中越来越核心

Anthropic 把 routing 明确列为基础 workflow 之一，说明在生产系统里，很多复杂性首先是通过“先分类、再走不同路径”来降低的。

这本质上就是目标理解层的重要职责。

### 3.3 多模态、多轮上下文和系统事件让目标理解更复杂

现在输入不再只是单轮文本，还可能包括：

- 历史对话
- 工具返回
- 页面上下文
- 截图
- 系统状态
- 任务恢复信息

因此目标理解层越来越像“任务语义解释层”。

## 4. 目标理解层解决什么问题

### 4.1 识别真实目标

判断当前用户真正想要的是：

- 回答
- 推荐
- 生成
- 诊断
- 执行
- 复合任务

### 4.2 识别任务边界

判断：

- 任务是一次性还是多步骤
- 是否需要 retrieval
- 是否需要 tools
- 是否需要人工确认

### 4.3 为任务建模提供结构化输入

目标理解层的输出通常会成为 Agent 任务模型的起点。

## 5. 目标理解层的核心输入

建议至少考虑以下输入来源：

- 当前用户请求
- 会话历史
- 页面 / 资源上下文
- 用户身份与角色
- 当前任务状态（如果是续跑）
- 多模态输入（若有）
- 系统事件 / 工具结果（若由系统触发）

## 6. 目标理解层的核心输出

建议至少输出：

- normalizedGoal
- taskType
- intentType
- requiredCapabilities
- riskLevel
- recommendedFlow
- structuredEntities
- confidence

这会直接影响：

- routing
- retrieval strategy
- tool set selection
- runtime entry point

## 7. 目标理解与 Query Understanding 的区别

两者相关但不相同。

### 7.1 Query Understanding

更偏 retrieval 场景，关注：

- query rewrite
- entity extraction
- query expansion
- recall optimization

### 7.2 Goal Understanding

更偏任务场景，关注：

- 想做什么
- 需要什么能力
- 进入哪条任务链路
- 是否涉及执行或高风险动作

可以理解为：

query understanding 更偏检索理解，goal understanding 更偏任务理解。

## 8. 目标理解与 Planner 的区别

- 目标理解层决定“任务是什么”
- Planner 决定“下一步做什么”

如果目标理解错了，Planner 再强也会在错误目标下优化。

## 9. 推荐的目标理解流程

### 9.1 输入标准化

先统一当前输入上下文。

### 9.2 intent / task classification

判断当前属于哪种任务类型。

### 9.3 entity extraction

识别关键业务对象。

### 9.4 risk detection

判断是否涉及高风险动作或高敏感知识。

### 9.5 route recommendation

为后续 Runtime 提供建议入口。

## 10. 实现路径建议

### 10.1 规则驱动识别

适用于：

- 显式操作意图
- 高风险动作识别
- 固定场景入口

### 10.2 模型驱动理解

适用于：

- 模糊、多义、复合任务
- 多轮上下文推断

### 10.3 混合式目标理解

企业中更推荐：

- 规则先做硬边界识别
- 模型再做复杂任务语义理解

## 11. 面向低代码平台的目标理解示例

### 11.1 “帮我生成一个采购审批页面”

目标类型：生成型

需要能力：

- retrieval
- structured generation
- validation

### 11.2 “这个字段为什么没显示出来”

目标类型：诊断型

需要能力：

- retrieval
- schema/tool validation
- reasoning

### 11.3 “查一下这个页面最近发布失败原因”

目标类型：查询 + 运维分析型

需要能力：

- tool query
- retrieval of rules / logs
- summary generation

### 11.4 “帮我直接发布这个页面”

目标类型：执行型高风险任务

需要能力：

- tool access
- permission check
- HITL / approval

## 12. 常见误区

### 12.1 认为目标理解只是 prompt 写得好不好

### 12.2 不把风险识别纳入目标理解层

### 12.3 目标理解层输出不结构化

### 12.4 所有任务都走同一个默认 agent loop

## 13. 一句话总结

目标理解层的核心，是在 Retrieval、Planning、Tool Use 和 Runtime 之前，把用户或系统输入转化为结构化、可路由、带风险意识的任务目标表达；近两年的 agent 工程实践也越来越清楚，这一层不是前置小技巧，而是决定整个 agentic system 是否走对路的第一个正式架构层。
