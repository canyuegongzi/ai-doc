# 执行审计与回放

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 Agent 系统中的执行审计与回放能力，重点回答：

- 为什么生产级 Agent 必须具备可审计、可回放能力
- 审计记录和执行回放分别解决什么问题
- 近两年行业为什么越来越强调 trace、replay、durable execution 和 run history

## 2. 什么是执行审计与回放

### 2.1 执行审计

执行审计指的是：

对 Agent 一次完整任务执行过程中发生的关键事件、状态变化、工具调用、人工决策、输出结果和风险动作进行结构化记录，以满足：

- 责任追踪
- 风险审计
- 排障分析
- 合规检查
- 效果评估

### 2.2 执行回放

执行回放指的是：

基于已记录的执行轨迹，在事后重新查看、重建或部分复现一次 Agent 任务执行过程，以支持：

- 故障复盘
- 行为解释
- 人工复核
- 调试优化
- 评测与对比

用一句话概括：

审计回答“发生了什么、是谁批准的、为什么这样做”，回放回答“系统当时是怎样一步步走到这个结果的”。

## 3. 为什么 Agent 必须具备审计与回放

### 3.1 Agent 不是单步模型调用，而是长链路执行系统

一旦系统具备：

- retrieval
- planner
- tool use
- memory
- HITL
- stateful runtime

就意味着问题不再只会出在最终回答上，而可能出在中间任何一步。

如果没有审计与回放：

- 无法判断错误出在哪个节点
- 无法解释为什么调用了某个工具
- 无法还原谁批准了高风险动作
- 无法判断问题来自模型、工具、知识还是运行时

### 3.2 企业需要责任归属和风险复盘

对于发布、配置修改、对外通知、数据写入等动作，企业必须回答：

- 是谁发起的
- 系统依据是什么
- 是否经过人工确认
- 哪一步发生了异常
- 为什么最终执行了该动作

### 3.3 近两年的运行时体系正在把审计能力前置

LangGraph、OpenAI Agents SDK tracing、LangSmith 等都在强调 execution trace；Anthropic 也强调 long-running agents 需要良好的 harness、checkpoint 和可恢复链路。

这说明行业共识已经比较明确：

生产级 Agent 的可追踪性，不是附加功能，而是基础设施。

## 4. 执行审计解决什么问题

### 4.1 风险动作留痕

记录高风险动作的输入、输出、审批与结果。

### 4.2 责任可追溯

明确用户、系统、审批人和工具执行者之间的责任边界。

### 4.3 异常定位

帮助判断故障到底来自：

- planner 决策错误
- retrieval 召回问题
- tool 调用失败
- validation 规则缺失
- human decision 错误

### 4.4 合规与审计检查

支持内部审计、权限审查、数据访问追溯。

## 5. 执行回放解决什么问题

### 5.1 故障复盘

还原任务执行过程，分析错误是如何发生的。

### 5.2 行为解释

帮助产品、研发、业务方理解：

- 为什么系统做了这个决策
- 为什么走了这条路径
- 为什么没有停在某一步

### 5.3 调试优化

将某次失败 run 作为调试样本，用于：

- prompt 优化
- planner 优化
- tool schema 优化
- retrieval 优化
- guardrails 优化

### 5.4 评测与回归测试

高质量运行轨迹本身就可以成为后续 eval dataset 的来源。

## 6. 近两年的关键变化

### 6.1 Trace 正在成为 Agent 平台的标准能力

OpenAI Agents SDK、LangSmith、LangGraph Studio 等都在强化 trace 作为正式能力，而不只是开发阶段日志。

### 6.2 审计对象从“最终输出”扩展到“中间链路”

现代 Agent 系统中需要审计的不只是最终结果，还包括：

- query understanding
- planner outputs
- retrieval evidence
- tool selection
- tool parameters
- validation result
- HITL decisions

### 6.3 回放越来越强调“结构化 replay”而不是“看日志”

简单日志只能帮助读问题，不能真正复原状态机、工具输入输出和状态转移。

更成熟的回放能力通常要求：

- 可查看状态机流转
- 可查看每一步输入输出
- 可跳到具体节点
- 可对比不同 run
- 可基于旧轨迹重跑局部节点

## 7. 应审计哪些对象

建议至少覆盖以下七类对象：

### 7.1 任务元信息

包括：

- runId
- traceId
- taskId
- tenantId
- userId
- appId
- sceneType
- startTime / endTime
- runtimeVersion

### 7.2 目标与输入

包括：

- 原始用户请求
- 结构化任务模型
- planner 初始判断
- 输入上下文摘要

### 7.3 检索与证据

包括：

- rewrite / expansion 结果
- 召回 source
- topK 结果摘要
- rerank 结果
- citation 列表

### 7.4 工具调用

包括：

- tool name
- input params
- sanitized output
- latency
- retry count
- error code
- risk level

### 7.5 状态转换

包括：

- fromState
- toState
- trigger
- transitionReason
- timestamp

### 7.6 人工介入

包括：

- review type
- reviewer identity
- decision
- comment
- approved constraints
- decision time

### 7.7 最终输出与结果

包括：

- final answer / final action
- validation result
- success / failed / escalate
- failure reason

## 8. 审计记录应满足什么原则

### 8.1 结构化优先

不要只存自然语言日志，应优先结构化事件。

### 8.2 时间顺序明确

每个关键事件必须可按时间和状态顺序重建。

### 8.3 关键信息去敏后可追踪

审计记录应可用于排障，但要考虑：

- 敏感字段脱敏
- 密钥与凭证隐藏
- PII 最小化保存

### 8.4 业务可解释而不是只对研发可读

审计视图不应只有技术日志，也应能让产品、运营或审计人员理解。

## 9. 回放系统应具备哪些能力

### 9.1 轨迹浏览

按步骤查看一次 run 的完整执行链路。

### 9.2 节点详情查看

展开查看某一步的：

- 输入
- 输出
- 状态
- 证据
- tool payload
- validation result

### 9.3 状态机视图

用状态图或步骤图呈现任务流转。

### 9.4 差异对比

对比两次 run 的计划、证据、工具路径、输出差异。

### 9.5 局部重跑

在安全条件下，对局部节点进行 replay 或重新执行。

注意：

回放不一定等于重新调用外部工具。很多场景下，更安全的是：

- replay recorded outputs
- mock tool results
- 只重跑 planner / generation 节点

## 10. 审计与回放在 Runtime 中应如何落位

推荐把它们设计成 runtime 的内建能力，而不是外围日志系统。

建议链路：

1. runtime 在任务开始时创建 run record
2. 每个状态转换写入 event store
3. 每次 retrieval、tool call、HITL 决策都生成审计事件
4. 最终汇总 run result
5. replay service 基于 event store 和 checkpoint 恢复执行轨迹

换句话说：

trace、checkpoint、state transition、tool event 应共享同一条运行时骨架。

## 11. 与 Evals、Observability、HITL 的关系

### 11.1 与 Evals 的关系

执行轨迹可以沉淀成：

- 失败样本集
- 高价值问题集
- 回归测试集
- planner / tool 评测样本

### 11.2 与 Observability 的关系

observability 更关注实时运行指标和排障信号；
审计与回放更关注任务级、责任级、复盘级记录。

### 11.3 与 HITL 的关系

HITL 决策如果不进入审计与回放链路，后续就无法回答“谁在什么证据基础上批准了什么”。

## 12. 面向低代码平台的典型场景

### 12.1 配置纠错 Agent

应审计：

- 查了哪些组件规则
- 调用了哪些校验工具
- 自动修复建议是什么
- 是否经过人工确认
- 最终修复是否执行

### 12.2 页面生成 Agent

应审计：

- 检索了哪些模板和物料
- 生成了哪些 schema 草稿
- 哪一步校验未通过
- 是否做过自动修复

### 12.3 运维分析 Agent

应审计：

- 查了哪些日志、发布记录、配置差异
- 根因分析的证据链是什么
- 是否建议回滚或切换配置
- 是否由人工批准执行

### 12.4 逻辑编排 Agent

应审计：

- 自动生成了哪些 DAG 节点
- 哪些依赖被新增或修改
- 类型校验结果如何
- 是否经过人工复核

## 13. 常见误区

### 13.1 把普通应用日志当成审计系统

日志能排查技术错误，但不能天然满足责任、审批和回放需求。

### 13.2 只记录最终答案，不记录中间轨迹

这会让复杂 Agent 行为几乎不可解释。

### 13.3 审计信息过度原始，导致无法阅读或泄露敏感数据

应在完整性、可读性和脱敏之间平衡。

### 13.4 回放等于重新执行所有步骤

很多动作不可重放，也不应该重放。真正需要的是可控 replay。

### 13.5 审计与状态机脱节

如果没有状态转移骨架，回放通常只能停留在零散事件浏览。

## 14. 一句话总结

执行审计与回放的核心，是把 Agent 从“能运行”提升到“能解释、能追责、能复盘、能优化”的生产系统；近两年的最佳实践也越来越明确，Agent Runtime 不仅要管理任务执行，还要内建 trace、event、checkpoint 和 replay 机制，才能真正支撑企业级落地。
