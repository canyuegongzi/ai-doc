# 幻觉分析

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Hallucination Analysis 设计，重点回答：

- 什么是企业场景中的“幻觉”，以及它和普通错误有什么不同
- 如何从 RAG、Tool、Agent、输出约束多个层面分析幻觉来源
- 近两年为什么行业开始更强调 uncertainty-aware evaluation，而不是只看准确率

## 2. 什么是幻觉

幻觉通常指：

模型输出了看似合理、语气自信，但实际上不真实、无依据、与事实不符或超出证据范围的内容。

在企业场景里，幻觉不只表现为“编造知识”，还可能表现为：

- 编造配置字段
- 编造组件能力
- 编造不存在的日志结论
- 编造工具结果或审批状态
- 在证据不足时给出确定性建议

用一句话概括：

幻觉的本质，是模型在缺乏充分证据或约束时，仍然生成了超出真实依据范围的结果。

## 3. 为什么幻觉分析必须单独做

### 3.1 准确率高不代表幻觉率低

近两年的研究和实践都在强调：

模型如果被鼓励在不确定时继续猜测，可能在某些任务上提高“答对率”，但同时显著提高错误且自信的输出比例。

### 3.2 幻觉在企业系统中的成本往往比普通错误更高

因为幻觉常常伴随：

- 高自信表达
- 结构化输出
- 动作建议
- 下游系统消费

这意味着一条幻觉不仅误导用户，还可能触发后续错误执行。

### 3.3 RAG 和 Agent 并不会天然消除幻觉

它们只是把问题拆散了：

- RAG 可以降低知识缺失型幻觉
- Tool 可以降低外部状态幻觉

但如果检索错、工具错、总结错、约束弱，幻觉仍然会发生。

## 4. 近两年的关键变化

### 4.1 行业开始强调“不确定时拒答”优于“错误猜测”

OpenAI 2025 年关于幻觉的研究强调，很多评测和训练会奖励模型猜测，从而压低了拒答率但抬高了错误率。

这说明：

评测和产品设计需要给“不知道”“证据不足”以正式位置。

### 4.2 幻觉分析正在从最终答案层扩展到过程层

成熟系统越来越关注：

- query rewrite 是否偏离
- retrieval 证据是否不足
- tool output 是否被误读
- planner 是否在不确定条件下继续推进

### 4.3 幻觉类型正在从纯文本知识错误扩展到结构化与动作错误

在 Agent 系统里，幻觉可能表现为：

- 错误的 JSON 字段
- 虚构的 citation
- 虚构的审批状态
- 虚构的环境结果

## 5. 推荐的幻觉分类

建议至少区分以下类型：

### 5.1 知识型幻觉

编造事实、概念、术语、组件能力。

### 5.2 证据越界型幻觉

检索到了部分证据，但结论超出了证据本身支持范围。

### 5.3 引用型幻觉

引用不存在的 source，或引用与结论不匹配。

### 5.4 结构型幻觉

输出结构化对象时编造字段、枚举值或层级关系。

### 5.5 状态型幻觉

错误描述系统状态、执行结果、审批状态或环境 outcome。

### 5.6 动作型幻觉

在没有充分依据时建议执行高风险动作，或虚构动作前提已经满足。

## 7. 推荐的分析框架

建议按四层分析：

### 7.1 输入与任务理解层

看是否：

- 误解用户意图
- 误抽取实体
- 误扩写 query

### 7.2 证据获取层

看是否：

- 没有召回正确证据
- 召回证据不足
- 证据排序不合理
- 权限过滤误伤关键证据

### 7.3 推理与整合层

看是否：

- 结论超出了证据范围
- 忽略了冲突信息
- 未显式表达不确定性

### 7.4 输出与执行层

看是否：

- 结构化输出编造字段
- 错误生成动作建议
- 错误总结 tool / state 结果

## 8. 在 RAG 中如何分析幻觉

对于 RAG 场景，建议重点看：

### 8.1 evidence missing

没有召回到 gold evidence。

### 8.2 evidence mismatch

召回了内容，但并不真正支持答案。

### 8.3 citation mismatch

答案与引用不一致。

### 8.4 over-generalization

模型从局部证据推到过度确定结论。

## 9. 在 Tool / Agent 中如何分析幻觉

在 Tool / Agent 场景中，建议重点看：

### 9.1 tool result misread

工具返回了正确信息，但模型理解错了。

### 9.2 state hallucination

模型错误描述当前执行状态或审批状态。

### 9.3 unsupported action suggestion

系统在证据不足时建议或发起动作。

### 9.4 fabricated completion

模型声称“任务已完成”，但环境结果并未达到。

## 10. 推荐的幻觉指标

可以结合以下指标：

- hallucination rate
- unsupported claim rate
- citation mismatch rate
- uncertainty omission rate
- fabricated action / state rate

注意：

这些指标通常需要结合人工标注、模型 grader 和环境结果判断。

## 13. 常见误区

### 13.1 把所有错误都归为幻觉

有些问题其实是检索失败、权限过滤或工具故障。

### 13.2 只看最终回答，不看证据链

无法区分“有依据但总结差”与“完全无依据”。

### 13.3 用准确率替代幻觉率

模型可能因为猜测而维持准确率，但幻觉仍然很高。

### 13.4 只靠 RAG 解决幻觉

RAG 只能缓解部分知识型幻觉，不能替代约束与验证。

### 13.5 不允许模型拒答

会显著鼓励不确定时继续编造。

## 14. 一句话总结

幻觉分析的核心，是把“错误但自信的输出”从普通错误中单独拆出来，并沿输入理解、证据获取、推理整合和输出执行四层追踪其来源与传导路径；近两年的最佳实践也越来越明确，降低幻觉并不只是让模型更聪明，更重要的是让系统在不确定时学会约束、证据化和拒答。
