# 在线质量监控

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Online Quality Monitoring 设计，重点回答：

- 为什么离线评测不能替代线上质量监控
- 在线监控应如何覆盖 RAG、Tool、Agent、HITL 和成本链路
- 近两年为什么行业越来越强调 production monitoring、real-user feedback 和 transcript review

## 2. 什么是在线质量监控

在线质量监控可以理解为：

在 AI 系统真实上线运行后，对请求质量、行为路径、异常模式、风险触发、用户反馈和资源消耗进行持续观测、告警和分析的一整套机制。

它不只关心“服务有没有挂”，还关心：

- 用户得到的结果是否仍然可靠
- 是否出现新的失败模式
- 是否发生分布漂移
- 是否开始频繁触发升级人工或高风险阻断

用一句话概括：

在线质量监控的本质，是让 AI 系统上线后不再“飞盲”，而是持续知道真实世界里发生了什么。

## 3. 为什么离线评测不能替代在线监控

### 3.1 真实用户分布总会变化

线上请求会出现：

- 新业务场景
- 模糊表达
- 越权尝试
- 非预期工具组合
- 极端长尾问题

这些常常不会完整出现在离线评测集中。

### 3.2 线上问题往往来自系统交互，而不是单点模型输出

例如：

- 某些 tool 近期超时率上升
- 某个索引版本切换后召回质量下降
- 某类审批请求异常增多
- 某个租户的预算耗尽导致体验退化

这些都需要线上监控才能及时发现。

### 3.3 近两年的最佳实践越来越强调多层监控而不是单次离线验证

Anthropic 在 agent eval 文中明确提到：

自动化 evals、production monitoring、A/B test、用户反馈和人工 transcript review 应共同构成“瑞士奶酪模型”式多层质量防线。

## 4. 在线质量监控要解决什么问题

### 4.1 发现真实失败模式

### 4.2 监控回归和漂移

### 4.3 监控风险与越界事件

### 4.4 支撑运营、排障和持续优化

## 5. 推荐的在线监控分层

建议至少分成五层：

### 5.1 服务健康层

- availability
- latency
- error rate
- queue backlog

### 5.2 质量结果层

- success rate
- groundedness proxy
- tool success rate
- task completion rate
- escalation rate

### 5.3 行为路径层

- average turns
- average tool calls
- planner branch distribution
- retry distribution

### 5.4 风险治理层

- high-risk action attempts
- approval trigger count
- policy blocks
- sensitive data redaction hits

### 5.5 成本运营层

- per request cost
- per scene cost
- per tenant budget usage
- token / rerank / tool cost distribution

## 6. 推荐的核心线上指标

### 6.1 通用质量指标

- 请求成功率
- 平均响应时长
- 错误率
- 重试率
- 升级人工率

### 6.2 RAG 指标

- citation 出现率
- insufficient evidence 率
- topK 使用分布
- retrieval timeout rate

### 6.3 Tool / Agent 指标

- tool call success rate
- invalid parameter rate
- average tool calls per task
- planner failure rate
- task completion rate

### 6.4 安全与治理指标

- 权限阻断次数
- 高风险审批触发次数
- 审批通过率 / 拒绝率
- 脱敏命中次数

### 6.5 成本指标

- 平均每请求成本
- 平均每任务成本
- 模型使用占比
- 超预算降级次数

## 7. 线上监控不应只看数值，还应看轨迹样本

很多 AI 质量问题不能只靠聚合指标看出来。

建议同时建立：

- 高频失败样本池
- 高风险事件样本池
- 用户差评样本池
- 高成本长链路样本池

这样才能把指标异常映射回真实 transcript 和运行路径。

## 8. 推荐的线上监控机制

### 8.1 指标监控

看整体趋势和波动。

### 8.2 异常告警

如：

- 某类 tool success rate 突降
- 某个租户高风险审批骤增
- 某模型成本异常上升

### 8.3 样本抽检

定期人工查看高价值 transcript。

### 8.4 用户反馈接入

将：

- thumbs up / down
- 工单
- 直接投诉
- 回退操作

接入质量分析链路。

## 9. 在线监控与离线评测的关系

### 9.1 离线评测

适合：

- 回归测试
- 发布前验证
- 版本对比

### 9.2 在线监控

适合：

- 发现新问题
- 监测分布变化
- 捕捉真实用户影响

### 9.3 最佳实践

把线上失败样本回流为离线评测任务，把离线评测发现的问题映射为线上重点监控指标。

## 10. 面向低代码平台的重点线上监控建议

### 10.1 组件问答

重点监控：

- citation 缺失率
- 用户追问率
- “无帮助”反馈率

### 10.2 Schema 生成

重点监控：

- schema 校验失败率
- 自动修复触发率
- 草稿采纳率

### 10.3 配置诊断 Agent

重点监控：

- root cause 命中率 proxy
- 修复建议采纳率
- 错误升级人工率

### 10.4 运维与动作助手

重点监控：

- 高风险动作审批频次
- 审批后执行成功率
- 高成本多工具链路占比

## 11. 推荐的反馈闭环

建议形成这条闭环：

1. 线上监控发现异常指标
2. 关联具体 trace / transcript
3. 标注失败模式
4. 转入离线评测集
5. 修复后再用离线评测验证
6. 上线后继续观察同类指标

这会让监控和评测形成正循环。

## 12. 常见误区

### 12.1 线上只监控接口可用性

这无法覆盖 AI 质量退化。

### 12.2 线上监控没有样本回看能力

指标异常后仍然无法定位原因。

### 12.3 不采集用户反馈

会失去最直接的真实质量信号。

### 12.4 指标过多但没有分层重点

团队容易噪声淹没。

### 12.5 监控结果不回流到评测集

同类问题会反复出现。

## 13. 一句话总结

在线质量监控的核心，是在真实用户环境中持续观测 AI 系统的质量、风险、行为路径和成本表现，并把异常指标、失败样本和用户反馈不断回流到离线评测和系统优化中；近两年的最佳实践也越来越明确，AI 系统上线并不是结束，而是评测与监控真正开始发挥价值的时候。
