# 离线评测集设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 Offline Eval Dataset / Task Suite 设计，重点回答：

- 离线评测集应该如何构建，而不是简单堆一批问答样本
- 在 RAG、Tool、Agent 场景下，任务、trial、grader、outcome 应如何设计
- 近两年为什么行业越来越强调从真实失败案例中构建 eval suite

## 2. 什么是离线评测集

离线评测集可以理解为：

一组在非真实线上环境中运行的标准化任务集合，用于稳定地测试 AI 系统在特定能力、特定场景或特定风险边界下的表现。

它不只是“题库”，而通常包括：

- 任务输入
- 环境上下文
- 期望结果或成功标准
- 打分逻辑
- 多次 trial 运行记录

用一句话概括：

离线评测集的本质，是为 AI 系统建立一个可以反复运行、对比和回归验证的标准实验场。

## 3. 为什么离线评测集是 AI 工程的关键资产

### 3.1 没有固定任务集，就无法稳定比较版本

如果每次只靠临时试几个例子：

- 无法判断版本差异
- 无法判断优化是否真实生效
- 无法做模型切换和索引切换

### 3.2 真实线上用户样本不适合直接替代离线评测

线上样本分布复杂、噪声高、缺少稳定标注。离线评测集的价值在于：

- 可重复
- 可控
- 可分层
- 可做发布门禁

### 3.3 近两年的共识是：从真实失败案例构建 eval 更有效

Anthropic 在 agent eval 实战中强调，一个高价值起点不是追求上百个合成样本，而是先从真实失败案例和高价值任务里抽取 20-50 个任务起步。

## 4. 离线评测集要解决什么问题

### 4.1 回归检测

检测系统更新后是否退化。

### 4.2 能力测量

评估系统是否已经具备某类能力。

### 4.3 模型 / 索引 / Prompt 对比

同一任务集上比较不同版本的表现。

### 4.4 失败模式聚类

识别哪些类型问题仍然薄弱。

## 5. 推荐的评测任务结构

建议一个任务至少包含：

- `taskId`
- `sceneType`
- `input`
- `context`
- `constraints`
- `expectedOutcome`
- `graderConfig`
- `riskTag`
- `difficulty`

对于 Agent 任务，建议额外记录：

- `availableTools`
- `environmentState`
- `terminationCriteria`
- `maxSteps`

## 6. Task、Trial、Grader、Outcome 的推荐概念边界

### 6.1 Task

单个测试问题或任务定义。

### 6.2 Trial

同一 task 的一次实际运行尝试。

因为模型具有随机性，关键任务通常需要多次 trial。

### 6.3 Grader

对 trial 结果进行判断的逻辑，可能是：

- 规则 grader
- 模型 grader
- 环境 outcome grader

### 6.4 Outcome

任务结束时环境或结果对象的最终状态。

尤其在 Agent 场景中，最终 outcome 常比回答文本更重要。

## 7. 推荐的离线评测集分类

### 7.1 基础问答评测集

适合：

- FAQ
- 知识问答
- grounded answer

### 7.2 RAG 检索评测集

适合：

- recall
- rerank
- citation
- context relevance

### 7.3 结构生成评测集

适合：

- schema 生成
- JSON / DSL 输出
- 表单配置草稿生成

### 7.4 Tool / Agent 任务评测集

适合：

- tool selection
- planner quality
- multi-step completion
- approval triggering

### 7.5 风险与治理评测集

适合：

- 敏感信息拦截
- 高风险动作审批
- 权限越界阻断

## 8. RAG 任务集应如何设计

建议至少覆盖：

- 明确问答
- 多跳检索
- 含歧义 query
- 权限受限 query
- citation 必须准确的 query
- 低证据场景

同时给出：

- gold documents / gold chunks
- acceptable answer rubric
- citation expectations

## 9. Agent 任务集应如何设计

对 Agent 类任务，仅给“标准答案文本”通常不够。

更合理的是定义：

- 环境初始状态
- 可用工具
- 成功条件
- 不允许行为
- 风险边界
- 最终状态断言

例如：

- 是否成功创建页面草稿
- 是否触发了不该触发的 prod 发布
- 是否在应审批场景下进入 HITL

## 10. 推荐的 Grader 设计方式

### 10.1 规则型 Grader

适合：

- JSON schema 合法性
- tool 参数正确性
- state transition 正确性
- outcome 存在性

### 10.2 模型型 Grader

适合：

- 回答质量
- 覆盖度
- 解释与总结质量
- 主观生成质量

### 10.3 环境型 Grader

适合：

- 资源是否真的创建
- 配置是否真的被应用
- 审批是否真的被触发

更成熟的评测通常会组合这三类 grader。

## 11. 评测集样本来源建议

推荐优先级：

1. 真实线上失败案例
2. 真实高价值业务任务
3. 产品需求中的关键边界案例
4. 人工构造的补充对抗样本

不建议长期只依赖纯合成样本，因为它们很容易过于干净、缺少真实分布特征。

## 12. 面向低代码平台的重点任务集建议

### 12.1 组件问答任务集

覆盖：

- 组件用途问答
- slot / props 问题
- setter 配置问题
- 组件推荐问题

### 12.2 Schema 生成任务集

覆盖：

- 页面草稿生成
- 表单配置生成
- 查询条件生成
- 结构校验失败样本

### 12.3 配置诊断任务集

覆盖：

- 数据绑定缺失
- props 类型错误
- slot 配置异常
- 层级结构异常

### 12.4 运维 Agent 任务集

覆盖：

- 构建失败分析
- 发布记录查询
- 回滚建议
- 敏感动作审批触发

## 13. 常见误区

### 13.1 只做几十道普通问答题就叫评测集

这无法覆盖 Agent、RAG、治理等复杂场景。

### 13.2 样本全是理想路径，没有失败路径

这样无法测出真实鲁棒性。

### 13.3 只看最终回答，不看环境 outcome

对于 Tool / Agent 场景，这是非常不够的。

### 13.4 没有版本基线和多 trial

很难区分真实回归和随机波动。

### 13.5 评测集长期不更新

会逐渐脱离真实用户分布。

## 14. 一句话总结

离线评测集设计的核心，是围绕真实高价值任务和真实失败样本，把 AI 系统需要稳定做到的能力、边界和风险条件编码成可重复运行的标准任务集合，并通过 task、trial、grader、outcome 的分层设计形成长期回归资产；近两年的最佳实践也越来越明确，成熟团队的 eval suite 不是题库，而是产品能力地图。
