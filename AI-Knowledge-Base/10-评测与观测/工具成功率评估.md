# 工具成功率评估

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 Tool Success Evaluation 设计，重点回答：

- 为什么 Tool success rate 是 Agent 系统最关键的质量指标之一
- 工具成功率不能只看“接口有没有返回 200”
- 如何从选择、填参、执行、结果可用性、风险边界多个层次评估工具使用质量

## 2. 什么是工具成功率

工具成功率可以理解为：

在给定任务上下文下，系统选择并调用工具后，是否以正确的工具、正确的参数、正确的边界和正确的结果推动任务向目标完成的一类综合指标。

它不只是“工具是否执行成功”，还包括：

- 选没选对工具
- 参数填得对不对
- 调用是否真的成功
- 返回结果是否可用
- 是否触发了不该触发的动作

用一句话概括：

工具成功率的本质，是衡量 Agent 将“调用外部能力”这件事做得是否真正有效且可控。

## 3. 为什么工具成功率是 Agent 质量核心变量

### 3.1 Agent 很多失败并不是推理失败，而是工具失败

在很多真实系统里，任务失败常见原因是：

- 选错工具
- 工具参数生成错误
- 工具超时或异常
- 返回结果不可消费
- 高风险动作被错误触发或错误阻断

### 3.2 工具链路是从“知道”走向“做到”的关键桥梁

没有高工具成功率，Agent 只能停留在建议层。

### 3.3 近两年的行业共识越来越明确：tool quality 决定 agent quality

Anthropic 在 tools 实战文章中已明确指出，高质量工具设计和稳定工具使用，是 agent 成功率的核心因素之一。

## 4. 工具成功率要解决什么问题

### 4.1 发现错误的工具选择

### 4.2 发现参数生成质量问题

### 4.3 发现工具执行可靠性问题

### 4.4 发现结果消费与任务推进问题

### 4.5 发现风险治理是否过松或过紧

## 5. 推荐的工具成功率分层

建议至少分成五层：

### 5.1 Tool Selection Success

当前任务是否选对了工具。

### 5.2 Parameter Success

模型生成的参数是否正确、完整、合法。

### 5.3 Execution Success

工具调用是否成功执行。

### 5.4 Result Usefulness Success

工具返回结果是否真的帮助了后续决策或完成任务。

### 5.5 Safety Success

工具使用是否遵守权限、风险和审批边界。

## 6. 为什么不能只看接口成功率

很多系统会只统计：

- HTTP 200
- 无异常返回

但这远远不够。因为：

- 选错工具也可能返回 200
- 参数语义错也可能返回 200
- 返回结果为空也可能返回 200
- 本不该执行的高风险动作即使执行成功，也算失败

所以更合理的 success 定义应基于任务语义，而不是技术返回码。

## 7. 推荐的核心指标

### 7.1 工具选择正确率

在可用工具集合中，是否选择了正确工具或正确工具族。

### 7.2 参数正确率

包括：

- 必填字段完整率
- 枚举值正确率
- 目标资源绑定正确率
- 环境参数正确率

### 7.3 执行成功率

包括：

- 无异常完成率
- timeout rate
- retry success rate

### 7.4 结果可用率

工具结果是否被后续流程真正使用，并推动任务前进。

### 7.5 风险合规率

是否：

- 未越权
- 未跳过审批
- 未误触发高风险动作

## 8. 推荐的工具成功定义方式

不同工具应有不同成功定义。

### 8.1 查询类工具

成功标准通常是：

- 返回正确对象
- 数据可读且完整
- 在授权范围内

### 8.2 检索类工具

成功标准通常是：

- 召回到相关对象
- 结果足够可用于后续决策

### 8.3 校验类工具

成功标准通常是：

- 输出结果正确
- 错误信息足够清晰
- 可供后续修复链路消费

### 8.4 执行类工具

成功标准通常是：

- 真正改变到目标状态
- 在正确边界内执行
- 具备 receipt / audit
- 必要时已通过审批

## 9. 工具成功率与 Agent 任务完成率的关系

两者相关，但不是同一个指标。

### 9.1 工具成功率高，任务未必完成

可能 planner 错、链路错、总结错。

### 9.2 任务完成率高，工具成功率未必健康

系统可能靠偶然路径完成任务，但工具使用不稳定。

因此建议同时看：

- tool-level success
- task-level success

## 10. 近两年的关键变化

### 10.1 Tool Success 开始从接口指标走向语义指标

成熟系统越来越关注：

- correct tool
- correct params
- useful result
- safe action

而不是只看调用有没有报错。

### 10.2 多工具编排让 success 评估必须看链路上下文

某个工具本身成功，不代表它在当前链路中使用得合理。

### 10.3 高风险动作让“错误执行成功”成为重点失败类型

在执行型工具中，误执行成功是最危险的失败模式之一。

## 11. 面向低代码平台的典型工具评估建议

### 11.1 `searchComponentMaterial`

关注：

- 是否命中相关组件
- 返回信息是否可用于回答或生成

### 11.2 `validatePageSchema`

关注：

- 是否识别出真实结构问题
- 错误位置和原因是否清晰

### 11.3 `generateSchemaPatch`

关注：

- patch 是否语法正确
- 是否对准正确节点
- 是否产生副作用风险

### 11.4 `publishPage`

关注：

- 是否在正确环境执行
- 是否经过审批
- 是否有 receipt
- 是否成功验证发布结果

## 12. 推荐的评测方式

### 12.1 离线任务集评测

对 tool selection、param generation、outcome success 做回归评测。

### 12.2 线上监控

持续看：

- tool success trend
- invalid param rate
- timeout / retry rate
- blocked-by-policy rate

### 12.3 失败样本复盘

对高频失败工具做 transcript review 和 schema / doc 优化。

## 13. 常见误区

### 13.1 把 200 返回当成工具成功

这太粗糙，无法指导真正优化。

### 13.2 不区分工具类型的成功标准

查询类、校验类、执行类工具的 success 定义完全不同。

### 13.3 不统计参数质量

很多问题就会被隐藏在“接口成功但结果错误”里。

### 13.4 不统计被 guardrails 阻断的情况

这会看不出系统到底是在安全阻断还是误阻断。

### 13.5 不把工具失败样本回流到文档和 schema 优化

工具质量难以持续提升。

## 14. 一句话总结

工具成功率评估的核心，是从工具选择、参数生成、执行结果、结果可用性和安全边界五个层面衡量 Agent 是否真正把外部能力用对、用稳、用安全；近两年的最佳实践也越来越明确，Agent 的上限不仅由模型决定，更由工具成功率决定。
