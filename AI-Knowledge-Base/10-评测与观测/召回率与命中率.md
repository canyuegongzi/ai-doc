# 召回率与命中率

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 RAG 检索链路中的 Recall / Hit Rate 指标设计，重点回答：

- 为什么召回率和命中率是 RAG 评测中的核心指标之一
- 这两个指标分别在什么层次上有意义
- 近两年为什么行业越来越强调分层评估 retrieval，而不是只看最终回答质量

## 2. 什么是召回率与命中率

### 2.1 召回率（Recall）

召回率用于衡量：

在所有“本应被找回”的相关知识对象中，系统实际找回了多少。

### 2.2 命中率（Hit Rate）

命中率通常用于衡量：

在一次查询的前 K 个结果中，是否至少命中了一个或多个关键正确对象。

例如：

- `Hit@1`
- `Hit@3`
- `Hit@5`

用一句话概括：

召回率更强调“找全”，命中率更强调“前排有没有找对”。

## 3. 为什么这两个指标对 RAG 很关键

### 3.1 检索没找对，后面生成再强也无济于事

RAG 的生成质量有一个硬前提：

至少要先把正确证据召回来。

### 3.2 最终答案正确不代表检索链路健康

有时模型会凭常识或偶然信息答对，但检索其实并没有稳定命中正确来源。

如果只看 final answer，会掩盖 retrieval 层问题。

### 3.3 近两年的最佳实践越来越强调 retrieval 分层评测

现代 RAG 系统通常会单独评估：

- rewrite 质量
- recall
- rerank
- context relevance
- answer groundedness

这已经成为成熟系统的共识。

## 4. 召回率与命中率分别解决什么问题

### 4.1 召回率

用于判断：

- 检索范围是否足够覆盖相关知识
- recall stage 是否漏掉关键证据

### 4.2 命中率

用于判断：

- 前 K 结果是否有用
- 是否值得进入后续 rerank / context builder

### 4.3 两者结合

可以同时判断：

- 系统有没有找回来
- 找回来的内容是不是足够靠前、足够有用

## 5. 推荐的指标层次

建议至少区分以下几层：

### 5.1 Document-level Recall / Hit

以文档为单位评估。

### 5.2 Chunk-level Recall / Hit

以切片为单位评估，更适合 chunk-based RAG。

### 5.3 Object-level Recall / Hit

以结构化对象为单位评估，例如：

- 组件定义
- 页面模板
- API 协议
- 规则条目

### 5.4 Citation-level Hit

看最终引用中是否包含 gold evidence。

## 6. 推荐的常用指标形式

### 6.1 Recall@K

看前 K 个结果中召回了多少相关对象。

### 6.2 Hit@K

看前 K 个结果中是否至少命中一个正确对象。

### 6.3 MRR / NDCG（可选）

当需要更细粒度评估排序质量时，可引入：

- MRR
- NDCG

但对于很多企业 RAG 场景，Recall@K + Hit@K 已经非常有价值。

## 7. 为什么不能只看一个 K

不同场景下 K 的意义不同：

- `Hit@1` 更看第一命中是否准确
- `Hit@3 / Hit@5` 更适合评估前排是否可用
- `Recall@10 / Recall@20` 更适合评估召回覆盖能力

因此建议按：

- 小 K 看前排质量
- 大 K 看覆盖能力

组合使用。

## 8. 近两年的关键变化

### 8.1 hybrid retrieval 让指标解释更复杂但更必要

dense + sparse + metadata filter 混合后，检索性能不再能靠单一经验判断，必须用分层指标拆解。

### 8.2 query rewrite / expansion 使召回指标前移

现在很多 retrieval 质量变化不是索引本身问题，而是 query understanding 质量变化，因此召回指标应结合 rewrite 版本一起分析。

### 8.3 agentic retrieval 引入多步命中问题

在多轮检索或多子问题检索中，需要评估：

- 每个 subquery 的 hit rate
- 整体 evidence coverage

## 9. RAG 场景下如何设计 gold 数据

要评估召回率与命中率，前提是为任务定义 gold evidence。

可以按以下方式构建：

### 9.1 单一 gold

只有一个标准正确文档或对象。

### 9.2 多 gold

允许多个来源都算正确。

### 9.3 层级 gold

例如：

- 文档级正确
- chunk 级更精确
- citation 级最精确

更成熟的系统往往采用多 gold 和层级 gold。

## 10. 在低代码平台场景中的典型应用

### 10.1 组件问答

评估：

- 是否召回正确组件文档
- 是否命中正确 props / slot 说明

### 10.2 页面模板推荐

评估：

- 是否命中相关页面模板
- 前 3 个推荐是否覆盖正确模板类型

### 10.3 配置诊断

评估：

- 是否命中正确规则条目
- 是否命中相关历史案例

### 10.4 运维分析

评估：

- 是否命中正确发布记录
- 是否命中对应日志和规则文档

## 11. 指标使用建议

### 11.1 Recall@K 适合作为索引 / query 侧优化指标

### 11.2 Hit@K 适合作为产品可用性指标

### 11.3 需要与后续指标联动看

例如：

- recall 高但 answer 仍差，说明后续 rerank / context / generation 有问题
- hit 低但 answer 偶尔答对，说明系统不稳定

## 12. 常见误区

### 12.1 只看最终回答准确率，不看 recall

这样无法定位问题是否出在 retrieval。

### 12.2 召回率高就认为系统好

如果命中不在前排，用户体验仍然会很差。

### 12.3 只定义一个 gold document

真实企业场景中往往存在多个正确来源。

### 12.4 不区分 document、chunk、object 粒度

这会让指标解释失真。

### 12.5 指标不和具体 query 类型分组

不同 query 类型对 recall 的要求差异很大。

## 13. 一句话总结

召回率与命中率的核心，是从检索链路本身衡量系统有没有把正确知识找回来、找得够不够靠前，并据此为 RAG 优化提供可定位的信号；近两年的最佳实践也越来越明确，成熟的 RAG 系统不能只看最终回答，而必须把 retrieval 指标单独拆出来长期跟踪。
