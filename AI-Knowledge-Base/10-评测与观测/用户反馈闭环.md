# 用户反馈闭环

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 User Feedback Loop 设计，重点回答：

- 为什么用户反馈不是附加功能，而是评测体系的重要输入
- 如何把 thumbs up/down、工单、投诉、回退操作等信号接入质量改进链路
- 近两年为什么行业越来越强调 production feedback 和 transcript review 的组合价值

## 2. 什么是用户反馈闭环

用户反馈闭环可以理解为：

从用户在真实使用过程中的显式和隐式反馈中提取质量信号，并将这些信号转化为问题分类、优先级排序、评测样本、规则优化和产品迭代动作的一套持续机制。

它不只关心“用户满不满意”，更关心：

- 哪类问题在真实环境里频繁出现
- 哪些失败模式离线评测没有覆盖
- 哪些风险虽然系统看起来成功，但用户仍然不信任

用一句话概括：

用户反馈闭环的本质，是把真实使用中的主观感受和客观失败样本，持续转化为系统优化资产。

## 3. 为什么用户反馈是评测体系的重要组成

### 3.1 离线评测永远无法完整覆盖真实分布

产品上线后，用户会暴露出：

- 长尾需求
- 新表达方式
- 复杂上下文组合
- 对结果风格和可解释性的真实要求

### 3.2 很多质量问题只有用户才会感知到

例如：

- 回答虽“正确”但不够有帮助
- 诊断建议太啰嗦
- schema 草稿虽合法但不实用
- 运维建议虽谨慎但缺少行动性

### 3.3 近两年的 agent eval 实践越来越强调线上反馈价值

Anthropic 明确指出：

生产监控、用户反馈、人工 transcript review 与自动化 eval 是互补关系。单独依赖某一种方式都不够。

## 4. 用户反馈要解决什么问题

### 4.1 发现离线评测未覆盖的问题

### 4.2 提供真实任务优先级信号

### 4.3 帮助校准模型 grader 和人工判断标准

### 4.4 为评测集扩充和产品路线提供输入

## 5. 推荐的反馈信号分类

建议至少分成四类：

### 5.1 显式正负反馈

例如：

- thumbs up / down
- 有帮助 / 无帮助
- 满意 / 不满意

### 5.2 文本反馈

例如：

- 用户留言
- 工单描述
- 投诉与建议

### 5.3 行为反馈

例如：

- 用户立即追问
- 用户复制结果后未采用
- 用户手动回退系统建议
- 用户中断自动流程

### 5.4 结果反馈

例如：

- 生成草稿是否被采纳
- 修复建议是否被采用
- 审批后是否仍被人工修改

## 6. 推荐的反馈闭环链路

一条较稳的闭环通常包括：

1. 采集反馈信号
2. 绑定 trace / request / scene
3. 做问题分类和严重度判断
4. 回看 transcript 和系统链路
5. 决定是：
   - 加入离线评测集
   - 调整 prompt / tool / policy
   - 调整产品交互
6. 修复后再次观察线上反馈变化

## 7. 近两年的关键变化

### 7.1 用户反馈正从“产品满意度”升级为“AI 质量数据源”

过去反馈更多用于客服和产品满意度；现在在 AI 场景里，反馈越来越直接成为 eval 和优化的高价值输入。

### 7.2 反馈与 Trace 绑定越来越重要

单独一条“没用”反馈价值有限；如果能关联：

- 当时 query
- 检索结果
- tool path
- 输出结果

问题定位效率会显著提升。

### 7.3 显式反馈和隐式行为反馈正在结合

很多用户不会主动点踩，但会通过：

- 重复追问
- 手动修复
- 放弃系统建议

表达不满意。

## 8. 推荐的反馈记录字段

建议至少包含：

- `feedbackId`
- `requestId`
- `traceId`
- `sceneType`
- `tenantId`
- `userId`
- `feedbackType`
- `feedbackValue`
- `feedbackText`
- `severity`
- `linkedOutcome`
- `createdAt`

## 9. 面向低代码平台的典型反馈信号

### 9.1 组件问答

关注：

- “无帮助”比例
- 同类问题是否高频追问
- citation 是否被用户点击后仍觉得不对

### 9.2 Schema 生成

关注：

- 草稿是否被采纳
- 用户是否大量手工修改
- 哪些字段经常被回退

### 9.3 配置诊断

关注：

- 修复建议是否被接受
- 建议是否经常需要人工重写
- 用户是否仍需工单升级

### 9.4 运维与动作助手

关注：

- 用户是否信任系统建议
- 审批人是否频繁拒绝某类动作
- 审批后执行结果是否仍需人工补救

## 10. 反馈如何回流到评测体系

### 10.1 负反馈样本回流离线评测集

把高价值失败样本写入 regression suite。

### 10.2 高反馈频次问题回流产品优先级

### 10.3 反馈校准 model grader

如果 model grader 与用户实际感受长期不一致，说明评分逻辑需要调整。

### 10.4 反馈驱动 guardrail 和交互设计优化

有些问题不是模型问题，而是：

- 输出太确定
- 引导不够
- 操作确认不清楚

## 11. 用户反馈与人工复核的关系

反馈只是入口，真正高价值的是：

- 反馈聚类
- transcript 回看
- 根因分类
- 是否转入长期资产

如果没有后续复盘机制，反馈很容易沦为噪声堆积。

## 12. 常见误区

### 12.1 只收集点赞点踩，不看具体 transcript

信号太浅，难以定位问题。

### 12.2 只处理投诉最严重的问题

会错过很多高频但中等严重度的问题。

### 12.3 不采集隐式行为反馈

很多真实不满意不会被显式表达。

### 12.4 反馈不回流评测集

同类问题会长期反复出现。

### 12.5 产品、模型、平台团队各自处理反馈，不共享

会导致问题分类和优化动作碎片化。

## 13. 一句话总结

用户反馈闭环的核心，是把真实用户在生产环境中的显式意见、隐式行为和结果采纳情况，系统化地绑定到 trace、场景和任务链路上，并不断回流到评测、规则、工具和交互优化中；近两年的最佳实践也越来越明确，AI 产品真正成熟的标志之一，就是它能把用户反馈高效转化为模型和系统改进资产。
