# 评测体系总览

> 状态：第一版草案

## 1. 文档定位

本文档用于定义企业 AI 系统中的 Evaluation & Monitoring 总体框架，重点回答：

- 为什么 AI 系统不能只靠体感和人工试用来判断效果
- 评测体系、在线监控、人工复核、用户反馈之间的关系是什么
- 近两年为什么行业越来越强调 eval-driven development 和 production monitoring 双轮驱动

## 2. 什么是评测体系

评测体系可以理解为：

围绕 AI 系统从开发、测试、上线到持续迭代的全生命周期，建立一套可定义成功标准、可量化衡量质量、可发现回归、可指导优化的机制集合。

它通常包括：

- 离线评测集
- 自动化打分器
- 在线监控指标
- 人工复核机制
- 用户反馈闭环
- 回归对比与发布门禁

用一句话概括：

评测体系的本质，是把“模型和 Agent 好不好”从模糊感觉转化为持续可测、可比、可改进的工程事实。

## 3. 为什么企业 AI 系统必须建设评测体系

### 3.1 AI 系统天然易漂移、易回归

只要发生以下变化，质量都可能变化：

- 模型切换
- prompt 调整
- tool schema 变更
- 知识索引更新
- runtime 策略调整
- 权限和 guardrails 策略变化

如果没有评测，团队很难知道变好还是变差。

### 3.2 RAG、Agent、Tool 系统是多模块组合体

最终效果不仅取决于模型，还取决于：

- query understanding
- recall
- rerank
- context building
- tool selection
- planner
- HITL
- action execution

因此评测不能只看最终答案。

### 3.3 近两年的行业共识正在从“Prompt 调优”转向“Evals 驱动迭代”

Anthropic 在 2026 年的 agent evals 文章中明确强调：

真正成熟的团队会把 evals 当作定义能力、发现回归、加速模型切换和产品迭代的核心基础设施，而不是开发末尾的补充工作。

## 4. 评测体系要解决什么问题

### 4.1 定义成功标准

先明确什么叫“答对了”“做成了”“风险可接受”。

### 4.2 发现质量问题

在上线前或变更后发现回归与失败模式。

### 4.3 指导优化方向

知道问题是在：

- 检索
- 规划
- 工具使用
- 输出约束
- 审批链路

哪个环节。

### 4.4 建立发布门禁

为模型升级、prompt 改动、索引切换提供可量化判断依据。

## 5. 近两年的关键变化

### 5.1 从单轮问答评测转向多轮 Agent 评测

传统评测更多聚焦单轮 prompt-response；现在越来越多系统需要评估：

- 多轮轨迹
- tool use
- state transition
- 最终环境 outcome

### 5.2 从单一分数转向分层评测

更成熟的系统不会只看一个总体得分，而会拆开看：

- 召回质量
- 路由正确率
- 工具成功率
- 动作完成率
- 幻觉率
- 审批触发准确率

### 5.3 线上监控与离线评测正在结合

离线评测适合回归测试，线上监控适合发现分布漂移和真实用户问题。两者缺一不可。

## 6. 推荐的评测体系分层

建议至少分成五层：

### 6.1 基础能力评测

针对：

- 模型回答
- 结构化输出
- reasoning / routing

### 6.2 RAG 评测

针对：

- query rewrite
- recall
- rerank
- context quality
- grounded answer

### 6.3 Tool / Agent 评测

针对：

- tool selection
- parameter generation
- planner quality
- task completion
- action safety

### 6.4 在线质量监控

针对真实线上请求的：

- success rate
- latency
- escalation rate
- user feedback
- abnormal traces

### 6.5 人工复核与反馈闭环

针对主观质量、边缘案例和高风险任务的最终校准。

## 7. 推荐的评测方法组合

### 7.1 规则型打分器

适合：

- 格式校验
- schema 合法性
- tool 参数验证
- 状态结果验证

### 7.2 模型型打分器

适合：

- 回答质量
- 覆盖度
- 解释质量
- 主观任务评分

### 7.3 环境结果打分

尤其适合 Agent / Tool / Action：

- 任务是否真的完成
- 环境状态是否真的变化到目标结果

### 7.4 人工评分与抽检

用于：

- 校准 model grader
- 检查高风险输出
- 处理复杂主观任务

## 8. 能力评测与回归评测的区别

### 8.1 能力评测

关注：

- 这个系统现在能不能处理更复杂的任务
- 上限在哪里

### 8.2 回归评测

关注：

- 原来能做好的事情现在是否退化了

更成熟的团队通常同时维护：

- capability suite
- regression suite

## 9. 评测体系与发布流程的关系

建议将评测接入：

- prompt 变更评审
- 模型升级评审
- 索引切换评审
- tool 版本升级评审
- runtime 策略发布门禁

换句话说：

评测结果不应只停留在报表，而应影响是否允许发布。

## 10. 面向低代码平台的重点评测对象

### 10.1 组件知识问答

关注：

- groundedness
- citation 正确性
- 推荐组件相关性

### 10.2 Schema 生成

关注：

- schema 合法性
- 组件搭配合理性
- 数据绑定正确率

### 10.3 配置诊断 Agent

关注：

- 问题定位准确率
- 修复建议有效率
- 自动修复安全性

### 10.4 运维分析与动作建议

关注：

- 证据链完整性
- 根因分析质量
- 是否错误建议高风险动作

## 11. 常见误区

### 11.1 只靠人工试用判断效果

很难发现回归，也不具备可比性。

### 11.2 只有总分，没有分层指标

无法定位问题来源。

### 11.3 只做离线评测，不做线上监控

真实用户分布和内部测试分布往往不同。

### 11.4 只做线上监控，不建离线评测集

无法稳定做回归测试和模型切换对比。

### 11.5 没有把评测结果接入发布门禁

那评测体系很难真正发挥治理作用。

## 12. 一句话总结

评测体系总览的核心，是为 AI 系统建立一套从离线任务集、自动打分、环境结果验证到线上监控与人工校准的全生命周期质量闭环；近两年的最佳实践也越来越明确，真正能持续进化的 AI 系统，不是只会调 Prompt，而是会围绕 Evals 和 Monitoring 迭代。
