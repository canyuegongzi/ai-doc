# AI应用总体架构

> 状态：第一版草案

## 1. 文档目标

本文用于定义 AI 应用知识库中的总体架构主线，统一后续 RAG、Agent、模型网关、知识工程、Tool 系统、评测治理等专题文档的视角。

该架构不是面向单点 Demo，而是面向企业内部可复用、可治理、可演进的 AI 应用平台，适用于问答、搜索推荐、内容生成、流程协作、诊断修复、低代码平台智能助手等场景。

## 2. 行业主流架构趋势

结合近一阶段行业主流实践，AI 应用架构正在呈现以下共识：

1. 从“单模型调用”走向“系统化编排”。
AI 应用不再只是 Prompt + LLM，而是由模型、检索、工具、工作流、状态管理、观测与治理共同构成。

2. 从“通用聊天”走向“场景化 AI 系统”。
企业更关注具体任务闭环，例如知识问答、表单生成、工单处理、运维诊断、配置修复，而不是开放式聊天本身。

3. 从“自由 Agent”走向“受控 Agent / Workflow”。
主流工程实践越来越强调可控性、状态机、审批、人机协作、幂等执行和错误恢复，而不是完全自治。

4. 从“向量检索”走向“完整 RAG 链路”。
RAG 的重点不再只是 embedding 和向量库，而是 query 理解、混合召回、重排、上下文构建、引用和评测。

5. 从“效果优先”走向“治理优先”。
生产级 AI 系统必须具备权限控制、租户隔离、成本治理、审计、追踪、评测与回放能力。

6. 从“单体能力”走向“平台化沉淀”。
企业会逐步形成统一的模型网关、知识处理管线、检索引擎、工具注册中心、Agent Runtime 和评测中心。

## 3. 总体设计目标

AI 应用总体架构的目标不是追求最复杂的智能，而是建立一套：

- 可复用：同一套基础设施支持多个业务场景复用。
- 可扩展：能够从 FAQ 助手演进到 RAG、Workflow Agent、多工具协作。
- 可治理：具备权限、审计、成本、评测、观测、风险控制能力。
- 可集成：方便接入企业现有系统、知识源、低代码平台与业务 API。
- 可演进：允许从规则流程逐步演进到更强的 Agent 化架构。

## 4. 总体分层架构

建议将 AI 应用总体架构划分为七层，两侧由横向治理能力贯穿。

### 4.1 七层架构

#### L1. 交互与应用入口层

负责承接真实业务入口与用户交互。

典型入口包括：

- AI 助手
- 搜索框
- 页面生成面板
- 低代码设计器智能入口
- 运维控制台智能入口
- API / SDK / Webhook

这一层的重点不是智能本身，而是统一请求上下文，包括用户身份、租户信息、业务空间、会话上下文、权限边界、调用来源等。

#### L2. 应用编排层

负责根据场景选择合适的处理模式。

典型编排模式包括：

- 直接问答
- 检索增强问答
- 生成辅助流程
- Tool 调用流程
- Workflow Agent
- 多步骤审批流程

这一层是业务场景和底层能力的衔接层，负责确定本次请求是走 Prompt 直答、RAG 流程、工具执行，还是进入 Agent Runtime。

#### L3. Agent / Workflow Runtime 层

负责执行复杂任务的状态管理和步骤编排。

核心能力包括：

- 目标理解
- 计划生成
- 工具选择
- 执行状态机
- 中间结果记录
- 人工确认节点
- 失败恢复与重试
- 长流程任务管理

这一层是“行动中枢”，适合承接多步骤、带副作用、需要审计和回放的任务。

#### L4. 检索与认知增强层

负责让模型具备业务知识、上下文补充和事实约束能力。

核心模块包括：

- Query Rewrite
- Query Expansion
- 意图识别
- 多路召回
- Hybrid Search
- Rerank
- Context Builder
- Citation / 来源保留

这一层是 RAG 的核心，不只是“查向量”，而是负责把原始查询转换成高质量上下文。

#### L5. Tool / Action 能力层

负责把 AI 从“能回答”扩展到“能做事”。

典型工具类型包括：

- 检索工具
- 查询工具
- 配置校验工具
- 页面生成工具
- 发布工具
- 运维工具
- 通知工具
- 数据处理工具

这一层需要清晰的协议、参数 Schema、风险分级、权限校验、幂等设计和调用审计。

#### L6. 知识与数据处理层

负责将企业数据加工成 AI 可消费的知识资产。

典型能力包括：

- 数据源接入
- 文档解析
- 清洗标准化
- Chunk 切分
- Metadata 抽取
- 向量化
- 索引构建
- 增量更新
- 权限同步

这一层是 RAG 和知识增强的基础，重点在知识工程质量，而不是单一存储组件。

#### L7. 模型接入与推理层

负责统一接入各类模型能力。

包括：

- 通用大模型
- Embedding 模型
- Rerank 模型
- 多模态模型
- 语音模型
- 结构化输出能力

核心职责包括：

- 多供应商接入
- 模型路由
- 成本统计
- 限流与重试
- 熔断与降级
- Fallback 策略

这一层相当于 AI 时代的模型网关和推理基础设施。

### 4.2 横向治理底座

除七层主链路外，还需要一组横向治理能力贯穿全局：

- 身份与权限治理
- 租户隔离
- Prompt 资产管理
- 评测体系
- 观测与 Trace
- 成本与配额治理
- 审计与回放
- 安全策略与风控

## 5. 核心调用链路

从请求进入到结果返回，主流 AI 应用通常遵循以下链路：

1. 用户从业务入口发起请求。
2. 应用编排层识别请求类型和场景模式。
3. 若为知识型请求，进入检索增强流程。
4. 若为执行型请求，进入 Agent / Workflow Runtime。
5. Runtime 按需调用检索、模型和工具能力。
6. 系统在执行过程中持续记录状态、轨迹、成本和中间结果。
7. 输出在返回前经过结构化校验、权限校验和风险控制。
8. 最终结果返回业务入口，并沉淀日志、反馈和评测样本。

可以用一句话概括：

业务请求先被识别为某种“任务模式”，再由编排层选择检索、生成、执行等能力进行组合，最终在治理体系约束下完成交付。

## 6. 四类核心能力域

从总体架构视角，AI 应用主要围绕四个能力域建设。

### 6.1 认知增强能力

解决“模型不知道”的问题。

对应能力：

- 知识处理
- RAG
- 多源检索
- 引用与事实约束

### 6.2 执行协作能力

解决“模型不能稳定做事”的问题。

对应能力：

- Tool Calling
- Workflow
- Agent Runtime
- 人机协同

### 6.3 输出约束能力

解决“模型输出不稳定、不可接入”的问题。

对应能力：

- 结构化输出
- JSON Schema
- DSL 生成
- 结果校验
- Fallback 修复

### 6.4 工程治理能力

解决“系统不可运营、不可审计、不可优化”的问题。

对应能力：

- Trace
- 评测
- 成本分析
- 权限控制
- 审计回放
- 指标监控

## 7. 主流架构模式映射

当前行业主流前沿架构，大致可以归纳为以下几类模式：

### 7.1 Prompt Application

最轻量模式，适合简单改写、摘要、分类、抽取。

特征：

- 流程短
- 无复杂状态
- 无外部工具
- 依赖 Prompt 和结构化输出约束

### 7.2 RAG Application

适合知识问答、文档搜索、推荐引导、生成辅助。

特征：

- 重点在知识工程和检索质量
- 强依赖 Metadata、召回策略和上下文治理
- 通常需要评测闭环

### 7.3 Workflow AI Application

适合流程明确、步骤可控、需审计的企业场景。

特征：

- 采用预定义节点或有边界的计划机制
- 模型只负责局部决策
- 易监控、易调试、易审批

### 7.4 Agentic Application

适合复杂任务、多工具协作、长链路执行。

特征：

- 依赖状态机和任务管理
- 需要记忆、恢复、回放和人机介入
- 必须有风险治理和权限约束

## 8. RAG 与 Agent 的架构关系

在总体架构中，RAG 和 Agent 不是并列孤立的两个系统，而是互相协作的两类能力。

- RAG 负责知识增强，是 Agent 的认知底座。
- Agent 负责行动执行，是 RAG 的动作扩展。
- Workflow 负责把 RAG、模型、Tool、人工确认串联为稳定流程。

因此更合理的企业级形态不是“单独做一个 RAG 系统，再单独做一个 Agent 系统”，而是：

- 以知识工程和检索引擎构建统一认知底座
- 以 Tool 中心和 Runtime 构建统一执行底座
- 由应用编排层按场景进行组合

## 9. 面向企业落地的关键设计原则

### 9.1 先平台化，再场景化

先统一模型网关、检索链路、工具协议、评测与观测，再落具体业务场景，避免能力碎片化。

### 9.2 先 Workflow，后自治 Agent

优先建设可控工作流和受限工具调用，再逐步开放更复杂的 Agent 自主决策能力。

### 9.3 先知识工程，后复杂推理

企业场景效果不佳，往往根因在知识质量、检索质量和上下文构造，而不是推理链不够长。

### 9.4 先治理闭环，后规模扩张

没有评测、观测、权限和审计体系的 AI 应用，很难进入生产级扩展阶段。

### 9.5 先低风险场景，后高风险自动化

先从问答、推荐、草稿生成、配置校验等低风险能力切入，再进入发布、审批、写入、通知等高风险动作。

## 10. 面向低代码平台的架构映射

结合低代码平台场景，该总体架构可优先映射到以下方向：

- 组件知识问答：以知识工程 + RAG 为核心。
- Schema 生成增强：以检索增强 + 结构化输出为核心。
- 配置纠错助手：以领域知识 + 校验工具 + Workflow 为核心。
- 逻辑编排助手：以 Planner + Tool + 结构化 DSL 生成 为核心。
- 平台运维助手：以检索、查询工具、风险控制和审计为核心。

这意味着低代码 AI 的重点不是做一个泛化聊天框，而是构建一个围绕“页面、组件、Schema、数据源、流程、发布、运维”展开的领域智能系统。

## 11. 推荐的核心子系统

为了支撑上述总体架构，建议逐步沉淀以下平台子系统：

1. Model Gateway
2. Prompt Center
3. Knowledge Pipeline
4. Retrieval Engine
5. Tool Registry / Action Center
6. Agent Runtime / Workflow Engine
7. Evaluation & Observability Center
8. AI Application Layer

这八类子系统共同构成企业级 AI 平台的核心骨架。

## 12. 建议的建设顺序

### Phase 1：模型与知识底座

先建设：

- 模型网关
- 文档处理管线
- 向量化与索引
- 基础 RAG 链路
- Prompt 模板体系
- Trace 与日志

### Phase 2：高价值 RAG 场景

优先建设：

- 组件知识问答
- Schema 生成增强
- 文档搜索与推荐

### Phase 3：工具化与 Workflow 化

逐步建设：

- Tool 协议
- Tool 注册中心
- 校验工具
- 查询工具
- Workflow Agent

### Phase 4：治理与评测闭环

重点建设：

- 离线评测集
- 在线监控
- 成本分析
- 幻觉分析
- 审计回放

### Phase 5：平台化与多场景复用

最终形成统一 AI 能力平台，并在多个业务场景中复用。
