# 建设原则

> 状态：第一版草案

## 1. 文档定位

本文档用于明确 AI 架构知识库与 AI 应用基础设施建设过程中应当遵循的核心原则。

如果说 [AI架构建设目标.md](D:/low-code/ai-doc/AI-Knowledge-Base/00-总览与路线/AI架构建设目标.md) 回答的是“为什么建设”，[AI应用总体架构.md](D:/low-code/ai-doc/AI-Knowledge-Base/00-总览与路线/AI应用总体架构.md) 回答的是“整体如何设计”，那么本文档回答的是“建设过程中应当坚持什么取向、如何做关键取舍、如何避免常见偏差”。

建设原则的意义在于，为后续知识沉淀、架构设计、工程实现和场景落地提供一致的方法论约束。

## 2. 为什么需要建设原则

AI 相关建设天然容易出现以下问题：

- 被热点牵着走，方向不断漂移
- 概念很多，但没有形成稳定分层
- 快速做出 Demo，但难以复用和治理
- 过早追求复杂 Agent，忽略基础设施建设
- 重模型能力，轻知识工程、轻安全治理
- 文档很多，但结构混乱、难以复用

因此，需要用一组稳定原则来统一判断标准，确保这套知识体系与后续能力建设可以持续演进，而不是碎片化堆积。

## 3. 总体原则

这套 AI 架构建设，建议以一句话概括总体原则：

以业务价值为牵引，以平台能力为底座，以知识工程和受控执行为核心，以治理闭环为保障，采用渐进演进方式沉淀企业级 AI 能力体系。

这句话可以拆成若干条具体原则。

## 4. 业务导向原则

### 4.1 以业务问题为起点，而不是以技术热点为起点

AI 架构建设首先应回答“业务痛点是什么”，而不是“现在流行什么框架、什么模型、什么 Agent 玩法”。

后续所有能力沉淀，都应尽量映射到具体问题，例如：

- 如何让平台用户快速理解组件能力
- 如何辅助生成更稳定的页面 Schema
- 如何帮助定位配置错误和发布问题
- 如何提升内部研发和运维效率

如果脱离问题域谈 AI 架构，最终很容易变成技术名词堆积。

### 4.2 以高价值场景优先，而不是追求全面铺开

应优先选择高频、标准化程度较高、风险可控、收益明确的场景切入，例如：

- FAQ 问答
- 组件知识检索
- Schema 生成增强
- 配置纠错助手
- 平台运维问答

先形成可验证价值，再逐步扩展复杂场景。

### 4.3 以真实落地为标准，而不是以概念完备为标准

判断一项能力是否应优先建设，不取决于它在概念上是否高级，而取决于它是否能进入真实使用链路、产生稳定价值、形成复用资产。

## 5. 分层设计原则

### 5.1 先建立清晰分层，再展开细节

AI 应用是一套系统工程，必须先建立总体分层，再分别沉淀模型、知识、检索、工具、执行、治理等专题。

建议始终维持以下层次感：

- 概念层
- 架构层
- 工程层
- 领域层
- 案例层

分层不清，后续知识和实现都会互相缠绕。

### 5.2 先定义边界，再定义能力

每一层都需要先明确自身解决什么、不解决什么，再展开能力建设。

例如：

- 模型网关负责统一接入与路由，不负责业务语义理解
- 知识工程负责知识加工，不直接负责问答效果本身
- Tool 体系负责动作标准化，不负责复杂任务整体规划
- Agent Runtime 负责任务闭环，不应越俎代庖承载所有治理职责

### 5.3 先抽象共性，再沉淀场景特性

基础能力应优先沉淀可复用的通用抽象，例如：

- Model 接口
- Retrieval Pipeline
- Tool 协议
- 状态机模型
- Trace 结构

在此基础上，再按低代码平台场景补领域特性。

## 6. 平台化原则

### 6.1 优先建设共用底座，而不是重复造场景轮子

多个 AI 场景不应各自独立建设一套模型接入、一套知识处理、一套工具调用逻辑。

应优先沉淀共用能力，例如：

- Model Gateway
- Prompt Center
- Knowledge Pipeline
- Retrieval Engine
- Tool Registry
- Agent Runtime
- Evaluation & Observability Center

### 6.2 优先形成能力中心，而不是散点实现

每新增一个 AI 场景时，都应优先考虑：

- 是否复用已有知识管线
- 是否复用已有检索链路
- 是否复用已有 Tool 协议
- 是否复用已有 Trace 与评测能力

避免每个场景都有一套私有实现。

### 6.3 平台能力要做到可替换、可扩展、可治理

平台化不是把功能堆在一起，而是让能力具备明确边界、统一接口和可演进空间。

## 7. 知识工程优先原则

### 7.1 先建设知识工程，再追求复杂智能

在企业场景中，很多效果问题首先不是模型能力不足，而是知识工程不到位。

常见根因包括：

- 文档没有清洗好
- Chunk 切分不合理
- Metadata 缺失
- 权限没有同步
- 索引更新不及时
- 上下文构造混乱

因此，知识工程是高优先级基础设施，而不是附属模块。

### 7.2 RAG 的重点不在“上向量库”，而在完整检索链路

应避免把 RAG 简化为 embedding + vector store。

RAG 真正需要关注的是：

- Query 理解
- 多路召回
- 重排
- 上下文治理
- Citation
- 评测

### 7.3 领域知识质量优先于泛化知识规模

对于低代码平台这类场景，真正决定效果的是：

- 组件协议是否被结构化沉淀
- Schema 模型是否被清晰描述
- 页面和物料样本是否可检索
- 常见错误和经验是否被沉淀

不是简单堆更多外部资料。

## 8. 受控执行原则

### 8.1 先 Workflow，后自治 Agent

在企业环境中，早期应优先建设边界清晰、流程稳定、可审计的 Workflow 能力，而不是直接追求开放式自治 Agent。

原因包括：

- 更稳定
- 更可解释
- 更容易调试
- 更容易接入人工确认
- 更适合企业权限控制与审批体系

### 8.2 Tool 必须标准化，而不能随意拼接

Tool 不是简单函数列表，而是 AI 执行系统的动作协议。

每个 Tool 都应具备：

- 名称
- 描述
- 输入参数 Schema
- 输出定义
- 风险等级
- 权限边界
- 超时与重试策略
- 审计信息

### 8.3 高风险动作必须显式加控制点

对于写入、发布、删除、通知、审批等动作，应优先考虑：

- 人工确认
- 二次校验
- 审批流
- 沙箱执行
- 幂等保障

## 9. 治理优先原则

### 9.1 没有治理闭环的 AI 系统，不应视为生产级系统

AI 能力如果没有以下配套机制，就只能视为实验能力：

- Trace
- 评测
- 监控
- 成本分析
- 审计
- 权限控制

### 9.2 先建立可观测性，再扩大使用范围

在大规模推广前，应确保至少能够看到：

- 请求链路
- 检索结果
- 模型输入输出
- Tool 调用过程
- 错误节点
- 成本消耗

### 9.3 先建立评测机制，再持续优化

没有评测集和指标体系，优化只能依赖体感和个例，容易误判方向。

### 9.4 先建立安全边界，再开放能力边界

权限、租户隔离、敏感信息保护、风险动作审批应前置设计，而不是上线后补救。

## 10. 结构化输出原则

### 10.1 优先面向结构化结果设计，而不是自由文本结果

只要结果需要被系统消费，就应优先追求：

- JSON Schema
- DSL
- 类型约束
- 明确字段定义
- 结果校验机制

### 10.2 生成只是第一步，校验和修复同样重要

生成结果必须进入后续校验链路，包括：

- 格式校验
- 协议校验
- 业务规则校验
- 自动修复或回退

### 10.3 能约束就不要完全依赖 Prompt 自觉

Prompt 很重要，但不应把所有稳定性都寄托在 Prompt 上。只要能通过协议、Schema、函数调用、规则引擎显式约束，就应优先使用显式约束机制。

## 11. 演进式建设原则

### 11.1 先搭骨架，再填内容

无论是知识库建设还是系统建设，都应优先搭好分层骨架与统一模板，再逐步补内容。

### 11.2 先从低风险场景起步，再逐步扩展

建议优先顺序：

- 问答与检索
- 推荐与生成草稿
- 校验与诊断
- 低风险执行
- 高风险执行

### 11.3 先形成最小闭环，再扩展复杂能力

相比一开始追求大而全，更有效的方式是先跑通一个小闭环，例如：

- 文档采集 -> RAG 问答
- 检索增强 -> Schema 生成
- 错误信息 -> 诊断建议

### 11.4 允许迭代演进，不追求一次性定型

AI 技术变化快，架构应保持模块化和可替换性，允许随着场景增长和技术变化不断修正。

## 12. 文档沉淀原则

### 12.1 文档要服务于设计与实现，而不是只服务于阅读

每篇文档都应尽量回答以下一类问题：

- 这个概念到底是什么
- 这个模块为什么存在
- 这个能力应该如何设计
- 这个问题在工程上如何落地
- 这个场景为什么这么拆

### 12.2 文档要结构统一，便于后续复用和抽取

文档应尽量使用统一模板，例如概念模板、架构模式模板、工程设计模板、案例复盘模板。

### 12.3 文档要保留边界与取舍，而不是只写正向描述

一篇真正有价值的文档，不只是写“应该怎么做”，还应写清楚：

- 不适用场景
- 风险点
- 权衡关系
- 常见误区
- 失败经验

### 12.4 文档要兼顾人读与机读

当前主要面向人类阅读和沉淀，但应尽量保持结构清晰，便于未来做：

- 标签化
- 结构化抽取
- 全文检索
- 向量索引
- AI 知识助手接入

## 13. 低代码场景特化原则

### 13.1 通用 AI 能力必须映射到平台领域对象

不能只讲通用 RAG、通用 Agent，而要把能力具体映射到：

- 页面 Schema
- 组件物料
- 插槽机制
- Setter 协议
- 数据源模型
- 逻辑编排 DAG
- 发布与运维流程

### 13.2 优先建设“平台智能助手”，而不是“泛聊天助手”

低代码场景真正有价值的方向，是围绕平台对象和平台流程提供智能辅助。

### 13.3 优先建设“理解 + 生成 + 校验”闭环

与其做纯粹生成，不如优先做：

- 理解平台规则
- 生成结构化结果
- 用规则和工具做校验
- 在必要时给出修复建议

这条路线更稳，也更符合平台型系统特点。

## 14. 判断一项建设是否值得优先推进的标准

后续无论是写文档还是做能力建设，都可以用以下问题做判断：

1. 它是否解决真实业务问题。
2. 它是否能够被多个场景复用。
3. 它是否有助于后续平台化沉淀。
4. 它是否具备明确边界和治理方式。
5. 它是否能够形成可验证的价值闭环。

如果这五点中大部分都不能满足，就不应作为高优先级建设项。

## 16. 一句话总结

AI 架构建设应坚持业务导向、分层清晰、平台复用、知识工程优先、受控执行、治理先行、结构化输出和渐进演进的原则，最终沉淀成一套真正能落地、能复用、能持续演进的企业级 AI 能力体系。
