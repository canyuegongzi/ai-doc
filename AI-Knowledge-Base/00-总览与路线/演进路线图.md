# 演进路线图

> 状态：第一版草案

## 1. 文档定位

本文档用于描述 AI 架构与知识库建设的阶段性演进路线，回答“应该按什么顺序建设、每个阶段重点做什么、阶段之间如何衔接”这几个问题。

如果说 [AI架构建设目标.md](D:/low-code/ai-doc/AI-Knowledge-Base/00-总览与路线/AI架构建设目标.md) 回答的是“为什么做”，[建设原则.md](D:/low-code/ai-doc/AI-Knowledge-Base/00-总览与路线/建设原则.md) 回答的是“应该怎么做”，那么本文档回答的是“应该先做什么、后做什么、如何逐步演进到目标形态”。

本文档既适用于知识库建设节奏规划，也适用于后续 AI 能力平台的工程落地节奏规划。

## 2. 为什么需要演进路线图

AI 相关建设很容易出现两个极端：

- 过于保守，只停留在概念层和文档层，迟迟不进入真实场景。
- 过于激进，一上来就想同时建设模型网关、RAG、Agent、评测平台和多个业务场景，最终失去重点。

演进路线图的作用，就是帮助在“认知建设”和“工程落地”之间建立合理节奏，做到：

- 先打基础，再做扩展
- 先验证价值，再做平台化
- 先低风险场景，再高风险执行
- 先形成闭环，再扩大范围

## 3. 总体演进思路

这套 AI 架构建议采用“六阶段递进式演进”路径：

1. 知识骨架阶段
2. 基础能力阶段
3. 场景验证阶段
4. 工具与受控执行阶段
5. 治理闭环阶段
6. 平台化扩展阶段

这六个阶段并不是绝对串行的，但在优先级上应遵循由浅入深、由稳到难的顺序。

## 4. Phase 0：知识骨架阶段

### 4.1 阶段目标

先完成 AI 知识库骨架、总纲结构和统一模板，建立一套可持续扩展的知识组织方式。

### 4.2 重点工作

- 建立知识库目录结构
- 建立模板体系
- 补齐总纲文档
- 建立统一术语与分类框架
- 明确 AI 应用总体架构和问题域

### 4.3 阶段产出

- 结构完整的知识库骨架
- `00-总览与路线` 总纲文档集
- 核心模板文件
- 第一版知识地图

### 4.4 阶段价值

这一阶段的价值不在于直接产出 AI 能力，而在于把后续的认知、架构和实现建立在统一骨架上，避免未来内容与方案四处散落。

## 5. Phase 1：基础能力阶段

### 5.1 阶段目标

建立 AI 应用最小可复用底座，重点打好“模型接入 + 知识处理 + 基础 RAG”三项基础能力。

### 5.2 重点工作

- 建立模型接入抽象
- 梳理模型网关设计
- 梳理 Embedding、Rerank、Vision 等能力边界
- 建立知识工程体系
- 梳理文档采集、解析、切分、Metadata、向量化、索引更新
- 梳理基础 RAG 检索链路
- 建立 Prompt 资产沉淀意识
- 建立基础日志与 Trace 记录意识

### 5.3 阶段产出

- `03-模型接入层` 第一批核心文档
- `04-知识工程` 第一批核心文档
- `05-RAG专题` 的总览与主链路文档
- 第一版基础 RAG 架构认知闭环

### 5.4 阶段价值

这一阶段的目标，是先解决“AI 能否知道正确内容”的问题，也就是认知底座问题。

### 5.5 成功标志

- 能清晰描述模型接入边界
- 能清晰描述知识处理链路
- 能清晰描述基础 RAG 架构
- 后续场景已经有统一底座可依赖

## 6. Phase 2：场景验证阶段

### 6.1 阶段目标

围绕少量高价值场景，验证基础能力是否真正产生业务价值。

### 6.2 场景选择原则

优先选择以下类型场景：

- 高频
- 知识密集
- 边界较清晰
- 风险较低
- 便于构造评估标准

### 6.3 建议优先场景

- 组件知识问答
- 平台使用 FAQ
- Schema 生成增强
- 页面与组件推荐
- 低风险配置诊断

### 6.4 重点工作

- 建立场景级方案文档
- 将知识工程与 RAG 映射到具体业务对象
- 梳理低代码平台领域知识
- 补齐 `12-领域知识-低代码平台` 的核心对象文档
- 建立第一批案例复盘文档

### 6.5 阶段产出

- 至少 1 到 2 个高价值 AI 场景设计方案
- 第一批领域知识文档
- 第一批案例与复盘文档
- 初步沉淀业务价值验证依据

### 6.6 阶段价值

这一阶段的核心，不是追求平台完备，而是验证“这套架构是否能在真实场景中产生价值”。

### 6.7 成功标志

- 有明确场景闭环
- 有清晰问题与方案映射
- 有可复用的场景设计模式
- 有实践反馈进入知识库

## 7. Phase 3：工具与受控执行阶段

### 7.1 阶段目标

在基础认知增强能力之上，建设 Tool 体系与受控执行能力，让 AI 从“可答”逐步走向“可做”。

### 7.2 为什么这一阶段放在场景验证之后

如果没有先跑通高价值知识场景，直接做 Agent 往往会出现两个问题：

- 不清楚到底要让 Agent 解决什么问题
- 没有稳定知识底座和领域模型，执行效果很难可靠

因此，应先明确场景和知识，再逐步进入执行闭环建设。

### 7.3 重点工作

- 梳理 Tool 协议
- 设计 Tool 注册中心
- 梳理 Tool 参数 Schema 与风险分级
- 梳理 Workflow Agent 模式
- 设计 Agent 任务模型和状态机
- 定义人工确认、幂等、重试、失败恢复机制

### 7.4 建议优先执行能力

- 查询类工具
- 校验类工具
- 生成草稿类工具
- 低风险配置修复建议

不建议一开始就做高风险自动发布、删除、通知等动作。

### 7.5 阶段产出

- `07-Agent专题` 第一批核心文档
- `08-Tool与Action体系` 第一批核心文档
- Workflow Agent 的架构认知闭环
- 第一版受控执行模式设计

### 7.6 阶段价值

这一阶段让 AI 从“知识服务能力”升级为“受控执行能力”，是从 RAG 应用走向 Agent 应用的关键转折点。

### 7.7 成功标志

- 能清晰定义 Tool 协议
- 能清晰定义 Agent 执行边界
- 能设计受控 Workflow
- 能描述人机协同与风险控制策略

## 8. Phase 4：治理闭环阶段

### 8.1 阶段目标

让 AI 系统从“可运行”进入“可治理、可优化、可审计”的阶段。

### 8.2 重点工作

- 建立离线评测集
- 建立在线质量监控机制
- 建立 Trace 结构与链路追踪能力
- 建立成本分析与配额治理机制
- 建立审计与回放机制
- 梳理权限、租户隔离和敏感信息保护方案

### 8.3 阶段产出

- `09-安全与治理` 的核心文档集
- `10-评测与观测` 的核心文档集
- 治理层方法论闭环

### 8.4 阶段价值

这是从 Demo 级能力迈向生产级系统的关键阶段。

没有这一阶段，AI 能力虽然可以展示，但很难在企业中持续扩展和被信任。

### 8.5 成功标志

- 能追踪请求全链路
- 能衡量效果变化
- 能识别成本消耗
- 能回放关键执行过程
- 能对风险动作进行控制与审计

## 9. Phase 5：平台化扩展阶段

### 9.1 阶段目标

在前几阶段能力逐步成熟后，将零散能力抽象为平台级共用服务，支撑多个业务线或多个场景复用。

### 9.2 重点工作

- 抽象 Model Gateway
- 抽象 Prompt Center
- 抽象 Knowledge Pipeline
- 抽象 Retrieval Engine
- 抽象 Tool Registry / Action Center
- 抽象 Agent Runtime / Workflow Engine
- 抽象 Evaluation & Observability Center

### 9.3 阶段产出

- 平台级能力清单
- 统一接口与协议
- 跨场景复用机制
- 多业务线适配策略

### 9.4 阶段价值

这一阶段的目标不是多做几个场景，而是把能力真正从“项目能力”升级为“平台能力”。

### 9.5 成功标志

- 多个场景共享统一底座
- 新场景接入成本显著下降
- 关键能力具备标准接口和边界
- 平台资产可以持续复用与扩展

## 10. Phase 6：持续演进与资产化阶段

### 10.1 阶段目标

让知识库、平台能力和场景方案形成持续演进的资产体系，而不是一次性建设成果。

### 10.2 重点工作

- 持续补齐案例与复盘
- 持续完善术语表与分类法
- 持续更新领域知识
- 根据实践反馈修正架构模式
- 逐步探索更复杂的 Agent 协作模式
- 逐步建设更丰富的自动化与智能协同能力

### 10.3 阶段产出

- 稳定增长的知识资产库
- 稳定复用的 AI 架构模式库
- 更成熟的领域智能能力体系

### 10.4 阶段价值

这一阶段的意义，在于确保这套体系具备长期生命力，可以随着业务、组织和技术变化持续演进。

## 11. 各阶段之间的依赖关系

### 11.1 Phase 0 是所有阶段的认知基础

没有知识骨架，后续内容和方案会持续散乱。

### 11.2 Phase 1 是 Phase 2 和 Phase 3 的基础

没有模型接入、知识工程和基础 RAG，就很难形成可靠场景，也很难让 Agent 真正具备领域理解能力。

### 11.3 Phase 2 提供真实问题验证

没有场景验证，后续 Tool 和 Agent 很容易脱离实际需求。

### 11.4 Phase 3 是从“可答”到“可做”的能力跃迁

它建立执行闭环，但执行越强，对治理要求越高。

### 11.5 Phase 4 决定系统能否真正推广

没有治理闭环，系统很难规模化进入生产环境。

### 11.6 Phase 5 和 Phase 6 是长期价值放大器

它们决定的是体系的复用能力和持续生命力。

## 13. 推荐的场景演进顺序

从业务场景角度，建议采用由低风险到高风险的演进顺序。

### 13.1 第一阶段场景

- FAQ 问答
- 组件知识问答
- 文档检索与推荐

### 13.2 第二阶段场景

- Schema 生成增强
- 页面和组件推荐
- 配置诊断建议

### 13.3 第三阶段场景

- Tool 辅助查询
- 逻辑编排草稿生成
- 配置修复建议

### 13.4 第四阶段场景

- 受控执行 Workflow
- 人工确认下的低风险操作
- 运维辅助执行链路

### 13.5 第五阶段场景

- 更复杂的多工具协作
- 更复杂的跨系统 Agent 协同
- 更高成熟度的平台智能助手

## 14. 路线推进中的关键风险

### 14.1 过早追求大而全

容易导致文档写很多、能力做很多，但没有一个稳定闭环真正跑通。

### 14.2 过早追求开放式 Agent

容易在知识、工具、治理都不成熟时，把复杂性提前引入系统。

### 14.3 只关注技术能力，不验证场景价值

会导致架构看起来完整，但缺少实际业务牵引。

### 14.4 治理建设滞后

会导致能力越扩展，风险越大，后续补治理成本更高。

### 14.5 领域知识沉淀不足

会导致通用 AI 能力无法真正落到低代码平台场景中。

## 15. 路线调整原则

虽然本文给出了推荐路线，但在实际推进时仍应遵守几个调整原则：

### 15.1 可以并行推进，但不能颠倒优先级

例如可以在写 RAG 文档的同时开始补低代码领域知识，但不建议在知识工程还不清晰时先大规模做 Agent。

### 15.2 可以因场景价值调整顺序，但不能破坏基础能力建设

如果某个场景价值极高，可以适当提前验证，但仍应尽量复用基础能力骨架，而不是走临时方案。

### 15.3 每一阶段都要有明确可见产出

不要让阶段目标停留在方向层。每一阶段都应至少产出可见文档、可复用方案、可验证场景或可观测能力。

## 17. 一句话总结

这套 AI 架构应按“先知识骨架、再基础能力、再场景验证、再受控执行、再治理闭环、再平台化扩展、最后持续资产化”的节奏推进，确保每一步都建立在可复用、可验证、可治理的基础之上。
