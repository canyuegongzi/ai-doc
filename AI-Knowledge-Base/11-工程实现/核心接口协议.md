# 核心接口协议

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 平台工程实现中的核心接口协议，重点回答：

- 平台内部哪些接口需要优先标准化
- 为什么接口协议应围绕能力对象而不是围绕底层实现细节设计
- 如何为后续多服务协作、替换、回放、评测和版本治理提供稳定契约

## 2. 为什么接口协议是工程落地关键

前面的知识库已经定义了多个核心能力：

- Chat / Embedding / Rerank / Vision
- Knowledge Pipeline
- Retrieval Engine
- Tool Registry / Tool Execution
- Agent Runtime
- Evals / Trace / Audit

如果这些能力之间没有稳定协议，后期会出现：

- 各服务之间字段语义不一致
- 评测与 replay 无法复用
- 模块替换成本高
- 线上问题难定位

因此，工程上最重要的不是先把功能做多，而是先把核心接口稳定下来。

用一句话概括：

核心接口协议的本质，是把平台内部高频交互沉淀成长期可复用的能力契约。

## 3. 协议设计的基本原则

### 3.1 以能力对象为中心

接口应围绕：

- retrieval request
- tool call request
- agent task request
- eval run request

而不是围绕底层数据库或第三方 SDK 字段直接暴露。

### 3.2 统一 request / response 结构风格

建议所有内部接口都保持：

- metadata
- input / params
- options
- output / result
- errors

相对一致。

### 3.3 支持版本化与扩展

关键协议应包含：

- version
- extensible metadata
- stable enum / status definitions

### 3.4 结构化优先

避免过多自然语言字段作为唯一输入输出。只要后续需要自动消费，就应优先结构化。

## 4. 推荐优先标准化的接口族

建议至少优先定义以下六类接口：

### 4.1 模型调用协议

例如：

- `generateResponse`
- `embedDocuments`
- `rerankCandidates`
- `analyzeVisionInput`

### 4.2 检索协议

例如：

- `retrieveContext`
- `searchKnowledge`
- `buildCitations`

### 4.3 Tool 协议

例如：

- `resolveAvailableTools`
- `invokeTool`
- `simulateAction`

### 4.4 Agent Runtime 协议

例如：

- `startTask`
- `resumeTask`
- `submitHumanDecision`
- `getTaskState`

### 4.5 评测与观测协议

例如：

- `runEvalSuite`
- `recordTraceSpan`
- `submitFeedback`

### 4.6 审计与治理协议

例如：

- `recordAuditEvent`
- `checkPolicy`
- `requestApproval`

## 5. 推荐的统一包络结构

建议内部接口响应统一采用类似结构：

```ts
interface PlatformResponse<T> {
  success: boolean;
  code: string;
  message?: string;
  data?: T;
  error?: {
    type: string;
    detail?: string;
    retryable?: boolean;
  };
  meta?: {
    requestId: string;
    traceId?: string;
    version?: string;
  };
}
```

这样有利于：

- trace 打通
- error 统一处理
- SDK 封装
- replay 与 mock

## 6. 检索协议建议

检索接口应避免只暴露一个 query string，而应至少支持：

- `query`
- `sceneType`
- `filters`
- `tenantContext`
- `topK`
- `retrievalMode`
- `citationRequired`

返回结构建议包含：

- `hits`
- `citations`
- `traceMeta`
- `queryRewriteInfo`

## 7. Tool 调用协议建议

Tool 调用协议建议分为：

### 7.1 发现协议

用于返回可用工具集合与政策裁剪结果。

### 7.2 执行协议

建议至少包含：

- `toolName`
- `input`
- `identityContext`
- `riskContext`
- `dryRun`
- `approvalToken`

返回建议包含：

- `status`
- `output`
- `receipt`
- `warnings`
- `policyHits`

## 8. Agent Runtime 协议建议

Agent Runtime 的关键协议不应只是“问一句，回一句”，而应是任务制。

建议至少支持：

- `startTask(request)`
- `getTask(taskId)`
- `resumeTask(taskId, input)`
- `cancelTask(taskId)`
- `submitHumanDecision(taskId, decision)`

任务对象建议显式包含：

- `taskId`
- `state`
- `currentStep`
- `result`
- `riskStatus`
- `traceId`

## 9. 评测与观测协议建议

建议至少标准化：

- 评测任务定义协议
- trace span 上报协议
- 反馈提交协议
- 指标采样协议

如果这些协议不统一，后续很难形成平台级 eval / trace 资产。

## 10. 面向低代码平台的领域协议建议

建议在通用平台协议之上，再定义低代码领域对象协议，例如：

- `PageSchemaRef`
- `ComponentMaterialRef`
- `DataModelRef`
- `PublishPlan`
- `SchemaPatch`

这样可以让：

- Tool 输入输出更稳定
- Generator 输出更结构化
- 评测与 replay 更容易复用

## 11. 错误协议建议

内部接口应统一错误类型，至少区分：

- validation error
- auth error
- policy denied
- not found
- retryable infra error
- execution conflict
- human approval required

错误协议统一后，Runtime、SDK 和前端交互都会显著更稳定。

## 12. 常见误区

### 12.1 直接透传第三方 SDK 协议

后期替换成本会很高。

### 12.2 各服务自行定义字段名和错误码

很快会失去跨服务一致性。

### 12.3 协议只定义 happy path

实际系统更依赖错误、审批、中断和恢复路径。

### 12.4 不做版本字段

后续模型升级、tool schema 变更会非常痛苦。

### 12.5 全靠自然语言字段承载结构信息

会让 Runtime 和评测难以稳定消费。

## 13. 一句话总结

核心接口协议的核心，是把平台内部最重要的能力交互沉淀为统一、版本化、结构化、可回放的工程契约，并优先围绕 retrieval、tool、runtime、eval 和治理对象设计，而不是围绕具体实现细节设计；对 AI 平台来说，稳定协议往往比一时多做几个功能更有长期价值。
