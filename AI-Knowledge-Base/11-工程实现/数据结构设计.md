# 数据结构设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义企业 AI 平台的核心数据结构设计，重点回答：

- 前面提到的模型、知识、检索、工具、任务、审计、评测对象，在工程上应沉淀成哪些核心结构
- 哪些对象适合做稳定主模型，哪些适合作为扩展元数据
- 如何兼顾平台通用性和低代码领域特性

## 2. 为什么要先定义核心数据结构

很多 AI 系统前期能跑，但后期很难维护，一个常见原因是：

- 核心对象语义不稳定
- 不同模块各自有一套字段
- 相同概念在检索、工具、评测、审计里名字不同

因此，数据结构设计的目标不是一次性把所有表都设计完，而是先定义平台级主对象。

用一句话概括：

数据结构设计的本质，是为 AI 平台建立统一的对象语言，让服务、接口、评测、审计都围绕同一套核心对象协作。

## 3. 推荐的核心对象分层

建议把核心对象分成六层：

### 3.1 运行上下文对象

描述一次请求或任务的上下文。

### 3.2 知识对象

描述文档、chunk、索引、citation 等。

### 3.3 Tool / Action 对象

描述工具、动作请求、执行结果。

### 3.4 Agent / Task 对象

描述任务、状态、步骤、HITL、审批。

### 3.5 观测与治理对象

描述 trace、audit、feedback、eval run。

### 3.6 领域对象

描述低代码平台相关对象。

## 4. 推荐的平台级主对象

建议优先稳定以下对象：

### 4.1 `RuntimeContext`

用于贯穿请求身份、租户、场景、trace。

### 4.2 `KnowledgeItem`

用于统一描述可检索知识对象。

### 4.3 `RetrievalResult`

用于承载 recall / rerank 后的候选结果。

### 4.4 `ToolDefinition`

用于描述 Tool 协议与治理属性。

### 4.5 `ToolExecutionRecord`

用于记录一次具体工具执行。

### 4.6 `AgentTask`

用于描述一个可追踪任务实例。

### 4.7 `TraceSpan`

用于描述一段链路执行片段。

### 4.8 `AuditEvent`

用于记录责任和治理事实。

### 4.9 `EvalTask` / `EvalRun`

用于定义评测任务和实际评测运行。

## 5. 运行上下文对象建议

例如：

```ts
interface RuntimeContext {
  requestId: string;
  traceId: string;
  tenantId: string;
  userId?: string;
  appId?: string;
  sceneType: string;
  environment?: 'dev' | 'test' | 'prod';
  metadata?: Record<string, unknown>;
}
```

这个对象建议贯穿：

- retrieval
- tool execution
- agent runtime
- audit / trace

## 6. 知识对象建议

### 6.1 `KnowledgeItem`

建议至少包含：

- `knowledgeId`
- `sourceType`
- `title`
- `content`
- `tenantId`
- `appId`
- `tags`
- `permissionScope`
- `updatedAt`
- `metadata`

### 6.2 `KnowledgeChunk`

建议包含：

- `chunkId`
- `knowledgeId`
- `chunkText`
- `chunkIndex`
- `embeddingRef`
- `citationMeta`

### 6.3 `IndexVersion`

建议包含：

- `indexName`
- `version`
- `status`
- `embeddingModel`
- `chunkStrategy`
- `builtAt`

## 7. 检索对象建议

### 7.1 `RetrievalRequest`

用于承载：

- query
- filters
- topK
- retrievalMode
- sceneType

### 7.2 `RetrievalHit`

建议包含：

- `hitId`
- `sourceId`
- `score`
- `rank`
- `summary`
- `citation`

### 7.3 `RetrievalResult`

建议包含：

- `hits`
- `rewrittenQuery`
- `usedFilters`
- `latencyMs`

## 8. Tool / Action 对象建议

### 8.1 `ToolDefinition`

至少包含：

- `toolName`
- `description`
- `category`
- `inputSchema`
- `outputSchema`
- `riskLevel`
- `authScope`
- `version`

### 8.2 `ToolCall`

用于描述一次意图层调用。

### 8.3 `ToolExecutionRecord`

用于描述一次真实执行，建议包含：

- `toolName`
- `inputSummary`
- `status`
- `receipt`
- `riskLevel`
- `retryCount`
- `policyHits`

### 8.4 `ActionRequest`

建议包含：

- `actionType`
- `targetResource`
- `targetEnvironment`
- `approvalRequired`
- `idempotencyKey`

## 9. Agent / Task 对象建议

### 9.1 `AgentTask`

建议包含：

- `taskId`
- `taskType`
- `goal`
- `state`
- `runtimeContext`
- `riskStatus`
- `result`
- `traceId`

### 9.2 `TaskStep`

建议包含：

- `stepId`
- `taskId`
- `stepType`
- `status`
- `inputRef`
- `outputRef`
- `toolName?`
- `startedAt`
- `finishedAt`

### 9.3 `HumanDecision`

建议包含：

- `decisionId`
- `taskId`
- `decisionType`
- `decision`
- `comment`
- `approverId`

## 10. 观测与治理对象建议

### 10.1 `TraceSpan`

至少包含：

- `traceId`
- `spanId`
- `parentSpanId`
- `spanType`
- `status`
- `latencyMs`
- `costMetrics`

### 10.2 `AuditEvent`

至少包含：

- `auditId`
- `eventType`
- `actor`
- `resource`
- `decision`
- `riskLevel`
- `timestamp`

### 10.3 `FeedbackRecord`

至少包含：

- `feedbackId`
- `traceId`
- `sceneType`
- `feedbackType`
- `feedbackValue`
- `feedbackText`

### 10.4 `EvalTask` / `EvalRun`

用于定义样本与实际跑分结果。

## 11. 低代码领域对象建议

建议额外稳定这些领域对象：

### 11.1 `PageSchemaRef`

描述页面 schema 的定位与版本。

### 11.2 `ComponentMaterialRef`

描述组件物料定义。

### 11.3 `SchemaPatch`

描述自动修复或生成差异。

### 11.4 `PublishPlan`

描述发布建议、风险摘要与审批状态。

这些对象不一定属于平台通用层，但应成为低代码场景的领域标准对象。

## 12. 结构设计建议

### 12.1 核心对象字段保持稳定

不要把频繁变化配置直接塞进主表字段，优先放 metadata / extension。

### 12.2 关键 ID 全链路统一

至少统一：

- requestId
- traceId
- taskId
- tenantId
- userId

### 12.3 时间与状态显式化

所有长任务对象建议显式记录：

- createdAt
- updatedAt
- status
- startedAt
- finishedAt

### 12.4 对高风险对象保留治理字段

例如：

- riskLevel
- approvalRequired
- auditLevel
- policyRef

## 13. 常见误区

### 13.1 每个服务各自定义一套相似对象

后期会导致语义漂移和大量转换代码。

### 13.2 一开始把所有字段都做成强耦合主字段

扩展成本会很高。

### 13.3 trace / audit / eval 对象与主业务对象脱节

后续很难形成真正闭环。

### 13.4 领域对象只存在于 prompt 里，不进入结构化模型

会降低系统可验证性和复用性。

### 13.5 身份与租户字段不是一等属性

安全治理很快会变得脆弱。

## 14. 一句话总结

数据结构设计的核心，是把 AI 平台中真正稳定的运行上下文、知识、检索、工具、任务、观测和领域对象沉淀成统一的数据语言，并让这些对象同时服务于服务协作、评测、审计和回放；对工程落地来说，稳定对象模型往往是平台能否持续演进的根基。
