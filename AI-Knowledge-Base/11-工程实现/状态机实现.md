# 状态机实现

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 Agent Runtime 中的状态机实现方案，重点回答：

- 前面概念层的 Agent 状态机在工程上如何落地
- 如何用显式状态、事件、转移和持久化实现可恢复、可中断、可审计的运行时
- 在低代码平台场景中，如何为问答、生成、诊断、动作执行提供统一状态骨架

## 2. 为什么状态机实现是 Runtime 内核

前面的文档已经说明，生产级 Agent 不应只是模型循环，而应是显式状态系统。

真正到工程实现层时，核心问题变成：

- 状态如何定义
- 转移如何触发
- 失败如何恢复
- 人工节点如何插入
- trace 和 audit 如何挂接

如果这层实现不稳定，Planner、Tool、HITL、Approval 等能力都会变成分散逻辑。

用一句话概括：

状态机实现的本质，是把 Agent Runtime 从一段“流程代码”提升为可管理的任务操作系统。

## 3. 状态机实现要解决什么问题

### 3.1 生命周期显式化

### 3.2 故障恢复与 resume

### 3.3 长任务与人工节点插入

### 3.4 Trace / Audit / Eval 的统一挂载点

## 4. 推荐的状态机实现模型

建议采用：

- 状态（State）
- 事件（Event）
- 转移（Transition）
- 动作（Action）
- 持久化快照（Snapshot）

这五个核心概念。

### 4.1 State

表示任务当前所处阶段。

### 4.2 Event

表示驱动状态变化的触发器。

### 4.3 Transition

定义从一个状态迁移到另一个状态的规则。

### 4.4 Action

状态进入、退出或转移时执行的具体逻辑。

### 4.5 Snapshot

记录某一时刻可恢复的任务状态。

## 5. 推荐的通用状态集合

建议至少支持：

- `INIT`
- `UNDERSTAND`
- `PLAN`
- `RETRIEVE`
- `ACT`
- `OBSERVE`
- `VALIDATE`
- `HUMAN_REVIEW`
- `RETRY`
- `COMPLETE`
- `FAILED`
- `ESCALATE`

不同场景可以在此基础上扩展子状态，但不建议完全失去通用主骨架。

## 6. 推荐的事件类型

建议至少包含：

- `TASK_CREATED`
- `UNDERSTAND_DONE`
- `PLAN_READY`
- `RETRIEVE_DONE`
- `TOOL_EXECUTED`
- `VALIDATION_FAILED`
- `HUMAN_DECISION_RECEIVED`
- `RETRY_REQUESTED`
- `TASK_COMPLETED`
- `TASK_FAILED`

这使状态机不依赖“代码执行顺序”，而依赖显式事件。

## 7. 推荐的状态对象结构

一个运行中的任务状态建议包含：

```ts
interface TaskStateSnapshot {
  taskId: string;
  state: string;
  stepIndex: number;
  contextRef?: string;
  currentPlan?: unknown;
  latestToolResultRef?: string;
  retryCount: number;
  riskStatus?: string;
  traceId: string;
  updatedAt: string;
}
```

关键点是：

- 不把所有中间数据都塞进状态对象
- 只保留恢复所需最小核心信息
- 大对象通过 ref 引用外部存储

## 8. 状态转移实现建议

建议每条转移规则显式声明：

- from state
- event
- guard condition
- action
- to state

简化示例：

```ts
{
  from: 'PLAN',
  event: 'PLAN_READY',
  guard: hasToolAction,
  action: enqueueToolExecution,
  to: 'ACT'
}
```

这样做的价值是：

- 易测试
- 易审计
- 易 replay
- 易插政策规则

## 9. 持久化与恢复建议

### 9.1 持久化粒度

建议在以下时机持久化：

- 状态切换前
- 状态切换后
- 高风险动作前后
- HITL 节点进入前
- 长耗时工具调用前后

### 9.2 恢复策略

恢复时建议：

1. 读取最新 snapshot
2. 恢复当前 state
3. 校验是否存在未完成 side effect
4. 决定继续、重试、查询 receipt 或转人工

### 9.3 避免“恢复即重复执行”

对带副作用动作，恢复时必须先确认是否已执行成功。

## 10. 与 Planner、Tool、HITL 的工程关系

### 10.1 与 Planner 的关系

Planner 负责生成下一步决策；状态机负责允许或拒绝进入某状态。

### 10.2 与 Tool 的关系

Tool 调用通常作为 `ACT` 状态下的 action，被状态机统一调度和记录。

### 10.3 与 HITL 的关系

HITL 应作为正式状态，例如 `HUMAN_REVIEW`，而不是流程外分叉。

## 12. 常见误区

### 12.1 状态和代码步骤完全一一绑定

会导致状态机失去抽象价值。

### 12.2 所有状态都塞大量业务数据

会让恢复与迁移变得臃肿。

### 12.3 没有显式事件层

状态转移就会重新退化成 if/else 流程代码。

### 12.4 不对状态切换做持久化

长任务和人工节点会变得不可靠。

### 12.5 高风险动作没有特殊 guard

容易在错误分支里被直接执行。

## 13. 一句话总结

状态机实现的核心，是把任务运行过程拆成显式状态、事件和转移规则，并通过持久化快照、guard 条件和统一 action 调度支撑恢复、审批、追踪和治理；对 Agent Runtime 来说，状态机不是附属能力，而是让系统真正可控的内核。
