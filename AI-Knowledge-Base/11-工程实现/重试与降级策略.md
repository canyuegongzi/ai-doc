# 重试与降级策略

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 平台中的 Retry & Degradation 策略，重点回答：

- 在模型、检索、工具、Agent 链路中，哪些失败适合重试，哪些更适合降级
- 如何避免“无限重试”与“直接失败”这两种极端
- 如何让重试、熔断、fallback、降级和 HITL 形成协同机制

## 2. 为什么 AI 系统必须显式设计重试与降级

AI 系统天然容易遇到：

- 模型超时
- provider rate limit
- 向量检索超时
- rerank 服务抖动
- tool 执行失败
- 审批节点等待超时
- 结构化输出校验失败

如果没有策略，系统很容易陷入两种极端：

- 一出错就直接失败，体验很差
- 不加区分地反复重试，成本和风险失控

用一句话概括：

重试与降级策略的本质，是让系统在失败时仍然以可控方式继续提供价值，而不是崩掉或乱试。

## 3. 重试与降级分别解决什么问题

### 3.1 重试

解决可恢复的临时失败，例如：

- 瞬时网络异常
- provider 短暂限流
- 短时超时

### 3.2 降级

解决“即使继续重试也未必值得”或“当前条件不允许继续完整能力”的情况，例如：

- 切换低成本模型
- 从 Agent 模式降为建议模式
- 跳过 rerank
- 进入人工处理

## 4. 近两年的关键变化

### 4.1 AI 链路中的失败类型比传统 API 更复杂

因为失败可能来自：

- 模型
- 检索
- 工具
- 结构校验
- 审批
- 状态恢复

### 4.2 行业越来越强调 graceful degradation

成熟系统不追求“所有功能都永远可用”，而是追求：

- 核心价值尽量保留
- 风险边界不被突破
- 成本不会失控

### 4.3 Retry / Fallback / Circuit Breaker / HITL 正在形成协同链路

也就是说：

- 先判断是否可重试
- 再判断是否可 fallback
- 再判断是否需降级
- 再判断是否需人工介入

## 5. 推荐的失败分类

建议至少分成四类：

### 5.1 临时基础设施失败

例如：

- timeout
- transient network error
- rate limit

通常适合有限重试。

### 5.2 可预测业务失败

例如：

- schema validation error
- no relevant docs found
- permission denied

通常不应盲目重试，应直接转降级或反馈。

### 5.3 高风险不确定失败

例如：

- Action 请求超时但不确定是否已执行
- 审批状态不明

通常不应自动重试，应转人工确认或查询 receipt。

### 5.4 系统性失败

例如：

- 某 provider 大面积故障
- 索引服务不可用
- 工具后端整体异常

应优先熔断和降级，而不是逐请求重试。

## 6. 推荐的重试策略

### 6.1 模型调用重试

适用于：

- rate limit
- transient timeout
- transport failure

建议：

- 限制次数
- 指数退避
- 保留 trace
- 必要时切换 fallback model

### 6.2 检索链路重试

适用于：

- search service timeout
- rerank service transient error

建议：

- recall 可短重试
- rerank 失败可降级为不 rerank

### 6.3 Tool 执行重试

必须结合幂等性判断。

- 只读工具可较大胆重试
- 写工具必须看 idempotency / receipt

### 6.4 结构化输出重试

适用于：

- parse error
- schema mismatch

建议：

- 使用原请求上下文 + 明确修复提示
- 限制 1-2 次

## 7. 推荐的降级策略

### 7.1 模型降级

- 高性能模型降到中档模型
- 多模态降到纯文本

### 7.2 检索降级

- 跳过 rerank
- 缩小 topK
- 退化到关键词检索或 FAQ 缓存

### 7.3 Agent 降级

- 从 Tool-Based Agent 降为 Workflow
- 从自动执行降为建议输出
- 从多工具分析降为单工具摘要

### 7.4 动作降级

- 从 execute 降为 dry-run
- 从 auto-fix 降为 patch proposal
- 从直接发布降为生成发布计划

## 8. 推荐的决策顺序

一条较稳的失败处理链路通常是：

1. 识别失败类型
2. 判断是否可重试
3. 判断是否存在安全 fallback
4. 判断是否可接受降级
5. 判断是否需人工介入
6. 记录 trace / audit / metrics

## 9. 与熔断、HITL、成本治理的关系

### 9.1 与熔断的关系

当某类请求持续失败，应进入熔断，避免单请求层面的无意义重试。

### 9.2 与 HITL 的关系

在高风险或状态不明场景下，HITL 往往比继续自动重试更合适。

### 9.3 与成本治理的关系

重试是成本放大器，因此必须受：

- token budget
- step budget
- retry budget

约束。

## 10. 面向低代码平台的典型建议

### 10.1 组件问答

- 检索失败可短重试
- rerank 失败可降级为 direct recall answer

### 10.2 Schema 生成

- 结构化输出失败可短重试
- 校验不过时可降级为“草稿 + warning”

### 10.3 配置诊断

- 规则检索失败可降级为案例检索 + 不确定性说明
- 自动修复不可执行时应降级为 patch proposal

### 10.4 发布与运维动作

- 不确定执行状态时禁止自动重试
- 应转 receipt 查询或 HITL

## 11. 常见误区

### 11.1 所有错误统一三次重试

这既浪费成本，又可能扩大风险。

### 11.2 只做重试，不做降级

一旦长期依赖服务异常，系统体验会非常差。

### 11.3 高风险动作也盲目 fallback

可能导致误执行。

### 11.4 失败后不记录决策路径

后续无法复盘“为什么这样重试或降级”。

### 11.5 降级后不显式告诉用户和下游系统

会造成错误信任。

## 12. 一句话总结

重试与降级策略的核心，是按失败类型、风险等级、幂等性和成本边界设计分层响应机制，让系统在失败时既不过度冲动也不过早放弃，而是尽量以安全、可解释、可控的方式继续提供价值；对 AI 平台来说，真正成熟的稳定性设计不是“永不失败”，而是“失败时知道如何体面地退”。
