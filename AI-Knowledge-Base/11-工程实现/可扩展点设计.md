# 可扩展点设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 平台中的可扩展点设计，重点回答：

- 平台哪些边界应该为后续变化预留扩展点
- 如何在不过度抽象的前提下支持模型、检索、工具、评测、领域能力持续演进
- 结合低代码平台场景，哪些扩展点最值得优先建设

## 2. 为什么要设计可扩展点

AI 平台的变化速度非常快，常见变化包括：

- 模型替换
- 检索策略变化
- Tool 增加与协议升级
- 新增 Agent 模式
- 新增评测器
- 新增低代码领域对象

如果系统没有扩展点，后期会出现：

- 每次变化都改主链路核心代码
- 旧场景被新能力影响
- 评测、治理和审计难以跟上

用一句话概括：

可扩展点设计的本质，是把“未来一定会变的地方”从一开始就从主骨架中抽离出来。

## 3. 扩展点设计的基本原则

### 3.1 先找高变化点，再谈扩展

不是所有模块都要做成插件系统，优先关注真正高变化区域。

### 3.2 核心骨架稳定，局部策略可替换

例如：

- Runtime 主状态机应稳定
- 但 Planner 策略、retrieval policy、tool routing policy 可替换

### 3.3 扩展点必须有协议，不只是“留个回调”

没有明确输入输出契约，扩展点很快会失控。

### 3.4 扩展点要能被观测和治理

插件化之后，仍然需要：

- trace
- audit
- policy check
- version control

## 4. 推荐优先建设的扩展点

建议至少优先考虑以下几类：

### 4.1 模型提供者扩展点

支持：

- OpenAI
- Anthropic
- Qwen
- 自建模型网关

### 4.2 检索策略扩展点

支持：

- dense recall
- hybrid recall
- metadata-first recall
- agentic retrieval policy

### 4.3 Tool 注册与执行扩展点

支持：

- 新工具类型接入
- 新风险策略挂载
- 新 schema validator

### 4.4 Planner / Routing 策略扩展点

支持不同任务和场景采用不同 planning policy。

### 4.5 评测器扩展点

支持新增：

- 规则 grader
- 模型 grader
- 环境 grader

### 4.6 领域对象适配扩展点

用于承接低代码平台中的：

- PageSchema
- ComponentMaterial
- DataModel
- PublishPlan

## 5. 模型层扩展点建议

建议通过统一接口抽象：

- `ChatModel`
- `EmbeddingModel`
- `RerankModel`
- `VisionModel`

扩展点要允许：

- 新 provider 接入
- 路由策略切换
- 配置热更新
- usage / cost 统一上报

## 6. 检索层扩展点建议

建议把以下能力做成可替换策略：

- query rewrite strategy
- recall strategy
- rerank strategy
- context builder strategy
- citation builder strategy

这样可以在不同场景中快速切换而不改主链路。

## 7. Tool / Action 扩展点建议

建议分成两类扩展：

### 7.1 Tool Definition 扩展

允许新增：

- tool category
- schema validator
- policy hook

### 7.2 Tool Execution 扩展

允许新增：

- executor
- result adapter
- receipt resolver
- idempotency policy

## 8. Agent Runtime 扩展点建议

不建议把核心状态机做成任意插件式扩展，但可以开放以下扩展点：

- planner provider
- state guard hook
- approval policy hook
- step evaluator hook
- fallback / degradation hook

这样既保证骨架稳定，又允许局部变化。

## 9. Eval / Observability 扩展点建议

建议支持扩展：

- grader registry
- metric collector
- trace exporter
- feedback classifier
- sample sampler

这是因为评测体系本身也会持续演进。

## 10. 面向低代码平台的领域扩展点建议

这是非常关键的一类。

建议专门预留：

- schema generator adapter
- component knowledge adapter
- publish workflow adapter
- diagnosis rule adapter
- data model adapter

这样新业务对象和平台新能力进来时，不需要改通用 AI 平台主骨架。

## 11. 推荐的扩展实现方式

可按不同层次选择：

### 11.1 配置化扩展

适合：

- 路由规则
- 风险策略
- prompt 选择

### 11.2 注册表扩展

适合：

- tools
- graders
- exporters
- validators

### 11.3 策略接口扩展

适合：

- retrieval policy
- planner policy
- fallback policy

### 11.4 领域适配器扩展

适合：

- 不同业务域接入通用平台

## 12. 常见误区

### 12.1 一开始把所有能力都做成插件系统

会显著增加复杂度和维护负担。

### 12.2 没有稳定主骨架就先设计高度可扩展架构

扩展点会失去锚点。

### 12.3 扩展点没有统一协议和版本治理

后续很容易变成隐式耦合。

### 12.4 扩展逻辑绕开 trace / audit / policy

会破坏平台治理能力。

### 12.5 把低代码领域能力直接硬编码到通用平台层

后续新领域接入会越来越困难。

## 13. 一句话总结

可扩展点设计的核心，是在模型、检索、工具、规划、评测和领域适配这些高变化区域预留明确协议化边界，同时保持 Runtime 主骨架和治理主链路稳定；对 AI 平台而言，真正好的扩展性不是“哪里都能插”，而是“该稳定的稳定，该变化的可替换”。
