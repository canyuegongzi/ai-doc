# 表结构与索引设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 平台中的表结构与索引设计原则，重点回答：

- 前面定义的核心对象在持久化层如何落表
- 哪些对象适合关系型表，哪些适合对象存储、向量索引或事件存储
- 如何兼顾在线查询性能、长任务可追踪性、审计保留和多租户隔离

## 2. 为什么这部分重要

前面的数据结构设计定义了平台主对象，但工程落地还要继续回答：

- 这些对象如何持久化
- 哪些字段需要主表字段，哪些适合 metadata
- 如何建索引支持高频查询
- 如何避免在线链路和审计链路互相拖累

如果表结构和索引设计不合理，系统很快会出现：

- trace 查询慢
- task 恢复慢
- 审计检索难
- 多租户隔离薄弱
- 索引切换成本高

用一句话概括：

表结构与索引设计的本质，是把 AI 平台的核心对象转化为长期可维护、可查询、可隔离、可扩展的持久化结构。

## 3. 持久化层推荐分工

建议不要所有对象都塞进同一类数据库，通常可按以下方式分工：

### 3.1 关系型数据库

适合：

- 任务状态
- 审批记录
- 工具注册元数据
- 评测任务定义
- 审计事件索引表

### 3.2 向量数据库 / 向量索引

适合：

- chunk embeddings
- object embeddings
- hybrid retrieval 支撑数据

### 3.3 对象存储

适合：

- 原始文档
- 大块解析结果
- replay snapshot
- 高体积 trace 明细附件

### 3.4 事件存储 / 日志系统

适合：

- trace span 流
- audit event 流
- online metrics event

## 4. 推荐优先落表的核心实体

建议优先设计以下表或表组：

### 4.1 `ai_agent_task`

记录任务主对象。

### 4.2 `ai_task_step`

记录任务步骤和状态机流转。

### 4.3 `ai_tool_definition`

记录工具注册元数据。

### 4.4 `ai_tool_execution`

记录工具实际执行。

### 4.5 `ai_knowledge_item`

记录知识对象主元数据。

### 4.6 `ai_knowledge_chunk`

记录 chunk 元数据与索引引用。

### 4.7 `ai_index_version`

记录索引版本。

### 4.8 `ai_trace_span`

记录 trace 主表或索引表。

### 4.9 `ai_audit_event`

记录审计事件。

### 4.10 `ai_eval_task` / `ai_eval_run`

记录评测任务与运行结果。

## 5. 任务与步骤表设计建议

### 5.1 `ai_agent_task`

建议至少包含：

- `task_id`
- `tenant_id`
- `app_id`
- `scene_type`
- `task_type`
- `goal_summary`
- `state`
- `risk_status`
- `trace_id`
- `request_id`
- `created_at`
- `updated_at`
- `finished_at`

建议索引：

- `(tenant_id, created_at desc)`
- `(trace_id)`
- `(state, updated_at)`
- `(scene_type, created_at)`

### 5.2 `ai_task_step`

建议至少包含：

- `step_id`
- `task_id`
- `step_type`
- `tool_name`
- `status`
- `input_ref`
- `output_ref`
- `started_at`
- `finished_at`
- `retry_count`

建议索引：

- `(task_id, started_at)`
- `(status, started_at)`

## 6. Tool 表设计建议

### 6.1 `ai_tool_definition`

建议至少包含：

- `tool_name`
- `version`
- `category`
- `description`
- `risk_level`
- `auth_scope`
- `status`
- `schema_ref`
- `updated_at`

建议唯一索引：

- `(tool_name, version)`

建议查询索引：

- `(status, category)`
- `(risk_level, status)`

### 6.2 `ai_tool_execution`

建议至少包含：

- `execution_id`
- `task_id`
- `tool_name`
- `tool_version`
- `status`
- `receipt`
- `risk_level`
- `latency_ms`
- `retry_count`
- `created_at`

建议索引：

- `(task_id, created_at)`
- `(tool_name, created_at desc)`
- `(status, created_at)`

## 7. 知识与索引表设计建议

### 7.1 `ai_knowledge_item`

建议至少包含：

- `knowledge_id`
- `tenant_id`
- `app_id`
- `source_type`
- `title`
- `permission_scope`
- `visibility_level`
- `content_ref`
- `updated_at`

建议索引：

- `(tenant_id, app_id, source_type)`
- `(updated_at desc)`

### 7.2 `ai_knowledge_chunk`

建议至少包含：

- `chunk_id`
- `knowledge_id`
- `tenant_id`
- `index_version`
- `chunk_index`
- `embedding_ref`
- `citation_ref`

建议索引：

- `(knowledge_id, chunk_index)`
- `(tenant_id, index_version)`

### 7.3 `ai_index_version`

建议至少包含：

- `index_name`
- `version`
- `status`
- `embedding_model`
- `chunk_strategy`
- `built_at`
- `activated_at`

建议唯一索引：

- `(index_name, version)`

## 8. Trace 与审计表设计建议

### 8.1 `ai_trace_span`

如果量级可控，可用宽表 + TTL；若量级大，建议主索引表 + 原始事件存储分离。

建议字段：

- `trace_id`
- `span_id`
- `parent_span_id`
- `tenant_id`
- `scene_type`
- `span_type`
- `status`
- `latency_ms`
- `cost_metrics_json`
- `created_at`

建议索引：

- `(trace_id, created_at)`
- `(tenant_id, created_at desc)`
- `(scene_type, created_at)`

### 8.2 `ai_audit_event`

建议字段：

- `audit_id`
- `tenant_id`
- `actor_id`
- `event_type`
- `resource_type`
- `resource_id`
- `decision`
- `risk_level`
- `trace_id`
- `created_at`

建议索引：

- `(tenant_id, created_at desc)`
- `(trace_id)`
- `(event_type, created_at)`
- `(actor_id, created_at)`

## 9. 评测与反馈表设计建议

### 9.1 `ai_eval_task`

用于存任务定义，建议字段：

- `eval_task_id`
- `scene_type`
- `difficulty`
- `risk_tag`
- `task_payload_json`
- `grader_config_json`
- `updated_at`

### 9.2 `ai_eval_run`

用于存实际运行结果，建议字段：

- `eval_run_id`
- `eval_task_id`
- `system_version`
- `score`
- `outcome`
- `trace_id`
- `created_at`

### 9.3 `ai_feedback_record`

用于存反馈，建议字段：

- `feedback_id`
- `tenant_id`
- `scene_type`
- `trace_id`
- `feedback_type`
- `feedback_value`
- `feedback_text`
- `created_at`

## 10. 多租户与安全设计建议

### 10.1 所有核心表都显式带 `tenant_id`

不要把租户上下文只留在外部服务。

### 10.2 高敏感字段避免明文入表

如：

- secret
- token
- prompt internals
- full raw outputs

应使用引用、脱敏或对象存储隔离策略。

### 10.3 Trace / Audit / Feedback 查询也必须做租户和角色过滤

## 11. 索引设计原则

### 11.1 面向主要查询场景建索引，而不是面向所有字段建索引

### 11.2 时间维度几乎是所有治理表的核心索引维度

### 11.3 高频过滤字段前置

例如：

- tenant_id
- state
- event_type
- scene_type

### 11.4 避免在大 JSON 字段上做过多依赖查询

关键字段应显式拉平。

## 12. 常见误区

### 12.1 所有对象都只存 JSON

短期快，长期非常难治理和查询。

### 12.2 所有字段都扁平展开

会导致扩展困难和表爆炸。

### 12.3 Trace、Audit、Task 不共享关键 ID

后续链路串联会非常痛苦。

### 12.4 不做时间和状态索引

长任务和审计查询性能会迅速恶化。

### 12.5 向量索引版本与主数据版本脱节

检索结果会难以解释和回放。

## 13. 一句话总结

表结构与索引设计的核心，是围绕任务、工具、知识、Trace、审计、评测这些平台主对象建立稳定持久化模型，并按查询场景、租户隔离、时间维度和版本治理设计索引，而不是把所有信息都混在少数 JSON 表里；对 AI 平台来说，能否查得清、追得回、切得稳，往往取决于这层设计是否扎实。
