# 服务拆分设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义企业 AI 平台的服务拆分设计，重点回答：

- AI 能力平台化后，哪些模块适合独立成服务，哪些更适合作为共享库或模块内聚
- 服务边界应该按技术类型拆，还是按能力职责拆
- 结合低代码平台场景，如何在可演进和不过度微服务之间取得平衡

## 2. 为什么要讨论服务拆分

到目前为止，前面的文档已经定义了：

- 模型网关
- 知识工程与 RAG
- Tool / Action 体系
- Agent Runtime
- 安全治理
- 评测与观测

但这些能力如果都混在一个应用里，后期会很快出现问题：

- 发布边界不清
- 模型调用与知识处理耦合过深
- Tool 注册与执行逻辑分散
- 评测、观测、审计难以统一
- 不同业务场景难复用

因此，服务拆分的目标不是“拆得越多越高级”，而是把稳定复用的能力边界沉淀成平台服务。

用一句话概括：

服务拆分设计的本质，是为 AI 平台确定长期稳定的能力边界和协作边界。

## 3. 服务拆分要解决什么问题

### 3.1 能力复用

让多个 AI 场景共享同一套底座能力。

### 3.2 发布解耦

避免某个知识管线变更影响所有在线应用。

### 3.3 责任边界清晰

明确哪个服务负责：

- 调模型
- 建知识
- 检索
- 执行工具
- 跑 Agent
- 做评测

### 3.4 可观测与可治理

每层能力都能独立监控、限流、审计和治理。

## 4. 服务拆分的基本原则

### 4.1 优先按能力职责拆，而不是按技术组件名词拆

例如：

- 不建议仅按“向量库服务”“Prompt 服务”这样技术名词拆
- 更建议按“Knowledge Pipeline”“Retrieval Engine”“Agent Runtime”这类能力边界拆

### 4.2 在线链路与离线链路分离

知识构建、离线评测、索引重建通常和在线问答/Agent 执行的负载特征不同，建议拆开。

### 4.3 高变更模块和高稳定模块分离

例如：

- Prompt 资产、评测样本、检索策略变化较频繁
- 身份透传、审计、核心协议相对稳定

### 4.4 先模块化，再服务化

不是所有能力一开始都必须是独立微服务。对于早期系统，可以先在单体或模块化架构中形成明确边界，再逐步独立。

## 5. 推荐的一阶服务版图

对于企业内部 AI 平台，建议优先考虑以下服务边界：

### 5.1 `ai-gateway-service`

职责：

- 模型接入
- 路由
- fallback
- usage 统计
- 配额控制

### 5.2 `ai-knowledge-pipeline-service`

职责：

- 文档采集
- 解析
- 清洗
- chunk
- embedding
- 索引构建与更新

### 5.3 `ai-retrieval-service`

职责：

- query rewrite
- query expansion
- recall
- rerank
- context builder

### 5.4 `ai-tool-registry-service`

职责：

- tool 注册与发现
- tool 协议管理
- tool policy 元数据
- 版本治理

### 5.5 `ai-agent-runtime-service`

职责：

- state machine
- planner orchestration
- tool execution coordination
- HITL / approval node integration
- task lifecycle management

### 5.6 `ai-eval-observability-service`

职责：

- offline eval runs
- online metrics
- trace storage
- feedback loop
- failure sample management

### 5.7 `ai-application-service`

职责：

- 具体业务场景编排
- 组件问答
- schema 生成
- 配置诊断
- 运维助手

## 6. 这些服务不一定都要物理独立

建议分三个阶段理解：

### 6.1 阶段一：模块化单体

把这些边界先落实为模块。

### 6.2 阶段二：高负载或高变更模块独立服务化

优先独立：

- gateway
- knowledge pipeline
- retrieval
- runtime

### 6.3 阶段三：平台化成熟后再细拆

例如：

- eval center
- tool registry
- approval integration

## 7. 推荐的在线调用链路

以“配置诊断 Agent”为例：

1. `ai-application-service` 接收请求
2. 调用 `ai-agent-runtime-service`
3. runtime 调 `ai-retrieval-service` 获取规则和案例
4. runtime 调 `ai-tool-registry-service` 解析工具与策略
5. runtime 通过 tool executor 调业务后端
6. 模型调用统一走 `ai-gateway-service`
7. trace / metrics / feedback 写入 `ai-eval-observability-service`

这条链路里最核心的原则是：

- 业务应用层不直接拼所有细节
- 共享能力通过平台服务提供

## 8. 推荐的离线链路

以“知识重建 + 评测”为例：

1. `ai-knowledge-pipeline-service` 采集和构建索引
2. 构建完成后发布索引版本
3. `ai-eval-observability-service` 跑离线评测集
4. 结果达标后，在线服务切换新索引版本

这比在线服务直接触发索引重建要稳定得多。

## 9. 面向低代码平台的具体建议

### 9.1 不建议把低代码业务对象逻辑散落在平台服务中

平台服务应处理通用 AI 能力；
页面、组件、schema、数据源这类领域语义应更多沉淀在业务应用层和领域适配层。

### 9.2 需要一个领域适配层

建议在 `ai-application-service` 或单独 adapter 模块中承接：

- PageSchema
- ComponentMaterial
- Setter
- DataSource
- PublishWorkflow

这样平台服务保持通用，业务层承接低代码特性。

## 10. 服务之间推荐的协议风格

建议：

- 在线接口：同步 API + 明确 request/response schema
- 长任务：异步任务 + 状态查询
- 高风险动作：任务单 / approval token 模式
- trace / audit：异步事件上报

## 11. 常见误区

### 11.1 一开始就拆很多微服务

会极大增加协作和调试成本。

### 11.2 所有 AI 功能都塞进业务 BFF

后期很难复用和治理。

### 11.3 把离线知识构建和在线检索放在同一高耦合服务里

负载、发布和故障域都会变差。

### 11.4 Tool Registry 和 Runtime 混在一起

会让 tool 元数据治理和执行协调耦合过深。

### 11.5 观测与评测不作为独立能力设计

后期很难形成持续优化闭环。

## 12. 一句话总结

服务拆分设计的核心，是围绕模型接入、知识构建、检索、工具治理、Agent 运行时、评测观测和业务应用编排建立长期稳定的能力边界，并按阶段逐步服务化，而不是一开始追求过度微服务；对企业 AI 平台而言，真正重要的不是“拆多少服务”，而是“边界是否清晰且可演进”。
