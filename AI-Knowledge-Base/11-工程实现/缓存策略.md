# 缓存策略

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 平台中的缓存策略，重点回答：

- 在模型、RAG、Tool、Agent 场景下，哪些内容适合缓存，哪些不适合
- 缓存应如何同时服务性能、成本和稳定性，而不破坏权限与数据一致性
- 如何设计多层缓存与失效机制

## 2. 为什么 AI 系统必须认真设计缓存

AI 系统的高成本和高延迟，常常来自重复工作：

- 重复 query rewrite
- 重复检索同样知识
- 重复 embedding 同一文本
- 重复工具查询稳定状态
- 重复生成相同 FAQ 答案

如果没有缓存：

- 成本会上升
- 延迟会升高
- 高峰时更容易触发限流和降级

但如果缓存设计不好，又会引入：

- 脏数据
- 权限泄露
- 跨租户污染
- replay 与审计失真

用一句话概括：

缓存策略的本质，是在性能、成本、一致性和安全边界之间做有意识的平衡。

## 3. 缓存要解决什么问题

### 3.1 降低重复计算

### 3.2 降低调用成本

### 3.3 缩短响应时间

### 3.4 在高峰期提供缓冲能力

## 4. 推荐的缓存分层

建议至少分成五层：

### 4.1 模型调用缓存

缓存稳定场景下的生成结果。

### 4.2 检索缓存

缓存 rewrite、recall、rerank 或 context building 结果。

### 4.3 embedding 缓存

避免重复向量化相同内容。

### 4.4 tool result 缓存

对只读工具和低变化数据做缓存。

### 4.5 运行时短缓存

对任务执行中的临时中间结果做短时缓存。

## 5. 哪些内容适合缓存

### 5.1 FAQ 类稳定问答结果

前提是：

- 场景稳定
- 权限边界清晰
- 引用内容变化不频繁

### 5.2 query rewrite / expansion 结果

通常对同类 query 很有复用价值。

### 5.3 embedding 结果

高复用且天然适合缓存。

### 5.4 只读工具结果

例如：

- 组件定义
- 静态规则
- 发布历史摘要（短期）

### 5.5 结构化生成中间结果

例如：

- schema 草稿前置候选
- retrieval context package

## 6. 哪些内容不适合直接缓存

### 6.1 高风险动作结果

例如：

- 发布状态确认
- 删除动作结果
- 审批状态实时值

### 6.2 高频变更数据的长期缓存

例如实时日志、瞬时配置状态。

### 6.3 跨租户共用缓存键的敏感内容

会造成严重隔离风险。

### 6.4 含有敏感字段的原始工具输出

除非做脱敏和严格作用域控制。

## 7. 缓存键设计原则

建议缓存键至少考虑：

- tenantId
- appId / scope
- sceneType
- model / prompt version
- index version
- tool version
- query hash / input hash

这是为了避免：

- 跨租户污染
- 模型切换后误复用旧结果
- 索引切换后命中旧 retrieval 缓存

## 8. RAG 场景下的缓存建议

### 8.1 Embedding Cache

高优先级，性价比很高。

### 8.2 Rewrite Cache

对高频 query 可以显著降低额外模型调用成本。

### 8.3 Retrieval Result Cache

适合：

- 稳定 FAQ
- 低变化文档库

但必须绑定：

- tenant scope
- index version
- retrieval strategy version

### 8.4 Context Package Cache

对高频问题可以缓存最终进入模型的 context package，而不是每次重新组装。

## 9. Tool 场景下的缓存建议

### 9.1 只读工具结果缓存

适合：

- 组件协议
- 规则文档
- 静态元信息

### 9.2 短期状态缓存

适合：

- 构建状态摘要
- 发布历史摘要

但必须 TTL 很短，并保留 freshness 策略。

### 9.3 动作型工具避免直接复用旧执行结果

对于 Action，应优先依赖 receipt / status query，而不是简单结果缓存。

## 10. Agent 运行时中的缓存建议

Agent 运行时更适合：

### 10.1 任务内短缓存

复用同一次任务中已拿到的信息，避免重复 tool call。

### 10.2 步骤级 memoization

对 planner 和 tool execution 中间结果做局部复用。

### 10.3 任务外共享缓存要非常谨慎

尤其涉及：

- 权限边界
- 环境状态
- 审批状态
- 高风险动作前置条件

## 11. 缓存失效策略建议

### 11.1 TTL

适合时效性明确的结果。

### 11.2 版本失效

例如：

- prompt version 变更
- index version 切换
- tool version 更新

### 11.3 事件驱动失效

例如：

- 文档更新
- 配置变更
- 页面发布
- 审批状态变化

### 11.4 显式手动失效

用于运维和紧急修复。

## 12. 面向低代码平台的典型建议

### 12.1 组件问答

适合缓存：

- 组件协议摘要
- FAQ 问答结果
- query rewrite 结果

### 12.2 Schema 生成

适合缓存：

- 模板检索结果
- 组件物料摘要
- 数据模型字段定义

### 12.3 配置诊断

适合短期缓存：

- 规则条目
- 案例检索结果
- 页面静态结构信息

不适合长期缓存：

- 实时运行状态
- 最新构建异常详情

## 13. 常见误区

### 13.1 只要贵就缓存

缓存还要看时效性、权限和一致性。

### 13.2 缓存键不带租户和版本信息

极易造成脏缓存和越权。

### 13.3 对高风险动作结果做强缓存

会误导后续执行判断。

### 13.4 不记录缓存命中率和失效原因

后续无法优化缓存策略。

### 13.5 把缓存当作数据源真相

缓存只应是加速层，不应替代 authoritative source。

## 14. 一句话总结

缓存策略的核心，是围绕模型调用、检索、embedding、工具只读结果和任务内中间状态建立多层缓存，并通过租户作用域、版本绑定和事件失效机制控制一致性与安全边界；对 AI 平台来说，真正好的缓存不是“缓存得多”，而是“缓存得准、失效得稳、不会越权”。
