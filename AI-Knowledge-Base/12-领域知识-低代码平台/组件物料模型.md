# 组件物料模型

> 状态：第一版草案

## 1. 文档定位

本文档用于定义低代码平台中的组件物料模型，重点回答：

- 什么是组件物料，它与普通前端组件库对象有什么不同
- 组件物料应包含哪些关键语义，才能同时服务编辑器、渲染器和 AI 系统
- 为什么组件物料模型是低代码 RAG、推荐、生成和诊断场景中的核心知识对象

## 2. 什么是组件物料

组件物料可以理解为：

在低代码平台中，对某一类可被页面 Schema 引用、可被编辑器配置、可被渲染器消费的组件能力对象进行结构化封装后的平台资产。

它通常不仅包含前端组件本身，还包含：

- 元信息
- props 协议
- slots 协议
- Setter 配置方式
- 示例用法
- 适用场景
- 版本信息

用一句话概括：

组件物料的本质，是“可被平台理解和配置的组件能力单元”，而不是单纯的 UI 代码包。

## 3. 为什么组件物料模型是低代码核心对象

### 3.1 页面 Schema 依赖物料模型

页面 Schema 中大量节点本质上是“某个物料的一个实例化配置”。

### 3.2 编辑器依赖物料模型做可编辑性

编辑器不是靠读组件源码渲染配置面板，而是依赖物料模型中的协议和 Setter 描述。

### 3.3 AI 系统依赖物料模型做推荐、生成和诊断

无论是：

- 组件推荐
- 页面生成
- schema 修复
- 组件问答

都必须理解组件物料的能力边界。

## 4. 组件物料模型要解决什么问题

### 4.1 描述组件能做什么

### 4.2 描述组件怎么配

### 4.3 描述组件适合在哪些场景使用

### 4.4 描述组件与其他物料之间的关系

## 5. 推荐的组件物料分层

建议至少分成五层：

### 5.1 基础元信息层

描述物料身份和分类。

### 5.2 协议层

描述 props、events、slots、children 结构。

### 5.3 编辑层

描述 Setter 与配置能力。

### 5.4 场景层

描述适用业务场景、限制和推荐搭配。

### 5.5 运行层

描述版本、依赖和渲染要求。

## 6. 推荐的核心字段结构

一个简化的组件物料模型可以表示为：

```ts
interface ComponentMaterial {
  componentName: string;
  title: string;
  category: string;
  description?: string;
  propsSchema?: Record<string, unknown>;
  slotSchema?: Record<string, unknown>;
  eventSchema?: Record<string, unknown>;
  setterSchema?: Record<string, unknown>;
  tags?: string[];
  examples?: unknown[];
  version?: string;
  meta?: Record<string, unknown>;
}
```

这个对象的目标不是穷尽所有实现细节，而是为平台提供“可理解、可配置、可推荐”的描述。

## 7. 物料元信息建议

建议至少包含：

- `componentName`
- `title`
- `category`
- `description`
- `tags`
- `version`
- `status`
- `owner`

这些字段主要服务于：

- 搜索
- 分类
- 推荐
- 版本治理

## 8. Props 协议建议

Props 协议应描述：

- 字段名
- 字段类型
- 默认值
- 是否必填
- 可选枚举
- 示例
- 是否支持绑定

这对 AI 场景非常关键，因为：

- 生成要知道能填什么
- 诊断要知道哪里填错了
- 推荐要知道组件是否满足需求

## 9. Slot 协议建议

很多复杂低代码组件并不是简单 children 树，而是 slot 驱动。

建议明确描述：

- slot 名称
- slot 是否必填
- slot 可接受节点类型
- slot 数量限制
- slot 的语义用途

对于 AI 生成和诊断来说，slot 协议常常比普通 props 更难理解，也更容易出错。

## 10. Event 与行为协议建议

若组件支持交互事件，建议显式建模：

- event 名称
- event 参数
- 常见绑定动作
- 风险级别（如有）

这有助于后续逻辑编排 Agent 理解“这个组件可触发什么”。

## 11. Setter 与编辑语义建议

组件物料不应只面向运行时，还应面向编辑器。

建议物料模型中保留：

- setter 入口定义
- 配置分组
- 字段编辑方式
- 条件显隐规则

这使得 AI 后续不只是“会生成 props”，还可能“会推荐如何在编辑器里配置”。

## 12. 场景知识建议

这是很多物料模型容易缺失但对 AI 特别重要的一层。

建议补充：

- 适用场景
- 不适用场景
- 常见搭配组件
- 常见错误用法
- 替代组件

这些内容对于：

- RAG 问答
- 推荐系统
- schema 生成约束
- 配置诊断

都很有价值。

## 13. 组件物料与页面 Schema 的关系

可以理解为：

- 物料模型定义模板和能力边界
- 页面 Schema 是具体实例化配置

因此 AI 在生成 Schema 时，理想过程通常是：

先理解需求 -> 再匹配物料 -> 再按物料协议生成节点配置

## 14. AI 场景下为什么要把组件物料单独建成知识对象

### 14.1 组件问答依赖它

### 14.2 组件推荐依赖它

### 14.3 页面生成依赖它

### 14.4 配置诊断依赖它

因为很多平台问题，本质上都是“某个组件该怎么用、能不能这么用”的问题。

## 15. 面向低代码平台的典型问题类型

围绕组件物料，AI 常见要处理的问题包括：

- 某组件适不适合审批表单
- 某 slot 应该放什么
- 某 props 是否支持表达式绑定
- 某组件与哪个组件更适合组合使用
- 某组件为什么渲染异常

## 16. 常见误区

### 16.1 只记录组件源码信息，不记录平台语义

对低代码编辑器和 AI 来说远远不够。

### 16.2 props 协议不完整

后续生成和诊断都会很不稳定。

### 16.3 不建 slot / setter 模型

复杂组件会变得难配置、难生成、难排错。

### 16.4 不补充场景知识

AI 只能做“组件搜索”，很难做“组件推荐”。

### 16.5 版本信息缺失

物料升级后会严重影响历史 Schema 和 AI 生成结果。

## 17. 一句话总结

组件物料模型的核心，是把低代码组件从“可渲染实现”提升为“可被平台和 AI 理解的能力对象”，并围绕元信息、props、slots、setter、场景知识和版本治理形成稳定描述；对低代码 AI 来说，它既是知识库的主对象，也是生成和诊断的硬约束来源。
