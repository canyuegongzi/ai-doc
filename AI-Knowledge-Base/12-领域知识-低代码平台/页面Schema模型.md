# 页面Schema模型

> 状态：第一版草案

## 1. 文档定位

本文档用于定义低代码平台中的页面 Schema 模型，重点回答：

- 页面 Schema 在平台中的语义边界是什么
- 它与组件物料、数据源、逻辑编排、发布流程是什么关系
- 为什么页面 Schema 是低代码 AI 场景里最关键的领域对象之一

## 2. 什么是页面 Schema

页面 Schema 可以理解为：

对一个低代码页面在结构、组件树、属性配置、数据绑定、交互行为、布局关系和运行时所需元信息的结构化描述。

它既是：

- 页面编辑器的中间表达
- 渲染器的输入
- AI 生成和修复的目标对象
- 平台审计、校验、发布的核心载体

用一句话概括：

页面 Schema 的本质，是低代码页面的“可计算表示”。

## 3. 为什么页面 Schema 是核心领域对象

### 3.1 页面不是代码文件，而是结构化配置系统

在低代码平台中，页面最终通常不是直接由开发者手写组件树，而是由 Schema 驱动渲染和配置。

### 3.2 AI 生成和诊断最终都要落到 Schema

无论是：

- 页面草稿生成
- 布局推荐
- 组件推荐
- 配置纠错
- 自动修复

最终都需要理解和操作页面 Schema。

### 3.3 Schema 是连接多种平台对象的中心

页面 Schema 通常同时引用：

- 组件物料
- Setter 配置
- 数据源定义
- 逻辑节点
- 发布元信息

因此，它天然是低代码平台知识图中的中心节点。

## 4. 页面 Schema 要解决什么问题

### 4.1 描述页面结构

### 4.2 描述组件树与布局关系

### 4.3 描述属性和数据绑定

### 4.4 描述页面运行所需的元信息

## 5. 推荐的页面 Schema 分层

建议至少分成五层：

### 5.1 页面元信息层

描述页面身份与全局配置。

### 5.2 结构树层

描述页面中的节点树和层级关系。

### 5.3 组件配置层

描述每个节点引用什么物料，以及配置了哪些属性。

### 5.4 数据绑定层

描述页面和数据源、字段、变量之间的绑定关系。

### 5.5 行为扩展层

描述动作、事件、交互触发点等附加能力。

## 6. 推荐的核心字段结构

一个简化的页面 Schema 通常至少包含：

```ts
interface PageSchema {
  pageId: string;
  pageName: string;
  pageType: string;
  version: string;
  meta?: Record<string, unknown>;
  layout?: Record<string, unknown>;
  nodes: SchemaNode[];
  dataBindings?: DataBinding[];
  actions?: SchemaAction[];
}
```

其中最核心的是 `nodes`，因为它承载页面主体结构。

## 7. 推荐的节点模型

页面 Schema 的节点通常可抽象成：

```ts
interface SchemaNode {
  nodeId: string;
  componentName: string;
  props?: Record<string, unknown>;
  slots?: Record<string, SchemaNode[]>;
  children?: SchemaNode[];
  style?: Record<string, unknown>;
  bindings?: Record<string, unknown>;
  events?: Record<string, unknown>;
}
```

### 7.1 `nodeId`

页面内唯一定位节点。

### 7.2 `componentName`

引用组件物料主标识。

### 7.3 `props`

承载组件属性配置。

### 7.4 `slots` / `children`

承载嵌套关系与插槽内容。

### 7.5 `bindings`

承载和数据源、变量、表达式的绑定。

## 8. 页面 Schema 与组件物料的关系

Schema 不应直接定义组件实现细节，而应引用物料模型。

也就是说：

- 物料模型定义“这个组件支持什么”
- 页面 Schema 定义“这个页面用了什么、怎么配置的”

如果没有清晰分工，页面 Schema 很容易变成杂糅实现细节的大对象。

## 9. 页面 Schema 与 Setter 的关系

Setter 不是页面 Schema 的一部分，但会影响 Schema 的可编辑性。

可以理解为：

- Schema 是运行时配置对象
- Setter 是编辑时配置界面和输入机制

因此，同一个 Schema 字段背后可能对应一套复杂 Setter 体系。

## 10. 页面 Schema 与数据源的关系

页面 Schema 中的很多字段不是静态值，而是：

- 常量
- 变量引用
- 数据源字段绑定
- 表达式结果

因此，Schema 必须允许描述绑定关系，而不是只存纯字面值。

## 11. AI 场景下为什么要把页面 Schema 单独建模

### 11.1 RAG 需要它作为知识对象

页面模板、页面案例、节点结构、常见布局都应能作为检索对象存在。

### 11.2 生成需要它作为目标结构

AI 页面生成不应只输出文案，而应最终输出可校验的 Schema。

### 11.3 诊断需要它作为分析对象

配置错误、绑定缺失、层级异常，本质上都是 Schema 问题。

### 11.4 修复需要它作为差异对象

很多自动修复最终表现为：

- schema patch
- 节点替换
- props 修正

## 12. 推荐的 AI 友好建模建议

### 12.1 节点 ID 稳定且可追踪

有利于：

- patch
- diff
- replay
- 审计

### 12.2 组件名与类型语义明确

不要只存内部缩写。

### 12.3 配置字段显式结构化

不要大量把关键配置埋进字符串表达式。

### 12.4 绑定关系显式建模

便于 AI 做生成、校验和修复。

## 13. 面向低代码平台的典型问题类型

围绕页面 Schema，AI 系统常见要处理的问题包括：

- 页面结构生成
- 布局优化建议
- 组件替换推荐
- 字段绑定校验
- props 类型错误修复
- slot 层级异常诊断

## 14. 常见误区

### 14.1 把页面 Schema 当成“随便一个 JSON”

这会让后续校验、生成和诊断都非常困难。

### 14.2 页面元信息、结构树、绑定逻辑完全混在一起

对象边界会非常模糊。

### 14.3 不为节点提供稳定 ID

后续 patch 和定位都会变差。

### 14.4 Schema 结构和物料结构没有清晰分工

系统会变得难演进。

### 14.5 只为渲染设计，不为 AI 和治理设计

后期会发现很难做生成、修复和评测。

## 15. 一句话总结

页面 Schema 模型的核心，是把低代码页面的结构、组件配置、绑定关系和运行元信息沉淀成稳定的可计算对象，并作为渲染、编辑、生成、诊断和发布的共同基础；对 AI 场景而言，它不是页面的附属产物，而是最关键的操作对象之一。
