# 插槽机制

> 状态：第一版草案

## 1. 文档定位

本文档用于定义低代码平台中的插槽（Slot）机制，重点回答：

- 插槽在组件物料与页面 Schema 中分别扮演什么角色
- 为什么插槽机制比普通 children 结构更适合低代码平台表达复杂组件嵌套关系
- 在 AI 生成、组件推荐和配置诊断场景下，如何把插槽机制知识化

## 2. 什么是插槽机制

插槽机制可以理解为：

组件对外暴露的一种结构化内容承载位，用于约定“哪些内容可以被放进组件内部、放在哪里、以什么类型存在、满足什么约束”。

它不只是普通前端里的 `children` 扩展，而是低代码平台中：

- 组件嵌套结构的协议化入口
- 编辑器可视化布局的重要基础
- 页面 Schema 中节点树和容器关系的关键语义点

用一句话概括：

插槽机制的本质，是把组件内部可嵌套区域从“实现细节”提升为“平台可理解的结构协议”。

## 3. 为什么插槽机制是低代码平台的重要领域知识

### 3.1 复杂组件通常不是单层 props 配置，而是结构容器

例如：

- 表单容器
- 卡片头尾区域
- Tab 内容区域
- 列表项模板区域
- 表格列自定义渲染区域

这些能力很难只靠 props 表达。

### 3.2 插槽是页面 Schema 树结构的关键连接点

很多页面不是单纯父子节点嵌套，而是：

- 某节点进入某个特定 slot
- 不同 slot 对内容类型有不同约束
- slot 本身有数量和语义限制

### 3.3 AI 场景非常依赖插槽知识

无论是：

- 页面生成
- 布局推荐
- 组件替换
- slot 配置诊断

都必须理解 slot 语义。

## 4. 插槽机制要解决什么问题

### 4.1 描述组件内部可承载内容的区域

### 4.2 约束可放入的节点类型和数量

### 4.3 为编辑器提供结构化嵌套模型

### 4.4 为运行时和 AI 提供组合语义

## 5. 插槽与普通 children 的区别

### 5.1 普通 children

通常只是一个通用嵌套位，语义较弱。

### 5.2 插槽

通常具备：

- 名称
- 语义用途
- 可接受节点类型
- 数量限制
- 条件显隐

因此，插槽更适合作为低代码平台中的正式协议对象。

## 6. 推荐的插槽定义结构

一个简化的 slot 定义可以表示为：

```ts
interface SlotDefinition {
  slotName: string;
  title: string;
  description?: string;
  accepts?: string[];
  required?: boolean;
  multiple?: boolean;
  minItems?: number;
  maxItems?: number;
  conditions?: unknown[];
}
```

其中关键字段包括：

- `slotName`
- `accepts`
- `required`
- `multiple`

这些字段直接影响：

- 编辑器拖拽约束
- AI 生成可行性
- 配置诊断规则

## 7. 插槽在页面 Schema 中的表达建议

在页面 Schema 里，slot 通常不是单独的页面级对象，而是节点上的结构字段。

例如：

```ts
interface SchemaNode {
  nodeId: string;
  componentName: string;
  props?: Record<string, unknown>;
  slots?: Record<string, SchemaNode[]>;
}
```

这里的 `slots` 表示：

- 不同 slotName 对应不同节点集合
- 节点不只是“是孩子”，而是“是某个 slot 的孩子”

## 8. 推荐的插槽语义类型

建议至少区分：

### 8.1 容器型 Slot

用于承载任意子节点或布局区域。

### 8.2 模板型 Slot

用于承载特定结构模板，例如列表项、表格单元格模板。

### 8.3 功能型 Slot

用于承载头部、底部、操作区、空态区等语义位置。

### 8.4 限制型 Slot

只接受特定物料类型或单节点内容。

## 9. AI 场景下为什么要单独知识化插槽机制

### 9.1 页面生成依赖它

AI 若不理解 slot，很容易：

- 把组件放错位置
- 生成不合法层级
- 忽略必填 slot

### 9.2 配置诊断依赖它

很多页面渲染问题本质上是：

- slot 缺失
- slot 内容类型不对
- slot 层级不对

### 9.3 组件推荐依赖它

推荐某组件时，不仅要知道它“能用”，还要知道它“应嵌在哪个 slot 中”。

## 10. 推荐的插槽知识内容

为了让 AI 真正理解 slot，建议知识化时至少补充：

- slot 名称与用途
- slot 可接受节点类型
- 是否必填
- 常见合法组合
- 常见错误配置
- 示例结构

## 11. 面向低代码平台的典型问题类型

围绕插槽，AI 常见要处理的问题包括：

- 为什么这个 slot 没渲染出来
- 某个 slot 是否支持多个子节点
- 某个组件能否放到当前 slot
- 某个 slot 为什么必须填
- slot 中能否放交互型组件或模板型组件

## 12. AI 友好的插槽建模建议

### 12.1 slot 名称要稳定且语义明确

### 12.2 接受类型要结构化

### 12.3 slot 规则要能被校验器和生成器共同消费

### 12.4 为模板型 slot 提供示例结构

## 13. 常见误区

### 13.1 把插槽当成普通 children

这会丢失大量结构语义。

### 13.2 slot 协议不建模 accepts 和数量限制

后续生成和校验会非常不稳定。

### 13.3 页面 Schema 中只保留 children，不保留 slot 维度

会让复杂组件结构无法被准确表达。

### 13.4 不记录 slot 常见错误

配置诊断会缺少高价值规则。

### 13.5 组件物料与 slot 机制脱节

平台很难统一做组合推荐和自动修复。

## 14. 一句话总结

插槽机制的核心，是把组件内部的可嵌套区域沉淀成带名称、带约束、带语义的位置协议，并作为页面结构表达、编辑器拖拽规则、AI 生成约束和配置诊断规则的共同基础；对低代码 AI 来说，真正懂布局组合，必须先懂 slot。
