# 数据源模型

> 状态：第一版草案

## 1. 文档定位

本文档用于定义低代码平台中的数据源模型，重点回答：

- 数据源在平台中扮演什么角色，它与页面 Schema、组件配置、逻辑编排分别如何关联
- 为什么数据源模型是低代码 AI 生成、绑定校验、诊断和运维分析的关键领域对象
- 如何把数据源从“接口配置”提升为可被知识化、可被生成和可被校验的结构对象

## 2. 什么是数据源模型

数据源模型可以理解为：

在低代码平台中，对页面运行时可获取、可绑定、可计算、可驱动交互的数据能力进行结构化描述的一组对象与关系。

它通常不仅包括接口地址或查询语句，还包括：

- 输入参数
- 输出结构
- 调用方式
- 缓存与刷新策略
- 权限与环境信息
- 与页面字段的绑定关系

用一句话概括：

数据源模型的本质，是页面与外部数据世界之间的契约层。

## 3. 为什么数据源模型是低代码核心对象

### 3.1 页面中的很多行为都依赖数据源驱动

例如：

- 表单回填
- 列表查询
- 下拉选项加载
- 条件联动
- 按钮提交

### 3.2 页面 Schema 中大量配置本质上是绑定到数据源

页面不是只有静态 props，还包含：

- 字段绑定
- 表达式依赖
- 动态选项
- 查询条件
- 请求动作

### 3.3 AI 场景高度依赖数据源结构理解

无论是：

- 页面生成
- 数据绑定推荐
- 配置诊断
- 发布风险分析

都必须理解数据源模型。

## 4. 数据源模型要解决什么问题

### 4.1 描述可获取哪些数据

### 4.2 描述如何请求这些数据

### 4.3 描述数据结构和字段语义

### 4.4 描述数据如何绑定到页面和逻辑节点

## 5. 推荐的数据源模型分层

建议至少分成五层：

### 5.1 数据源身份层

描述数据源名称、类型、归属和版本。

### 5.2 调用协议层

描述请求方式、参数、环境和认证信息。

### 5.3 输出结构层

描述返回字段结构与语义。

### 5.4 绑定关系层

描述和页面字段、组件 props、表达式的绑定方式。

### 5.5 运行策略层

描述缓存、刷新、重试和错误处理方式。

## 6. 推荐的核心字段结构

一个简化的数据源对象可表示为：

```ts
interface DataSourceDefinition {
  dataSourceId: string;
  name: string;
  type: string;
  request?: Record<string, unknown>;
  responseSchema?: Record<string, unknown>;
  bindings?: unknown[];
  runtimePolicy?: Record<string, unknown>;
  meta?: Record<string, unknown>;
}
```

其中最关键的是：

- 请求结构
- 返回字段结构
- 绑定关系

## 7. 数据源类型建议

建议至少区分：

### 7.1 接口型数据源

例如 HTTP / RPC 接口。

### 7.2 查询型数据源

例如数据库查询、报表查询。

### 7.3 字典型数据源

例如枚举值、选项列表。

### 7.4 状态型数据源

例如页面上下文、会话变量、用户信息。

### 7.5 组合型数据源

由多个源计算或拼接而来。

## 8. 请求协议建议

建议数据源模型显式描述：

- method / operation type
- path / endpoint / query key
- input params
- required params
- auth / env scope
- timeout / retry policy

这对 AI 生成和诊断非常关键，因为很多问题不是“有没有数据源”，而是“请求协议能不能配对”。

## 9. 返回结构建议

建议显式建模：

- 字段名
- 类型
- 业务含义
- 是否可绑定
- 是否敏感
- 是否可作为主键 / 显示字段 / 搜索字段

如果没有稳定输出结构，AI 很难做：

- 字段匹配
- 表单回填
- 列表列推荐
- 查询条件生成

## 10. 数据绑定关系建议

数据源模型不应停留在“能调接口”，还应描述：

- 哪个返回字段绑定到哪个页面字段
- 哪些字段支持双向绑定
- 哪些字段只能展示不能写入
- 哪些字段需要表达式转换

这层信息对生成和诊断场景价值非常高。

## 11. 运行策略建议

建议建模：

- 是否缓存
- 刷新时机
- 自动加载或懒加载
- 错误重试
- 空数据处理方式

因为很多真实问题并不是 schema 配置错，而是运行策略不合理。

## 12. AI 场景下为什么要单独知识化数据源模型

### 12.1 页面生成依赖它

AI 生成页面时，必须决定：

- 需要哪些数据源
- 查询条件如何生成
- 字段如何映射

### 12.2 配置诊断依赖它

很多问题本质上是：

- 绑定字段不存在
- 数据结构不匹配
- 请求参数没传对
- 返回结构和组件需求不一致

### 12.3 运维和发布风险分析依赖它

数据源变化可能直接影响页面运行和发布安全。

## 13. 面向低代码平台的典型问题类型

围绕数据源，AI 常见要处理的问题包括：

- 为什么某个字段绑定不生效
- 某组件应该绑定哪个返回字段
- 某数据源适不适合表单回填
- 某个查询条件应该如何生成
- 某页面为什么在发布后拿不到数据

## 14. AI 友好的数据源建模建议

### 14.1 请求和响应都要结构化

### 14.2 字段语义要补齐，而不是只给技术类型

### 14.3 绑定关系要单独建模

### 14.4 敏感字段和只读字段要显式标识

这样更适合：

- 生成
- 诊断
- 权限分析
- 发布风险检查

## 15. 常见误区

### 15.1 把数据源当成简单接口配置

这会丢失大量页面运行语义。

### 15.2 没有返回结构模型

后续字段推荐和绑定校验会很弱。

### 15.3 不建模绑定关系

AI 很难稳定生成可运行页面。

### 15.4 不区分状态型和远程型数据源

运行语义会混乱。

### 15.5 不标注敏感字段

后续安全和发布治理会变脆弱。

## 16. 一句话总结

数据源模型的核心，是把低代码页面与外部数据交互所需的请求、响应、绑定和运行策略沉淀成结构化契约，并作为页面生成、字段映射、诊断和治理的共同基础；对低代码 AI 来说，真正懂页面，不只是懂组件和 Schema，还要懂数据源。
