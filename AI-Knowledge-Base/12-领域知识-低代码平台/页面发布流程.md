# 页面发布流程

## 1. 定义
页面发布流程是低代码平台中将页面从设计态资产转变为可访问、可运行、可回滚的线上版本的完整过程。它不仅仅是一个“点击发布”动作，而是一条贯穿校验、构建、制品生成、权限控制、环境投放、灰度验证和回滚恢复的工程链路。

对于 AI 而言，页面发布流程不是外围运维知识，而是平台级核心知识对象。因为很多高价值场景都与发布强相关：
- 发布前风险检查
- 发布失败诊断
- 发布记录检索与解释
- 自动生成发布建议
- 发布审批与执行增强 Agent

## 2. 它解决什么问题
页面发布流程主要解决以下问题：
- 把编辑态 Schema 转换为可运行的正式页面版本
- 确保发布对象经过必要的校验与审核
- 管理不同环境中的版本状态
- 为回滚、灰度、审计和问题追踪提供依据
- 把页面资产与构建产物、运行环境和租户配置连接起来

## 3. 它不解决什么问题
- 不负责页面本身的结构设计，这属于页面 Schema
- 不负责组件能力定义，这属于组件物料
- 不替代运维平台和 CI/CD 系统，但会与它们协同
- 不天然保证业务正确性，仍需要测试、审批和监控

## 4. 发布流程的典型阶段
一个完整的低代码页面发布流程通常包括：
- Draft：编辑中的草稿态
- Validate：结构校验、依赖校验、权限校验、风险检查
- Build：构建或编译页面运行产物
- Package：生成页面制品、资源索引、依赖清单
- Approve：进入审批或人工确认
- Deploy：投放到目标环境
- Verify：发布后验证与健康检查
- Activate：切换正式生效版本
- Rollback：异常时回滚到上一个稳定版本

这条链路对平台和 AI 都应该是显式可见的，而不是隐藏在一条后端接口里。

## 5. 核心对象
页面发布流程至少涉及以下对象：
- 页面定义对象：页面 ID、Schema、页面类型、所属应用
- 发布任务对象：发布请求、操作者、目标环境、发布时间
- 构建对象：构建状态、日志、产物、耗时、错误信息
- 版本对象：版本号、环境状态、是否生效、来源版本
- 审批对象：审批人、审批结论、审批意见、风险等级
- 运行对象：部署地址、资源路径、依赖版本、健康状态

可以用一个简化结构表达：

```ts
interface PageReleaseTask {
  releaseTaskId: string;
  pageId: string;
  appId: string;
  targetEnv: string;
  sourceVersion: string;
  status: string;
  riskLevel?: string;
  buildInfo?: Record<string, unknown>;
  approvalInfo?: Record<string, unknown>;
  deploymentInfo?: Record<string, unknown>;
  operator?: string;
  createdAt?: string;
}
```

## 6. 与页面 Schema 的关系
页面发布流程的输入通常是页面 Schema 的某个版本，而不是“当前编辑器画面”。因此平台需要明确：
- 发布的是哪个 schema snapshot
- 该 snapshot 对应哪些组件版本和依赖
- schema 是否通过静态校验
- schema 是否引用了未发布或非法物料

对 AI 来说，这意味着发布相关问答和诊断必须能够关联：
- 页面 schema 版本
- 依赖组件清单
- 数据源与逻辑图配置
- 发布失败日志

## 7. 与逻辑编排和数据源的关系
页面发布流程不只是前端页面资源投放，它往往还要确认：
- 逻辑 DAG 是否可执行
- 数据源配置是否完整
- 权限点是否匹配目标环境
- 外部依赖是否可达
- 变量与密钥是否在目标环境存在

如果这些关系没有被显式建模，AI 很难准确回答“为什么这个页面发布失败”。

## 8. 发布前校验
成熟平台在发布前至少应包括：
- Schema 结构校验
- 组件依赖校验
- Setter 配置校验
- Slot 合法性校验
- 逻辑 DAG 校验
- 数据源配置校验
- 权限与租户边界校验
- 风险动作与审批要求检查

AI 可以在这一阶段提供：
- 预检查解释
- 风险摘要
- 缺失项定位
- 自动修复建议

## 9. 发布审批与治理
并非所有页面都应允许直接发布。尤其在企业场景中，需要根据以下因素决定是否审批：
- 页面是否面向生产环境
- 是否涉及高权限入口
- 是否新增高风险动作
- 是否涉及敏感数据展示
- 是否影响多个租户或多个应用

因此，发布流程通常需要与审批系统、审计系统和运行时风险治理对接。

## 10. 发布后的验证与回滚
真正成熟的发布流程不应在“部署成功”时结束。还应包括：
- 页面加载可用性检测
- 资源是否可访问
- 关键接口可用性检测
- 错误率与异常日志观察
- 用户反馈与回滚入口

对于 AI，这一阶段非常适合做：
- 发布后异常总结
- 构建/部署日志解释
- 回滚建议生成
- 相似失败案例检索

## 11. AI 场景中的价值
页面发布流程是 AI 在低代码平台中的高价值切入点之一，因为它具备明确状态、明确风险和高频问题：
- 运维问答助手：查询某页面最近一次发布记录
- 诊断助手：解释为什么构建失败或部署失败
- 审批助手：总结本次发布变更风险
- 发布前检查助手：列出发布阻塞项
- 修复建议助手：结合日志与依赖关系给出排查顺序

## 12. 建议沉淀的知识条目
知识库中建议围绕发布流程沉淀：
- 发布状态机定义
- 环境模型
- 发布校验规则
- 构建错误码与常见原因
- 依赖关系说明
- 审批策略规则
- 回滚机制与条件
- 典型发布事故案例

## 13. 常见误区
- 把发布当成简单保存动作，而不是受控工程流程
- 不记录发布前的 schema snapshot，导致无法回溯
- 只记录最终状态，不记录中间构建和校验信息
- 不把逻辑 DAG、数据源、组件依赖纳入发布前校验
- 出现问题后没有可读的构建日志和诊断信息

## 14. 小结
页面发布流程是低代码平台从“设计态资产”走向“生产态服务”的关键通道。它连接了 Schema、组件物料、逻辑图、数据源、环境、审批和运维治理。对于 AI 系统来说，这是一类非常适合做诊断、问答、审计和受控执行增强的核心领域知识。
