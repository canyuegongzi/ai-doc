# 逻辑编排DAG

## 1. 定义
逻辑编排 DAG（Directed Acyclic Graph）是低代码平台中用于表达页面逻辑、事件响应、数据处理和动作执行关系的有向无环图模型。它的核心作用不是“把节点画出来”，而是把业务逻辑组织成一组可配置、可验证、可执行、可追踪的计算单元与依赖关系。

在低代码平台中，DAG 一般承载以下几类逻辑：
- 页面初始化逻辑
- 用户事件触发逻辑
- 数据查询与转换逻辑
- 条件分支与规则判断逻辑
- 提交、审批、通知等动作逻辑
- 页面间联动与状态同步逻辑

从 AI 视角看，逻辑编排 DAG 是“业务流程语义”和“运行时执行结构”之间的关键桥梁。AI 如果要帮助用户生成逻辑流、检查配置、修复问题或辅助运维，必须理解 DAG，而不能只理解页面 Schema 或组件物料。

## 2. 它解决什么问题
逻辑编排 DAG 主要解决以下问题：
- 把自然语言业务需求转换成可执行逻辑结构
- 把复杂业务流程拆成可组合、可复用的节点与边
- 让逻辑执行路径可显式表达、可校验、可回放
- 让平台能够对逻辑进行静态分析、调试和诊断
- 为 AI 生成、优化和修复逻辑流提供结构化目标对象

例如用户提出“提交前校验表单，成功后保存数据，再发送通知”，平台不能只返回一段描述，而需要生成一张带有触发、校验、写入、通知节点和依赖关系的 DAG。

## 3. 它不解决什么问题
逻辑编排 DAG 不直接解决以下问题：
- 不负责组件渲染与页面布局，这属于页面 Schema
- 不定义组件配置能力，这属于组件物料与 Setter 体系
- 不天然提供权限和审批能力，这属于治理层与 Runtime
- 不等于 BPMN 或完整流程引擎，它更偏页面内与应用内的逻辑编排
- 不天然保证业务正确性，仍需要输入输出约束、类型校验和运行时验证

## 4. 核心组成
一个可被平台和 AI 理解的逻辑编排模型，通常包括以下几部分：
- 图元信息：图标识、名称、场景、版本、作者、租户等
- 节点定义：每个逻辑单元的类型、输入、输出、配置、状态
- 边定义：节点之间的数据依赖或控制依赖关系
- 触发器：图的入口条件，如页面加载、按钮点击、定时器、外部回调
- 变量与上下文：全局变量、页面状态、环境信息、用户信息
- 校验规则：环检测、输入输出兼容性、必填参数、风险动作校验
- 执行策略：同步/异步、并行/串行、失败策略、重试策略

可以用一个简化结构来描述：

```ts
interface LogicGraph {
  graphId: string;
  name: string;
  scene?: string;
  triggers?: GraphTrigger[];
  nodes: LogicNode[];
  edges: LogicEdge[];
  variables?: GraphVariable[];
  meta?: Record<string, unknown>;
}

interface LogicNode {
  nodeId: string;
  nodeType: string;
  title?: string;
  config?: Record<string, unknown>;
  inputSchema?: Record<string, unknown>;
  outputSchema?: Record<string, unknown>;
  runtimePolicy?: Record<string, unknown>;
}

interface LogicEdge {
  edgeId: string;
  sourceNodeId: string;
  targetNodeId: string;
  sourcePort?: string;
  targetPort?: string;
  condition?: Record<string, unknown>;
}
```

## 5. 节点类型设计
低代码平台里的 DAG 节点通常至少应支持以下几类：
- Trigger 节点：页面加载、点击事件、数据变化、外部事件
- Query 节点：调用数据源、加载远程数据、查询字典
- Transform 节点：字段映射、聚合、过滤、表达式计算
- Condition 节点：条件判断、分支选择、规则校验
- Action 节点：提交、更新、删除、跳转、发布、通知
- State 节点：读写页面状态、上下文变量、临时缓存
- Integration 节点：调用外部系统、Webhook、消息系统
- Human/Approval 节点：人工确认、审批、人工补录

从工程和 AI 角度看，节点类型不能只靠名称区分，至少应具备：
- 输入输出结构
- 配置参数
- 风险等级
- 幂等属性
- 可重试性
- 执行副作用说明

## 6. 边与依赖关系
DAG 的价值不仅在节点，更在依赖关系。平台至少要区分两种边：
- 控制流边：谁先执行、成功后走向哪里、失败后走向哪里
- 数据流边：上游输出映射到下游哪个输入字段

如果只保留“连接关系”而没有输入输出映射，后续 AI 很难进行：
- 参数自动补全
- 类型兼容性检查
- 节点替换建议
- 错误定位与修复

因此，边最好至少表达：
- 来源节点与目标节点
- 来源输出口与目标输入口
- 条件表达式
- 分支标签
- 数据映射规则

## 7. 输入输出契约
逻辑编排系统稳定性的核心之一是节点间契约。每个节点都应显式声明：
- 它接收什么输入
- 它输出什么结构
- 哪些字段必填
- 哪些字段敏感
- 哪些字段可能为空

这对于 AI 尤其重要。AI 在生成 DAG 时，如果看不到节点 I/O 契约，只能靠名称猜测；一旦有结构化契约，就能做更可靠的：
- 自动补线
- 参数映射
- 类型校验
- 错误提示
- 节点替换

## 8. 与页面 Schema 的关系
逻辑编排 DAG 与页面 Schema 是相互关联但边界不同的两个对象：
- 页面 Schema 描述页面长什么样、有什么组件
- 逻辑 DAG 描述页面在什么条件下做什么事

两者通常通过以下关系连接：
- 页面中的组件事件触发 DAG
- DAG 节点读取或写入页面状态
- DAG 查询结果回填到页面组件
- DAG 动作影响页面路由、弹窗、表单状态等

因此，在知识库建模时，建议显式记录：
- 哪些组件事件可以作为 trigger
- 哪些节点可以读写哪些 schema 节点
- 哪些逻辑图挂载在哪个页面或模块下

## 9. 与数据源模型的关系
很多逻辑图中的核心节点都依赖数据源，因此 DAG 必须理解数据源模型，而不是把数据请求写成自由脚本。至少要能表达：
- 使用哪个数据源
- 输入参数如何来自上游或页面状态
- 输出结果如何绑定回页面或传给下游节点
- 失败时如何处理

在 AI 场景里，这意味着系统可以从自然语言需求中自动推出：
- 需要查询哪个数据源
- 输入参数从哪些字段映射
- 查询结果进入哪个组件或节点
- 是否需要分页、排序、鉴权或缓存

## 10. 与 Agent / Tool 的关系
逻辑编排 DAG 和 Agent / Tool 并不是互斥关系。

更合理的理解是：
- DAG 是低代码平台内显式、可治理的逻辑执行图
- Tool 是可被 Agent 或 Workflow 调用的标准动作对象
- Agent 可以辅助生成 DAG，也可以在 DAG 的某些节点中承担智能决策角色

例如：
- 用户用自然语言描述流程，AI 生成一张 DAG
- 某个“诊断建议”节点内部调用 Agent 分析日志
- 某个“组件推荐”节点先通过 RAG 检索物料知识，再输出候选组件

因此，未来平台更适合的做法不是“DAG 替代 Agent”，而是“DAG 作为受控骨架，Agent 作为局部增强能力”。

## 11. AI 场景下为什么要单独沉淀 DAG 知识
逻辑编排 DAG 是最值得沉淀成领域知识的对象之一，因为它直接决定了 AI 在低代码平台里的几个高价值场景能否成立：
- 自然语言生成逻辑流
- 自动检查逻辑链路是否缺节点、断链或死链
- 自动补全参数映射与节点配置
- 自动识别高风险动作并建议审批节点
- 根据运行报错回溯到节点配置错误
- 对已有逻辑图进行重构、压缩或推荐复用模板

如果没有结构化的 DAG 知识对象，AI 很容易停留在“描述流程”，而无法真正进入“生成和修复执行图”。

## 12. 建议沉淀的 DAG 知识条目
知识库里建议围绕 DAG 沉淀以下类型的知识：
- 节点类型定义
- 节点输入输出 schema
- 常见节点组合模板
- 典型业务流程图模式
- 条件分支规则样例
- 数据映射规则样例
- 常见错误与修复建议
- 高风险动作与审批建议
- 性能瓶颈与并行优化经验

## 13. 常见误区
- 把逻辑图当成纯可视化编辑对象，不建模输入输出契约
- 只建节点，不建控制流和数据流的区别
- 允许自由脚本绕开节点协议，导致 AI 无法理解与治理
- 不做类型检查和环检测，导致生成的 DAG 不可执行
- 不区分高风险动作节点与普通查询节点
- 不沉淀常见模板，导致每次都从零构图

## 14. 小结
逻辑编排 DAG 是低代码平台中的核心领域对象之一。它把页面行为、数据流转、业务动作和执行约束组织成可计算、可验证、可回放的结构。对于 AI 而言，DAG 不是附属能力，而是自然语言需求、平台能力和运行时执行之间的关键桥梁。

只有把 DAG 作为正式知识对象建模，AI 才能从“理解页面”进一步走向“生成逻辑、诊断问题和辅助执行”。
