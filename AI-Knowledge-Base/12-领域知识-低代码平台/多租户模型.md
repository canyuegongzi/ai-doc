# 多租户模型

## 1. 定义
多租户模型是低代码平台用来在同一套平台能力之上服务多个租户、组织或业务域的隔离与共享机制。它决定了平台中的页面、组件、数据源、逻辑图、发布记录、权限和运行资源如何在不同租户之间分配、复用与隔离。

对于 AI 系统而言，多租户模型不是基础设施细节，而是所有知识检索、Agent 执行、审计回放和缓存设计必须依赖的基础边界。如果租户模型不清晰，AI 系统几乎一定会在知识召回、上下文构造或工具调用上出现越权风险。

## 2. 它解决什么问题
多租户模型主要解决：
- 不同租户之间的数据与配置隔离
- 平台能力的复用与共享控制
- 租户级资源、权限、配额和成本治理
- 租户级发布、环境与运行时隔离
- AI 系统中的检索边界和执行边界控制

## 3. 它不解决什么问题
- 不等于权限系统，租户边界和角色权限需要同时存在
- 不等于工作空间模型，但常常会叠加使用
- 不天然保证数据安全，仍需要权限、脱敏、审计等治理能力
- 不仅是数据库字段设计，而是平台级对象关系模型

## 4. 常见租户分层
企业低代码平台通常会存在多层边界，而不只是单一 tenantId：
- Tenant：客户或业务实体的顶层边界
- Workspace / Project / App：租户内的工作区或应用边界
- Environment：同一应用在不同环境中的隔离边界
- User / Role：租户内身份与操作边界

因此，很多对象至少要带以下范围信息：
- tenantId
- workspaceId 或 appId
- envId
- ownership / visibility / shareScope

## 5. 多租户模型的常见形态
常见的租户模型包括：
- 完全隔离型：每个租户独立资源、独立存储、独立运行
- 逻辑隔离型：共享基础设施，按 tenantId 做逻辑隔离
- 混合型：核心数据隔离，部分物料和模板共享

对低代码平台来说，最常见的是混合型：
- 租户自己的页面、数据源、逻辑图和配置隔离
- 平台基础组件、通用模板、通用知识可以共享
- 共享对象仍需要明确可见范围与复制策略

## 6. 领域对象如何体现租户边界
多租户模型必须体现在平台核心对象中，包括：
- 页面 Schema
- 组件物料与模板
- Setter 配置
- 数据源定义
- 逻辑编排 DAG
- 发布记录
- 审计日志
- 知识库文档与索引
- Agent Runtime 上下文

一个简化示例：

```ts
interface TenantScopedObject {
  tenantId: string;
  workspaceId?: string;
  appId?: string;
  envId?: string;
  visibility?: 'private' | 'workspace' | 'tenant' | 'public';
  ownerId?: string;
}
```

## 7. 与知识库和 RAG 的关系
AI 知识检索必须显式理解租户模型。至少要控制：
- 检索时只能查到本租户允许访问的知识
- 共享模板和公共知识应与私有知识分层管理
- 检索结果进入上下文前仍需做二次过滤
- citation、trace、audit 中不能暴露其他租户信息

在低代码平台中，容易跨租户泄漏的知识包括：
- 页面 schema 片段
- 组件配置案例
- 数据源结构
- 发布日志
- 运维分析记录

## 8. 与 Agent / Tool 的关系
Agent 在多租户场景中需要明确多层身份：
- 请求发起者属于哪个租户
- 运行时允许访问哪个租户范围
- tool 执行时在哪个 tenant/app/env 下生效

这意味着 Tool 和 Action 不应只接收业务参数，还应携带租户范围与作用域约束。

## 9. 共享与复用策略
多租户平台不能只强调隔离，还需要考虑复用：
- 基础物料是否全平台共享
- 模板是否允许跨租户复制
- 通用规则和 Prompt 是否租户共享
- 知识库中的平台文档是否公共可见
- 私有案例是否可以抽象后进入公共知识池

因此，共享对象最好显式区分：
- 平台级公共对象
- 租户级私有对象
- 租户内共享对象
- 复制生成对象

## 10. 版本、发布与运行时隔离
多租户模型不仅影响设计态数据，还影响运行时：
- 不同租户可能使用不同组件版本
- 不同租户可能有不同环境变量、密钥和数据源连接
- 发布、回滚和运维操作必须限制在租户边界内
- 缓存、索引、trace、metrics 也要具备租户维度

## 11. AI 场景中的价值
多租户模型是 AI 在企业平台中最重要的约束之一。它直接影响：
- RAG 检索是否越界
- Agent 工具调用是否越权
- Prompt 是否引入其他租户案例
- 审计和回放是否可对外展示
- 成本是否能按租户归集

因此，它不是“安全附录”，而是 AI 平台设计的正式主对象。

## 12. 建议沉淀的知识条目
- 租户层级模型说明
- 对象可见性规则
- 公共物料共享策略
- 租户级索引与缓存策略
- 租户级审计与成本归集规则
- 跨租户复制与脱敏规则
- 租户隔离事故案例

## 13. 常见误区
- 只在数据库层加 tenantId，而不贯穿检索、缓存、日志和运行时
- 把公共知识和私有知识混在同一索引中且缺少过滤
- 共享模板没有复制边界，导致后续变更互相影响
- Agent 执行时没有携带租户作用域
- 回放、日志和导出报告泄漏其他租户信息

## 14. 小结
多租户模型是低代码平台和企业 AI 系统的基础边界模型。它不仅决定平台对象如何隔离和共享，也直接决定 AI 的知识检索、工具执行、审计回放和成本治理能否安全落地。只有把租户模型显式建模并贯穿全链路，平台级 AI 才能真正可控。
