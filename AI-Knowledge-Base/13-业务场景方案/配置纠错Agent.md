# 配置纠错Agent

## 1. 场景定义
配置纠错 Agent 是指在低代码平台中，围绕页面 Schema、组件配置、数据绑定、逻辑编排和运行报错信息，自动识别问题、解释原因并给出修复建议，必要时在受控边界内执行自动修复的 Agent 场景。

它不是普通问答助手，也不是通用代码修复 Agent，而是一类高度依赖平台领域知识和运行时上下文的垂直诊断系统。它的核心价值在于帮助用户快速回答：
- 为什么页面渲染失败
- 哪个组件配置错了
- 哪个字段绑定不存在
- 哪个 slot / setter / props 不匹配
- 哪个逻辑节点断链或参数错误
- 应该如何最小化修复

## 2. 为什么这个场景值得优先建设
配置问题是低代码平台中最常见、最影响体验的一类问题。相比从零生成页面，配置纠错 Agent 有几个明显优势：
- 问题明确，诊断目标清晰
- 有现成错误上下文和已有配置可供分析
- 领域知识边界相对明确
- 能快速体现平台 AI 的专业能力
- 更容易与日志、Schema、RAG、规则校验结合

对企业内部平台来说，这通常是非常高 ROI 的场景。

## 3. 典型输入
配置纠错 Agent 的输入通常包括：
- 页面 schema 或局部 schema 片段
- 组件物料协议
- setter / slot / props 定义
- 数据源与字段绑定信息
- 逻辑 DAG 配置
- 错误日志、运行时异常、堆栈摘要
- 用户描述的现象
- 历史类似问题案例

与通用 Agent 最大的不同在于，这里输入不是纯自然语言，而是“自然语言 + 结构化配置 + 运行时证据”。

## 4. 输出目标
一个成熟的配置纠错 Agent 通常输出：
- 问题定位结果
- 原因解释
- 影响范围说明
- 修复建议
- 参考依据或引用
- 风险提示
- 必要时的修复补丁草稿或自动修复动作

在高治理要求场景下，它还应支持：
- 生成可审阅的 diff
- 进入人工确认
- 记录审计轨迹

## 5. 为什么它适合 Agent 而不仅是 RAG
RAG 可以帮助“查知识”，但配置纠错通常还需要：
- 收集上下文
- 读取页面配置
- 校验组件协议
- 分析运行时报错
- 尝试局部修复
- 再次验证修复结果

因此它天然更接近 Agent 闭环：
- Understand：识别错误类型
- Retrieve：检索相关协议和历史案例
- Inspect：读取相关 schema、数据源、逻辑图
- Diagnose：形成可能原因列表
- Propose：生成修复方案
- Validate：校验修复是否合理
- Escalate：必要时提交人工确认

## 6. 推荐的技术架构
一个成熟的配置纠错 Agent 通常包括：
- 问题分类器：先判断是渲染、绑定、slot、setter、逻辑还是数据源问题
- Retrieval 模块：召回组件知识、规则、历史案例
- Context collector：读取相关 schema 节点、数据源定义、日志和 trace
- Diagnosis planner：规划诊断顺序
- Validator：规则校验、协议校验、类型校验
- Repair generator：生成修复建议或补丁草稿
- Execution guard：限制自动修复范围，必要时走人工确认

## 7. 典型问题分类
配置纠错场景中常见问题包括：
- 组件不存在或版本不兼容
- props 类型错误
- setter 配置与字段类型不匹配
- slot 配置结构非法
- 数据绑定字段不存在
- 数据源参数缺失或返回结构不匹配
- 逻辑 DAG 节点断链
- 事件引用了不存在的动作
- 页面 schema 节点层级异常

如果先把问题类型分类清楚，后续检索和修复成功率会高很多。

## 8. 配置纠错中的知识源
这个场景依赖的知识源包括：
- 组件物料模型
- setter 体系
- slot 机制
- 数据源模型
- 逻辑编排 DAG 模型
- 发布与运行时日志
- 常见错误码与错误模式
- 历史修复案例
- 平台校验规则

这说明它本质上是“领域知识 + 运行时证据 + 规则引擎”的联合问题。

## 9. 自动修复的边界
并不是所有配置问题都适合自动修复。较适合自动修复的包括：
- 明显的字段命名错误
- 缺失的默认值补齐
- 已知兼容性迁移规则
- 安全的结构修复

而以下情况更适合只给建议或进入人工确认：
- 会改变页面行为的修复
- 涉及数据写入和逻辑动作的修复
- 无法确定唯一正确解的修复
- 生产环境中的高风险页面修复

## 10. 与治理体系的关系
配置纠错 Agent 要真正上线，必须接入治理：
- 权限：只能读取和修改有权限的页面与应用
- 审批：高风险修复需要人工确认
- 审计：记录诊断过程、引用依据、补丁内容
- 回放：出现误修时可追溯
- Evals：持续衡量诊断准确率和修复采纳率

## 11. AI 带来的实际价值
这个场景的业务价值通常体现在：
- 缩短从报错到定位的时间
- 降低对平台专家的依赖
- 让新手也能理解配置错误原因
- 将重复性问题沉淀成可复用经验
- 逐步形成“诊断 -> 修复 -> 验证 -> 复盘”的闭环能力

## 12. 成功指标
建议重点关注：
- 问题定位准确率
- 修复建议采纳率
- 自动修复成功率
- 平均问题处理时长
- 相同问题重复发生率是否下降
- 用户满意度与人工升级率

## 13. 常见误区
- 只拿日志做问答，不读取真实配置对象
- 不做问题分类，所有问题都走一条诊断链
- 没有修复边界，直接让 Agent 改配置
- 只给结论，不给依据和影响范围
- 不把历史案例和规则校验纳入诊断过程

## 14. 小结
配置纠错 Agent 是低代码平台最有实用价值的 AI 场景之一。它把领域知识、运行时证据、规则校验和受控执行结合起来，不只是“解释为什么错”，更是帮助用户“更快修对”。如果边界控制得当，这个场景非常适合作为平台从问答式 AI 走向执行增强 AI 的关键一步。
