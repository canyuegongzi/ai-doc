# 逻辑编排Agent

## 1. 场景定义
逻辑编排 Agent 是指在低代码平台中，围绕页面行为、业务流程、数据流转和动作执行，帮助用户理解需求、生成逻辑 DAG、补全节点配置、检查链路问题并在受控边界内辅助修复的 Agent 场景。

它不是简单的流程问答，也不是通用流程自动化机器人，而是一类高度依赖平台节点协议、数据源模型、页面 Schema 和执行约束的垂直 Agent。它的重点不在“能不能自由思考”，而在：
- 能否把自然语言目标拆成逻辑步骤
- 能否生成可执行的 DAG 草稿
- 能否识别输入输出依赖关系
- 能否发现断链、死链、类型不匹配等问题
- 能否在治理边界内建议或执行修复

## 2. 为什么这个场景有价值
低代码平台中的逻辑编排通常比页面布局更难，因为它需要同时处理：
- 触发条件
- 数据输入输出
- 条件分支
- 动作顺序
- 异常处理
- 审批与人工确认

对很多用户来说，真正的门槛不在“拖一个组件上来”，而在“页面该怎么动起来”。因此，逻辑编排 Agent 往往是平台 AI 能力从“好用”走向“有生产力”的关键场景之一。

## 3. 典型用户需求
这类场景中常见的需求包括：
- 当审批通过后更新状态并发送通知
- 页面加载时先查详情，再回填表单
- 点击按钮后校验输入，成功后提交数据并跳转
- 当某个字段变化时联动另一个组件
- 需要一个审批流程的逻辑骨架
- 帮我看这张逻辑图为什么不执行

这些需求天然适合从自然语言进入结构化任务建模，再进入 DAG 生成或诊断。

## 4. 场景目标
逻辑编排 Agent 的目标通常包括：
- 把用户需求转换成结构化任务描述
- 规划节点类型和执行顺序
- 生成逻辑 DAG 草稿
- 自动映射数据源和变量输入输出
- 做静态校验与风险检查
- 对已有 DAG 进行诊断、重构和修复建议

更进一步，它还可以支持：
- 逻辑模板推荐
- 局部节点替换
- 并行/串行优化建议
- 高风险动作审批建议

## 5. 为什么它比普通 RAG 更像 Agent
RAG 可以回答“怎么配置逻辑”，但逻辑编排 Agent 还需要做闭环动作：
- 理解目标
- 拆解任务
- 选择节点类型
- 生成图结构
- 读取已有 DAG 和数据源
- 校验输入输出兼容性
- 修复局部配置

因此它天然适合按 Agent Runtime 设计，而不是停留在问答层。

## 6. 推荐的总体架构
一个成熟的逻辑编排 Agent 通常包括：
- Goal understanding：把自然语言需求转成流程任务模型
- Retrieval：召回节点定义、DAG 模板、案例和规则
- Planner：规划逻辑阶段、节点类别和分支结构
- Graph builder：生成 DAG 草稿或局部 patch
- Validator：做环检测、类型检查、依赖检查和风险校验
- Repair / refine：补全参数、纠正链路、提出优化建议
- Human-in-the-loop：对高风险动作或不确定方案进入人工确认

## 7. 依赖的知识源
这个场景依赖的知识通常包括：
- 逻辑编排 DAG 模型
- 节点类型定义
- 节点输入输出 schema
- 数据源模型
- 页面 Schema 与组件事件模型
- 常见业务逻辑模板
- 逻辑错误案例与修复经验
- 风险动作分类与审批规则

如果这些知识没有结构化沉淀，Agent 很容易生成“看起来像流程，但跑不起来”的图。

## 8. 典型执行链路
一个典型逻辑编排 Agent 的执行链路可以是：
1. 识别用户是“新建逻辑”还是“诊断已有逻辑”
2. 提取触发条件、核心动作、依赖数据、成功/失败分支
3. 召回相似逻辑模板和相关节点
4. 规划图骨架
5. 生成节点和边
6. 做输入输出契约校验
7. 标出缺失参数或需要人工确认的部分
8. 输出 DAG 草稿或修复建议

## 9. 与页面 Schema、数据源、Tool 的关系
逻辑编排 Agent 不能孤立存在，它至少要理解：
- 页面 Schema：有哪些组件事件可以作为 trigger
- 数据源模型：查询节点和动作节点如何取数与回填
- Tool / Action：哪些节点具有真实副作用，哪些需要审批
- 权限系统：谁可以创建、修改、发布逻辑图

因此它实际上是低代码平台多个领域模型的交汇点。

## 10. 生成与诊断两种模式
这个场景通常应明确区分两种工作模式：
- 生成模式：从需求描述生成 DAG 草稿
- 诊断模式：分析已有逻辑图的问题并给建议

生成模式更依赖模板、检索和规划；诊断模式更依赖上下文读取、规则校验和错误分类。把两者分开建模，系统会稳定很多。

## 11. 自动执行边界
逻辑编排 Agent 可以辅助生成和修复，但不是所有操作都适合自动落地。较适合自动执行的包括：
- 安全的参数补全
- 节点默认值补齐
- 已知模板的骨架生成
- 可验证的连接修复

更高风险的操作应进入人工确认：
- 新增写数据或发通知类节点
- 修改审批或发布链路
- 删除已有关键节点
- 大范围重构逻辑图

## 12. 成功指标
建议重点关注：
- 逻辑草稿可执行率
- 生成后人工修改量
- DAG 诊断准确率
- 修复建议采纳率
- 问题定位耗时下降幅度
- 高风险误修率

## 13. 常见误区
- 把逻辑编排 Agent 当成自由文本问答
- 不做节点和边的结构化建模
- 不检查输入输出契约和节点风险
- 只让模型生成图，不做校验和修复
- 不把高风险动作和审批边界纳入设计

## 14. 小结
逻辑编排 Agent 是低代码平台中最能体现“AI 不只是回答，而是帮助做事”的场景之一。它连接自然语言需求、DAG 结构、数据源、页面事件和受控执行边界。只有把它设计成“检索 + 规划 + 生成 + 校验 + HITL”的闭环系统，才能真正稳定落地。
