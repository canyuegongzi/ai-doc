# Schema生成增强

## 1. 场景定义
Schema 生成增强是指在低代码平台中，利用大模型结合知识检索、模板约束、组件协议和页面上下文，辅助生成页面 Schema、表单配置、布局结构、数据绑定和逻辑骨架的场景。

它不是“让模型自由生成一段 JSON”，而是让模型在平台约束下生成可落地、可校验、可继续编辑的结构化页面定义。因此，这个场景的核心并不是模型文采，而是：
- 是否使用了正确的组件
- 是否遵守了页面 schema 协议
- 是否正确映射了数据模型
- 是否避免了常见配置错误
- 是否输出了可继续编排和修正的结构化结果

## 2. 为什么这个场景重要
在低代码平台里，Schema 生成增强往往是用户感知价值最强的 AI 能力之一。因为它可以显著降低：
- 页面搭建门槛
- 表单和列表页面的重复配置成本
- 新手理解平台协议的难度
- 从需求描述到可用页面草稿的时间

同时，它也非常适合沉淀成平台级能力，因为它天然会复用：
- 组件物料知识
- 页面模板知识
- 数据源模型知识
- 逻辑编排知识
- Prompt 模板、结构化输出与评测体系

## 3. 典型输入与输出
典型输入包括：
- 自然语言需求描述
- 数据模型字段定义
- 页面类型或业务场景
- 参考页面/模板
- 交互约束与风格要求

典型输出包括：
- 页面 schema 草稿
- 组件树结构
- 字段与数据源绑定结果
- 初步逻辑流或交互骨架
- 结构化校验结果与修复建议

## 4. 为什么不能只靠 Prompt
很多团队初做这个场景时，会尝试“写一个很长的 Prompt，让模型直接吐出 Schema”。这通常会很不稳定，原因包括：
- 组件协议复杂，难靠记忆准确生成
- 页面布局和字段映射依赖大量领域知识
- 不同租户、不同场景可能有不同模板和规范
- 输出结构越复杂，纯 Prompt 越容易出现局部错误
- 少量错误就可能导致页面不可渲染或不可编辑

因此，成熟方案通常是“生成增强”而不是“纯生成”，核心依赖：
- RAG 检索模板和物料知识
- 结构化输出约束
- 生成后校验与修复
- 多阶段生成而不是一步生成

## 5. 推荐的总体架构
一个较成熟的 Schema 生成增强链路通常包括：
- Requirement understanding：理解页面类型、目标对象、核心字段、交互要求
- Retrieval：召回页面模板、组件案例、字段映射规则、平台规范
- Planning：先决定页面骨架和组件组合，再决定细节配置
- Structured generation：按 schema 协议输出结构化草稿
- Validation：校验组件、字段、slot、setter、数据源绑定是否合法
- Repair：对局部错误自动修复或提出人工补充项

这条链路比单轮生成复杂，但稳定性会高很多。

## 6. 核心知识源
这个场景最关键的知识源包括：
- 页面 schema 模型
- 组件物料定义
- setter 体系
- slot 机制
- 数据源模型
- 常见页面模板
- 历史成功页面样例
- 字段映射规则与命名规范
- 逻辑 DAG 模板
- 发布与校验规则

如果这些知识没有结构化沉淀，模型很难稳定生成可执行结果。

## 7. 场景中的 RAG 角色
RAG 在这个场景中不是为了“回答问题”，而是为了给生成过程提供约束和参考。它通常承担：
- 检索相似页面模板
- 检索适合当前场景的组件组合
- 检索字段与组件的映射案例
- 检索某类页面的布局范式
- 检索常见校验规则和禁忌配置

换句话说，这里的检索结果更像“生成参考包”，而不是最终答案。

## 8. 推荐的生成粒度
Schema 生成通常不适合一步到位生成全部内容，更适合分阶段：
- 第一步：确定页面类型与布局骨架
- 第二步：确定组件树与字段映射
- 第三步：补充 props / setter / slots 细节
- 第四步：补充数据源与逻辑骨架
- 第五步：做校验和修复

这种分层生成方式可以显著降低错误扩散。

## 9. 与低代码领域模型的关系
Schema 生成增强必须显式复用以下领域模型：
- 页面 Schema 模型：决定输出协议
- 组件物料模型：决定能选哪些组件
- Setter 体系：决定编辑态配置如何补齐
- Slot 机制：决定组件嵌套与结构插入方式
- 数据源模型：决定数据绑定与接口接入
- 逻辑编排 DAG：决定行为和执行骨架

如果脱离这些领域模型，生成结果最多只是“看起来像 JSON”，而不是可用页面定义。

## 10. 校验与修复的重要性
这个场景里，后置校验非常关键。平台至少应检查：
- 组件是否存在
- props 是否符合协议
- setter 配置是否匹配字段类型
- slot 内容是否合法
- 数据绑定是否存在
- 数据源是否配置完整
- 逻辑节点是否引用了不存在对象

生成系统真正可用的标志，不是“能生成”，而是“生成后能校验、能修复、能继续编辑”。

## 11. AI 能带来的增量价值
在这个场景中，AI 的价值不只是减少手工配置，还包括：
- 根据业务描述快速给出页面初稿
- 把结构化领域知识整合成可落地页面
- 帮助用户理解字段为什么映射到某个组件
- 在页面草稿基础上做迭代式修改
- 自动指出生成结果中的缺口和待补充项

## 12. 成功指标
建议重点关注：
- 首次生成页面可渲染率
- 生成结果可编辑率
- 用户采纳率
- 生成后人工修改量
- 生成到可发布草稿的时间
- 校验错误率和自动修复率

## 13. 常见误区
- 把这个场景理解成“让模型吐 JSON”
- 不做模板和案例召回
- 不把组件协议和 setter 约束纳入生成过程
- 只看生成流畅度，不看渲染成功率和编辑成功率
- 不做生成后校验与修复

## 14. 小结
Schema 生成增强是低代码平台里最有业务感知价值的 AI 场景之一，但它本质上是“知识约束下的结构化生成问题”，而不是开放式创作问题。只有把模板、物料、数据源、逻辑编排和校验能力都接进来，这个场景才能稳定落地，并逐步演进成真正可用的页面生成能力。
