# 组件知识问答

## 1. 场景定义
组件知识问答是低代码平台中最典型、最容易先落地的一类 AI 应用场景。它的核心目标是围绕组件物料提供准确、可引用、可解释的知识服务，帮助用户理解组件能做什么、适用于什么场景、如何配置、常见问题是什么。

它不是泛化聊天，也不是单纯文档搜索，而是一种围绕“组件对象”组织的领域问答系统。回答的重点不在于语言多自然，而在于：
- 能否找到正确组件
- 能否解释组件能力边界
- 能否给出正确配置建议
- 能否引用权威来源
- 能否结合低代码平台自己的物料协议和 Schema 约束

## 2. 为什么优先做这个场景
组件知识问答适合作为低代码 AI 的第一批落地场景，原因在于：
- 用户需求高频，几乎所有设计器用户都会遇到
- 组件知识天然结构化，适合做领域知识建模
- 业务风险相对低，先从问答切入更容易建立信任
- 可直接复用 RAG、Metadata、Citation、权限过滤等底座能力
- 后续可自然延伸到组件推荐、Schema 生成、配置纠错等更复杂能力

## 3. 典型用户问题
这类场景下用户常见问题包括：
- 这个组件是做什么的
- 哪个组件适合审批表单/列表展示/多选输入
- 这个组件有哪些 props 和 slots
- 某个属性应该怎么配
- 为什么某个 setter 配置后没有生效
- 有哪些同类组件可以替代
- 是否有示例 schema
- 这个组件常见报错是什么

这些问题看起来像 FAQ，但实际上往往需要跨文档、跨物料协议、跨案例进行综合回答。

## 4. 场景目标
组件知识问答的目标通常包括：
- 快速定位正确组件或相关组件
- 提供准确的组件能力说明
- 给出配置方法、示例与注意事项
- 返回可追溯引用
- 支持进一步跳转到物料详情、案例或示例 schema

在成熟阶段，它还应支持：
- 组件对比
- 场景化推荐
- 与页面 Schema 上下文结合的问答
- 与配置诊断联动

## 5. 适合接入的知识源
该场景的知识源通常包括：
- 组件物料定义
- props schema
- setter schema
- slot 定义
- 组件使用文档
- 示例页面 schema
- 组件分类与标签
- 常见问题与历史问题单
- 组件版本变更记录
- 设计规范与场景建议

比起通用文档问答，这类场景更强调“对象化知识”而不是纯文本块。

## 6. 推荐的知识建模方式
组件知识不建议只按文档 chunk 存储，而应该以组件为核心对象组织知识。建议至少建模：
- componentName
- title
- category
- description
- propsSchema
- setterSchema
- slots
- events
- tags
- scenarios
- examples
- commonIssues
- relatedComponents
- versionInfo

这样做的价值在于，检索时不仅能召回描述文本，还能召回结构化协议对象，后续更方便做：
- 属性问答
- 组件比较
- 推荐与过滤
- 配置生成

## 7. 典型架构
一个成熟的组件知识问答系统通常包括：
- Query understanding：识别用户在问什么类型的问题
- Entity extraction：识别组件名、属性名、场景词、错误词
- Retrieval：混合召回组件定义、文档、示例和 FAQ
- Rerank：按组件相关性、版本、租户可见范围排序
- Context builder：组装组件摘要、协议片段、示例与出处
- Answer generation：生成准确回答，并提供 citation

如果平台已有组件目录服务，通常还应补一层对象查询：
- 先判断能否命中明确组件对象
- 再补文档和案例检索

## 8. 检索设计建议
这类场景下，检索最好组合多种方式：
- 组件名精确匹配
- 分类/标签过滤
- 关键词匹配 props / slots / setter 名称
- 向量检索场景描述与组件说明
- 元数据过滤版本、租户、适用范围
- 示例 schema 关联召回

很多时候“哪个组件适合做 X”并不是单纯文档问题，而是“场景 -> 组件对象”的检索与排序问题。

## 9. 回答结构建议
相比自由回答，这类场景更适合结构化回答。一个较好的回答通常包括：
- 推荐或命中的组件名称
- 简短说明
- 关键能力点
- 适用场景
- 关键配置项
- 注意事项
- 示例或下一步建议
- 引用来源

例如可以返回：
- 推荐组件
- 为什么推荐
- 需要重点配置的 3 个字段
- 是否有替代方案
- 查看示例 schema 的入口

## 10. 与其他场景的关系
组件知识问答通常是多个更复杂场景的上游能力：
- 组件推荐：在问答基础上增强排序与比较
- Schema 生成：把组件知识转化为生成约束
- 配置纠错：把组件协议和错误案例用于诊断
- 页面与组件推荐：在场景描述下做对象选择

因此，这一场景做得好，后面很多能力都会更稳。

## 11. 面向低代码平台的特殊点
与通用 SaaS 文档问答不同，低代码平台中的组件知识问答有几个特殊要求：
- 要能理解 props / setter / slots / events 等平台语义
- 要能识别组件协议中的结构化字段
- 要能处理组件版本差异
- 要能关联示例 schema 和页面模板
- 要能根据租户和权限过滤组件可见性

## 12. 成功指标
这类场景建议重点看以下指标：
- 问题命中正确组件的比例
- 属性与配置建议的准确率
- citation 覆盖率
- 推荐组件被采纳率
- 用户继续追问率
- “查文档跳出率”是否下降

## 13. 常见误区
- 把组件知识问答做成纯文档聊天，不做对象建模
- 只检索描述，不检索 props / setter / slot 协议
- 不做版本和租户过滤
- 回答里没有明确引用和下一步行动建议
- 不与后续推荐、生成、诊断场景共享知识底座

## 14. 小结
组件知识问答是低代码平台最适合优先建设的 AI 场景之一。它风险低、价值高、知识边界清晰，而且能直接为组件推荐、Schema 生成和配置诊断提供上游能力。真正成熟的实现方式不是简单文档问答，而是基于组件对象、结构化协议和场景语义的领域化 RAG 系统。
