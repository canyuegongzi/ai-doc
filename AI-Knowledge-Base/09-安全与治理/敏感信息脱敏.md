# 敏感信息脱敏

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Sensitive Data Redaction / Masking 设计，重点回答：

- 为什么 AI 系统中的脱敏不能只依赖数据库或前端展示层
- 脱敏与权限过滤、身份透传、审计治理的边界是什么
- 在 RAG、Tool、Agent Runtime、Replay 场景下如何做全链路脱敏

## 2. 什么是敏感信息脱敏

敏感信息脱敏可以理解为：

对系统在检索、工具调用、上下文装配、模型输入、模型输出、日志、审计、回放等环节中处理到的敏感数据进行识别、裁剪、替换、掩码、摘要化或降精度处理，以降低泄露风险的机制。

常见敏感信息包括：

- PII：姓名、手机号、邮箱、身份证号
- 凭证类：AK/SK、token、cookie、password
- 业务敏感字段：成本、价格、内部配置、客户名单
- 平台敏感信息：system prompt、内部策略、审批口径

用一句话概括：

脱敏的本质，是在“允许系统继续工作”的前提下，把不必要暴露的敏感内容降到最少。

## 3. 为什么 AI 系统更需要脱敏

### 3.1 AI 会把数据带入更多不可忽视的中间层

传统系统里，敏感数据通常只在：

- 接口请求
- 后端处理
- 数据库存储

中流转。

但在 AI 系统中，还会额外流入：

- prompt context
- retrieval result set
- tool outputs
- chain state
- trace logs
- audit payload
- replay snapshot

因此，脱敏不再只是展示层问题，而是运行时问题。

### 3.2 LLM 输出会放大敏感信息泄露风险

模型会：

- 重组敏感数据
- 归纳隐藏信息
- 生成本不该直接展示的摘要

所以即使原始字段没有原样输出，也可能通过回答被间接暴露。

### 3.3 近两年的安全共识越来越强调敏感信息披露风险

OWASP LLM Top 10 已明确把 Sensitive Information Disclosure 列为核心风险之一。OpenAI 在 agents guardrails 指南中也明确强调数据隐私风险和多层 guardrails 的必要性。

## 4. 脱敏要解决什么问题

### 4.1 不把不该进入模型的内容送入模型

### 4.2 不把不该展示给用户的内容显示出来

### 4.3 不把不该进入日志和审计的内容原样保留

### 4.4 让工具调用和 replay 仍然可用但不暴露敏感细节

## 5. 脱敏与权限过滤的区别

两者常常一起出现，但边界不同。

### 5.1 权限过滤

决定“能不能看”。

### 5.2 脱敏

决定“即使能看，看到多少、以什么形式看”。

更成熟的系统通常同时支持：

- 不可访问：直接过滤掉
- 可访问但脱敏：部分掩码、摘要化
- 审计可访问但模型不可访问：只保留后台记录

## 6. 近两年的关键变化

### 6.1 脱敏正从展示层策略扩展为 Prompt / Tool / Trace 策略

AI 链路中，敏感信息可能在最终 UI 之前就已经泄露到模型上下文或 trace 中，因此脱敏必须前移。

### 6.2 自动识别 + 规则脱敏 + 模型辅助脱敏开始组合使用

成熟系统通常不会只依赖一种方式，而会结合：

- regex / pattern
- schema-based field policy
- classifier / detector
- 人工审查规则

### 6.3 脱敏对象从用户数据扩展到系统自身资产

除了用户业务数据，还要关注：

- prompt 泄露
- policy 泄露
- tool credential 泄露
- hidden instructions 泄露

## 7. 推荐的脱敏分层

建议至少分成五层：

### 7.1 输入层脱敏

在用户输入或外部数据进入系统时做初步检查和遮罩。

### 7.2 检索层脱敏

在 retrieval result 进入 context builder 前处理敏感字段。

### 7.3 Tool 输出脱敏

工具执行结果返回给 runtime / model / UI 前做字段级 redaction。

### 7.4 输出层脱敏

对最终回答、citation、结构化输出做展示级敏感信息检查。

### 7.5 审计与回放脱敏

日志、trace、replay snapshot 需单独做保留策略与 redaction。

## 8. 推荐的脱敏方式

### 8.1 全遮蔽

例如：

- password
- token
- secretKey

### 8.2 局部掩码

例如：

- 手机号显示前三后四
- 邮箱部分星号替换

### 8.3 降精度

例如：

- 金额区间化
- 时间粒度降级
- 地址模糊到城市级

### 8.4 摘要化

例如只保留“存在异常配置”而不暴露完整值。

### 8.5 哈希 / 标识替换

用于审计关联但不暴露原值。

## 9. 在 RAG 中如何落地脱敏

### 9.1 ingestion 阶段标注敏感字段

在知识对象 metadata 中明确：

- `sensitivityLevel`
- `fieldPolicy`
- `displayPolicy`

### 9.2 recall 前优先过滤，必要时 recall 后再脱敏

对完全不可访问数据，应直接过滤；对可访问但应有限展示的数据，再做脱敏。

### 9.3 context builder 阶段避免原样拼接敏感内容

例如：

- 不把密钥和配置值直接塞进 prompt
- 对内部备注摘要化

### 9.4 citation 输出时避免暴露敏感原文

必要时只返回 source title 或安全摘要，而不是原始片段。

## 10. 在 Tool / Action 中如何落地脱敏

### 10.1 工具返回结果默认不直接透出内部字段

### 10.2 高风险工具输出应分成：

- machine-safe output
- user-safe output
- audit-only output

### 10.3 Action 执行前后都应防止凭证泄露

例如：

- 不把 approval token、internal receipt 原样传给模型
- 不把系统密钥写入 trace

## 11. 在日志、审计、回放中如何落地脱敏

### 11.1 Trace 脱敏

记录链路，但不保留全部原始数据。

### 11.2 Audit 脱敏

满足可追责，但避免保存明文敏感字段。

### 11.3 Replay 脱敏

回放时可使用：

- sanitized snapshot
- mock outputs
- token/secret replacement

## 12. 面向低代码平台的典型场景

### 12.1 组件与页面问答

应避免暴露：

- 内部备注
- 租户私有配置
- 内部接口地址

### 12.2 Schema 生成与修复

应避免将：

- secret 字段默认值
- 私有数据源连接信息
- 内部审批说明

直接带入模型上下文。

### 12.3 运维分析

应对：

- 日志中的 token
- 配置中的 AK/SK
- 内部告警接收人

做自动掩码或摘要化。

## 13. 常见误区

### 13.1 认为权限控制做好了就不需要脱敏

权限和最小暴露是两回事。

### 13.2 只对最终 UI 做脱敏

敏感信息可能早已进入模型上下文或 trace。

### 13.3 所有敏感字段一刀切删除

这会让系统失去可用性。更合理的是分级脱敏。

### 13.4 审计和 replay 不做脱敏

后台链路常常是最容易被忽略的泄露点。

### 13.5 把系统 prompt、策略和审批规则排除在敏感数据之外

这些同样可能属于高价值内部信息。

## 14. 一句话总结

敏感信息脱敏的核心，是把“最小暴露”原则贯彻到 AI 执行全链路中，在检索、工具、上下文、输出、日志和回放各层分别控制敏感数据以何种形式存在；近两年的最佳实践也越来越明确，企业级 AI 系统的安全性，不只取决于能否控制访问，还取决于能否控制暴露。
