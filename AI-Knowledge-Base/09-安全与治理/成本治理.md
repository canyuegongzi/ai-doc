# 成本治理

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Cost Governance 设计，重点回答：

- 为什么 AI 系统的成本控制不能只看模型账单
- 成本治理应如何贯穿模型、检索、工具、缓存、Agent Runtime 和场景层
- 近两年为什么行业越来越强调 usage visibility、budget control、cost-aware routing 和 graceful degradation

## 2. 什么是成本治理

成本治理可以理解为：

对 AI 系统在模型调用、向量化、检索、存储、工具执行、长任务运行、日志与回放等环节产生的资源消耗和费用进行可见化、可预算化、可限制、可优化和可退化控制的一整套机制。

它不只是“月底看账单”，而是要实时回答：

- 钱花在了哪里
- 哪类场景成本最高
- 哪类用户或租户消耗异常
- 当前调用是否值得继续
- 超预算时系统如何退化

用一句话概括：

成本治理的本质，是把 AI 系统的资源消耗从事后财务问题，前移成运行时治理问题。

## 3. 为什么企业 AI 系统必须重视成本治理

### 3.1 AI 成本来源远不止 LLM token

实际成本通常包括：

- chat / response model
- embedding
- rerank
- vector storage
- search service
- tool execution
- long-running agent runtime
- logging / tracing / replay storage

如果只盯模型单价，很容易低估整体成本。

### 3.2 Agent 和 RAG 会把单次请求扩展成多步链路

一次复杂任务可能包含：

- 多次检索
- 多轮模型调用
- 多个工具调用
- 多次重试
- HITL 等待和恢复

这意味着复杂场景的边际成本会快速上升。

### 3.3 近两年的最佳实践越来越强调 cost-aware systems

OpenAI、AWS 等公开实践都在强调 usage 可见性、预算控制、分场景路由和降级策略，说明成本治理已成为从 demo 走向生产的分水岭之一。

## 4. 成本治理要解决什么问题

### 4.1 成本可见性

知道每一类成本发生在哪里。

### 4.2 成本归因

知道是哪个：

- 场景
- 租户
- 应用
- 用户
- Agent 任务
- Tool 链路

消耗了资源。

### 4.3 成本控制

在运行时限制超预算行为。

### 4.4 成本优化

通过缓存、路由、压缩、降级等方式降低单次成本。

## 5. 近两年的关键变化

### 5.1 成本治理正在从离线报表转向运行时控制

过去更多是月底导出 usage；现在更成熟的系统会在运行时做：

- token budget
- step budget
- tool budget
- max cost per task

### 5.2 cost-aware routing 成为主流实践

模型选择已经不只是比质量，还要比：

- 单次成本
- latency
- 成功率
- 任务复杂度匹配度

### 5.3 长任务和 Agent 让成本上限控制变得更重要

没有步数、预算和降级边界的 Agent，很容易成为成本黑洞。

## 6. 推荐的成本分层

建议至少分成六层：

### 6.1 模型成本

- 输入 token
- 输出 token
- reasoning token / hidden reasoning overhead
- multimodal input cost

### 6.2 检索成本

- embedding
- rerank
- 搜索服务
- 数据读取

### 6.3 存储成本

- 向量存储
- 文档存储
- cache
- trace / replay retention

### 6.4 Tool 执行成本

- 内部服务调用
- 外部 API 调用
- 长任务执行

### 6.5 Runtime 成本

- state persistence
- queue / scheduler
- human review backlog

### 6.6 场景成本

从业务角度看每种场景的平均成本与 ROI。

## 7. 推荐的成本归因维度

建议至少按以下维度归因：

- `tenantId`
- `appId`
- `sceneType`
- `userId`
- `taskId`
- `provider`
- `model`
- `toolName`
- `runtimeVersion`
- `requestDate`

这样才能区分：

- 哪个场景值得继续优化
- 哪个租户超常消耗
- 哪类工具链路成本失控

## 8. 推荐的成本控制策略

### 8.1 token budget

限制单次请求、单轮回答、单任务的 token 上限。

### 8.2 step budget

限制 Agent 最大步数。

### 8.3 tool budget

限制某类高成本工具调用次数。

### 8.4 task budget

为整条任务链设置最大成本上限。

### 8.5 tenant / app quota

按租户、应用或团队配置配额和预算。

## 9. 推荐的成本优化手段

### 9.1 路由优化

简单任务使用更低成本模型，复杂任务才升级高性能模型。

### 9.2 上下文优化

通过：

- query rewrite
- better retrieval
- context compression
- citation pruning

减少无效 token。

### 9.3 缓存优化

对：

- embedding
- retrieval result
- grounded answer
- tool result

做可控缓存。

### 9.4 编排优化

避免：

- 重复工具调用
- 无意义循环
- 高风险重复重试

### 9.5 生命周期优化

控制：

- trace retention
- replay retention
- old index cleanup

## 10. 成本治理与模型路由的关系

模型层应支持：

- quality tier
- cost tier
- latency tier
- fallback tier

更成熟的系统会按：

- FAQ 问答
- 生成辅助
- Tool routing
- 高风险审批链

分别配置不同模型档位。

## 11. 成本治理与 RAG 的关系

RAG 不只是效果优化，也直接影响成本。

例如：

- 检索更准，能减少无效上下文
- rerank 更稳，能减少过长 prompt
- chunk 策略合理，能减少重复召回
- metadata filter 前置，能减少无意义搜索范围

换句话说：

好的知识工程同时也是成本工程。

## 12. 成本治理与 Agent 的关系

在 Agent 场景下，建议至少控制：

- max turns
- max tool calls
- max retries
- max planning loops
- max cost per run

并对以下情况设置降级：

- 预算耗尽后转建议模式
- 禁止进入高成本多工具分析
- 改用低成本模型总结结果

## 13. 面向低代码平台的典型建议

### 13.1 FAQ / 组件问答

优先用：

- 更轻量模型
- 强检索
- 强缓存

### 13.2 Schema 生成

优先优化：

- 上下文裁剪
- 模板召回
- 结构化输出重试次数

### 13.3 配置诊断 / 运维分析

优先控制：

- 多工具并发上限
- 日志拉取范围
- 高成本模型调用次数

### 13.4 长运行 Agent

必须设置：

- 任务预算
- 步数上限
- 自动降级策略
- 超预算升级人工

## 14. 成本治理与可观测性的关系

成本治理必须建立在 usage visibility 基础上。

建议把成本指标纳入：

- per request
- per scene
- per tenant
- per tool chain
- per model route

否则就无法判断优化是否有效。

## 15. 常见误区

### 15.1 只统计模型 token，不统计整体链路成本

会低估真实运行成本。

### 15.2 成本报表只有月度汇总

缺少实时控制能力。

### 15.3 不区分高价值场景和低价值场景

会把预算浪费在边际价值低的请求上。

### 15.4 没有超预算降级策略

系统容易在成本失控时直接停摆或持续烧钱。

### 15.5 把成本治理与效果治理割裂

真正合理的系统应同时看：

- 质量
- 成本
- 延迟
- 风险

## 16. 一句话总结

成本治理的核心，是把 AI 系统在模型、检索、工具、存储和长任务运行上的消耗全部纳入可观测、可归因、可预算、可降级的运行时控制体系中；近两年的最佳实践也越来越明确，企业级 AI 平台的成熟度，不只看它能做多少事，还看它能否用可接受的成本持续做这些事。
