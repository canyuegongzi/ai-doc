# 身份透传

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的身份透传（Identity Propagation）设计，重点回答：

- 为什么企业 Agent / RAG 系统不能只使用“平台统一服务账号”执行所有操作
- 用户身份、租户身份、应用身份、运行时身份应如何在链路中传递
- 近两年为什么行业越来越强调 runtime context、authorization-aware tools 和 end-user identity binding

## 2. 什么是身份透传

身份透传可以理解为：

在一次 AI 请求从入口层进入，到检索、工具调用、审批、执行、审计的完整链路中，将发起者及相关上下文身份信息持续、准确、结构化地传递给各层组件，并据此执行权限判断、数据裁剪和动作治理。

它不只是“带一个 userId”，而是要明确：

- 谁发起了请求
- 代表哪个租户
- 属于哪个应用或工作空间
- 当前是否处于代理执行 / 系统托管执行模式
- 哪些动作是用户本人权限，哪些是系统代执行权限

用一句话概括：

身份透传的本质，是让 AI 系统始终知道“当前是在替谁做事、能做到哪一步、责任属于谁”。

## 3. 为什么身份透传是企业 AI 系统的基础能力

### 3.1 AI 系统不是独立用户，而是代表真实用户或系统主体执行

企业内部绝大多数 AI 调用都不是匿名行为，而是代表：

- 某个登录用户
- 某个租户下的成员
- 某个业务应用上下文
- 某个后台任务或机器人身份

如果链路中身份丢失，就会导致：

- 检索结果越权
- 工具调用越权
- 审计无法追责
- HITL 无法绑定审批主体

### 3.2 “统一服务账号执行一切”会迅速破坏权限边界

很多系统在早期会偷懒，让 Agent 统一使用平台超级账号查询和操作。这种方式短期上手快，但很快会产生严重问题：

- 模型能看到不该看到的数据
- 普通用户可间接触发高权限动作
- 回放和审计只能看到系统账号，无法看到真实发起人

### 3.3 近两年的运行时设计越来越强调 context binding

LangChain / LangGraph 近年的 runtime 文档越来越强调 `context` 是 tools 和 middleware 的正式输入，意味着：

用户身份、配置、依赖不应靠全局变量或隐式状态传递，而应显式作为 runtime context 输入。

## 4. 身份透传要解决什么问题

### 4.1 检索层授权

在 RAG 中决定：

- 可检索哪些索引
- 可读取哪些文档
- 可看到哪些字段

### 4.2 工具层授权

在 Tool / Action 中决定：

- 可见哪些工具
- 可调用哪些工具
- 可作用于哪些资源范围

### 4.3 动作责任归属

记录真实发起人、批准人、代执行主体和最终执行上下文。

### 4.4 审计与合规

让系统在事后回答：

- 谁发起的
- 谁批准的
- 系统是以谁的身份执行的
- 哪一步做了权限放行

## 5. 近两年的关键变化

### 5.1 身份正在从 API 边界概念，升级为 AI Runtime 上下文概念

过去身份信息更多只在 API 网关或后端服务层处理；现在随着 Agent Runtime 的出现，身份还必须进入：

- tool discovery
- retrieval filter
- planner 决策
- HITL payload
- audit trace

### 5.2 end-user identity binding 越来越重要

尤其在企业 AI 助手和内部代理场景中，系统越来越需要区分：

- 用户本人权限
- 系统代执行权限
- 审批后临时提权

### 5.3 多主体身份链越来越常见

一个动作可能同时涉及：

- requester：请求发起人
- runtime actor：当前执行主体
- approver：审批人
- system principal：底层技术账号

因此，身份模型不能再只用一个字段表示。

## 6. 推荐的身份模型

建议至少显式建模以下身份维度：

### 6.1 用户身份

- `userId`
- `userName`
- `roles`
- `department`
- `permissions`

### 6.2 组织身份

- `tenantId`
- `workspaceId`
- `appId`
- `environment`

### 6.3 会话 / 请求身份

- `sessionId`
- `requestId`
- `traceId`
- `conversationId`

### 6.4 执行身份

- `actingPrincipal`
- `executionMode`
- `delegationType`
- `approvalToken`

建议区分：

- `requesterIdentity`
- `runtimeIdentity`
- `effectiveIdentity`

这样后续权限判断和审计才不会混淆。

## 7. 推荐的身份透传链路

一个较稳的链路通常包括：

1. 入口层认证获取用户和租户身份
2. 编排层将身份写入 runtime context
3. 检索层根据身份做索引与数据过滤
4. Tool Registry 根据身份裁剪工具集合
5. Tool Runtime 根据身份做授权判断
6. 审计层记录真实请求者与代执行主体

换句话说：

身份不应只在入口层存在一次，而应贯穿整条 AI 调用链路。

## 8. 身份透传与 RAG 的关系

在 RAG 中，身份透传至少影响三件事：

### 8.1 可访问知识源范围

不同租户、不同角色、不同应用可能看到不同知识源。

### 8.2 metadata filter 构造

例如：

- `tenantId = currentTenant`
- `appId in currentScopes`
- `permissionScope contains currentRole`

### 8.3 citation 与回答解释

当答案引用受限内容时，系统需要知道引用是否对当前用户可见。

## 9. 身份透传与 Tool / Action 的关系

对于 Tool / Action 层，身份透传应支撑：

- tool discovery
- authorization check
- parameter scope validation
- approval binding
- audit attribution

特别是在动作执行中，应明确区分：

- 这是用户本人有权执行的动作
- 还是系统在审批后代用户执行的动作

## 10. 推荐的运行时上下文设计

在 Runtime 中，建议把身份作为正式 context schema 的一部分，而不是零散散落在请求头或全局变量中。

简化示例：

```ts
interface RuntimeIdentityContext {
  requester: {
    userId: string;
    tenantId: string;
    roles: string[];
  };
  resourceScope: {
    appId?: string;
    workspaceId?: string;
    environment?: 'dev' | 'test' | 'prod';
  };
  execution: {
    mode: 'user' | 'delegated' | 'system';
    approvalToken?: string;
  };
  traceId: string;
}
```

## 11. 面向低代码平台的典型场景

### 11.1 组件问答

身份透传决定：

- 可查哪些组件物料
- 可读哪些内部文档
- 是否能看到租户内定制组件配置

### 11.2 页面生成

身份透传决定：

- 可使用哪些模板
- 可绑定哪些数据模型
- 可创建到哪个应用 / 工作空间

### 11.3 发布与修复动作

身份透传决定：

- 是否具备发布权限
- 是否仅允许 test 环境执行
- 是否需审批令牌才能进入 prod

### 11.4 运维问答与动作

身份透传决定：

- 是否可读取构建日志
- 是否可查看跨租户配置差异
- 是否可触发回滚建议或工单创建

## 12. 常见误区

### 12.1 只传 userId，不传租户与作用域

这样无法做正确的数据隔离和资源约束。

### 12.2 入口认证后，中间链路不再传身份

检索、工具、审计都会失去授权依据。

### 12.3 统一使用系统账号执行所有动作

这会迅速破坏最小权限原则。

### 12.4 审计只记录技术账号，不记录真实请求者

后续责任归属会失真。

### 12.5 不区分 requester 与 approver

审批链路中的责任边界会变得模糊。

## 13. 一句话总结

身份透传的核心，是让用户、租户、作用域和执行主体信息从入口一路贯穿到检索、工具、动作和审计全链路，并据此驱动授权、隔离、审批和追责；近两年的最佳实践也越来越明确，企业级 AI 系统要做到可控，第一件事不是把模型接进来，而是把身份上下文真正传对、传全、传到底。
