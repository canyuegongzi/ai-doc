# 输出约束设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Output Constraint / Structured Output / Guardrails 设计，重点回答：

- 为什么企业 AI 系统不能把模型输出当作自由文本直接消费
- 输出约束应如何在问答、RAG、Tool、Action、Agent Runtime 中落地
- 近两年为什么结构化输出、response schemas、policy-based guardrails 越来越成为主流做法

## 2. 什么是输出约束

输出约束可以理解为：

通过结构、类型、枚举、策略、校验和运行时拦截等方式，对模型最终产出的内容进行限制，使其满足业务可消费、可验证、可审计、可集成的要求。

输出约束不仅关心“说得像不像”，更关心：

- 格式是否正确
- 字段是否完整
- 值是否在允许范围内
- 是否出现不应出现的内容
- 是否足以驱动下游系统继续执行

用一句话概括：

输出约束的本质，是把模型输出从“开放文本”收敛为“受控结果对象”。

## 3. 为什么企业 AI 系统必须做输出约束

### 3.1 自由文本无法稳定接入业务系统

如果模型输出要用于：

- 前端渲染
- 工具调用
- 任务状态推进
- 审批判断
- 动作执行

那么自由文本就会非常脆弱。

### 3.2 不受约束的输出会放大幻觉和越权风险

模型可能：

- 编造字段
- 生成无效 JSON
- 输出超出权限的信息
- 直接给出高风险动作建议

### 3.3 近两年的模型能力已经让结构化输出更可行

OpenAI Structured Outputs、Anthropic Tool Use、Gemini 的 function calling 等能力，都在推动行业从“解析自由文本”转向“直接让模型按结构输出”。

## 4. 输出约束要解决什么问题

### 4.1 结构稳定性

输出格式必须可预测。

### 4.2 类型正确性

字段类型、枚举值、嵌套结构应符合定义。

### 4.3 语义可验证性

即使结构正确，也要进一步验证业务语义是否合法。

### 4.4 安全与合规

输出中不能包含越权信息、敏感信息或危险指令。

## 5. 近两年的关键变化

### 5.1 行业从“Prompt 提示格式”转向“Schema 驱动输出”

过去更多依赖 prompt 写“请输出 JSON”；现在更成熟的系统会直接：

- 给模型 schema
- 让 runtime 校验
- 校验失败时重试或回退

### 5.2 输出约束开始成为 Agent Runtime 的正式控制点

在 Agent 场景中，Planner 输出、Tool 参数、Action 请求、HITL payload、final answer 都越来越倾向于结构化对象，而不是自然语言描述。

### 5.3 Guardrails 正从内容审查扩展为结构、策略和动作边界控制

输出约束现在通常同时包含：

- 结构约束
- 值域约束
- 策略约束
- 风险约束

## 6. 推荐的输出约束分层

建议至少分成四层：

### 6.1 格式约束

例如：

- JSON
- Markdown section schema
- DSL
- 表格结构

### 6.2 类型约束

例如：

- string / boolean / enum
- required fields
- object nesting

### 6.3 业务约束

例如：

- pageType 必须在允许范围内
- targetEnvironment 不能直接是 prod
- riskLevel 与 actionType 必须匹配

### 6.4 安全约束

例如：

- 不输出敏感字段
- 不生成高风险指令
- 不返回越权 citation

## 7. 推荐的输出约束策略

### 7.1 结构化输出优先

只要输出会被系统消费，就优先使用 schema / typed output。

### 7.2 双层校验

建议同时做：

- 模型侧结构约束
- Runtime 侧强校验

### 7.3 校验失败要有明确 fallback

例如：

- 自动重试
- 降级为建议型输出
- 转人工确认
- 退回普通问答

### 7.4 输出与权限策略联动

不同用户和场景应拿到不同粒度输出。

## 8. 在 RAG 中如何落地输出约束

### 8.1 grounded answer 结构化

可将回答拆成：

- answer
- evidence
- citations
- uncertainty
- nextSuggestion

### 8.2 citation 必须与 source 对齐

不能允许模型虚构 citation。

### 8.3 不确定性显式输出

例如：

- `confidence`
- `insufficientEvidence`
- `needsHumanReview`

这比强行生成确定答案更稳。

## 9. 在 Tool / Agent 中如何落地输出约束

### 9.1 Planner 输出必须结构化

例如：

- nextActionType
- targetTool
- riskFlags
- stopCondition

### 9.2 Tool 参数生成必须按 schema 输出

避免“看起来像 JSON”的伪结构。

### 9.3 Action 请求必须经过 runtime 校验

不能因为结构对了就直接执行。

### 9.4 HITL payload 也应结构化

让审批和回放可稳定处理。

## 10. 面向低代码平台的典型建议

### 10.1 Schema 生成

输出应为：

- 结构化 schema object
- 校验结果
- warnings
- 依赖组件列表

### 10.2 配置诊断

输出应为：

- issue list
- root cause summary
- suggested fix
- risk level
- whetherAutoFixAllowed

### 10.3 运维问答

输出应为：

- incident summary
- evidence chain
- suggested action
- approvalRequired

### 10.4 页面发布助手

输出应为：

- publish plan
- risk summary
- validation result
- required approvals

## 11. 推荐的运行时校验链路

1. 给模型输出 schema
2. 获取结构化结果
3. Runtime 进行 schema validation
4. 进行业务规则校验
5. 进行安全规则校验
6. 校验通过再进入后续流程

建议明确区分：

- parse error
- schema error
- business rule error
- security policy error

## 12. 常见误区

### 12.1 让模型“尽量输出 JSON”就算做了输出约束

这远远不够，必须有正式 schema 和 runtime 校验。

### 12.2 结构正确就认为可以执行

结构合法不代表业务合法或安全合法。

### 12.3 所有场景都要求过度严格结构化

用户可读问答和系统执行对象需要不同程度的约束。

### 12.4 校验失败后直接放弃

很多场景应优先重试、降级或转人工。

### 12.5 不把不确定性显式输出

这会迫使模型伪装确定，增加误用风险。

## 13. 一句话总结

输出约束设计的核心，是把模型生成结果收敛成结构可验、语义可判、安全可控的对象，并让这些约束真正进入 Runtime 与业务链路，而不是停留在 prompt 描述层；近两年的最佳实践也越来越明确，企业级 AI 系统的稳定性，很大程度上取决于它是否把输出约束当成正式基础设施来建设。
