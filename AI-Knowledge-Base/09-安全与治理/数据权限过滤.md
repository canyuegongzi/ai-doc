# 数据权限过滤

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Data Permission Filtering 设计，重点回答：

- 为什么 AI 系统的数据权限过滤不能只在最终回答阶段做
- 数据权限过滤在 RAG、Tool、Action、Audit 中应如何前置和分层
- 近两年为什么行业越来越强调 retrieval-time filtering、scope-aware tools 和 grounded access control

## 2. 什么是数据权限过滤

数据权限过滤可以理解为：

根据当前用户、租户、角色、资源范围和任务上下文，对系统中可被检索、读取、引用、操作和展示的数据进行动态裁剪和约束的机制。

它关心的不是“这个系统整体能不能访问数据”，而是：

- 当前请求能读哪些数据
- 当前任务能用哪些字段做上下文
- 当前工具能作用于哪些对象
- 当前回答可以展示哪些内容

用一句话概括：

数据权限过滤的本质，是把“可访问数据集合”从静态权限配置，变成每次 AI 执行时动态计算出来的结果。

## 3. 为什么数据权限过滤必须前置到 AI 链路中

### 3.1 AI 系统会在生成前就消费数据

RAG、Tool、Agent 都会在模型生成答案之前读取大量上下文。如果等到最终回答再过滤，已经太晚了。

### 3.2 检索结果和上下文本身就是泄露面

即使最终答案没直接输出受限内容，只要受限内容进入：

- 模型上下文
- citation 候选
- tool result cache
- trace / audit payload

都可能形成泄露风险。

### 3.3 动作型工具的权限问题本质上也是数据过滤问题

很多高风险动作并不是工具本身危险，而是作用对象越权。

例如：

- 修改了不属于当前应用的页面
- 查看了不属于当前租户的日志
- 对敏感字段生成了总结

## 4. 近两年的关键变化

### 4.1 权限过滤正在从“展示层控制”转向“检索层控制”

现代 RAG 和 Agent 系统越来越强调：

- 在 recall 前过滤
- 在 tool discovery 时过滤
- 在参数绑定时过滤

而不是把过滤放在最终 UI 展示层。

### 4.2 metadata-aware retrieval 让权限过滤更可落地

随着 metadata 设计成熟，系统越来越可以基于：

- tenantId
- appId
- permissionScope
- resourceOwner
- visibilityLevel

在检索阶段进行正式过滤。

### 4.3 数据权限开始与生成约束和审计联动

权限过滤不仅影响检索，还影响：

- context builder
- citation 可见性
- tool output redaction
- audit payload redaction

## 5. 数据权限过滤要解决什么问题

### 5.1 检索可见范围控制

限制可召回的知识对象和数据对象。

### 5.2 字段级可见性控制

限制模型和用户能看到哪些字段。

### 5.3 资源作用范围控制

限制工具和动作只作用于授权对象。

### 5.4 输出展示控制

限制最终回答、citation、日志和审计中出现的敏感信息。

## 6. 推荐的数据权限过滤分层

建议至少分成四层：

### 6.1 Source 级过滤

决定当前请求能访问哪些知识源、索引、表、对象集。

### 6.2 Object 级过滤

决定能访问哪些文档、页面、组件、发布记录、日志对象。

### 6.3 Field 级过滤

决定对象中哪些字段可见，例如：

- secretKey
- internalConfig
- ownerEmail
- privateNotes

### 6.4 Output 级过滤

决定最终回答、citation、审计日志中哪些内容可以展示或必须脱敏。

## 7. 在 RAG 中如何落地数据权限过滤

### 7.1 ingestion 阶段绑定权限 metadata

入库时建议至少写入：

- tenantId
- appId
- visibilityLevel
- permissionScope
- ownerId / ownerGroup

### 7.2 recall 阶段前置过滤

召回时应把权限条件纳入 filter，而不是召回后再删。

### 7.3 rerank 阶段保留权限上下文

防止无权限候选进入后续排序。

### 7.4 context builder 再次兜底

即便前面已过滤，也应在装配上下文前再做一次 final check。

### 7.5 citation 输出校验

引用的 source 对当前用户不可见时，不应直接暴露原始出处。

## 8. 在 Tool / Action 中如何落地数据权限过滤

### 8.1 tool discovery 先过滤可见工具

不同角色和场景可见不同工具。

### 8.2 参数绑定时验证资源归属

例如：

- `pageId` 是否属于当前 app
- `buildId` 是否属于当前 tenant
- `releaseId` 是否在当前环境可访问

### 8.3 tool output 做字段裁剪

即使工具查询成功，也不代表所有返回字段都可回传给模型或用户。

### 8.4 action 执行前做对象范围校验

确保动作不会越界作用到未授权资源。

## 9. 权限过滤与脱敏的区别

两者经常一起出现，但不是同一件事。

### 9.1 权限过滤

决定是否允许访问某类数据。

### 9.2 脱敏

在允许访问的前提下，对敏感字段做遮蔽或降精度处理。

更成熟的系统会同时支持：

- 不可见：直接不返回
- 可见但脱敏：返回摘要或掩码
- 仅审计可见：不进入模型上下文，只保留后台记录

## 10. 推荐的权限过滤链路

建议一条较稳的链路是：

1. 身份透传得到 user / tenant / scope
2. 查询层构造权限过滤条件
3. 检索或工具执行时前置过滤
4. 返回结果后做字段裁剪与脱敏
5. context builder / answer layer 再做输出级校验
6. audit payload 再做一次 redaction

换句话说：

数据权限过滤不应是单点逻辑，而应是分层重复校验的链路。

## 11. 面向低代码平台的典型场景

### 11.1 组件与页面知识问答

应限制：

- 仅检索当前租户可见的组件文档
- 仅显示当前应用可访问的页面与模板
- 过滤内部私有示例或管理员备注

### 11.2 Schema 生成与修复

应限制：

- 只可读取当前应用的数据模型
- 不暴露不属于当前角色的字段定义
- 不让模型引用其他租户页面样例

### 11.3 运维分析

应限制：

- 日志、发布记录、配置差异仅限授权范围
- 敏感配置值默认脱敏
- 高权限运维视图不默认暴露给普通助手

### 11.4 审计回放

应限制：

- 回放界面只显示当前租户和当前角色有权查看的运行数据
- 高敏感字段在 replay 中仍保持脱敏

## 12. 常见误区

### 12.1 只在最终答案层做过滤

这时敏感信息可能早已进入上下文、缓存或 trace。

### 12.2 召回后再删除无权限结果

这会增加泄露和排序污染风险。

### 12.3 工具输出原样返回给模型

很多工具结果里包含内部字段和敏感数据。

### 12.4 把脱敏当成权限控制

脱敏不能替代授权判断。

### 12.5 审计和 replay 不做再过滤

这会让后台系统成为新的泄露面。

## 13. 一句话总结

数据权限过滤的核心，是把可访问数据范围在检索、工具、动作、上下文装配、答案输出和审计回放全链路中前置并重复校验，而不是只在最终展示层做一次裁剪；近两年的最佳实践也越来越明确，企业级 AI 系统能否真正安全落地，很大程度上取决于权限过滤是否已经进入 RAG 和 Agent Runtime 的核心路径。
