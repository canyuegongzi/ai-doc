# 租户隔离

> 状态：第一版草案

## 1. 文档定位

本文档用于定义多租户 AI 系统中的 Tenant Isolation 设计，重点回答：

- 为什么 AI 系统中的租户隔离比传统 SaaS 更复杂
- 租户隔离应落在哪些层，而不是只靠数据库隔离
- 在 RAG、Tool、Agent Runtime 场景下如何落实租户边界

## 2. 什么是租户隔离

租户隔离可以理解为：

在同一套 AI 平台中，确保不同租户的身份、数据、知识索引、工具作用域、动作影响范围、审计记录和配置策略彼此边界清晰，互不越权访问或相互污染的一整套机制。

它不仅是数据库表里加一个 `tenantId`，还包括：

- 模型上下文隔离
- 检索索引隔离
- Tool 可见性隔离
- Action 执行范围隔离
- Trace 与审计隔离
- Prompt / 配置隔离

用一句话概括：

租户隔离的本质，是让 AI 系统在“共享基础设施”的前提下，仍能对每个租户呈现出“独立且安全的能力边界”。

## 3. 为什么 AI 场景中的租户隔离更复杂

### 3.1 AI 系统会跨越更多层访问信息

传统请求链路通常只涉及：

- API
- 业务服务
- 数据库

但 AI 系统还会额外涉及：

- knowledge pipeline
- retrieval engine
- prompt assembly
- tool discovery
- agent runtime
- audit / replay

因此，任何一层隔离失效，都可能造成跨租户泄露。

### 3.2 模型上下文本身就是潜在泄露面

即使底层数据库是隔离的，如果在 prompt / context builder 阶段把别的租户内容装进上下文，模型依然可能泄露信息。

### 3.3 工具和动作也存在跨租户风险

例如：

- 查询跨租户配置差异
- 读取其他租户日志
- 对其他租户页面执行修复或发布

这些都不能只靠数据库层防守。

## 4. 近两年的关键变化

### 4.1 租户隔离正从数据层概念扩展为 Runtime 概念

随着 Agent Runtime、RAG 和 Tool 系统发展，租户边界已经不只是“存储隔离”，而是“整个执行过程隔离”。

### 4.2 检索型系统越来越强调 tenant-aware indexing

在 RAG 场景里，更成熟的实践会把：

- tenantId
- appId
- permissionScope

作为索引和检索过滤的正式组成，而不是后置补丁。

### 4.3 动作型系统越来越强调 scope-bound execution

在 Tool / Action 场景中，系统越来越强调：

- 工具只在租户边界内可见
- 参数必须绑定当前 tenant scope
- 高风险跨边界动作默认禁止

## 5. 租户隔离要解决什么问题

### 5.1 数据不可见性

不同租户之间的数据、文档、知识对象不可互见。

### 5.2 工具作用域限制

工具调用不能跨越租户边界访问或修改资源。

### 5.3 配置与策略隔离

不同租户的模型策略、Prompt、风险规则、审批策略应能独立配置。

### 5.4 审计与回放隔离

不同租户的执行轨迹与审计记录应相互隔离。

## 6. 推荐的租户隔离分层

建议至少分成五层：

### 6.1 身份层隔离

所有请求必须绑定明确的 `tenantId` 和相关 scope。

### 6.2 数据与知识层隔离

文档、向量、索引、对象存储、缓存都要有租户边界。

### 6.3 检索层隔离

query rewrite、recall、rerank、context builder 都要 tenant-aware。

### 6.4 Tool / Action 层隔离

工具发现、参数校验、执行范围和动作影响都要限定在当前租户内。

### 6.5 观测与治理层隔离

日志、trace、audit、eval dataset、feedback 也应隔离。

## 7. 推荐的隔离策略

### 7.1 逻辑隔离

通过：

- `tenantId`
- scope metadata
- query filter
- auth policy

做隔离。

适合共享基础设施，但要求全链路治理成熟。

### 7.2 资源级隔离

为关键资源单独隔离，例如：

- 独立索引
- 独立存储桶
- 独立缓存 namespace

### 7.3 服务级隔离

对高敏感租户或高合规场景，使用更强的独立部署或独立服务实例。

### 7.4 混合隔离

多数企业场景更现实的方案是：

- 普通租户逻辑隔离
- 高敏感租户资源级或服务级隔离

## 8. 租户隔离在 RAG 中如何落地

在 RAG 中至少要控制：

### 8.1 ingestion 阶段

文档入库时必须绑定：

- tenantId
- appId
- permissionScope

### 8.2 index 阶段

索引设计应支持 tenant-aware filter，必要时可独立索引。

### 8.3 retrieval 阶段

召回前就应加租户条件，而不是召回后再删。

### 8.4 context builder 阶段

最终装配到上下文的内容必须再次确认均在当前租户可见范围内。

## 9. 租户隔离在 Tool / Action 中如何落地

### 9.1 tool discovery 隔离

只向当前租户暴露其可用工具。

### 9.2 参数作用域校验

例如：

- `pageId` 是否属于当前租户
- `appCode` 是否属于当前租户
- `targetEnvironment` 是否在租户允许范围内

### 9.3 action impact 校验

执行动作前再次确认影响对象不越过租户边界。

### 9.4 cross-tenant action 默认禁用

只有在极特殊且受控场景下，才允许显式声明跨租户分析或运维操作。

## 10. 租户隔离与 Prompt / Memory / Cache 的关系

### 10.1 Prompt 资产隔离

不同租户的：

- prompt 模板
- tool hints
- business instructions

应可独立管理。

### 10.2 Memory 隔离

长期记忆、会话记忆不能跨租户共享。

### 10.3 Cache 隔离

检索缓存、生成缓存、工具结果缓存也要带租户键，否则容易形成隐式泄露。

## 11. 面向低代码平台的典型建议

### 11.1 页面与组件知识库

至少应按：

- tenantId
- appId
- materialScope

进行索引和检索隔离。

### 11.2 页面生成与修复

生成、修复、发布动作必须限定在当前租户和当前应用范围内。

### 11.3 运维与配置分析

跨租户对比类能力默认不开放给普通助手；若必须支持，应定义专门运维工具和高风险审批策略。

### 11.4 审计回放

租户只能查看自身运行记录与审批记录。

## 12. 常见误区

### 12.1 认为数据库带 tenantId 就够了

AI 系统还涉及上下文装配、缓存、索引、工具和审计。

### 12.2 召回后再做租户过滤

这会增加越权泄露和性能浪费风险，正确做法应尽量前置过滤。

### 12.3 系统账号默认可跨租户查询一切

这会在 Tool / Agent 场景下迅速形成高危路径。

### 12.4 忽略缓存和 Memory 的租户边界

这是很多 AI 系统容易遗漏的隐蔽泄露点。

### 12.5 审计与 Trace 共用全局查询视图

容易让租户看到不属于自己的运行信息。

## 13. 一句话总结

租户隔离的核心，是在身份、数据、检索、工具、动作、缓存、记忆和审计全链路中建立 tenant-aware 边界，而不是只在数据库层面做一次过滤；近两年的最佳实践也越来越明确，AI 系统的多租户安全性，取决于租户边界是否真正进入了 Runtime、RAG 和 Tool 执行体系。
