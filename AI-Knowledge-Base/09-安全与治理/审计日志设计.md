# 审计日志设计

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 Audit Logging 设计，重点回答：

- 为什么企业 AI 系统不能只依赖普通应用日志
- 审计日志与 trace、observability、执行回放的边界是什么
- 在 RAG、Tool、Action、审批和多租户场景下应记录哪些关键审计事件

## 2. 什么是审计日志

审计日志可以理解为：

对 AI 系统中的关键访问、检索、工具调用、权限放行、审批决策、动作执行、配置变更和异常事件进行结构化、可追责、可查询记录的一种治理机制。

它不只是为了排障，而是为了回答：

- 谁做了什么
- 在什么上下文中做的
- 系统为什么允许这么做
- 最终产生了什么影响
- 出问题后由谁承担责任

用一句话概括：

审计日志的本质，是把 AI 系统中的关键行为沉淀成责任、合规和复盘可依赖的事实记录。

## 3. 为什么 AI 系统特别需要审计日志

### 3.1 AI 系统的执行链比传统应用更长、更隐式

一次请求可能跨越：

- 身份透传
- query rewrite
- retrieval
- context builder
- planner
- tool selection
- HITL
- action execute
- replay / audit view

如果没有结构化审计，很难还原责任链。

### 3.2 模型、检索、工具和审批共同决定了最终结果

AI 系统不是单一模块行为，而是多模块协同结果。出现问题时，企业需要判断：

- 是权限问题
- 是检索问题
- 是工具调用问题
- 是审批放行问题
- 还是模型越界问题

### 3.3 近两年的 Agent 体系正在强化 trace + audit 并存

Trace 更偏排障和链路理解；Audit 更偏责任、授权、审批、合规。生产系统通常两者都需要。

## 4. 审计日志要解决什么问题

### 4.1 责任追踪

识别请求发起人、审批人、代执行主体和实际影响对象。

### 4.2 合规检查

证明关键动作经过了正确授权、审批和控制。

### 4.3 安全调查

分析越权访问、敏感信息暴露、异常工具调用等安全事件。

### 4.4 复盘与治理优化

为 eval、policy 调整、tool 风险治理提供事实依据。

## 5. 审计日志与普通日志、Trace 的区别

### 5.1 与普通应用日志的区别

普通日志偏技术运行细节；审计日志偏责任和关键事实。

### 5.2 与 Observability Trace 的区别

Trace 侧重链路诊断和性能；审计日志侧重：

- 谁
- 什么动作
- 什么权限
- 什么审批
- 什么影响

### 5.3 与 Replay 的关系

Replay 依赖审计与执行事件，但审计日志并不等于完整回放数据。

## 6. 推荐的审计对象

建议至少审计以下八类对象：

### 6.1 身份与会话事件

- 登录用户
- tenant / app scope
- requestId / traceId
- effective identity

### 6.2 检索访问事件

- 访问了哪些知识源
- 使用了哪些 filter
- 是否命中受限数据

### 6.3 Tool 调用事件

- tool name
- input summary
- output summary
- permission decision
- risk level

### 6.4 Action 执行事件

- action type
- target resource
- target environment
- execution result
- receipt / operationId

### 6.5 权限与策略决策事件

- allow / deny
- policy matched
- dynamic risk escalation
- filter applied

### 6.6 审批与 HITL 事件

- approval requested
- approver identity
- decision
- attached constraints

### 6.7 配置与治理变更事件

- prompt policy change
- tool version change
- risk rule change
- index version switch

### 6.8 安全异常事件

- 越权尝试
- 脱敏拦截
- 输出约束失败
- 高风险动作阻断

## 7. 推荐的审计日志字段

建议至少包含：

- `auditId`
- `timestamp`
- `tenantId`
- `userId`
- `effectiveIdentity`
- `requestId`
- `traceId`
- `eventType`
- `resourceType`
- `resourceId`
- `action`
- `decision`
- `riskLevel`
- `policyRef`
- `status`
- `summary`

对于高风险事件，建议额外记录：

- `approvalId`
- `approverId`
- `executionReceipt`
- `impactScope`
- `rollbackAvailable`

## 8. 推荐的审计分层

### 8.1 访问审计

谁访问了什么数据和知识源。

### 8.2 决策审计

系统做了什么权限、风险、审批判断。

### 8.3 执行审计

真正执行了什么工具和动作。

### 8.4 治理审计

系统配置、策略、版本和模型路由发生了什么变化。

## 9. 在 RAG 中如何落地审计

### 9.1 检索访问审计

记录：

- query summary
- rewrite result summary
- index / source accessed
- permission filters used

### 9.2 citation 审计

记录最终回答引用了哪些来源以及是否经过可见性校验。

### 9.3 受限内容拦截审计

如果某些候选结果因权限或脱敏规则被过滤，也应留痕。

## 10. 在 Tool / Action 中如何落地审计

### 10.1 Tool 调用审计

记录工具名、输入摘要、参数校验、权限决策和结果状态。

### 10.2 Action 审计

重点记录：

- 动作目标
- 风险级别
- 是否审批
- 最终执行结果
- receipt

### 10.3 幂等与重试审计

每次 retry、dedupe、receipt 查询都应作为关键事件被审计。

## 11. 多租户和脱敏要求

审计日志本身也可能是敏感数据，因此必须：

- 租户隔离
- 字段脱敏
- 查询权限控制
- 长期保留策略分级

推荐策略：

- 原始敏感字段不直接写入审计
- 用 masked value / hashed id 替代
- 对高权限安全团队保留更完整视图

## 12. 面向低代码平台的典型场景

### 12.1 页面发布审计

记录：

- 谁发起发布
- 哪个页面
- 哪个环境
- 风险检查结果
- 谁审批通过
- 最终是否发布成功

### 12.2 配置修复审计

记录：

- 生成了什么 patch
- patch 作用范围
- 是否进入审批
- 是否真正应用成功

### 12.3 运维分析审计

记录：

- 查了哪些日志
- 是否访问敏感配置
- 是否建议回滚
- 是否最终触发动作

## 13. 常见误区

### 13.1 把 trace 当成审计日志

Trace 解决排障，不自动满足责任和合规。

### 13.2 只记录最终动作，不记录决策依据

后续很难解释为什么系统允许执行。

### 13.3 审计日志原样保存敏感内容

这会让审计系统本身变成泄露面。

### 13.4 不记录审批与策略命中信息

关键责任链会断裂。

### 13.5 不对审计查询本身做权限控制

会形成新的越权风险。

## 14. 一句话总结

审计日志设计的核心，是把 AI 系统中真正影响权限、风险、审批和动作执行的关键事实沉淀成结构化记录，并让这些记录可追责、可脱敏、可隔离、可查询；近两年的最佳实践也越来越明确，企业级 AI 系统不是“能跑起来”就够了，还必须“能被审计”。
