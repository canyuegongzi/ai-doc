# 高风险操作审批

> 状态：第一版草案

## 1. 文档定位

本文档用于定义 AI 系统中的 High-Risk Action Approval 设计，重点回答：

- 为什么企业 Agent 的高风险动作必须进入审批链，而不能仅依赖模型自我约束
- 高风险审批应如何与 HITL、风险分级、Action 治理、审计回放联动
- 近两年为什么行业越来越强调 approval gates 和 human oversight

## 2. 什么是高风险操作审批

高风险操作审批可以理解为：

对模型建议或发起的高风险动作，在真正落地执行前，由具备相应权限的人类角色进行确认、拒绝、修改约束或接管处理的一种治理机制。

这里的审批不是普通的“确认一下”，而是正式的执行闸门，典型会覆盖：

- 发布生产环境
- 删除资源
- 批量修改配置
- 向外部用户发送通知
- 跨租户或高敏感数据操作

用一句话概括：

高风险操作审批的本质，是把不可完全自动化托管的动作，重新收回到明确责任主体手中。

## 3. 为什么必须做高风险操作审批

### 3.1 模型输出再合理，也不等于具备业务执行责任

很多动作的风险不在“能不能做”，而在“谁来承担后果”。

### 3.2 高风险动作通常具有不可逆或高传播性

例如：

- 发布生产配置
- 删除关键页面
- 对外通知客户
- 批量修复线上对象

一旦误执行，影响远大于普通问答错误。

### 3.3 近两年的最佳实践越来越强调 human oversight

无论是 OpenAI 的 guardrails / agent 指南，还是 Anthropic 的 agent 工程实践，都越来越强调：

高风险动作不能默认自动执行，必须经过人工闸门。

## 4. 哪些动作应视为高风险

建议至少覆盖以下类型：

### 4.1 生产环境动作

- 发布
- 回滚
- 修改线上配置
- 切流 / 开关操作

### 4.2 删除或覆盖型动作

- 删除资源
- 覆盖更新
- 批量替换

### 4.3 外部影响型动作

- 给客户发通知
- 创建外部工单
- 对接外部系统触发动作

### 4.4 跨边界动作

- 跨租户分析或变更
- 管理员权限动作
- 访问高敏感资源

### 4.5 高不确定性动作

即使动作本身不是最高风险，但当前证据不足、参数不稳定、目标对象不清晰时，也应升级为审批。

## 5. 近两年的关键变化

### 5.1 审批正在从“外部业务流程”变成 Runtime 原生控制点

过去审批常常只是业务系统自己的工作流；现在在 Agent 系统里，审批越来越需要与：

- Planner
- State machine
- Action governance
- Audit / replay

深度整合。

### 5.2 审批不再只是 yes/no，而是结构化决策

更成熟的审批结果通常包括：

- approve / reject
- 附加约束
- 限定环境
- 限定作用范围
- 要求人接管

### 5.3 动态风险评估推动审批条件变得上下文敏感

同一个工具是否需要审批，不仅取决于工具本身，也取决于：

- 参数
- 环境
- 目标对象范围
- 当前证据充分性

## 6. 推荐的审批链路

一个较稳的高风险审批链路通常包括：

1. Planner / Runtime 识别动作意图
2. Risk engine 计算风险等级
3. 若命中审批条件，则生成 approval request
4. 进入 `HUMAN_REVIEW` / `APPROVAL_PENDING`
5. 审批人查看结构化证据与建议
6. 返回结构化审批结果
7. Runtime 根据审批结果继续、拒绝、修改约束或升级处理

## 7. 审批请求中应包含什么信息

给审批人的信息不应是完整 trace 倾倒，而应是结构化摘要，建议至少包括：

- actionType
- targetResource
- targetEnvironment
- reasonSummary
- evidenceSummary
- riskLevel
- potentialImpact
- suggestedPlan
- rollbackAvailability
- traceId

## 8. 审批结果应如何结构化

建议至少支持：

```json
{
  "decision": "approve",
  "constraints": {
    "targetEnvironment": "test",
    "allowBatch": false
  },
  "comment": "仅允许在测试环境执行",
  "approverId": "u_123"
}
```

这样 Runtime 才能继续消费，而不是依赖自然语言理解审批结论。

## 9. 审批与 HITL 的关系

### 9.1 HITL 是更广义概念

HITL 包括：

- 确认
- 审批
- 补充输入
- 人工接管

### 9.2 高风险审批是 HITL 的一种正式子类型

它更强调：

- 权限角色
- 风险责任
- 审批留痕
- 执行约束

## 10. 审批与 Action 治理的关系

高风险审批应成为 Action 治理中的正式节点，而不是外置流程。

更合理的关系是：

- Risk policy 决定是否进审批
- Runtime 决定何时暂停
- Approval node 产出结构化结果
- Action engine 决定后续执行或终止

## 11. 审批与审计回放的关系

每一次高风险审批都应记录：

- 谁发起
- 为什么进入审批
- 审批人是谁
- 审批时看到的证据是什么
- 审批结果和附加约束是什么
- 最终是否真的执行

否则审批就失去治理与追责意义。

## 12. 面向低代码平台的典型场景

### 12.1 页面发布

以下情况建议强制审批：

- prod 发布
- 高影响页面覆盖更新
- 校验存在 warning 但仍要继续

### 12.2 配置修复

以下情况建议强制审批：

- 自动修复涉及批量对象
- patch 影响核心数据绑定
- 修复目标不止一个应用

### 12.3 运维动作

以下情况建议强制审批：

- 回滚版本
- 变更关键配置开关
- 清理高价值资源

### 12.4 通知动作

以下情况建议强制审批：

- 面向外部租户发送通知
- 面向大量用户批量触达
- 内容来源存在模型生成成分

## 13. 常见误区

### 13.1 只有极端危险动作才审批

很多中高风险动作在特定上下文中也应进入审批。

### 13.2 审批只做 yes/no 二元确认

无法表达很多实际限制条件。

### 13.3 审批请求直接把全部日志丢给审批人

审批人需要的是高信噪比证据摘要，而不是原始 trace 洪流。

### 13.4 审批通过后不做后续验证

审批只是进入执行许可，不代表结果一定正确。

### 13.5 审批记录不进入审计回放

后续将无法追责和复盘。

## 14. 一句话总结

高风险操作审批的核心，是在模型发起真实高影响动作前插入正式的人类责任闸门，并让审批与风险分级、HITL、Action 执行治理和审计回放形成完整闭环；近两年的最佳实践也越来越明确，企业级 Agent 要想真正落地，关键不是“减少审批”，而是“让审批成为运行时中的高质量控制点”。
